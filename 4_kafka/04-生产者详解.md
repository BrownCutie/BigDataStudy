# 04 - 生产者详解：如何可靠地发送消息

## 一、生产者发送流程概览

在深入细节之前，我们先看一条消息从生产者到 Kafka Broker 的完整旅程：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    生产者发送消息完整流程                               │
│                                                                     │
│  应用代码                                                            │
│    │                                                                │
│    ▼                                                                │
│  ┌─────────────┐     ┌──────────────┐     ┌──────────────┐         │
│  │  Serializer  │───→│  Partitioner │───→│ RecordAccum. │         │
│  │  序列化器    │     │  分区器      │     │ 消息累加器    │         │
│  └─────────────┘     └──────────────┘     └──────┬───────┘         │
│                                                 │                   │
│  把对象转成字节数组   决定发到哪个分区              │                   │
│                                                 ▼                   │
│                                          ┌──────────────┐          │
│                                          │ Sender 线程   │          │
│                                          │              │          │
│                                          │ 按分区分组     │          │
│                                          │ 批量发送      │          │
│                                          └──────┬───────┘          │
│                                                 │                   │
│                                                 ▼                   │
│                                          ┌──────────────┐          │
│                                          │ Kafka Broker │          │
│                                          │ (Leader 副本) │          │
│                                          └──────────────┘          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**关键组件说明：**

| 组件 | 作用 |
|------|------|
| **Serializer** | 将消息对象序列化为字节数组（如 JSON → bytes） |
| **Partitioner** | 决定消息发到哪个分区（详见[03-topics-and-partitions.md](03-topics-and-partitions.md)） |
| **RecordAccumulator** | 消息缓冲区，按分区缓存消息，凑够一批再发送 |
| **Sender 线程** | 后台线程，从缓冲区取出消息，批量发送到 Broker |

## 二、三种发送模式

### 2.1 Fire-and-Forget（发后即忘）

最简单的方式，发送后不关心结果：

```python
from kafka import KafkaProducer
import json

producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)

# 发送消息，不等待确认
producer.send('blog-events', {
    'type': 'post_viewed',
    'postId': 1,
    'userId': 100,
    'timestamp': '2026-04-15T10:00:00'
})

# 可能在消息还没真正到达 Broker 时就继续执行了
# 如果发送失败，你不会知道！
```

```
┌──────────────────────────────────────────────────────────┐
│              Fire-and-Forget 流程                          │
│                                                          │
│  应用代码 ──→ producer.send() ──→ 放入缓冲区 ──→ 返回      │
│                                              │            │
│                                     后台 Sender 线程       │
│                                     （可能成功也可能失败）   │
│                                                          │
│  优点：吞吐量最高（不阻塞）                                 │
│  缺点：消息可能丢失，无法重试                               │
│  适用：日志采集等允许少量丢失的场景                           │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 2.2 同步发送（Sync）

发送后阻塞等待 Broker 的确认：

```python
from kafka import KafkaProducer
import json

producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)

try:
    # 同步发送，等待 Broker 返回确认
    future = producer.send('blog-events', {
        'type': 'post_published',
        'postId': 1,
        'title': 'Kafka 入门'
    })

    # get() 会阻塞，直到收到确认或超时
    record_metadata = future.get(timeout=10)

    print(f"消息发送成功！")
    print(f"  Topic: {record_metadata.topic}")
    print(f"  Partition: {record_metadata.partition}")
    print(f"  Offset: {record_metadata.offset}")

except Exception as e:
    print(f"消息发送失败: {e}")
    # 这里可以做重试或记录失败日志

producer.flush()
producer.close()
```

```
┌──────────────────────────────────────────────────────────┐
│                  同步发送流程                               │
│                                                          │
│  应用代码 ──→ producer.send() ──→ 放入缓冲区              │
│                                          │               │
│                              future.get() 阻塞等待        │
│                                          │               │
│                              Sender 线程发送到 Broker      │
│                                          │               │
│                              Broker 返回确认              │
│                                          │               │
│                              返回 metadata 给应用代码     │
│                                                          │
│  优点：可靠性高，能确认消息是否发送成功                       │
│  缺点：吞吐量低（每次发送都要等待）                          │
│  适用：文章创建、评论等不能丢消息的场景                       │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 2.3 异步发送（Async）—— 推荐

发送后不阻塞，通过回调函数处理结果：

```python
from kafka import KafkaProducer
import json
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    value_serializer=lambda v: json.dumps(v).encode('utf-8'),
    # Kafka 客户端内置重试
    retries=3,
    # 重试间隔（毫秒）
    retry_backoff_ms=100
)

def on_send_success(record_metadata):
    """发送成功回调"""
    logger.info(
        f"消息发送成功 → Topic: {record_metadata.topic}, "
        f"Partition: {record_metadata.partition}, "
        f"Offset: {record_metadata.offset}"
    )

def on_send_error(excp):
    """发送失败回调"""
    logger.error(f"消息发送失败: {excp}", exc_info=excp)
    # 可以在这里做补偿处理：
    # - 写入本地死信队列（Dead Letter Queue）
    # - 记录到数据库，后续人工处理
    # - 发送告警通知

# 异步发送 + 回调
producer.send('blog-events', {
    'type': 'comment_added',
    'postId': 1,
    'content': '学习 Kafka'
}).add_callback(on_send_success).add_errback(on_send_error)

# 主线程不阻塞，可以继续处理其他事情
print("消息已提交，继续处理其他请求...")

# 确保所有消息发送完毕
producer.flush()
producer.close()
```

```
┌──────────────────────────────────────────────────────────┐
│                  异步发送流程                               │
│                                                          │
│  应用代码 ──→ producer.send() ──→ 放入缓冲区 ──→ 立即返回  │
│                                          │               │
│                                     主线程继续执行...      │
│                                          │               │
│                              后台 Sender 发送到 Broker     │
│                                          │               │
│                              ┌───────────┴──────────┐    │
│                              │                      │    │
│                           成功                    失败     │
│                              │                      │    │
│                              ▼                      ▼    │
│                        on_success               on_error  │
│                        回调函数                 回调函数   │
│                                                          │
│  优点：高吞吐 + 可靠（通过回调知道结果）                      │
│  缺点：实现稍复杂，需要处理回调中的异常                       │
│  适用：绝大多数生产场景                                      │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 2.4 三种模式对比

| 模式 | 可靠性 | 吞吐量 | 复杂度 | 适用场景 |
|------|--------|--------|--------|---------|
| Fire-and-Forget | 低 | 最高 | 最低 | 日志采集、监控数据 |
| 同步发送 | 最高 | 最低 | 低 | 金融交易、文章 |
| 异步发送 | 高 | 高 | 中 | 绝大多数场景 |

## 三、acks 参数与可靠性保证

`acks` 是生产者最重要的配置参数，决定了 Kafka 在什么情况下算"发送成功"：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    acks 参数详解                                      │
│                                                                     │
│  acks=0（发后即忘）：                                                 │
│  ┌──────────┐     ┌──────────┐                                     │
│  │ Producer │────→│  Leader  │                                     │
│  └──────────┘     └──────────┘                                     │
│  发送后不等确认，直接认为成功。                                       │
│  最快，但 Leader 挂了消息就丢了。                                     │
│                                                                     │
│  acks=1（Leader 确认）：                                             │
│  ┌──────────┐     ┌──────────┐                                     │
│  │ Producer │────→│  Leader  │──→ 写入本地日志                      │
│  └──────────┘  ←──└──────────┘   返回确认                          │
│                     │                                               │
│                     │ Follower 还没同步                              │
│                     ▼                                               │
│               ┌──────────┐                                         │
│               │ Follower │  ← Leader 挂了，消息可能丢失              │
│               └──────────┘                                         │
│                                                                     │
│  acks=all / acks=-1（ISR 全部确认）：                                  │
│  ┌──────────┐     ┌──────────┐                                     │
│  │ Producer │────→│  Leader  │──→ 写入本地日志                      │
│  └──────────┘     └──────────┘                                     │
│                     │                                               │
│                     ├──→ 等待 ISR 中所有 Follower 同步               │
│                     │                                               │
│                     ▼                                               │
│               ┌──────────┐                                         │
│               │ Follower │──→ 同步完成                              │
│               └──────────┘                                         │
│                     │                                               │
│               ┌──────────┐                                         │
│               │ Leader   │──→ 返回确认给 Producer                   │
│               └──────────┘                                         │
│  最可靠，但延迟最高。                                                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

| acks 值 | 可靠性 | 延迟 | 说明 |
|---------|--------|------|------|
| `0` | 最低 | 最低 | 不等确认，可能丢消息 |
| `1` | 中等 | 中等 | Leader 写入成功即返回 |
| `all` / `-1` | 最高 | 最高 | ISR 中所有副本写入成功才返回 |

```python
# 数据分析项目推荐配置：acks=all
producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    acks='all',                    # 最高可靠性
    retries=3,                     # 失败重试 3 次
    retry_backoff_ms=100,          # 重试间隔 100ms
    max_in_flight_requests_per_connection=5,  # 控制并发请求数
    enable_idempotence=True,       # 开启幂等性（防止重试导致重复）
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)
```

## 四、批量发送与压缩

### 4.1 批量发送

Kafka 不会每条消息单独发送，而是攒够一批再发：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    批量发送机制                                       │
│                                                                     │
│  没有批量（每条单独发送）：                                            │
│  ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐                                    │
│  │ m1│→│ m2│→│ m3│→│ m4│→│ m5│  5 次网络请求                       │
│  └───┘ └───┘ └───┘ └───┘ └───┘                                    │
│                                                                     │
│  批量发送：                                                           │
│  ┌───┬───┬───┬───┬───┐                                             │
│  │ m1│ m2│ m3│ m4│ m5│ →  1 次网络请求                               │
│  └───┴───┴───┴───┴───┘                                             │
│                                                                     │
│  批量发送触发条件（满足任一即发送）：                                    │
│  1. 批量大小达到 batch.size（默认 16KB）                               │
│  2. 等待时间达到 linger.ms（默认 0ms，即立即发送）                      │
│                                                                     │
│  关键配置：                                                           │
│  ┌──────────────────────┬─────────┬───────────────────────────┐     │
│  │ 配置项                │ 默认值   │ 说明                      │     │
│  ├──────────────────────┼─────────┼───────────────────────────┤     │
│  │ batch.size           │ 16384   │ 批量最大字节数（16KB）      │     │
│  │ linger.ms            │ 0       │ 等待最大时间（毫秒）        │     │
│  │ buffer.memory        │ 33554432│ 缓冲区总大小（32MB）       │     │
│  │ max.block.ms         │ 60000   │ 缓冲区满时阻塞最大时间      │     │
│  └──────────────────────┴─────────┴───────────────────────────┘     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

```python
# 推荐的批量配置（提升吞吐量）
producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    batch_size=32768,       # 32KB（调大 batch.size）
    linger_ms=10,          # 最多等 10ms，凑够一批再发
    buffer_memory=67108864, # 64MB 缓冲区
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)
```

### 4.2 消息压缩

开启压缩可以减少网络传输和磁盘存储：

```
┌──────────────────────────────────────────────────────────┐
│                    压缩效果对比                            │
│                                                          │
│  压缩算法       压缩比    CPU 开销    适用场景              │
│  ─────────────────────────────────────────                │
│  none          1:1       最低        不压缩               │
│  gzip          3:1       高         存储空间敏感           │
│  snappy        2:1       低         通用（推荐）           │
│  lz4           2.5:1     最低        大数据场景（推荐）      │
│  zstd          3:1       低         Kafka 2.1+ 推荐       │
│                                                          │
│  推荐选择：                                                 │
│  - 通用场景：snappy 或 lz4（速度快，压缩比好）               │
│  - 存储成本敏感：gzip 或 zstd（压缩比高）                    │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

```python
producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    compression_type='lz4',  # 启用 LZ4 压缩
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)
```

## 五、幂等生产者与事务

### 5.1 为什么需要幂等性？

考虑这个场景：

```
┌──────────────────────────────────────────────────────────┐
│                 重复消息问题                                │
│                                                          │
│  Producer ──→ Broker（Leader 收到消息，写入成功）            │
│                     │                                     │
│                     └──→ 但网络超时，Producer 没收到确认     │
│                                                          │
│  Producer 以为失败了，触发重试 ──→ 同一条消息又发了一次       │
│                                                          │
│  结果：Broker 中有两条相同的消息！                           │
│  如果是评论消息，用户会看到两条相同的评论                     │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 5.2 幂等生产者（Idempotent Producer）

开启幂等性后，Kafka 会自动去重：

```python
producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    # 开启幂等性（Kafka 客户端 1.0+ 支持）
    enable_idempotence=True,
    # 开启幂等性后，以下参数会被自动设置：
    # acks=all
    # retries=Integer.MAX_VALUE
    # max.in.flight.requests.per.connection=5
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)
```

```
┌─────────────────────────────────────────────────────────────────────┐
│                    幂等性原理                                         │
│                                                                     │
│  每个 Producer 有一个唯一的 PID（Producer ID）                       │
│  每条消息有一个序列号（Sequence Number）                               │
│                                                                     │
│  Producer: PID=12345                                                │
│                                                                     │
│  第一次发送：  PID=12345, SeqNum=0, Value="评论A"                     │
│  ┌──────────┐     ┌──────────────────────┐                          │
│  │ Producer │────→│ Broker：SeqNum=0 ✓  │                          │
│  └──────────┘     │ 写入成功              │                          │
│        ↑          └──────────────────────┘                          │
│        │           网络超时，没收到确认                                │
│        │                                                            │
│  重试发送：  PID=12345, SeqNum=0, Value="评论A"                     │
│  ┌──────────┐     ┌──────────────────────┐                          │
│  │ Producer │────→│ Broker：SeqNum=0 已存在！                      │
│  └──────────┘     │ → 丢弃重复消息，返回成功  │                          │
│                   └──────────────────────┘                          │
│                                                                     │
│  注意：幂等性只能保证单分区内的 Exactly-Once                          │
│  跨分区的 Exactly-Once 需要使用事务                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 5.3 Kafka 事务（Transaction）

事务保证跨分区的原子性写入：要么全部成功，要么全部失败：

```python
from kafka import KafkaProducer

producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    # 事务 ID，全局唯一
    transactional_id='demo-event-producer-1',
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)

# 初始化事务
producer.init_transactions()

try:
    # 开启事务
    producer.begin_transaction()

    # 向多个 Topic/Partition 发送消息
    producer.send('blog-events', value={'type': 'post_published', 'postId': 1})
    producer.send('demo-notifications', value={'type': 'notify_author', 'postId': 1})
    producer.send('demo-search-index', value={'type': 'index_post', 'postId': 1})

    # 提交事务（要么全部成功）
    producer.commit_transaction()
    print("事务提交成功")

except Exception as e:
    # 回滚事务（要么全部失败）
    producer.abort_transaction()
    print(f"事务回滚: {e}")

finally:
    producer.close()
```

```
┌─────────────────────────────────────────────────────────────────────┐
│                    事务工作流程                                       │
│                                                                     │
│  begin_transaction()                                                │
│         │                                                           │
│         ▼                                                           │
│  ┌──────────────────────────────────────────────────────┐          │
│  │              事务中的消息                              │          │
│  │  send(Topic A) ──→ 消息标记为 "未提交"                  │          │
│  │  send(Topic B) ──→ 消息标记为 "未提交"                  │          │
│  │  send(Topic C) ──→ 消息标记为 "未提交"                  │          │
│  └──────────────────────────────────────────────────────┘          │
│         │                                                           │
│    ┌────┴────┐                                                      │
│    │         │                                                      │
│  成功      失败                                                     │
│    │         │                                                      │
│    ▼         ▼                                                      │
│  commit    abort                                                    │
│  │         │                                                        │
│  ▼         ▼                                                        │
│  所有消息    所有消息                                                 │
│  变为可见    被丢弃                                                   │
│                                                                     │
│  注意：消费者需要设置 isolation.level=read_committed                  │
│  才能只读到已提交的事务消息                                            │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 六、实战：Python 生产者发送数据平台行为事件

下面是一个完整的用户事件生产者，结合了我们前面讲的所有知识：

```python
"""
用户事件生产者 —— 完整示例
发送用户浏览、评论、点赞等事件到 Kafka
"""

import json
import time
import logging
from datetime import datetime
from kafka import KafkaProducer

logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s [%(levelname)s] %(message)s'
)
logger = logging.getLogger(__name__)

class BlogEventProducer:
    """用户事件生产者"""

    def __init__(self, bootstrap_servers='localhost:9092'):
        self.producer = KafkaProducer(
            bootstrap_servers=[bootstrap_servers],
            # 可靠性配置
            acks='all',
            retries=3,
            retry_backoff_ms=100,
            enable_idempotence=True,

            # 性能配置
            batch_size=32768,          # 32KB 批量
            linger_ms=10,              # 等待 10ms 凑批
            compression_type='lz4',    # LZ4 压缩
            buffer_memory=67108864,    # 64MB 缓冲区

            # 序列化
            key_serializer=lambda k: k.encode('utf-8') if k else None,
            value_serializer=lambda v: json.dumps(v, ensure_ascii=False).encode('utf-8')
        )

    def send_post_published(self, post_id, title, author):
        """发送文章发布事件"""
        event = {
            'type': 'post_published',
            'postId': post_id,
            'title': title,
            'author': author,
            'timestamp': datetime.now().isoformat()
        }
        # 用 postId 作为 Key，保证同一文章的事件有序
        self.producer.send(
            'blog-events',
            key=str(post_id),
            value=event
        ).add_callback(self._on_success).add_errback(self._on_error)
        logger.info(f"发送文章发布事件: postId={post_id}")

    def send_comment_added(self, post_id, user_id, content):
        """发送评论事件"""
        event = {
            'type': 'comment_added',
            'postId': post_id,
            'userId': user_id,
            'content': content,
            'timestamp': datetime.now().isoformat()
        }
        self.producer.send(
            'blog-events',
            key=str(post_id),  # 同一文章的事件在同一分区
            value=event
        ).add_callback(self._on_success).add_errback(self._on_error)
        logger.info(f"发送评论事件: postId={post_id}, userId={user_id}")

    def send_page_view(self, post_id, user_id, ip):
        """发送页面浏览事件（行为日志，不需要 Key）"""
        event = {
            'action': 'page_view',
            'postId': post_id,
            'userId': user_id,
            'ip': ip,
            'timestamp': datetime.now().isoformat()
        }
        self.producer.send(
            'demo-logs',
            value=event  # 无 Key，轮询分配分区
        ).add_callback(self._on_success).add_errback(self._on_error)

    def _on_success(self, metadata):
        """发送成功回调"""
        logger.debug(
            f"发送成功: topic={metadata.topic}, "
            f"partition={metadata.partition}, offset={metadata.offset}"
        )

    def _on_error(self, excp):
        """发送失败回调"""
        logger.error(f"发送失败: {excp}")

    def close(self):
        """关闭生产者"""
        self.producer.flush()
        self.producer.close()
        logger.info("生产者已关闭")


# 使用示例
if __name__ == '__main__':
    producer = BlogEventProducer()

    try:

        producer.send_post_published(post_id=1, title='Kafka 入门指南', author='browncutie')

        producer.close()
```

## 七、面试题

### Q1：Kafka 生产者如何保证消息不丢失？

需要从三个层面保证：

1. **生产者端**：`acks=all` + `retries>0` + `enable_idempotence=True`
2. **Broker 端**：`min.insync.replicas>=2` + `replication.factor>=3`
3. **消费者端**：手动提交 offset，处理完成后再提交

### Q2：`linger.ms` 设为 0 和设为 10ms 有什么区别？

`linger.ms=0`（默认）：消息一进入缓冲区就立即发送，延迟最低，但吞吐量也最低（每条消息都单独发送）。

`linger.ms=10`：最多等 10ms，如果这段时间内有更多消息进来，就一起打包发送。延迟增加了最多 10ms，但吞吐量大幅提升。

在高吞吐场景下，设置 `linger.ms=5~20` 是常见的优化手段。

### Q3：Kafka 的事务和数据库的事务有什么区别？

| 维度 | 数据库事务 | Kafka 事务 |
|------|-----------|-----------|
| 作用范围 | 单个数据库内 | 跨多个 Topic/Partition |
| 隔离级别 | 读未提交、读已提交、可重复读、串行化 | 读未提交、读已提交 |
| 回滚 | 支持完整回滚 | 只能标记消息为"已中止" |
| 适用场景 | 保证数据一致性 | 保证跨分区原子写入 |

Kafka 事务主要用于"流处理"场景，保证多个 Topic 之间的原子性。它不是替代数据库事务的方案。
