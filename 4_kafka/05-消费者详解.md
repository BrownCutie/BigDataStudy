# 05 - 消费者详解：如何高效可靠地消费消息

## 一、消费者组（Consumer Group）机制

消费者组是 Kafka 消费模型的核心。理解消费者组，就理解了 Kafka 消费的精髓。

### 1.1 什么是消费者组？

```
┌─────────────────────────────────────────────────────────────────────┐
│                    消费者组机制                                       │
│                                                                     │
│  Topic: blog-events（3 个分区）                                      │
│                                                                     │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                         │
│  │Partition0│  │Partition1│  │Partition2│                         │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘                         │
│       │             │             │                                │
│  ─────┼─────────────┼─────────────┼────── Consumer Group A          │
│       │             │             │                                │
│  ┌────▼─────┐  ┌────▼─────┐  ┌────▼─────┐                         │
│  │Consumer 1│  │Consumer 2│  │Consumer 3│                         │
│  │ 消费 P0  │  │ 消费 P1  │  │ 消费 P2  │                         │
│  └──────────┘  └──────────┘  └──────────┘                         │
│                                                                     │
│  规则 1：一个分区在同一时间只能被同一个组内的一个消费者消费              │
│  规则 2：一个消费者可以同时消费多个分区                                │
│  规则 3：组内消费者数量 ≤ 分区数量（多出来的消费者闲置）                │
│                                                                     │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  同一个 Topic 可以被多个消费者组同时消费（广播模式）：                   │
│                                                                     │
│  ┌────┬─────┐  ┌────┬─────┐  ┌────┬─────┐                         │
│  │Partition0│  │Partition1│  │Partition2│                         │
│  └──┬─┴──┬──┘  └──┬─┴──┬──┘  └──┬─┴──┬──┘                         │
│     │    │        │    │        │    │                            │
│     │  ──┼────────┼────┼────────┼────┼── Consumer Group B          │
│     │    │        │    │        │    │    (搜索索引更新)            │
│     │    │     ┌──▼──┐ ┌──▼──┐ ┌──▼──┐                            │
│     │    │     │ C4  │ │ C5  │ │ C6  │                            │
│     │    │     └─────┘ └─────┘ └─────┘                            │
│     │    │                                                        │
│  ───┼────┼────────┼────┼────────┼────┼── Consumer Group C          │
│     │    │        │    │        │    │    (实时统计)                │
│     │    │     ┌──▼──┐ ┌──▼──┐ ┌──▼──┐                            │
│     │    │     │ C7  │ │ C8  │ │ C9  │                            │
│     │    │     └─────┘ └─────┘ └─────┘                            │
│                                                                     │
│  Group B 和 Group C 都独立消费全部分区的消息（广播）                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 消费者组的核心概念

| 概念 | 说明 |
|------|------|
| **Consumer Group** | 一组消费者共同消费一个 Topic，组内竞争、组间广播 |
| **Group Coordinator** | Broker 中的一个特殊角色，负责管理消费者组 |
| **Group Leader** | 组内的第一个消费者，负责分区分配方案的制定 |
| **消费者数量 > 分区数量** | 多出来的消费者空闲（浪费资源） |
| **消费者数量 < 分区数量** | 部分消费者消费多个分区（正常情况） |

### 1.3 数据分析项目中的消费者组设计

```
┌──────────────────────────────────────────────────────────┐
│              数据分析项目消费者组设计                             │
│                                                          │
│  Topic: blog-events                                      │
│                                                          │
│  Consumer Group: "search-indexer"                        │
│  → 消费文章发布/评论事件，更新 Elasticsearch 索引          │

│  Consumer Group: "notification-service"                   │

│                                                          │
└──────────────────────────────────────────────────────────┘
```

## 二、分区分配策略

当消费者组中有新成员加入或离开时，需要重新分配分区。Kafka 提供了三种分配策略：

### 2.1 RangeAssignor（默认策略）

按 Topic 维度，将分区按范围分配给消费者：

```
┌──────────────────────────────────────────────────────────┐
│              Range 策略                                    │
│                                                          │
│  Topic: blog-events（5 个分区），2 个消费者                 │
│                                                          │
│  每个 Topic 独立计算：                                      │
│  分区数 / 消费者数 = 5 / 2 = 2 余 1                       │
│                                                          │
│  Consumer 0: P0, P1, P2（分到 3 个）                      │
│  Consumer 1: P3, P4  （分到 2 个）                        │
│                                                          │
│  特点：前面的消费者多分一个分区                              │
│  问题：多个 Topic 时，Consumer 0 总是分到更多分区            │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 2.2 RoundRobinAssignor

所有 Topic 的所有分区混在一起，轮询分配：

```
┌──────────────────────────────────────────────────────────┐
│              RoundRobin 策略                               │
│                                                          │
│  Topic A: P0, P1, P2                                     │
│  Topic B: P0, P1                                         │
│  2 个消费者                                               │
│                                                          │
│  按轮询分配（先排好所有分区，再轮流分配）：                    │
│  A-P0 → Consumer 0                                       │
│  A-P1 → Consumer 1                                       │
│  A-P2 → Consumer 0                                       │
│  B-P0 → Consumer 1                                       │
│  B-P1 → Consumer 0                                       │
│                                                          │
│  结果：                                                   │
│  Consumer 0: A-P0, A-P2, B-P1（3 个分区）                 │
│  Consumer 1: A-P1, B-P0  （2 个分区）                     │
│                                                          │
│  特点：分配更均匀                                          │
│  注意：要求所有消费者订阅相同的 Topic 集合                   │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 2.3 StickyAssignor（推荐）

在均匀分配的基础上，尽量保持原有分配不变（减少分区迁移）：

```
┌──────────────────────────────────────────────────────────┐
│              Sticky 策略                                   │
│                                                          │
│  初始状态（2 个消费者）：                                   │
│  Consumer 0: P0, P1, P2                                  │
│  Consumer 1: P3, P4                                      │
│                                                          │
│  Consumer 2 加入后：                                      │
│  → 需要重新分配，但尽量少移动                               │
│                                                          │
│  Range 策略结果（大量迁移）：                               │
│  Consumer 0: P0, P1  ← P2 被移走了                        │
│  Consumer 1: P2, P3  ← P4 被移走了                        │
│  Consumer 2: P4      ← 全新分配                           │
│                                                          │
│  Sticky 策略结果（最小迁移）：                              │
│  Consumer 0: P0, P1  ← 不变                               │
│  Consumer 1: P3, P4  ← 不变                               │
│  Consumer 2: P2      ← 只从 Consumer 0 拿走一个            │
│                                                          │
│  优势：减少 Rebalance 时的分区迁移，降低消费中断时间          │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

```python
# 配置分区分配策略
consumer = KafkaConsumer(
    'blog-events',
    bootstrap_servers=['localhost:9092'],
    group_id='demo-processor',
    # Kafka 2.4+ 推荐使用 CooperativeStickyAssignor
    # 支持增量 Rebalance，不需要停止所有消费者
    partition_assignment_strategy=['org.apache.kafka.clients.consumer.CooperativeStickyAssignor']
)
```

## 三、Offset 管理与提交策略

Offset 是消费者在分区中的消费进度标记。理解 Offset 管理是保证消息可靠消费的关键。

### 3.1 Offset 存储在哪里？

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Offset 存储演进                                    │
│                                                                     │
│  Kafka 0.9 之前：Offset 存储在 ZooKeeper 中                          │
│  → ZK 不适合高频写入，成为性能瓶颈                                    │
│                                                                     │
│  Kafka 0.9 之后：Offset 存储在 Kafka 内部的 __consumer_offsets Topic │
│  → 和普通消息一样存储，利用 Kafka 自身的高性能存储                     │
│                                                                     │
│  Kafka 4.x（KRaft）：                                               │
│  → 继续存储在内部 Topic 中，不再依赖 ZK                               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 自动提交 vs 手动提交

```
┌─────────────────────────────────────────────────────────────────────┐
│                    自动提交（默认）                                   │
│                                                                     │
│  配置：enable.auto.commit=true, auto.commit.interval.ms=5000         │
│                                                                     │
│  时间线：                                                            │
│  ────────────────────────────────────────────────────────            │
│  t=0s     t=1s     t=3s     t=5s     t=6s     t=7s                  │
│  │        │        │        │        │        │                     │
│  拉取P0    处理中    处理P0   自动提交   处理完成   拉取P1             │
│  offset=5 offset=5 offset=7 offset=7！       但如果 t=6s 崩溃...      │
│                                                                     │
│  问题：offset=7 被提交了，但 P0[5,6,7] 还没处理完                      │
│  → 如果此时消费者崩溃，重启后从 offset=7 开始，跳过了未处理的消息        │
│  → 导致消息丢失！                                                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────────────────────────────────┐
│                    手动提交（推荐）                                   │
│                                                                     │
│  配置：enable.auto.commit=false                                     │
│                                                                     │
│  时间线：                                                            │
│  ────────────────────────────────────────────────────────            │
│  t=0s     t=3s     t=5s     t=7s                                    │
│  │        │        │        │                                       │
│  拉取P0    处理中    处理完成   手动提交                               │
│  offset=5 offset=7  offset=7！ offset=7                              │
│                                                                     │
│  优势：处理完成后再提交，保证不丢消息                                  │
│                                                                     │
│  手动提交有两种方式：                                                  │
│  1. commit_sync()  —— 同步提交，阻塞等待确认                          │
│  2. commit_async() —— 异步提交，不阻塞（推荐）                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.3 手动提交代码示例

```python
from kafka import KafkaConsumer
import json
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

consumer = KafkaConsumer(
    'blog-events',
    bootstrap_servers=['localhost:9092'],
    group_id='demo-processor',
    auto_offset_reset='earliest',    # 没有已提交 offset 时，从头开始
    enable_auto_commit=False,        # 关闭自动提交！
    value_deserializer=lambda m: json.loads(m.decode('utf-8'))
)

try:
    for message in consumer:
        try:
            # 处理消息
            process_event(message.value)

            # 处理成功后，手动提交 offset
            consumer.commit()
            logger.info(f"处理成功并提交: offset={message.offset}")

        except Exception as e:
            logger.error(f"处理消息失败: offset={message.offset}, error={e}")
            # 处理失败，不提交 offset
            # 下次消费时还会拉到这条消息（可能重复消费）
            # 可以把失败消息写入死信队列

finally:
    consumer.close()
```

### 3.4 提交策略对比

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 自动提交 | 简单 | 可能丢消息 | 允许丢消息的日志场景 |
| 同步手动提交 | 可靠 | 阻塞，吞吐量低 | 关键业务 |
| 异步手动提交 | 可靠 + 高吞吐 | 可能重复提交 | 推荐 |

## 四、Rebalance 机制与触发条件

Rebalance（再平衡）是消费者组重新分配分区的过程。这个过程会导致消费暂停，理解它的触发条件非常重要。

### 4.1 Rebalance 触发条件

```
┌──────────────────────────────────────────────────────────┐
│              Rebalance 触发条件                            │
│                                                          │
│  1. 消费者加入                                            │
│     新的消费者加入消费者组                                  │
│     → 分区需要重新分配                                     │
│                                                          │
│  2. 消费者离开                                            │
│     消费者正常关闭或崩溃                                    │
│     → 它负责的分区需要分配给其他消费者                       │
│                                                          │
│  3. 消费者心跳超时                                         │
│     消费者长时间没发心跳（默认 10 秒）                       │
│     → 被判定为死亡，触发 Rebalance                        │
│                                                          │
│  4. 订阅的 Topic 数量变化                                  │
│     消费者订阅了新的 Topic                                  │
│     → 分区数变了，需要重新分配                              │
│                                                          │
│  5. 订阅的 Topic 分区数变化                                │
│     Topic 增加了分区                                       │
│     → 新分区需要分配给消费者                                │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 4.2 Rebalance 过程

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Rebalance 过程（Stop-the-World）                   │
│                                                                     │
│  正常消费状态：                                                       │
│  Consumer 0 ←→ Partition 0, 1    （正常消费中）                      │
│  Consumer 1 ←→ Partition 2, 3    （正常消费中）                      │
│                                                                     │
│  Consumer 2 加入 → 触发 Rebalance：                                   │
│                                                                     │
│  阶段 1：所有消费者停止消费（Stop-the-World）                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                          │
│  │Consumer 0│  │Consumer 1│  │Consumer 2│                          │
│  │ ⏸ 停止   │  │ ⏸ 停止   │  │ ⏸ 停止   │                          │
│  └──────────┘  └──────────┘  └──────────┘                          │
│                                                                     │
│  阶段 2：Group Leader 计算新的分区分配方案                              │
│                                                                     │
│  阶段 3：所有消费者开始消费新分配的分区                                │
│  Consumer 0 ←→ Partition 0, 1    （继续消费）                        │
│  Consumer 1 ←→ Partition 2       （减少了）                          │
│  Consumer 2 ←→ Partition 3       （新分配）                          │
│                                                                     │
│  问题：Rebalance 期间所有消费者都暂停了消费！                           │
│  如果 Rebalance 频繁发生 → 大量消费暂停 → 处理延迟                     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.3 如何减少 Rebalance？

```python
consumer = KafkaConsumer(
    'blog-events',
    bootstrap_servers=['localhost:9092'],
    group_id='demo-processor',

    # ===== 心跳相关配置（关键！）=====
    # 心跳间隔（默认 3 秒）
    heartbeat_interval_ms=3000,

    # 会话超时（默认 45 秒）
    # 消费者超过这个时间没发心跳 → 被踢出组 → 触发 Rebalance
    session_timeout_ms=45000,

    # 最大轮询间隔（默认 5 分钟）
    # 两次 poll() 调用的最大间隔
    # 如果处理消息太慢，超过这个时间 → 触发 Rebalance
    max_poll_interval_ms=300000,

    # ===== 拉取配置 =====
    # 每次拉取的最大条数
    max_poll_records=500,

    # 每次拉取的最大字节数
    max_poll_records_bytes=1048576,

    # 使用 CooperativeStickyAssignor（增量 Rebalance）
    partition_assignment_strategy=[
        'org.apache.kafka.clients.consumer.CooperativeStickyAssignor'
    ]
)
```

```
┌──────────────────────────────────────────────────────────┐
│              减少 Rebalance 的最佳实践                      │
│                                                          │
│  1. 不要在 poll() 循环中做耗时操作                          │
│     把消息处理丢到另一个线程池                               │
│                                                          │
│  2. 合理设置 max_poll_interval_ms                         │
│     如果单条消息处理需要 30 秒，设为 60000（60秒）            │
│                                                          │
│  3. 使用 CooperativeStickyAssignor                        │
│     支持增量 Rebalance，不需要 Stop-the-World              │
│                                                          │
│  4. 避免频繁启停消费者                                      │
│     滚动发布时，一次只重启一个消费者                          │
│                                                          │
│  5. 保持消费者数量稳定                                      │
│     不要随意扩缩消费者数量                                  │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## 六、实战：Python 消费者处理用户事件

```python
"""
用户事件消费者 —— 完整示例
从 Kafka 消费用户事件，处理并写入 Elasticsearch
"""

import json
import logging
import requests
from kafka import KafkaConsumer

logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s [%(levelname)s] %(message)s'
)
logger = logging.getLogger(__name__)

class BlogEventConsumer:
    """用户事件消费者"""

    def __init__(self, bootstrap_servers='localhost:9092',
                 group_id='demo-processor', topic='blog-events'):
        self.consumer = KafkaConsumer(
            topic,
            bootstrap_servers=[bootstrap_servers],
            group_id=group_id,
            auto_offset_reset='earliest',
            enable_auto_commit=False,

            # 拉取配置
            max_poll_records=100,
            fetch_max_wait_ms=500,

            # 使用增量 Rebalance
            partition_assignment_strategy=[
                'org.apache.kafka.clients.consumer'
                '.CooperativeStickyAssignor'
            ],

            # 反序列化
            value_deserializer=lambda m: json.loads(m.decode('utf-8'))
        )

        # Elasticsearch 地址
        self.es_url = 'http://localhost:9200'

    def start(self):
        """启动消费循环"""
        logger.info("用户事件消费者启动，等待消息...")

        try:
            for message in self.consumer:
                topic = message.topic
                partition = message.partition
                offset = message.offset
                value = message.value

                logger.info(
                    f"收到消息: topic={topic}, partition={partition}, "
                    f"offset={offset}, value={value}"
                )

                try:
                    # 根据事件类型分发处理
                    event_type = value.get('type', '')

                    if event_type == 'post_published':
                        self.handle_post_published(value)
                    elif event_type == 'comment_added':
                        self.handle_comment_added(value)
                    elif event_type == 'post_viewed':
                        self.handle_post_viewed(value)
                    else:
                        logger.warning(f"未知事件类型: {event_type}")

                    # 处理成功，提交 offset
                    self.consumer.commit()
                    logger.debug(f"提交 offset: {offset}")

                except Exception as e:
                    logger.error(
                        f"处理消息失败: offset={offset}, error={e}"
                    )
                    # 不提交 offset，下次还会消费这条消息
                    # 实际项目中应该写入死信队列

        except KeyboardInterrupt:
            logger.info("收到中断信号，停止消费")
        finally:
            self.consumer.close()
            logger.info("消费者已关闭")

    def handle_post_published(self, event):
        """处理文章发布事件 → 更新 ES 索引"""
        post_id = event['postId']
        logger.info(f"处理文章发布事件: postId={post_id}")

        # 更新 Elasticsearch 索引
        doc = {
            'postId': post_id,
            'title': event['title'],
            'author': event['author'],
            'timestamp': event['timestamp']
        }
        requests.put(
            f'{self.es_url}/posts/_doc/{post_id}',
            json=doc
        )

    def handle_comment_added(self, event):
        """处理评论事件 → 更新评论计数 + 通知作者"""
        post_id = event['postId']
        logger.info(f"处理评论事件: postId={post_id}")

        # 更新 Elasticsearch 中的评论数
        requests.post(
            f'{self.es_url}/posts/_update/{post_id}',
            json={'script': {'source': 'ctx._source.commentCount += 1'}}
        )

    def handle_post_viewed(self, event):
        """处理浏览事件 → 更新浏览量"""
        post_id = event['postId']
        logger.info(f"处理浏览事件: postId={post_id}")

        # 更新 Elasticsearch 中的浏览量
        requests.post(
            f'{self.es_url}/posts/_update/{post_id}',
            json={'script': {'source': 'ctx._source.viewCount += 1'}}
        )


if __name__ == '__main__':
    consumer = BlogEventConsumer()
    consumer.start()
```

## 七、面试题

### Q1：Kafka 如何保证消息不重复消费？

完全不重复消费很难做到，但可以尽量减少：

1. **消费者端幂等处理**：用唯一 ID 去重（如 `postId + eventType + timestamp`）
2. **手动提交 offset**：处理完再提交，避免自动提交导致重复
3. **数据库唯一约束**：利用数据库的主键/唯一索引天然去重
4. **Redis 缓冲去重**：用 Redis 的 SET 记录已处理的消息 ID

实际生产中，通常采用**"至少一次（at-least-once）+ 幂等处理"**的方案，而不是追求严格的"恰好一次"。

### Q2：消费者消费太慢导致 Rebalance 怎么办？

1. **增大 `max_poll_interval_ms`**：给消费者更多处理时间
2. **减少 `max_poll_records`**：每次拉取更少的消息，减少单次处理时间
3. **异步处理**：把消息处理放到线程池，poll() 循环快速返回
4. **增加消费者数量**：但要确保分区数 >= 消费者数
5. **优化处理逻辑**：批量写入数据库而不是逐条写入

### Q3：`auto_offset_reset` 的 `earliest` 和 `latest` 有什么区别？

- `earliest`：如果消费者组没有提交过 offset（首次消费），从分区的最早消息开始消费
- `latest`：如果消费者组没有提交过 offset，从分区的最新消息开始消费（只消费新消息）

注意：这个配置只在**消费者组第一次消费**时生效。如果消费者组已经有提交的 offset，会从已提交的 offset 继续消费。
