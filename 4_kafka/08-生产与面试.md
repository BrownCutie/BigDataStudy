### 2.1 消息丢失

```
┌─────────────────────────────────────────────────────────────────────┐
│                    消息丢失的三个环节及解决方案                         │
│                                                                     │
│  环节 1：生产者端                                                    │
│  ─────────────────────────────────────                              │
│  原因：acks=0 / 网络异常 / 缓冲区满                                  │
│  方案：                                                              │
│  - acks=all                                                         │
│  - retries=3（或更高）                                               │
│  - enable_idempotence=true                                          │
│  - 使用异步发送 + 回调，记录发送失败的消息                             │
│                                                                     │
│  环节 2：Broker 端                                                   │
│  ─────────────────────────────────────                              │

│  方案：                                                              │
│  - replication.factor >= 3                                          │
│  - min.insync.replicas >= 2                                         │

│  方案：                                                              │
│  - enable.auto.commit=false                                         │
│  - 处理完成后再手动提交 offset                                       │
│  - 幂等处理（防止重复消费）                                           │
│                                                                     │
│  完整防丢配置组合：                                                    │
│  Producer: acks=all + retries + idempotence                        │
│  Broker: replication.factor=3 + min.insync.replicas=2              │
│  Consumer: enable.auto.commit=false + 手动提交                       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 重复消费

```
┌─────────────────────────────────────────────────────────────────────┐
│                    重复消费的原因及解决方案                             │
│                                                                     │
│  原因：                                                              │
│  1. 生产者重试（网络超时导致重发）                                    │
│  2. 消费者 Rebalance（offset 重复提交）                               │

│  │                                                             │     │
│  │  方法 2：Redis 去重                                         │     │
│  │  用 eventId 作为 Key，SETNX 检查是否已处理                    │     │

│  │  （UPDATE 本身就是幂等的，执行多次结果一样）                    │     │
│  │                                                             │     │
│  │  方法 4：状态机                                               │     │
│  │  只接受状态向前的变更：草稿 → 发布 ✓，发布 → 草稿 ✗           │     │

│                                                                     │
```

### 2.4 Rebalance 风暴

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Rebalance 风暴的原因与解决                          │
│                                                                     │
│  什么是 Rebalance 风暴？                                             │
│  → 消费者组在短时间内频繁触发 Rebalance，导致消费反复暂停              │
│  → 所有消费者都在做 Rebalance，没有人在消费消息                       │
│  → Consumer Lag 飙升                                                │
│                                                                     │
│  常见原因：                                                          │
│  1. 消费者处理消息时间 > max.poll.interval.ms                        │
│     → 被踢出组 → 触发 Rebalance → 加入组 → 又被踢出 → 死循环         │
│                                                                     │
│  2. GC 停顿过长                                                     │
│     → 心跳超时 → 被踢出组 → Rebalance                               │
│                                                                     │
│  3. 网络抖动                                                        │
│     → 心跳丢失 → 被踢出组 → 网络恢复 → 加入组 → Rebalance           │
│                                                                     │
│  4. 消费者频繁启停                                                  │
│     → 每次启停都触发 Rebalance                                     │
│                                                                     │
│  解决方案：                                                          │
│  1. 增大 max.poll.interval.ms（根据实际处理时间设置）                 │
│  2. 减少 max.poll.records（每次拉取更少消息）                        │
│  3. 优化 GC 参数（减少 Full GC 停顿）                               │
│  4. 使用 CooperativeStickyAssignor（增量 Rebalance）                 │
│  5. 心跳线程和处理线程分离（不要在 poll 循环中做耗时操作）             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 Spark + Kafka

```python
# PySpark Structured Streaming 消费 Kafka
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, from_json, window

spark = SparkSession.builder \
    .appName("DemoAnalytics") \
    .config("spark.jars.packages", "org.apache.spark:spark-sql-kafka-0-10_2.13:3.5.8") \
    .getOrCreate()

# 从 Kafka 读取
df = spark.readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "localhost:9092") \
    .option("subscribe", "demo-logs") \
    .option("startingOffsets", "earliest") \
    .load()

# 解析 JSON
parsed = df.select(
    from_json(col("value").cast("string"), "eventType STRING, data STRUCT<postId INT>").alias("event")
).select("event.*")

# 窗口统计
view_counts = parsed.filter(col("eventType") == "post_viewed") \
    .withWatermark("timestamp", "1 minute") \
    .groupBy(window("timestamp", "5 minutes"), col("data.postId")) \
    .count()

# 写入控制台（也可以写入 Kafka、ES 等）
query = view_counts.writeStream \
    .outputMode("update") \
    .format("console") \
    .start()

query.awaitTermination()
```

## 五、15 道高频面试题

### Q1：Kafka 的架构是什么样的？

Kafka 采用分布式架构，核心组件包括：

- **Broker**：Kafka 服务节点，存储分区数据
- **Topic**：消息的逻辑分类
- **Partition**：Topic 的物理分片，实现并行处理
- **Producer**：消息生产者
- **Consumer / Consumer Group**：消息消费者/消费者组
- **Controller**（KRaft 模式）：管理集群元数据，替代 ZooKeeper

消息写入 Partition 的日志文件，通过 offset 追踪消费进度。同一 Consumer Group 内竞争消费，不同 Group 之间广播。

### Q2：Kafka 如何保证消息顺序性？

Kafka 只保证**同一 Partition 内消息有序**。全局有序需要：

1. 单分区（牺牲吞吐量，不推荐）
2. 相同 Key 的消息发到同一分区（推荐）：通过 `hash(Key) % 分区数` 实现

在数据分析项目中，用 `postId` 作为 Key，保证同一篇文章的事件（发布 → 评论 → 点赞）有序。

### Q3：Kafka 如何保证消息不丢失？

**生产者端**：`acks=all` + `retries>0` + `enable_idempotence=True`

**Broker 端**：
- `replication.factor>=3`
- `min.insync.replicas>=2`
- `unclean.leader.election.enable=false`

**消费者端**：
- `enable.auto.commit=false`
- 处理完成后再手动提交 offset

### Q4：Kafka 如何保证消息不重复消费？

完全避免重复消费很难，推荐**"至少一次 + 幂等处理"**：

1. 生产者开启幂等性（`enable_idempotence=true`），避免重试导致重复
2. 消费者手动提交 offset
3. 消费者端幂等处理：数据库唯一约束、Redis SETNX、UPDATE 操作天然幂等

### Q5：Kafka 的 acks 参数有什么作用？

- `acks=0`：不等确认，最快，可能丢消息
- `acks=1`：Leader 写入成功即返回，中等可靠性
- `acks=all`（或 `-1`）：ISR 中所有副本写入成功才返回，最可靠

推荐生产环境使用 `acks=all`。

### Q6：什么是 ISR？它和 AR 有什么区别？

- **AR（All Replicas）**：一个 Partition 的所有副本集合
- **ISR（In-Sync Replicas）**：与 Leader 保持同步的副本子集

Follower 如果落后 Leader 超过 `replica.lag.time.max.ms`（默认 30 秒），会被移出 ISR。Leader 挂了时，新 Leader 从 ISR 中选举，保证数据不丢失。

### Q7：Kafka 的消费者组机制是怎样的？

- 同一消费者组内的消费者**竞争消费**：一个分区同一时间只被组内一个消费者消费
- 不同消费者组**独立消费**：每组都能收到全部分区的消息（广播）
- 消费者数量 > 分区数量时，多出来的消费者空闲
- 消费者数量 <= 分区数量时，部分消费者消费多个分区

### Q8：什么是 Rebalance？如何避免频繁 Rebalance？

Rebalance 是消费者组重新分配分区的过程。期间所有消费者暂停消费。

触发条件：消费者加入/离开、心跳超时、订阅 Topic 变化。

避免频繁 Rebalance：
1. `max.poll.interval.ms` 设置合理（大于单批处理时间）
2. 优化消费逻辑，减少处理时间
3. 使用 `CooperativeStickyAssignor`（增量 Rebalance）
4. 避免频繁启停消费者
5. 调优 GC，避免长时间停顿

### Q9：Kafka 为什么这么快？

1. **顺序写磁盘**：只追加写入，磁盘顺序写速度可达 600MB/s
2. **零拷贝（Zero Copy）**：`sendfile()` 系统调用，减少数据在内核空间和用户空间的拷贝
3. **批量处理**：消息攒够一批再发送
4. **压缩**：减少网络传输和磁盘存储
5. **分区并行**：多分区并行读写
6. **页缓存**：利用操作系统页缓存，减少磁盘 IO

### Q10：Kafka 的零拷贝是什么？

传统数据传输路径（4 次拷贝）：
```
磁盘 → 内核缓冲区 → 用户缓冲区 → Socket 缓冲区 → 网卡
```

零拷贝路径（2 次拷贝）：
```
磁盘 → 内核缓冲区 → 网卡
```

通过 `java.nio.channels.FileChannel.transferTo()` 底层调用 `sendfile()` 系统调用，数据不经过用户空间，直接从页缓存发送到网卡，大幅提升性能。

### Q11：Kafka 如何处理消费积压？

1. **紧急处理**：增加消费者数量（需要先增加分区数）
2. **优化消费逻辑**：批量写入数据库、异步处理、减少外部调用
3. **临时方案**：写一个临时消费者，快速消费积压消息，后续补偿处理
4. **根本解决**：分析积压原因，针对性优化

### Q12：Kafka 事务的原理是什么？

Kafka 事务保证跨分区的原子性写入：

1. 生产者开启事务（`transactional_id`）
2. 调用 `begin_transaction()` 开始事务
3. 向多个分区发送消息，消息标记为"未提交"
4. 调用 `commit_transaction()` 提交事务，消息变为可见
5. 如果失败，调用 `abort_transaction()` 回滚

消费者需要设置 `isolation.level=read_committed`，才能只读到已提交的消息。

### Q13：Kafka 和 RabbitMQ 如何选择？

| 维度 | Kafka | RabbitMQ |
|------|-------|----------|
| 吞吐量 | 百万级/秒 | 万级/秒 |
| 延迟 | ms 级 | us 级 |
| 消息保留 | 持久化，可回溯 | 消费后删除 |
| 路由能力 | 弱（只有分区） | 强（多种 Exchange） |
| 生态 | 大数据（Flink/Spark） | 业务系统 |

选 Kafka：大数据场景、日志收集、流处理
选 RabbitMQ：业务消息、RPC、需要复杂路由

### Q14：Kafka 4.x 为什么去掉了 ZooKeeper？

1. **运维简化**：不需要维护两套系统（Kafka + ZooKeeper）
2. **元数据性能提升**：KRaft 使用 Raft 协议管理元数据，支持百万级分区
3. **消除单点故障**：ZooKeeper 的性能瓶颈和运维复杂度一直是 Kafka 的问题
4. **更快启动**：KRaft 模式下元数据加载更快

KRaft 通过内置的 Raft 协议实现元数据的共识，Controller 节点负责管理元数据，不再依赖外部系统。

### Q15：说一下你对 Kafka 的整体理解？

Kafka 是一个**分布式事件流平台**，核心设计是**分布式提交日志**。它的三个核心能力：

1. **发布/订阅消息流**：生产者发送消息到 Topic，消费者从 Topic 订阅
2. **持久化存储**：消息持久化到磁盘，支持回溯历史数据
3. **流处理**：配合 Kafka Streams / Flink / Spark 做实时计算

Kafka 的核心设计思想：
- 顺序写 + 零拷贝 → 高吞吐
- 分区 + 副本 → 可扩展 + 高可靠
- 消费者组 → 灵活的消费模型

在数据分析项目中，Kafka 作为事件总线，连接 Web 后端、Elasticsearch、通知服务、实时统计等子系统，实现系统解耦、异步处理和削峰填谷。
