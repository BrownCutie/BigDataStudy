# 02 - 安装与配置：搭建 Kafka 4.2.0 环境

## 一、Kafka 版本选择与下载

### 1.1 版本说明

Kafka 从 4.0 开始正式移除了对 ZooKeeper 的依赖，使用内置的 **KRaft（Kafka Raft）** 模式来管理集群元数据。这意味着：

```
┌──────────────────────────────────────────────────────────┐
│              Kafka 3.x vs Kafka 4.x 架构对比               │
│                                                          │
│  Kafka 3.x（旧模式）：                                      │
│                                                          │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐         │
│  │  Broker 1│◄───→│ZooKeeper │◄───→│  Broker 2│         │
│  └──────────┘     │  集群     │     └──────────┘         │
│                   │ (3~5节点) │                          │
│  ┌──────────┐     └──────────┘     ┌──────────┐         │
│  │  Broker 3│◄────────────────────→│  Broker N│         │
│  └──────────┘                     └──────────┘         │
│  问题：多了一个 ZooKeeper 集群要运维                        │
│                                                          │
│  Kafka 4.x（KRaft 模式）：                                  │
│                                                          │
│  ┌──────────────────────────────────────────┐            │
│  │           Kafka KRaft 集群                 │            │
│  │                                          │            │
│  │  ┌──────────┐  ┌──────────┐  ┌────────┐ │            │
│  │  │ Broker+  │  │ Broker+  │  │Broker+ │ │            │
│  │  │Controller│◄→│Controller│◄→│Control │ │            │
│  │  │  (节点1)  │  │  (节点2)  │  │(节点3) │ │            │
│  │  └──────────┘  └──────────┘  └────────┘ │            │
│  │                                          │            │
│  └──────────────────────────────────────────┘            │
│  优势：一个集群搞定，不需要 ZooKeeper！                      │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 1.2 下载 Kafka

```bash
# 从阿里云镜像下载（国内更快）
cd ~/Programme/claude-study
curl -O https://mirrors.aliyun.com/apache/kafka/4.2.0/kafka_2.13-4.2.0.tgz
```

> **说明**：`kafka_2.13-4.2.0` 中的 `2.13` 是 Scala 编译版本，`4.2.0` 是 Kafka 版本。

### 1.3 安装

```bash
# 创建安装目录
mkdir -p /opt/kafka

# 解压
tar -xzf kafka_2.13-4.2.0.tgz -C /opt/kafka/

# 重命名为简洁路径
mv /opt/kafka/kafka_2.13-4.2.0 /opt/kafka/4.2.0
```

### 1.4 配置环境变量

将以下内容添加到 `~/.zshrc`：

```bash
# Kafka
export KAFKA_HOME=/opt/kafka/4.2.0
export PATH=$PATH:$KAFKA_HOME/bin
```

使配置生效：

```bash
source ~/.zshrc
```

验证：

```bash
echo $KAFKA_HOME
# 输出：/opt/kafka/4.2.0

kafka-topics.sh --version
# 输出版本信息
```

## 二、KRaft 模式核心配置

Kafka 4.x 的配置文件位于 `/opt/kafka/4.2.0/config/server.properties`。我们来逐行理解关键配置：

### 2.1 核心配置详解

```properties
# ========================================
# 节点角色配置
# ========================================

# process.roles 定义节点的角色
# broker：处理客户端请求（生产者/消费者）
# controller：管理集群元数据（替代 ZooKeeper）
# 单机学习时用 broker,controller（一个节点身兼两职）
process.roles=broker,controller

# 节点 ID，集群中每个节点必须唯一
# 单机模式设为 1 即可
node.id=1

# ========================================
# Controller 配置（集群内部通信）
# ========================================

# Controller 的通信地址
# 格式：节点ID@主机:端口
controller.quorum.bootstrap.servers=localhost:9093

# ========================================
# 网络配置
# ========================================

# listeners：本机监听的协议和端口
# PLAINTEXT：客户端连接端口（9092）
# CONTROLLER：Controller 集群内部通信端口（9093）
listeners=PLAINTEXT://:9092,CONTROLLER://:9093

# advertised.listeners：对外广播的地址
# 客户端通过这个地址连接 Kafka
# 单机用 localhost，集群部署时改为实际 IP
advertised.listeners=PLAINTEXT://localhost:9092,CONTROLLER://localhost:9093

# Controller 的监听器名称映射
controller.listener.names=CONTROLLER

# 内部通信协议映射
inter.broker.listener.name=PLAINTEXT

# ========================================
# 存储配置
# ========================================

# 日志数据存储目录
log.dirs=/opt/kafka/4.2.0/data

# 每个分区日志段的最大大小（默认 1GB）
log.segment.bytes=1073741824

# ========================================
# 其他常用配置
# ========================================

# 自动创建 Topic（生产环境建议关闭）
auto.create.topics.enable=true

# 默认分区数量
num.partitions=1

# 默认副本因子（单机模式只能设为 1）
default.replication.factor=1

# Controller 所使用的元数据快照目录
metadata.log.dir=/opt/kafka/4.2.0/data/kraft-combined-logs
```

### 2.2 配置项速查表

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `process.roles` | - | 节点角色：`broker`、`controller`、`broker,controller` |
| `node.id` | - | 节点唯一 ID |
| `controller.quorum.bootstrap.servers` | - | Controller 通信地址 |
| `listeners` | - | 监听地址，格式 `协议://主机:端口` |
| `advertised.listeners` | - | 对外广播地址 |
| `log.dirs` | `/tmp/kafka-logs` | 数据存储目录 |
| `num.partitions` | 1 | 默认分区数 |
| `auto.create.topics.enable` | `true` | 是否自动创建 Topic |

## 三、启动与停止

### 3.1 首次启动：格式化存储

KRaft 模式在首次启动前需要**格式化存储目录**，生成集群 ID：

```bash
# 步骤 1：生成一个唯一的集群 ID
kafka-storage.sh random-uuid
# 输出类似：7x_xxAhIRVOPT9YhC4ZksQ

# 步骤 2：用生成的 ID 格式化存储目录
# 注意：替换 <集群ID> 为上一步的输出
kafka-storage.sh format \
  -t <集群ID> \
  -c $KAFKA_HOME/config/server.properties \
  --standalone
```

> `--standalone` 表示单节点模式。多节点集群部署时不需要这个参数。

### 3.2 启动 Kafka

```bash
# 前台启动（可以看到日志输出，适合调试）
kafka-server-start.sh $KAFKA_HOME/config/server.properties

# 后台启动（生产环境推荐）
nohup kafka-server-start.sh \
  $KAFKA_HOME/config/server.properties \
  > $KAFKA_HOME/logs/kafka.log 2>&1 &

# 查看进程
jps | grep Kafka
# 或
lsof -i :9092
```

### 3.3 停止 Kafka

```bash
# 优雅停止
kafka-server-stop.sh

# 如果优雅停止不生效，强制杀进程
# 先找到进程 ID
lsof -i :9092 -t
# 再杀掉
kill -9 <PID>
```

## 四、命令行工具验证

Kafka 提供了一组命令行工具，位于 `$KAFKA_HOME/bin/` 目录下。我们用这些工具来验证 Kafka 是否正常工作。

### 4.1 Topic 管理工具

```bash
# 列出所有 Topic
kafka-topics.sh --list --bootstrap-server localhost:9092

# 创建 Topic
kafka-topics.sh --create \
  --topic blog-events \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1

# 查看 Topic 详情
kafka-topics.sh --describe \
  --topic blog-events \
  --bootstrap-server localhost:9092

# 输出示例：
# Topic: blog-events  TopicId: xxx  PartitionCount: 3  ReplicationFactor: 1
# Topic: blog-events  Partition: 0  Leader: 1  Replicas: 1  Isr: 1
# Topic: blog-events  Partition: 1  Leader: 1  Replicas: 1  Isr: 1
# Topic: blog-events  Partition: 2  Leader: 1  Replicas: 1  Isr: 1

# 修改分区数（只能增加，不能减少）
kafka-topics.sh --alter \
  --topic blog-events \
  --bootstrap-server localhost:9092 \
  --partitions 6

# 删除 Topic
kafka-topics.sh --delete \
  --topic blog-events \
  --bootstrap-server localhost:9092
```

### 4.2 命令行生产消息

```bash
# 启动交互式生产者
kafka-console-producer.sh \
  --topic blog-events \
  --bootstrap-server localhost:9092

# 输入消息（每行一条）：
> {"type":"post_published","postId":1,"title":"Kafka入门指南","timestamp":"2026-04-15T10:00:00"}
> {"type":"comment_added","postId":1,"content":"写得不错！","timestamp":"2026-04-15T10:01:00"}
> {"type":"post_viewed","postId":1,"userId":100,"timestamp":"2026-04-15T10:02:00"}
> ^C  # Ctrl+C 退出
```

### 4.3 命令行消费消息

```bash
# 从最新消息开始消费（默认）
kafka-console-consumer.sh \
  --topic blog-events \
  --bootstrap-server localhost:9092

# 从头开始消费所有消息
kafka-console-consumer.sh \
  --topic blog-events \
  --bootstrap-server localhost:9092 \
  --from-beginning

# 指定消费者组（重要！）
kafka-console-consumer.sh \
  --topic blog-events \
  --bootstrap-server localhost:9092 \
  --group demo-processor \
  --from-beginning

# 只看某个分区的消息
kafka-console-consumer.sh \
  --topic blog-events \
  --bootstrap-server localhost:9092 \
  --partition 0 \
  --from-beginning
```

### 4.4 消费者组管理


## 五、数据分析项目：创建 blog-events Topic

根据我们的数据平台设计，创建两个核心 Topic：

```bash
# 1. 用户事件 Topic（文章发布、评论、浏览等）
kafka-topics.sh --create \
  --topic blog-events \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1

# 2. 用户行为日志 Topic（浏览记录、搜索记录等）
kafka-topics.sh --create \
  --topic demo-logs \
  --bootstrap-server localhost:9092 \
  --partitions 6 \
  --replication-factor 1
```

**为什么 demo-logs 分区数更多？** 因为用户行为日志的量级远大于用户事件（浏览量 >> 评论量），需要更多的分区来提升并行消费能力。

发送一些测试数据：

```bash
# 发送用户事件
kafka-console-producer.sh --topic blog-events --bootstrap-server localhost:9092
> {"type":"post_published","postId":1,"title":"我的第一篇数据平台","author":"browncutie"}
> {"type":"comment_added","postId":1,"userId":100,"content":"写得不错！"}
> {"type":"post_viewed","postId":1,"userId":101}

# 发送行为日志
kafka-console-producer.sh --topic demo-logs --bootstrap-server localhost:9092
> {"action":"page_view","url":"/posts/1","userId":101,"ip":"192.168.1.100","ts":"2026-04-15T10:00:00"}
> {"action":"search","keyword":"Kafka","userId":101,"ts":"2026-04-15T10:01:00"}
```

## 六、常见问题排查

### Q1：启动报错 "Node id 1 is already registered"

说明存储目录已经格式化过了，不需要重复执行 `kafka-storage.sh format`。如果需要重新格式化：

```bash
# 清空数据目录
rm -rf /opt/kafka/4.2.0/data

# 重新生成集群 ID 并格式化
CLUSTER_ID=$(kafka-storage.sh random-uuid)
kafka-storage.sh format -t $CLUSTER_ID -c $KAFKA_HOME/config/server.properties --standalone
```

### Q2：连接失败 "Connection refused"

```bash
# 检查 Kafka 是否在运行
lsof -i :9092

# 检查端口是否被防火墙拦截
telnet localhost 9092
```

### Q3：macOS Apple Silicon 兼容性

Kafka 4.2.0 基于 Java，在 Apple Silicon（M1/M2/M3）上运行完全没有问题，前提是安装了 Java 21：

```bash
java -version
# 应显示 java version "21.x.x"

# 如果没有安装 Java 21
brew install openjdk@21
```

## 七、面试题

### Q1：KRaft 模式和 ZooKeeper 模式有什么区别？

| 维度 | ZooKeeper 模式（3.x） | KRaft 模式（4.x） |
|------|----------------------|-------------------|
| 元数据存储 | 外部 ZooKeeper 集群 | Kafka 内部 Raft 协议 |
| 集群组件 | Kafka + ZooKeeper | 只需要 Kafka |
| 运维复杂度 | 高（两套系统） | 低（一套系统） |
| 元数据上限 | 受 ZK 性能限制 | 支持百万级分区 |
| Controller 选举 | 通过 ZK 选举 | 通过 Raft 协议选举 |

KRaft 的核心优势是**去掉了 ZooKeeper 依赖**，简化了运维，同时元数据管理性能大幅提升。

### Q2：`listeners` 和 `advertised.listeners` 有什么区别？

- `listeners`：Kafka **本机实际监听**的地址和端口。决定 Kafka 在哪个网络接口上接受连接。
- `advertised.listeners`：Kafka **告诉客户端**的地址和端口。客户端通过这个地址来连接 Kafka。

在单机模式下两者可以一样。但在集群部署时，如果 Kafka 部署在内网（如 Docker 网络或云内网），客户端在外网访问，就需要 `advertised.listeners` 设置为外网可达的地址。
