# 06 - 高级特性：深入 Kafka 的强大能力

## 一、消息保留策略（Retention）

Kafka 不是"消费完就删"的消息队列，消息会保留一段时间。这给了我们回溯历史数据的能力。

### 1.1 保留策略类型

```
┌─────────────────────────────────────────────────────────────────────┐
│                    消息保留策略                                       │
│                                                                     │
│  Kafka 提供两种保留策略（满足任一即删除）：                              │
│                                                                     │
│  1. 按时间保留                                                       │
│     retention.ms=604800000  (默认 7 天)                              │
│     → 超过 7 天的消息自动删除                                        │
│                                                                     │
│  2. 按大小保留                                                       │
│     retention.bytes=1073741824  (默认 -1，即不限制)                   │
│     → 分区总大小超过 1GB 时，删除最老的消息                           │
│                                                                     │
│  时间线示例（保留 7 天）：                                             │
│                                                                     │
│  Day 1    Day 3    Day 5    Day 7    Day 8    Day 9                  │
│  ┌───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┐               │
│  │ m │ m │ m │ m │ m │ m │ m │ m │ m │ m │ m │ m │               │
│  └───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┘               │
│  ↑ 保留范围 ↑                         ↑ 超过 7 天，被删除 ↑          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 配置保留策略

```bash
# 创建 Topic 时指定保留策略
kafka-topics.sh --create \
  --topic blog-events \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config retention.ms=86400000        # 保留 1 天
  --config retention.bytes=536870912    # 或保留 512MB

# 修改已有 Topic 的保留策略
kafka-configs.sh --alter \
  --bootstrap-server localhost:9092 \
  --entity-type topics \
  --entity-name blog-events \
  --add-config retention.ms=172800000   # 改为保留 2 天
```

### 1.3 数据分析项目的保留策略设计

| Topic | 保留时间 | 保留大小 | 理由 |
|-------|---------|---------|------|
| `blog-events` | 7 天 | 不限 | 事件需要保留足够时间供回溯 |
| `demo-logs` | 3 天 | 10GB | 行为日志量大，不需要保留太久 |
| `demo-notifications` | 1 天 | 不限 | 通知消息消费后无需长期保留 |

## 二、日志压缩（Log Compaction）

日志压缩是 Kafka 的一个独特功能，它只保留每个 Key 的最新值：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    日志压缩（Log Compaction）                         │
│                                                                     │
│  压缩前（同一个 Key 有多条消息）：                                     │
│                                                                     │
│  ┌────────────────────────────────────────────────────────┐        │
│  │ Key=post_1  Value="草稿"    offset=0                    │        │
│  │ Key=post_1  Value="审核中"  offset=1                    │        │
│  │ Key=post_1  Value="已发布"  offset=2                    │        │
│  │ Key=post_2  Value="草稿"    offset=3                    │        │
│  │ Key=post_1  Value="已更新"  offset=4                    │        │
│  └────────────────────────────────────────────────────────┘        │
│                                                                     │
│  压缩后（每个 Key 只保留最新值）：                                     │
│                                                                     │
│  ┌────────────────────────────────────────────────────────┐        │
│  │ Key=post_1  Value="已更新"  offset=4  ← post_1 最新值  │        │
│  │ Key=post_2  Value="草稿"    offset=3  ← post_2 最新值  │        │
│  └────────────────────────────────────────────────────────┘        │
│                                                                     │
│  适用场景：                                                          │
│  - 用户最新状态（最后登录时间、最后浏览的文章）                        │
│  - 配置变更历史（只关心最新配置）                                     │
│  - 数据库 binlog 同步（只关心最新数据）                               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

```bash
# 创建压缩 Topic
kafka-topics.sh --create \
  --topic demo-post-latest \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config cleanup.policy=compact          # 启用日志压缩
  --config delete.retention.ms=86400000    # 被压缩的消息再保留 1 天
  --config min.cleanable.dirty.ratio=0.01  # 1% 脏数据就触发压缩
```

```python
# 日志压缩示例：更新文章最新状态
producer.send('demo-post-latest',
    key=str(post_id),
    value={'title': '新标题', 'status': 'published'}
)

# 消费者总能读到每个 post_id 的最新状态
# 即使中间状态已经被压缩掉了
```

```
┌──────────────────────────────────────────────────────────┐
│              retention vs compaction 对比                  │
│                                                          │
│  retention（默认）：                                      │
│  - 按时间或大小删除旧消息                                   │
│  - 保留的是一个时间窗口内的所有消息                          │
│  - 适合日志、事件流场景                                     │
│                                                          │
│  compaction：                                             │
│  - 按 Key 保留最新值                                      │
│  - 保留的是每个 Key 的最终状态                              │
│  - 适合状态同步、变更日志场景                                │
│                                                          │
│  可以同时使用：                                            │
│  cleanup.policy=compact,delete                            │
│  → 先压缩，超过时间的再删除                                 │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## 三、消息格式与版本演进

### 3.1 消息格式

Kafka 的消息在磁盘上的存储格式经历了多次演进：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Kafka 消息格式（v2，当前版本）                      │
│                                                                     │
│  ┌─────────────────────────────────────────────────────┐            │
│  │                  Record Batch                        │            │
│  │  ┌───────────────────────────────────────────────┐  │            │
│  │  │  Batch Header（8 字节）                         │  │            │
│  │  │  - baseOffset: 第一条消息的 offset              │  │            │
│  │  │  - batchLength: batch 总长度                    │  │            │
│  │  │  - partitionLeaderEpoch: Leader 纪元            │  │            │
│  │  │  - magic: 版本号（v2 = 2）                     │  │            │
│  │  │  - crc: 校验和                                 │  │            │
│  │  └───────────────────────────────────────────────┘  │            │
│  │  ┌───────────────────────────────────────────────┐  │            │
│  │  │  Records                                       │  │            │
│  │  │  ┌─────────┐ ┌─────────┐ ┌─────────┐         │  │            │
│  │  │  │Record 0 │ │Record 1 │ │Record 2 │  ...    │  │            │
│  │  │  │offset   │ │offset+1 │ │offset+2 │         │  │            │
│  │  │  │key      │ │key      │ │key      │         │  │            │
│  │  │  │value    │ │value    │ │value    │         │  │            │
│  │  │  │headers  │ │headers  │ │headers  │         │  │            │
│  │  │  │timestamp│ │timestamp│ │timestamp│         │  │            │
│  │  │  └─────────┘ └─────────┘ └─────────┘         │  │            │
│  │  └───────────────────────────────────────────────┘  │            │
│  └─────────────────────────────────────────────────────┘            │
│                                                                     │
│  v2 格式的优势：                                                      │
│  1. 批量存储，减少开销                                               │
│  2. 支持更高效的压缩（整个 Batch 一起压缩）                            │
│  3. 支持事务标记                                                    │
│  4. 支持 Record Header（自定义元数据）                                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 消息版本兼容性

| 版本 | Kafka 版本 | 特点 |
|------|-----------|------|
| v0 | 0.10.0 之前 | 每条消息独立，无压缩优化 |
| v1 | 0.10.0 ~ 0.11.0 | 增加 timestamp 字段 |
| v2 | 0.11.0+（当前） | Record Batch，支持事务和幂等 |

Kafka 支持不同版本的消息格式共存，升级 Broker 时不需要一次性转换所有消息。

## 四、消费者滞后（Consumer Lag）监控

Consumer Lag 是消费者落后于生产者的消息数量，是 Kafka 运维中最重要的监控指标。

### 4.1 什么是 Consumer Lag？

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Consumer Lag 示意图                                │
│                                                                     │
│  Partition 0 的消息：                                                │
│                                                                     │
│  ┌───────┬───────┬───────┬───────┬───────┬───────┬───────┐         │
│  │Msg 0  │Msg 1  │Msg 2  │Msg 3  │Msg 4  │Msg 5  │Msg 6  │         │
│  │off=0  │off=1  │off=2  │off=3  │off=4  │off=5  │off=6  │         │
│  └───────┴───────┴───────┴───────┴───────┴───────┴───────┘         │
│  ↑                                                    ↑            │
│  LOG-END-OFFSET = 7                                    ↑            │
│  (最新消息的 offset + 1)                          CURRENT-OFFSET = 3 │
│                                                    (消费者已消费到)    │
│                                                                     │
│  Consumer Lag = LOG-END-OFFSET - CURRENT-OFFSET                     │
│               = 7 - 3                                              │
│               = 4  (还有 4 条消息没消费)                              │
│                                                                     │
│  Lag = 0 → 消费者完全跟上生产者（理想状态）                            │
│  Lag 持续增长 → 消费者处理不过来 → 可能积压                            │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 查看 Consumer Lag

```bash
# 查看消费者组的 Lag 详情
kafka-consumer-groups.sh --describe \
  --group demo-processor \
  --bootstrap-server localhost:9092

# 输出示例：
# GROUP           TOPIC        PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG     CONSUMER-ID
# demo-processor  blog-events  0          150             200             50      ...
# demo-processor  blog-events  1          120             180             60      ...
# demo-processor  blog-events  2          180             180             0       ...
```

### 4.3 Lag 告警策略

```
┌──────────────────────────────────────────────────────────┐
│              Consumer Lag 告警阈值建议                      │
│                                                          │
│  Lag < 1000       → 正常（绿色）                           │
│  Lag 1000 ~ 10000 → 注意（黄色），可能需要优化              │
│  Lag > 10000      → 告警（红色），消费者严重积压            │
│  Lag 持续增长      → 紧急，消费者已经跟不上生产速度          │
│                                                          │
│  处理积压的方法：                                          │
│  1. 增加消费者数量（前提：分区数 > 消费者数）               │
│  2. 优化消费逻辑（减少单条消息处理时间）                     │
│  3. 临时增加分区数（注意：增加分区后需要重新分配）            │
│  4. 批量处理（减少数据库写入次数）                           │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## 六、Kafka Streams 简介

Kafka Streams 是一个用 Java/Scala 编写的客户端库，用于处理 Kafka 中的实时数据流。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Kafka Streams 架构                                │
│                                                                     │
│  ┌──────────┐     ┌───────────────────────┐     ┌──────────┐       │
│  │  Kafka   │────→│   Kafka Streams App   │────→│  Kafka   │       │
│  │ (输入Topic)│    │                       │     │(输出Topic)│       │
│  └──────────┘     │  ┌───────┐ ┌───────┐  │     └──────────┘       │
│                   │  │Filter │→│ GroupBy│→ │                        │
│                   │  └───────┘ └───────┘  │     ┌──────────┐       │
│                   │  ┌───────┐ ┌───────┐  │     │  外部系统  │       │
│                   │  │Window │→│  Join │──│────→│  数据库   │       │
│                   │  └───────┘ └───────┘  │     └──────────┘       │
│                   └───────────────────────┘                        │
│                                                                     │
│  核心概念：                                                          │
│  - KStream：无界数据流（类似事件流）                                   │
│  - KTable：有状态的表（当前最新值，类似数据库表）                       │
│  - 时间窗口：滑动窗口、滚动窗口、会话窗口                               │
│  - 状态存储：RocksDB 本地状态存储                                     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 数据分析项目中的 Streams 示例

```java
// Java 示例：统计每篇文章的实时浏览量（5 分钟滚动窗口）
Properties props = new Properties();
props.put(StreamsConfig.APPLICATION_ID_CONFIG, "demo-view-counter");
props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
props.put(StreamsConfig.DEFAULT_KEY_SERDE_CLASS_CONFIG, Serdes.String().getClass());
props.put(StreamsConfig.DEFAULT_VALUE_SERDE_CLASS_CONFIG, Serdes.String().getClass());

StreamsBuilder builder = new StreamsBuilder();

// 从 demo-logs Topic 读取浏览事件
KStream<String, String> views = builder.stream("demo-logs");

// 过滤出浏览事件
KStream<String, String> pageViews = views
    .filter((key, value) -> value.contains("\"action\":\"page_view\""));

// 按 postId 分组，5 分钟滚动窗口统计
KTable<Windowed<String>, Long> viewCounts = pageViews
    .groupBy((key, value) -> extractPostId(value))
    .windowedBy(TimeWindows.of(Duration.ofMinutes(5)))
    .count();

// 将统计结果写入 output Topic
viewCounts.toStream()
    .map((windowedKey, count) -> new KeyValue<>(
        windowedKey.key(),
        String.format("{\"postId\":\"%s\",\"views\":%d,\"windowEnd\":%d}",
            windowedKey.key(), count, windowedKey.window().end())
    ))
    .to("demo-view-stats");

KafkaStreams streams = new KafkaStreams(builder.build(), props);
streams.start();
```

## 八、面试题

### Q1：Kafka 消息会被删除吗？什么时候删除？

会被删除。Kafka 通过保留策略自动删除旧消息：

- **按时间**：`retention.ms`（默认 7 天），超过时间的消息被删除
- **按大小**：`retention.bytes`，分区大小超过限制时删除最老的消息
- **日志压缩**：`cleanup.policy=compact`，相同 Key 只保留最新值

注意：Kafka 删除消息是**按日志段（Segment）删除的**，不是逐条删除。即使某条消息过期了，如果它所在的 Segment 中还有未过期的消息，它也不会被立即删除。

### Q2：Log Compaction 会导致消息丢失吗？

不会丢失"最终状态"。Log Compaction 只删除相同 Key 的旧版本消息，保留最新版本。如果业务只关心每个 Key 的最新值（比如用户最新状态），Log Compaction 不会丢失任何有意义的数据。

但如果业务需要保留所有历史变更记录（比如审计日志），就不应该使用 Log Compaction。

### Q3：Kafka Connect 和自己写消费者有什么区别？

| 维度 | Kafka Connect | 自己写消费者 |
|------|--------------|-------------|
| 开发成本 | 低（配置即可） | 高（需要写代码） |
| 灵活性 | 中（受限于 Connector 插件） | 高（完全自定义） |
| 运维 | 自动管理 offset、故障恢复 | 需要自己管理 |
| 并行度 | 自动分配 Task | 需要自己管理消费者组 |
| 适用场景 | 标准的数据导入导出 | 复杂的业务逻辑处理 |

建议：如果只是数据同步（Kafka → ES、Kafka → HDFS），优先用 Kafka Connect。如果有复杂的业务逻辑，自己写消费者。
