# 06 - 生产实战与面试：Iceberg 进阶指南

## 一、Iceberg 生产环境最佳实践

### 1.1 表设计规范

```
┌────────────────────────────────────────────────────────────────┐
│                   Iceberg 表设计最佳实践                        │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  1. 使用 format-version=2                                     │
│     - 支持 DELETE 操作的行级删除（不需要重写整个文件）         │
│     - 支持更高效的元数据管理                                   │
│                                                                │
│  2. 选择合适的分区策略                                         │
│     - 时间序列数据：days() 或 hours()                          │
│     - 高基数列：bucket(N, column)                              │
│     - 避免分区过细（分区数过多导致小文件）                     │
│     - 避免分区过粗（分区裁剪效果差）                           │
│                                                                │
│  3. 设置合理的目标文件大小                                     │
│     - 一般 128MB ~ 512MB                                      │
│     - 太小：文件数量多，元数据膨胀                             │
│     - 太大：并行度低，读取浪费                                 │
│                                                                │
│  4. 使用列默认值                                               │
│     - 新增列时指定 DEFAULT 值，旧数据也能有合理的默认值        │
│                                                                │
│  5. 为列添加注释                                               │
│     - 便于团队理解和维护                                       │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 1.2 写入优化

```sql
-- 1. 使用分布式写入
ALTER TABLE iceberg.demo.posts SET TBLPROPERTIES (
    'write.distribution-mode' = 'hash'   -- 按分区键哈希分布写入
);

-- 2. 设置合理的目标文件大小
ALTER TABLE iceberg.demo.posts SET TBLPROPERTIES (
    'write.target-file-size-bytes' = '268435456'   -- 256MB
);

-- 3. 使用 MERGE INTO 代替 DELETE + INSERT
-- 低效方式：
-- DELETE FROM iceberg.demo.posts WHERE id = 1;
-- INSERT INTO iceberg.demo.posts VALUES (1, 'new_title', ...);
--
-- 高效方式：
-- MERGE INTO iceberg.demo.posts t
-- USING (SELECT 1 AS id, 'new_title' AS title, ...) s
-- ON t.id = s.id
-- WHEN MATCHED THEN UPDATE SET *
-- WHEN NOT MATCHED THEN INSERT *;
```

### 1.3 查询优化

```sql
-- 1. 使用文件级过滤
-- Iceberg 自动做分区裁剪，确保查询条件中使用分区列
-- 好：WHERE created_at >= TIMESTAMP '2026-04-01'  (分区裁剪)
-- 差：WHERE title LIKE '%Kafka%'                 (全文件扫描)

-- 2. 利用 Iceberg 的元数据过滤
-- Iceberg 在 Manifest File 中记录了每列的 min/max 值
-- 查询条件中包含过滤条件时，Iceberg 可以跳过不包含匹配数据的文件

-- 3. 设置合理的读取参数
SET spark.sql.iceberg.planning.max-concurrent-file-scans = 50;

-- 4. 使用 Spark 缓存
-- 对于频繁查询的热数据
spark.sql("CACHE TABLE iceberg.demo.posts_cache AS SELECT * FROM iceberg.demo.posts");
```

### 2.2 JdbcCatalog 配置（本课程使用）

```properties
# spark-defaults.conf
spark.sql.catalog.iceberg=org.apache.iceberg.spark.SparkCatalog
spark.sql.catalog.iceberg.catalog-impl=org.apache.iceberg.jdbc.JdbcCatalog
spark.sql.catalog.iceberg.uri=jdbc:mysql://localhost:3306/iceberg
spark.sql.catalog.iceberg.jdbc.user=root
spark.sql.catalog.iceberg.jdbc.password=你的MySQL密码
spark.sql.catalog.iceberg.warehouse=hdfs://localhost:9000/iceberg-warehouse
spark.sql.catalog.iceberg.lock-impl=org.apache.iceberg.jdbc.JdbcLock
```

**JdbcCatalog 优势：**
- 部署简单，只需要一个 MySQL/PostgreSQL 实例
- 不依赖 Hive Metastore
- 通过 JDBC Lock 实现并发控制

**JdbcCatalog 限制：**
- 不支持分支（Branch）功能（需要 Nessie）
- 单个 Catalog 实例可能有性能瓶颈（大规模场景）

## 三、维护操作

### 3.1 rewrite_data_files（数据文件重写）

合并小文件、优化文件大小和排序。

```sql
-- 基本用法：合并小文件
CALL iceberg.system.rewrite_data_files('iceberg.demo.posts');

-- 指定目标文件大小
CALL iceberg.system.rewrite_data_files(
    table => 'iceberg.demo.posts',
    options => map(
        'target-file-size-bytes', '268435456',   -- 256MB
        'min-file-size-bytes', '16777216'          -- 小于 16MB 的文件才合并
    )
);

-- 按排序重写（提高查询性能）
CALL iceberg.system.rewrite_data_files(
    table => 'iceberg.demo.posts',
    strategy => 'sort',
    sort_order => 'created_at ASC',
    options => map('target-file-size-bytes', '268435456')
);

-- 按分区重写（只重写指定分区）
CALL iceberg.system.rewrite_data_files(
    table => 'iceberg.demo.posts',
    strategy => 'sort',
    sort_order => 'created_at ASC',
    options => map(
        'target-file-size-bytes', '268435456',
        'rewrite-all', 'false'
    )
);
```

### 3.2 expire_snapshots（快照过期）

```sql
-- 过期指定时间之前的快照
CALL iceberg.system.expire_snapshots(
    'iceberg.demo.posts',
    TIMESTAMP '2026-04-08 00:00:00'
);

-- 保留最近 N 个快照
CALL iceberg.system.expire_snapshots(
    'iceberg.demo.posts',
    TIMESTAMP '1970-01-01 00:00:00',
    10
);

-- 同时清理过期快照的元数据文件
CALL iceberg.system.expire_snapshots(
    'iceberg.demo.posts',
    TIMESTAMP '2026-04-08 00:00:00',
    100,
    true    -- retain_last：保留最后一个快照
);
```

### 3.3 remove_orphan_files（清理孤立文件）

```sql
-- 清理不被任何快照引用的数据文件
CALL iceberg.system.remove_orphan_files(
    table => 'iceberg.demo.posts',
    older_than => TIMESTAMP '2026-04-08 00:00:00'
);

-- 清理孤立的元数据文件
CALL iceberg.system.remove_orphan_files(
    table => 'iceberg.demo.posts',
    older_than => TIMESTAMP '2026-04-08 00:00:00',
    file_type => 'metadata'   -- 只清理元数据文件
);

-- 清理孤立的数据文件
CALL iceberg.system.remove_orphan_files(
    table => 'iceberg.demo.posts',
    older_than => TIMESTAMP '2026-04-08 00:00:00',
    file_type => 'data'   -- 只清理数据文件
);
```

### 3.4 rewrite_manifests（元数据文件重写）

```sql
-- 合并小 Manifest 文件（减少元数据文件数量）
CALL iceberg.system.rewrite_manifests(
    'iceberg.demo.posts'
);
```

### 3.5 完整维护脚本

```python
"""
Iceberg 表维护脚本
建议每天凌晨执行一次
"""
from pyspark.sql import SparkSession
from datetime import datetime, timedelta

spark = SparkSession.builder \
    .appName("IcebergMaintenance") \
    .config("spark.jars.packages", "org.apache.iceberg:iceberg-spark-runtime-3.5_2.12:1.10.1") \
    .config("spark.sql.extensions", "org.apache.iceberg.spark.extensions.IcebergSparkSessionExtensions") \
    .config("spark.sql.catalog.iceberg", "org.apache.iceberg.spark.SparkCatalog") \
    .config("spark.sql.catalog.iceberg.catalog-impl", "org.apache.iceberg.jdbc.JdbcCatalog") \
    .config("spark.sql.catalog.iceberg.uri", "jdbc:mysql://localhost:3306/iceberg") \
    .config("spark.sql.catalog.iceberg.jdbc.user", "root") \
    .config("spark.sql.catalog.iceberg.jdbc.password", "你的MySQL密码") \
    .config("spark.sql.catalog.iceberg.warehouse", "hdfs://localhost:9000/iceberg-warehouse") \
    .getOrCreate()

spark.sparkContext.setLogLevel("WARN")

# 需要维护的表列表
tables = [
    'iceberg.demo.posts',
    'iceberg.demo.comments',
    'iceberg.demo.page_views',
]

# 保留策略
retain_days = 7
retain_snapshots = 50
cutoff_time = datetime.now() - timedelta(days=retain_days)

for table in tables:
    print(f"\n===== 维护表: {table} =====")

    # 1. 查看当前状态
    files_count = spark.sql(f"SELECT count(*) FROM {table}.files").collect()[0][0]
    snapshots_count = spark.sql(f"SELECT count(*) FROM {table}.snapshots").collect()[0][0]
    print(f"当前文件数: {files_count}, 快照数: {snapshots_count}")

    # 2. 合并小文件
    print("合并小文件...")
    try:
        spark.sql(f"""
            CALL iceberg.system.rewrite_data_files(
                table => '{table}',
                options => map('target-file-size-bytes', '268435456')
            )
        """)
        print("合并完成")
    except Exception as e:
        print(f"合并跳过（可能没有需要合并的文件）: {e}")

    # 3. 过期旧快照
    print(f"过期 {retain_days} 天前的快照...")
    try:
        spark.sql(f"""
            CALL iceberg.system.expire_snapshots(
                '{table}',
                TIMESTAMP '{cutoff_time.strftime("%Y-%m-%d %H:%M:%S")}',
                {retain_snapshots}
            )
        """)
        print("过期完成")
    except Exception as e:
        print(f"过期跳过: {e}")

    # 4. 清理孤立文件
    print("清理孤立文件...")
    try:
        orphan_cutoff = datetime.now() - timedelta(hours=6)
        spark.sql(f"""
            CALL iceberg.system.remove_orphan_files(
                table => '{table}',
                older_than => TIMESTAMP '{orphan_cutoff.strftime("%Y-%m-%d %H:%M:%S")}'
            )
        """)
        print("清理完成")
    except Exception as e:
        print(f"清理跳过: {e}")

    # 5. 重写 Manifest 文件
    print("重写 Manifest...")
    try:
        spark.sql(f"CALL iceberg.system.rewrite_manifests('{table}')")
        print("重写完成")
    except Exception as e:
        print(f"重写跳过: {e}")

    # 6. 查看维护后的状态
    new_files_count = spark.sql(f"SELECT count(*) FROM {table}.files").collect()[0][0]
    new_snapshots_count = spark.sql(f"SELECT count(*) FROM {table}.snapshots").collect()[0][0]
    print(f"维护后 - 文件数: {new_files_count}, 快照数: {new_snapshots_count}")

print("\n===== 所有表维护完成 =====")
spark.stop()
```

## 七、10 道高频面试题

### 面试题 1：Iceberg 的元数据结构是怎样的？

**参考答案：**

Iceberg 的元数据采用**三级树形结构**：

1. **Metadata File（metadata.json）**：表级别的元数据，包含当前 Schema、分区规范、当前快照 ID、快照历史列表。这是 Iceberg 的入口文件。

2. **Manifest List（snap-xxx.avro）**：快照级别的元数据，记录了该快照包含哪些 Manifest File。每个 Manifest List 条目还包含分区范围、文件数量等摘要信息，用于快速跳过不相关的 Manifest。

3. **Manifest File（xxx-m{n}.avro）**：文件级别的元数据，记录了每个数据文件的路径、文件格式、记录数、文件大小、列级别的统计信息（min/max/null_count）等。

查询时，Iceberg 从 metadata.json 开始，通过三级索引逐层定位到需要读取的数据文件。这种设计的优势是：
- 元数据与数据分离，查询不需要列出所有文件
- 列级别的统计信息支持文件级过滤（谓词下推）
- 每个快照的 Manifest 是不可变的，保证了一致性

### 面试题 2：Iceberg 如何实现 ACID 事务？

**参考答案：**

Iceberg 的 ACID 事务基于**乐观并发控制（Optimistic Concurrency Control）**实现：

1. **原子性（Atomicity）**：写操作（INSERT/UPDATE/DELETE/MERGE）的流程是：先写入新的数据文件，然后创建新的 Manifest File 和 Manifest List，最后用新的 Manifest List 路径更新 metadata.json。metadata.json 的更新是原子性的（通过 Catalog 的 CAS 操作实现），要么全部成功，要么全部失败。

2. **一致性（Consistency）**：每个快照都是表的一个一致状态。读操作始终基于一个已提交的快照，不会看到部分写入的数据。

3. **隔离性（Isolation）**：读写操作互不阻塞。写操作在独立的新快照上进行，读操作在已提交的旧快照上进行。多个写操作通过乐观锁保证串行化。

4. **持久性（Durability）**：数据文件一旦写入 HDFS/S3 就是持久化的。metadata.json 通过 Catalog 持久化到 MySQL 等后端存储。

### 面试题 3：Iceberg 的读取性能为什么比 Hive 好？

**参考答案：**

Iceberg 的读取性能优势主要来自以下几个方面：

1. **文件级过滤**：Iceberg 在 Manifest File 中记录了每个数据文件的列级统计信息（min/max/null_count）。查询时，Iceberg 可以根据 WHERE 条件跳过不包含匹配数据的数据文件，而不需要实际读取文件内容。Hive 只能做分区级别的过滤。

2. **隐藏分区**：Iceberg 的分区对用户透明，用户不需要在 WHERE 子句中指定分区条件。Iceberg 会自动将查询条件转换为分区过滤条件。

3. **列式存储 + 列裁剪**：Iceberg 默认使用 Parquet 格式，支持列式存储和列裁剪，只读取查询涉及的列，减少 IO。

4. **元数据高效索引**：三级元数据树（Metadata → Manifest List → Manifest File → Data File）避免了列出所有文件的性能瓶颈（Hive 需要 NameNode 大量 RPC 调用来列出文件）。

5. **快照级文件追踪**：不需要扫描整个目录来找到相关文件，通过快照的 Manifest List 直接定位。

### 面试题 4：什么是 Iceberg 的 Format Version 1 和 Version 2？有什么区别？

**参考答案：**

Iceberg 的表格式有两个主要版本：

**Format Version 1：**
- 支持 Parquet、ORC、Avro 数据格式
- 支持 ACID 事务、时间旅行、Schema 演进
- DELETE 操作通过重写整个数据文件实现（COPY-ON-WRITE），成本高
- 不支持行级删除标记

**Format Version 2（推荐）：**
- 新增了 Equality Delete Files（等值删除文件）
- DELETE 操作不再需要重写整个数据文件，而是生成一个小的删除文件记录被删除的行
- 读取时，Iceberg 会合并数据文件和删除文件，过滤掉被删除的行
- 大幅降低了 UPDATE/DELETE 操作的 IO 开销
- 需要额外的 Compaction 操作来合并删除文件和数据文件

**建议：** 新建表使用 Format Version 2，除非有特殊的兼容性需求。

### 面试题 5：如何处理 Iceberg 的小文件问题？

**参考答案：**

小文件问题在数据湖中很常见，特别是高频写入（如 Flink 实时写入）的场景。Iceberg 提供了几种解决方案：

1. **设置合理的目标文件大小**：通过 `write.target-file-size-bytes` 属性控制写入时的目标文件大小（一般 128MB~512MB）。

2. **rewrite_data_files**：定期执行数据文件重写，将小文件合并为大文件。
   ```sql
   CALL iceberg.system.rewrite_data_files(
       table => 'my_table',
       options => map('target-file-size-bytes', '268435456')
   );
   ```

3. **使用 Binpack 策略**：Iceberg 支持多种重写策略，binpack 策略专门用于合并小文件而不改变排序。

4. **调整写入频率**：如果是实时写入，可以适当增加 Flink 的 checkpoint 间隔，减少写入次数。

5. **使用 Spark 的 coalesce/repartition**：在批量写入前先减少分区数。

### 面试题 6：Iceberg 的 MERGE INTO 和传统的 INSERT OVERWRITE 有什么区别？

**参考答案：**

**MERGE INTO：**
- 基于条件匹配（ON 子句），精确地更新或插入行
- 只修改匹配的行，不影响其他数据
- 生成新文件来替换受影响的文件，未受影响的文件保持不变
- 适合 Upsert 场景（存在则更新，不存在则插入）

**INSERT OVERWRITE：**
- 覆盖整个表或指定分区
- 所有数据都会被重写（即使没有变化）
- 适合全量刷新场景

性能对比：
- MERGE INTO 更精确，IO 开销更小（只修改需要变更的行）
- INSERT OVERWRITE 更简单，适合小表或需要完全替换的场景

实际生产中，推荐使用 MERGE INTO，因为它更精确、更高效。

### 面试题 7：如何监控 Iceberg 表的健康状态？

**参考答案：**

可以从以下几个维度监控 Iceberg 表：

1. **快照数量**：快照过多说明过期操作不及时
   ```sql
   SELECT count(*) FROM table.snapshots;
   ```

2. **小文件数量**：小文件过多说明需要合并
   ```sql
   SELECT count(*) FROM table.files WHERE file_size_in_bytes < 16777216;
   ```

3. **数据文件总大小**：监控存储增长趋势
   ```sql
   SELECT SUM(file_size_in_bytes) FROM table.files;
   ```

4. **孤儿文件检测**：通过 remove_orphan_files 的 dry-run 模式
   ```sql
   -- 设置为 dry-run 只报告不删除
   CALL iceberg.system.remove_orphan_files(
       table => 'my_table',
       older_than => TIMESTAMP '2026-04-08 00:00:00'
   );
   ```

5. **快照历史大小**：监控元数据的增长
   ```bash
   hdfs dfs -du -h /iceberg-warehouse/data/posts/metadata/
   ```

建议将这些监控指标集成到 Prometheus + Grafana 中，设置告警阈值。

### 面试题 8：Iceberg 在生产环境中遇到过什么坑？

**参考答案：**

1. **Hive Metastore 兼容性问题**：Hive 4.x 的 Thrift API 与 Iceberg 不兼容，导致 HiveCatalog 无法使用。解决方案是使用 JdbcCatalog 直连 MySQL（本课程就是这种方案）。

2. **快照过期导致查询失败**：如果有一个长时间运行的查询（如 Spark 作业跑了几个小时），期间快照被过期了，查询会报文件找不到的错误。建议保留足够的快照数量，或在作业完成后再执行过期操作。

3. **remove_orphan_files 误删文件**：如果 `older_than` 参数设置不当（太接近当前时间），可能会误删正在写入的文件。建议设置为比当前时间早至少 2-3 个小时。

4. **小文件爆炸**：高频写入（特别是 Flink 实时写入）会产生大量小文件，严重影响查询性能。需要建立定期合并机制。

5. **元数据文件膨胀**：频繁写入导致 Manifest File 和 Manifest List 数量激增。定期执行 `rewrite_manifests` 合并元数据文件。

6. **JAR 包冲突**：Iceberg Runtime JAR 与 Spark 自带的某些依赖可能冲突。建议使用 `--packages` 方式加载，而不是手动放入 jars 目录。

### 面试题 9：如何设计一个基于 Iceberg 的增量 ETL 管道？

**参考答案：**

基于 Iceberg 的增量 ETL 管道设计：

```
数据源 → Flink(实时捕获) → ODS(Iceberg) → Spark(增量转换) → DWD(Iceberg) → DWS(Iceberg)
```

关键设计点：

1. **ODS 层**：使用 Flink CDC 从 MySQL 实时捕获变更，通过 MERGE INTO 写入 Iceberg ODS 表。每次写入产生新快照。

2. **增量提取**：通过对比两次快照的 Manifest File，获取新增或修改的数据文件。Iceberg 的 `table.changes` 功能可以获取行级别的增量。

3. **DWD 层**：Spark 定期从 ODS 增量读取数据，做清洗和转换后写入 DWD 层。

4. **DWS 层**：基于 DWD 的增量数据做聚合，通过 MERGE INTO 更新 DWS 汇总表。

5. **调度**：使用 Airflow 或 DolphinScheduler 调度 Spark 作业，监控各层数据的 freshness。

```sql
-- 增量 ETL 示例：ODS → DWD
MERGE INTO iceberg.dwd.posts_detail AS target
USING iceberg.ods.posts_cdc AS source
ON target.id = source.id
WHEN MATCHED AND source.op_type = 'UPDATE' THEN
    UPDATE SET *
WHEN MATCHED AND source.op_type = 'DELETE' THEN
    DELETE
WHEN NOT MATCHED THEN
    INSERT *;
```

### 面试题 10：数据湖和 Lakehouse 架构中，Iceberg 的核心价值是什么？

**参考答案：**

Iceberg 在 Lakehouse 架构中的核心价值可以总结为以下三点：

1. **统一的数据管理**：Iceberg 作为开放表格式，为 HDFS/S3 等低成本存储提供了 ACID 事务、Schema 管理、时间旅行等传统数据仓库的能力。它弥合了数据湖和数据仓库之间的功能鸿沟。

2. **引擎无关性**：Iceberg 不绑定任何特定的计算引擎。Spark 做批量处理、Flink 做实时写入、Trino 做交互查询、Hive 做兼容查询，所有引擎读写同一份 Iceberg 数据，避免了数据复制和格式转换。

3. **数据治理能力**：通过快照历史实现数据审计和合规，通过 Schema 演进适应业务变化，通过时间旅行实现数据回滚和灾备。这些能力让数据湖不再是"数据沼泽"，而是一个可管理、可审计、可信赖的数据平台。

从战略角度看，选择 Iceberg 就是选择了开放和标准。它不依赖任何单一厂商（不像 Delta Lake 绑定 Databricks），代表了数据管理技术的未来方向。Apple、Netflix、Amazon 等大厂的选择也验证了这一点。
