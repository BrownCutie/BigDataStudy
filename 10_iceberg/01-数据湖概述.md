# 01 - 数据湖概述：从数据仓库到湖仓一体

## 一、先想一个问题：数据越来越多，存在哪里？

假设你运营着一个个人数据平台，每天都会产生大量的数据：

```
┌──────────────────────────────────────────────────────────────┐
│                    数据平台每天产生的数据                       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   文章数据          评论数据          用户行为日志             │
│   ┌──────────┐     ┌──────────┐     ┌──────────────┐        │
│   │ 新增文章  │     │ 新增评论  │     │ 页面浏览记录  │        │
│   │ 编辑历史  │     │ 点赞/收藏 │     │ 搜索关键词    │        │
│   │ 标签变更  │     │ 回复链路  │     │ 设备/地域信息 │        │
│   └──────────┘     └──────────┘     └──────────────┘        │
│                                                              │
│   数据量：从 MB 级 → GB 级 → TB 级，持续增长                 │
│   数据类型：结构化（文章）+ 半结构化（JSON 日志）+ 非结构化（图片）│
│   使用方式：实时查询 + 历史分析 + 机器学习                     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

传统的 MySQL 数据库能存这些数据吗？能，但随着数据量增长，查询越来越慢，成本越来越高。这时候，你就需要一个**数据湖**了。

## 二、什么是数据湖？

**数据湖**（Data Lake）是一种存储架构，它将**所有类型的数据**以**原始格式**存储在一个集中式的存储系统中，供后续分析和处理使用。

```
┌──────────────────────────────────────────────────────────────────┐
│                        数据湖概念图                               │
│                                                                  │
│   数据源                    数据湖                      数据消费   │
│                                                                  │
│   ┌─────────┐                                                    │
│   │ 数据库   │──┐                                                │
│   └─────────┘  │      ┌──────────────────────┐     ┌──────────┐  │
│   ┌─────────┐  │      │                      │     │ 数据分析  │  │
│   │ 日志文件 │──┼─────►│      数据湖           │────►│ 报表生成  │  │
│   └─────────┘  │      │                      │     │          │  │
│   ┌─────────┐  │      │  ┌────────────────┐  │     │ 机器学习  │  │
│   │ JSON/API│──┤      │  │ 结构化数据      │  │     │ 模型训练  │  │
│   └─────────┘  │      │  ├────────────────┤  │     │          │  │
│   ┌─────────┐  │      │  │ 半结构化数据    │  │     │ 实时查询  │  │
│   │ 图片/视频│──┘      │  ├────────────────┤  │     │ 即席分析  │  │
│   └─────────┘         │  │ 非结构化数据    │  │     └──────────┘  │
│                       │  └────────────────┘  │                   │
│                       │                      │                   │
│                       │  底层：HDFS / S3 / OSS │                   │
│                       └──────────────────────┘                   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

用一句通俗的话说：**数据湖就像一个巨大的水库，各种水源（数据）都可以流入，各种用途（分析、机器学习、报表）都可以从中取水。**

## 三、数据湖 vs 数据仓库

很多同学分不清"数据湖"和"数据仓库"，我们来做一下对比：

```
┌──────────────────────────────────────────────────────────────────────┐
│                     数据湖 vs 数据仓库 对比                           │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  数据仓库（Data Warehouse）            数据湖（Data Lake）              │
│  ─────────────────────────           ─────────────────               │
│                                                                      │
│  ┌──────────────┐                    ┌──────────────────────┐       │
│  │  结构化数据    │                    │  所有类型的数据        │       │
│  │  经过清洗转换  │                    │  原始格式存储          │       │
│  │  Schema on Write│                   │  Schema on Read      │       │
│  │  成本高        │                    │  成本低               │       │
│  │  适合 BI 报表  │                    │  适合数据分析/ML       │       │
│  └──────────────┘                    └──────────────────────┘       │
│                                                                      │
│  代表产品：Snowflake、                   代表技术：HDFS +              │
│           Redshift、BigQuery              Parquet/ORC +               │
│                                          Iceberg/Delta/Hudi           │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

用一张表来总结核心区别：

| 特性 | 数据仓库 | 数据湖 |
|------|---------|--------|
| **数据类型** | 仅结构化数据 | 结构化 + 半结构化 + 非结构化 |
| **处理方式** | ETL（先转换再加载） | ELT（先加载再转换） |
| **Schema** | 写入时定义（Schema on Write） | 读取时定义（Schema on Read） |
| **存储成本** | 高（专用存储） | 低（通用存储：HDFS/S3） |
| **灵活性** | 低（变更 Schema 困难） | 高（随时可以新增数据类型） |
| **用户群体** | 数据分析师、业务人员 | 数据工程师、数据科学家 |
| **典型场景** | BI 报表、经营分析 | 数据挖掘、机器学习、日志分析 |

**关键概念解释：**

- **Schema on Write**：数据写入时就确定好结构，不符合结构的数据会被拒绝或转换。就像入住酒店前必须先登记好所有信息。
- **Schema on Read**：数据先存进来，查询的时候再决定怎么解析。就像先把东西放进仓库，需要的时候再分类。

## 四、数据湖的发展历程

数据湖并不是一开始就这么成熟的，它经历了几代演进：


### 4.1 第一代：HDFS + Hive 表格式（2010-2016）

最原始的数据湖方案。数据以文件形式存储在 HDFS 上，用 Hive Metastore 管理元数据。

**核心问题：**
- Hive 表格式不支持 ACID 事务
- 修改数据需要覆盖整个分区
- 没有 Schema 演进能力
- 并发读写会互相阻塞
- 大量小文件导致性能急剧下降

### 4.2 第二代：开放表格式诞生（2017-2019）

三大开放表格式相继出现，解决了第一代的核心痛点：

```
┌────────────────────────────────────────────────────────────────┐
│                三大开放表格式的诞生                               │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  2017年：Delta Lake（Databricks 开源）                          │
│  └── 基于 Spark，与 Spark 深度绑定                              │
│                                                                │
│  2019年：Apache Iceberg（Netflix 开源 → Apache 顶级项目）        │
│  └── 引擎无关设计，支持多引擎                                   │
│                                                                │
│  2019年：Apache Hudi（Uber 开源 → Apache 顶级项目）             │
│  └── 专注于增量数据处理和近实时写入                              │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 4.3 第三代：Iceberg 成熟期（2020-2022）

Apache Iceberg 逐渐成为事实标准，被各大公司广泛采用：
- Apple、Netflix、Amazon、Adobe、LinkedIn、腾讯、阿里等大规模使用
- 支持 Spark、Flink、Trino、Presto、Hive 等多种计算引擎
- 引入了隐藏分区、Schema 演进等高级特性

### 4.4 第四代：湖仓一体（2023-至今）

数据湖和数据仓库的边界逐渐模糊，出现了**湖仓一体**（Lakehouse）架构。

## 五、Lakehouse 架构：湖仓一体

**湖仓一体**是把数据湖的低成本存储能力和数据仓库的 ACID 事务、Schema 管理能力结合在一起的新架构。

```
│  │  └──────────┘  └──────────┘  └──────────┘  └──────────────┘  │  │
│  └──────────────────────────┬─────────────────────────────────────┘  │
│                             │                                        │
│  ┌──────────────────────────▼─────────────────────────────────────┐  │
│  │                     查询引擎层                                  │  │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────┐  │  │
│  │  │  Spark   │  │  Flink   │  │  Trino   │  │    Hive      │  │  │
│  │  └──────────┘  └──────────┘  └──────────┘  └──────────────┘  │  │
│  └──────────────────────────┬─────────────────────────────────────┘  │
│                             │                                        │
│  ┌──────────────────────────▼─────────────────────────────────────┐  │
│  │                   开放表格式层（核心）                           │  │
│  │                                                                │  │

│  │     │           Apache Iceberg                 │              │  │

│  │     │  ✓ ACID 事务      ✓ 时间旅行             │              │  │
```

| 支持数据类型 | 结构化 | 全类型 | 全类型 |

| 实时分析 | 部分 | 弱 | 强 |
| 多引擎支持 | 弱 | 强 | 强 |


**本课程选择 Apache Iceberg**，因为它是目前最开放、最灵活的表格式，被越来越多的企业选为数据湖的统一标准。

## 七、Apache Iceberg 的核心优势

### 7.1 ACID 事务

Iceberg 在表级别提供完整的 ACID（原子性、一致性、隔离性、持久性）事务保证。

```
┌────────────────────────────────────────────────────────────────┐
│                    ACID 事务保证                                │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  传统 Hive 表：                                                │
│  ┌──────────────────────────────────────────┐                 │
│  │  Writer: 正在写入 partition=2026-04-15   │                 │
│  │  Reader: 正在读取 partition=2026-04-15   │                 │
│  │  结果: 读取可能不一致或直接报错！         │                 │
│  └──────────────────────────────────────────┘                 │
│                                                                │
│  Iceberg 表：                                                  │
│  ┌──────────────────────────────────────────┐                 │
│  │  Writer: 正在写入新快照                  │                 │
│  │  Reader: 读取的是上一个已提交的快照      │                 │
│  │  结果: 读写互不干扰，数据始终一致！      │                 │
│  └──────────────────────────────────────────┘                 │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 7.2 时间旅行（Time Travel）

每次对 Iceberg 表的修改都会生成一个新的**快照（Snapshot）**，你可以查询表在任意历史时间点的数据。

```
┌────────────────────────────────────────────────────────────────┐
│                     时间旅行原理                                │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  时间线:                                                       │
│  ──────────────────────────────────────────────────────────►   │
│  T1              T2              T3              T4           │
│  │               │               │               │            │
│  ▼               ▼               ▼               ▼            │
│  Snapshot 1      Snapshot 2      Snapshot 3      Snapshot 4   │
│  (3篇文章)       (新增1篇)       (修改标题)      (删除1篇)     │
│  [id:1,2,3]     [id:1,2,3,4]   [id:1,2,3,4]   [id:1,2,4]   │
│                                                                │
│  查询 T2 时刻的数据：SELECT * FROM t VERSION AS OF T2           │
│  结果：id 1, 2, 3, 4（4篇文章）                                │
│                                                                │
│  用途：                                                        │
│  ├── 数据审计：谁在什么时候修改了什么数据？                     │
│  ├── 数据回滚：发现错误修改，一键恢复                           │
│  ├── 增量处理：只处理两次快照之间的变化                         │
│  └── A/B 测试：对比不同版本的数据效果                           │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 7.3 Schema 演进

修改表结构不需要重写数据，这是 Iceberg 的一大亮点。

```
┌────────────────────────────────────────────────────────────────┐
│                   Schema 演进示例                               │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  Hive 表：改 Schema 需要重建表                                  │
│  ┌──────────────────────────────────────────────┐             │
│  │ 1. CREATE TABLE new_table AS SELECT ...       │             │
│  │ 2. DROP TABLE old_table                       │             │
│  │ 3. ALTER TABLE new_table RENAME TO old_table  │             │
│  │ → 几十TB的数据，这个过程要跑几个小时！          │             │
│  └──────────────────────────────────────────────┘             │
│                                                                │
│  Iceberg 表：改 Schema 只需要一条 SQL                           │
│  ┌──────────────────────────────────────────────┐             │
│  │ ALTER TABLE posts ADD COLUMN tags ARRAY<STRING>;             │
│  │ → 瞬间完成！旧数据中 tags 列为 null           │             │
│  └──────────────────────────────────────────────┘             │
│                                                                │
│  支持的操作：                                                  │
│  ├── 添加列（ADD COLUMN）                                      │
│  ├── 删除列（DROP COLUMN）                                     │
│  ├── 重命名列（RENAME COLUMN）                                  │
│  ├── 修改列类型（ALTER COLUMN TYPE）                            │
│  └── 调整列顺序（ALTER COLUMN FIRST/AFTER）                     │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 7.4 隐藏分区（Hidden Partitioning）

这是 Iceberg 独有的杀手级特性。

```
┌────────────────────────────────────────────────────────────────┐
│                   隐藏分区 vs 传统分区                           │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  传统 Hive 分区：                                              │
│  ┌──────────────────────────────────────────────┐             │
│  │ INSERT INTO posts PARTITION(dt='2026-04-15') │             │
│  │ VALUES (1, 'hello');                          │             │
│  │                                              │             │
│  │ 问题：                                       │             │
│  │ 1. 写入时必须手动指定分区值                    │             │
```

## 九、面试题

### 面试题 1：数据湖、数据仓库、湖仓一体有什么区别？

**参考答案：**

数据仓库是一种面向结构化数据的分析系统，数据在写入时经过 ETL 清洗和转换，具有强 Schema 约束。代表产品有 Snowflake、Redshift 等。它的优势是数据质量高、查询性能好，但存储成本高、灵活性差。

数据湖是一种存储架构，将所有类型的数据以原始格式存储在低成本的分布式存储（HDFS/S3）上。数据在读取时才进行解析（Schema on Read），支持结构化、半结构化和非结构化数据。它的优势是存储成本低、灵活性高，但缺乏事务保证和良好的数据管理能力。

湖仓一体（Lakehouse）是两者的融合。它在数据湖的低成本存储之上，通过开放表格式（如 Apache Iceberg）引入了 ACID 事务、Schema 管理、时间旅行等数据仓库的能力。同时保留了数据湖支持全类型数据、低成本存储的优势。核心代表是 Databricks 提出的 Lakehouse 架构，底层通常使用 Delta Lake 或 Iceberg 作为表格式。

简单来说：
- 数据仓库 = 高质量 + 高成本 + 低灵活
- 数据湖 = 低成本 + 高灵活 + 低管理
- 湖仓一体 = 低成本 + 高灵活 + 高管理（两全其美）

### 面试题 2：为什么选择 Apache Iceberg 而不是 Delta Lake 或 Hudi？

**参考答案：**

选择 Iceberg 主要基于以下考虑：

**引擎无关性**：Iceberg 的设计从一开始就是引擎无关的。它不依赖任何特定的计算引擎，可以被 Spark、Flink、Trino、Presto、Hive 等多种引擎同时读写。Delta Lake 深度绑定 Spark 生态，虽然现在也支持了其他引擎，但成熟度和性能不如 Iceberg。

**隐藏分区**：Iceberg 独有的隐藏分区特性让用户在写入和查询时都不需要关心分区字段，分区演进也不需要重写数据。Delta Lake 和 Hudi 仍然使用传统的显式分区方式。

**Schema 演进**：Iceberg 的 Schema 演进最完善，支持添加/删除/重命名列、修改列类型、调整列顺序等操作，而且不需要重写底层的数据文件。

**元数据管理**：Iceberg 的元数据设计非常优雅，通过多级元数据树（Metadata File → Manifest List → Manifest File → Data File）实现了高效的查询规划和文件级别的过滤。

**社区生态**：Iceberg 的社区非常活跃，Netflix、Apple、Amazon、腾讯、阿里等大厂都在大规模使用。它已经成为了事实上的数据湖表格式标准。

当然，如果你的场景主要是近实时增量处理和 CDC，Apache Hudi 可能更合适。如果你已经深度使用 Databricks 平台，Delta Lake 的集成度更高。

### 面试题 3：Iceberg 的快照（Snapshot）机制是怎么工作的？

**参考答案：**

Iceberg 采用**MVCC（多版本并发控制）**机制。每次对表的写操作（INSERT、UPDATE、DELETE 等）都会创建一个新的快照，而不会修改或删除之前的数据文件。

快照是表在某个时间点的完整状态的"指针"，它通过 Manifest List（清单列表）关联到一组 Manifest File（清单文件），每个 Manifest File 又记录了该快照包含的所有数据文件。

```
Snapshot → Manifest List → Manifest File(s) → Data File(s)
```

读操作总是基于一个已提交的快照进行，所以读写之间不会互相阻塞。每个快照都有唯一的 snapshot_id 和 committed_at（提交时间），用户可以通过 `VERSION AS OF` 语法查询历史快照的数据。

由于旧快照引用的数据文件不会被自动删除，所以 Iceberg 支持时间旅行。但这也意味着需要定期执行快照过期（expire_snapshots）和孤儿文件清理（remove_orphan_files）来释放存储空间。
