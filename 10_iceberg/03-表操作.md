# 03 - 表操作：Iceberg 的增删改查

## 一、回顾：Iceberg 表的文件结构

在开始操作之前，让我们再回顾一下 Iceberg 表在 HDFS 上的文件组织：

```
┌────────────────────────────────────────────────────────────────┐
│                  Iceberg 表的内部结构                            │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  iceberg.demo.posts                                            │
│  ┌────────────────────────────────────────────────────────┐   │
│  │                                                        │   │
│  │  metadata/v1.metadata.json                             │   │
│  │  ├── schema: 表的字段定义                              │   │
│  │  ├── partition-spec: 分区规则                          │   │
│  │  └── current-snapshot-id: 指向最新快照                 │   │
│  │                                                        │   │
│  │  metadata/snap-1234567890.avro  (Manifest List)        │   │
│  │  ├── manifest-1.avro  → 包含文件 A、B                  │   │
│  │  ├── manifest-2.avro  → 包含文件 C、D                  │   │
│  │  └── manifest-3.avro  → 包含删除的文件标记             │   │
│  │                                                        │   │
│  │  data/                                                  │   │
│  │  ├── created_at_day=2026-04-15/00000-xxx.parquet       │   │
│  │  └── created_at_day=2026-04-16/00001-xxx.parquet       │   │
│  │                                                        │   │
│  └────────────────────────────────────────────────────────┘   │
│                                                                │
│  多级索引：metadata.json → Manifest List → Manifest File → Data│
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

理解这个结构很重要，因为 Iceberg 的所有操作都是围绕这个多级元数据结构展开的。

## 二、创建与管理表

### 2.1 创建数据库（Namespace）

```sql
-- 创建数据库
CREATE NAMESPACE IF NOT EXISTS iceberg.demo;

-- 查看所有数据库
SHOW NAMESPACES IN iceberg;

-- 查看数据库属性
DESCRIBE NAMESPACE iceberg.demo;

-- 设置数据库属性（例如设置默认的表格式版本）
ALTER NAMESPACE iceberg.demo SET PROPERTIES (
    'owner' = 'browncutie'
);
```

### 2.2 创建表的三种方式

**方式一：SQL CREATE TABLE**

```sql
-- 创建博客文章表
CREATE TABLE iceberg.demo.posts (
    id          INT,
    title       STRING,
    content     STRING,
    author      STRING,
    tags        ARRAY<STRING>,
    status      STRING,
    created_at  TIMESTAMP,
    updated_at  TIMESTAMP,
    views       INT,
    word_count  INT
) USING iceberg
PARTITIONED BY (days(created_at))
TBLPROPERTIES (
    'format-version' = '2',
    'write.format.default' = 'parquet',
    'write.target-file-size-bytes' = '134217728'   -- 128MB 目标文件大小
);
```

**方式二：CREATE TABLE AS SELECT（CTAS）**

```sql
-- 从已有表创建新表
CREATE TABLE iceberg.demo.popular_posts
USING iceberg
PARTITIONED BY (bucket(8, author))
AS SELECT * FROM iceberg.demo.posts WHERE views > 100;
```

**方式三：REPLACE TABLE（替换表）**

```sql
-- 替换已有表（原子操作）
CREATE OR REPLACE TABLE iceberg.demo.posts_v2
USING iceberg
PARTITIONED BY (months(created_at))
AS SELECT * FROM iceberg.demo.posts;
```

### 2.3 表属性（Table Properties）

Iceberg 表支持丰富的属性配置：

```sql
-- 创建时设置属性
CREATE TABLE iceberg.demo.example (
    id INT, name STRING
) USING iceberg
TBLPROPERTIES (
    'format-version' = '2',                         -- 表格式版本（推荐 2）
    'write.format.default' = 'parquet',              -- 数据文件格式：parquet/orc/avro
    'write.target-file-size-bytes' = '134217728',   -- 目标文件大小（128MB）
    'write.metadata.delete-after-commit.enabled' = 'true',  -- 写入后自动清理旧元数据
    'read.split.target-size' = '134217728'          -- 读取时每个 task 的目标大小
);

-- 修改表属性
ALTER TABLE iceberg.demo.posts SET TBLPROPERTIES (
    'write.target-file-size-bytes' = '268435456'    -- 修改为 256MB
);

-- 查看表属性
SHOW TBLPROPERTIES iceberg.demo.posts;
```

### 2.4 常用的表格式属性

| 属性 | 默认值 | 说明 |
|------|--------|------|
| `format-version` | 2 | 表格式版本，推荐使用 2 |
| `write.format.default` | parquet | 数据文件格式 |
| `write.target-file-size-bytes` | 128MB | 目标文件大小 |
| `write.metadata.delete-after-commit.enabled` | false | 写入后自动删除旧元数据 |
| `write.distribution-mode` | none | 写入分布模式：none/hash/range |
| `history.expire.max-snapshots` | - | 保留的最大快照数 |
| `write.object-storage.enabled` | false | 是否针对对象存储优化 |

## 五、更新与删除

### 5.1 UPDATE（更新数据）

```sql
-- 更新单条记录
UPDATE iceberg.demo.posts
SET views = views + 1
WHERE id = 1;

-- 批量更新
UPDATE iceberg.demo.posts
SET status = 'archived', updated_at = CURRENT_TIMESTAMP()
WHERE created_at < TIMESTAMP '2026-01-01';

-- 条件更新
UPDATE iceberg.demo.posts
SET views = views + 10
WHERE array_contains(tags, '大数据');
```

**注意：** Iceberg 的 UPDATE 操作不会直接修改已有的数据文件，而是生成新的数据文件来替换。这就是 Iceberg 实现 MVCC 的方式。

```
┌────────────────────────────────────────────────────────────────┐
│                    UPDATE 操作原理                               │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  执行前：                                                      │
│  ┌──────────────────────────────┐                             │
│  │ data/00000.parquet           │                             │
│  │ id=1, title="Kafka...", views=150 │                       │
│  │ id=2, title="Spark...", views=230 │                       │
│  │ id=3, title="HDFS...",  views=89  │                       │
│  └──────────────────────────────┘                             │
│                                                                │
│  执行 UPDATE posts SET views = views + 1 WHERE id = 1:        │
│                                                                │
│  执行后：                                                      │
│  ┌──────────────────────────────┐                             │
│  │ data/00000.parquet           │ ← 旧文件（标记为删除）       │
│  │ (仍然存在，被旧快照引用)      │                             │
│  └──────────────────────────────┘                             │
│  ┌──────────────────────────────┐                             │
│  │ data/00002.parquet           │ ← 新文件                     │
│  │ id=1, title="Kafka...", views=151 │ ← 已更新              │
│  │ id=2, title="Spark...", views=230 │ ← 原样复制             │
│  │ id=3, title="HDFS...",  views=89  │ ← 原样复制             │
│  └──────────────────────────────┘                             │
│                                                                │
│  新快照引用 00002，旧快照引用 00000                             │
│  旧文件在快照过期前不会被物理删除                                │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 5.3 MERGE INTO（Upsert 操作）

MERGE INTO 是 Iceberg 最强大的操作之一，可以同时处理"存在则更新，不存在则插入"的逻辑。

```
┌────────────────────────────────────────────────────────────────┐
│                   MERGE INTO 工作流程                            │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  源数据（新数据）          目标表（Iceberg 表）                  │
│  ┌──────────────┐          ┌──────────────┐                   │
│  │ id=1, views=200│         │ id=1, views=150│  ← MATCHED    │
│  │ id=2, views=300│         │ id=2, views=230│  ← MATCHED    │
│  │ id=9, views=0  │         │ id=3, views=89 │  ← NOT MATCHED│
│  └──────────────┘          └──────────────┘                   │
│       │                          │                             │
│       └──────────┬───────────────┘                             │
│                  │                                             │
│                  ▼                                             │
│  ┌──────────────────────────────────────┐                     │
│  │           MERGE INTO 结果             │                     │
│  │                                      │                     │
│  │  MATCHED (id=1): UPDATE views=200    │                     │
│  │  MATCHED (id=2): UPDATE views=300    │                     │
│  │  NOT MATCHED (id=9): INSERT 新记录   │                     │
│  │  NOT MATCHED (id=3): 不做处理        │                     │
│  │                                      │                     │
│  └──────────────────────────────────────┘                     │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

```sql
-- 基础 MERGE INTO 示例
MERGE INTO iceberg.demo.posts AS target
USING (
    SELECT 1 AS id, 'Kafka 入门指南 V2' AS title, 200 AS views
    UNION ALL
    SELECT 9 AS id, 'Go 语言入门' AS title, 0 AS views
) AS source
ON target.id = source.id
WHEN MATCHED THEN
    UPDATE SET
        target.title = source.title,
        target.views = source.views,
        target.updated_at = CURRENT_TIMESTAMP()
WHEN NOT MATCHED THEN
    INSERT (id, title, author, status, created_at, updated_at, views, word_count)
    VALUES (source.id, source.title, 'new_author', 'published',
            CURRENT_TIMESTAMP(), CURRENT_TIMESTAMP(), source.views, 0);

-- 带 DELETE 的 MERGE
MERGE INTO iceberg.demo.posts AS target
USING (SELECT 3 AS id) AS source
ON target.id = source.id
WHEN MATCHED AND target.views < 100 THEN
    DELETE
WHEN MATCHED THEN
    UPDATE SET target.views = target.views + 1;
```

**PySpark 中使用 MERGE INTO：**

```python
# 通过 Spark SQL 执行 MERGE
spark.sql("""
    MERGE INTO iceberg.demo.posts AS target
    USING source_df AS source
    ON target.id = source.id
    WHEN MATCHED THEN UPDATE SET *
    WHEN NOT MATCHED THEN INSERT *
""")
```

## 六、数据文件组织

### 6.1 支持的数据文件格式

Iceberg 支持三种数据文件格式：

| 格式 | 压缩率 | 读取速度 | 写入速度 | 推荐场景 |
|------|--------|---------|---------|---------|
| **Parquet** | 高 | 快 | 中 | 通用场景（推荐） |
| **ORC** | 高 | 快 | 中 | Hive 生态兼容 |
| **Avro** | 低 | 慢 | 快 | Schema 频繁变更 |

```sql
-- 指定写入格式
ALTER TABLE iceberg.demo.posts SET TBLPROPERTIES (
    'write.format.default' = 'parquet'
);

-- 也可以在写入时指定
INSERT INTO iceberg.demo.posts
/*+ FORMAT('parquet') */
VALUES (10, '格式测试', '...', 'browncutie', ARRAY('test'), 'published',
    CURRENT_TIMESTAMP(), CURRENT_TIMESTAMP(), 0, 100);
```

### 6.2 文件大小优化

```sql
-- 查看当前文件大小
SELECT
    file_path,
    file_format,
    record_count,
    file_size_in_bytes,
    CAST(file_size_in_bytes / 1024 / 1024 AS DECIMAL(10,2)) AS size_mb

);
```

## 七、快照（Snapshot）机制

### 7.1 快照是什么？

快照是 Iceberg 表在某个时间点的完整状态的不可变引用。每次写操作都会生成一个新的快照。

```
┌────────────────────────────────────────────────────────────────┐
│                     快照链示意                                   │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  metadata.json（当前版本）                                      │
│  └── current-snapshot-id: 789                                  │
│                                                                │
│  Snapshot 链：                                                  │
│                                                                │
│  123 (INSERT 3篇) ──► 456 (INSERT 2篇) ──► 789 (UPDATE 1篇)   │
│  │                   │                    │                    │
│  ▼                   ▼                    ▼                    │
│  文件: A,B           文件: A,B,C,D         文件: C,D,E          │
│  (A,B为3篇文章)     (C,D为新增的2篇)      (E为更新的1篇,       │
│                                           A,B被标记删除)       │
│                                                                │
│  每个快照都是不可变的，旧快照的数据文件不会被立即删除              │
│  这就是时间旅行的基础                                           │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 7.2 查看快照信息

```sql

    snapshot_id,
    parent_id,
    committed_at,

```

### 7.3 快照的生命周期

```
┌────────────────────────────────────────────────────────────────┐
│                    快照生命周期                                  │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  写操作 → 生成新快照 → 旧快照保留（供时间旅行使用）             │
│                                                                │
│  ┌──────────────────────────────────────────────────────┐     │
│  │                                                       │     │
│  │  时间 ──────────────────────────────────────►         │     │
│  │                                                       │     │
│  │  当前快照     保留快照        已过期快照               │     │
│  │  ┌───┐     ┌───┐ ┌───┐     ┌───┐ ┌───┐ ┌───┐      │     │
│  │  │789│ ◄── │456│ │123│ ◄── │xxx│ │xxx│ │xxx│      │     │
│  │  └───┘     └───┘ └───┘     └───┘ └───┘ └───┘      │     │
│  │    ▲                         ▲                       │     │
│  │    │                         │                       │     │
│  │  可读写                    已执行 expire_snapshots   │     │
│  │                            旧快照的数据文件被删除     │     │
│  │                                                       │     │
│  └──────────────────────────────────────────────────────┘     │
│                                                                │
│  关键操作：                                                    │
│  - expire_snapshots: 过期旧快照，释放数据文件                   │
│  - remove_orphan_files: 清理不被任何快照引用的数据文件         │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 面试题：Iceberg 的 UPDATE 操作是原地更新还是生成新文件？

**参考答案：**

Iceberg 的 UPDATE 操作是**生成新文件**，而不是原地更新。这是由 Iceberg 的 MVCC（多版本并发控制）机制决定的。

具体流程：
1. 根据过滤条件（WHERE 子句）找到需要更新的数据文件
2. 读取这些文件中匹配的行
3. 对匹配的行进行更新
4. 将更新后的行和未匹配的行一起写入新的数据文件
5. 创建新的快照，引用新文件，标记旧文件为"删除"

旧文件不会立即被物理删除，而是保留在存储中，直到快照过期操作（expire_snapshots）被触发。这确保了时间旅行的可用性。

这种设计的优势：
- 读写隔离：读操作始终基于已提交的快照，不受写操作影响
- 数据安全：任何修改都可以通过时间旅行回滚
- 并发安全：多个写操作可以同时进行，通过乐观锁保证一致性
