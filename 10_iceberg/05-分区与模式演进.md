# 05 - 分区演进与模式演进：让数据表随业务生长

## 一、为什么需要分区？

假设你的数据平台有 100 万篇文章，每天有几十万次查询。如果不做分区，每次查询都要扫描所有数据文件，性能会越来越差。

分区（Partitioning）的核心思想是：**把数据按照某个规则分成多个子集，查询时只扫描相关的子集。**

```
┌────────────────────────────────────────────────────────────────┐
│                    分区的价值                                    │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  没有分区：查询 2026-04-15 的文章                               │
│  ┌──────────────────────────────────────────────┐             │
│  │ 扫描全部文件: 文件1 + 文件2 + ... + 文件100  │             │
│  │ 读取 100 万行 → 过滤出 200 行               │             │
│  │ 浪费 99.98% 的 IO                            │             │
│  └──────────────────────────────────────────────┘             │
│                                                                │
│  有分区：查询 2026-04-15 的文章                                 │
│  ┌──────────────────────────────────────────────┐             │
│  │ 只扫描: created_at_day=2026-04-15 目录下的文件│             │
│  │ 读取 200 行 → 全部命中                       │             │
│  │ IO 利用率 100%                               │             │
│  └──────────────────────────────────────────────┘             │
│                                                                │
│  分区裁剪（Partition Pruning）可以跳过 99% 的无关数据           │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 二、隐藏分区 vs 显式分区

这是 Iceberg 最独特的特性之一，我们先通过对比来理解。

### 2.1 传统 Hive 分区（显式分区）

```sql
-- Hive 表：创建分区表
CREATE TABLE hive.demo.posts (
    id INT,
    title STRING,
    author STRING,
    views INT
) PARTITIONED BY (dt STRING, hour STRING);

-- 写入时必须手动指定分区值
INSERT INTO hive.demo.posts PARTITION (dt='2026-04-15', hour='10')
VALUES (1, 'Kafka 入门', 'browncutie', 100);

-- 查询时需要知道分区字段
SELECT * FROM hive.demo.posts WHERE dt = '2026-04-15' AND hour = '10';

-- 改分区策略？需要重写所有数据！
-- 比如从按天分区改为按月分区，需要重新 INSERT ... SELECT
```

**问题：**
- 写入时必须手动指定分区值，容易出错
- 分区字段是物理存在的，会混在数据列中
- 修改分区策略需要重写所有数据

### 2.2 Iceberg 隐藏分区

```sql
-- Iceberg 表：使用隐藏分区
CREATE TABLE iceberg.demo.posts (
    id INT,
    title STRING,
    author STRING,
    created_at TIMESTAMP,
    views INT
) USING iceberg
PARTITIONED BY (days(created_at));  -- 按天分区，created_at 本身就是数据列

-- 写入时不需要指定分区，Iceberg 自动处理
INSERT INTO iceberg.demo.posts VALUES
    (1, 'Kafka 入门', 'browncutie', TIMESTAMP '2026-04-15 10:00:00', 100);

-- 查询时也不需要关心分区
SELECT * FROM iceberg.demo.posts WHERE created_at >= TIMESTAMP '2026-04-15 00:00:00';
-- Iceberg 自动做分区裁剪！

-- 改分区策略？一条 ALTER 语句搞定，无需重写数据！
ALTER TABLE iceberg.demo.posts REPLACE PARTITION FIELD days(created_at) WITH months(created_at);
```

```
┌────────────────────────────────────────────────────────────────┐
│                 隐藏分区 vs 显式分区对比                         │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  Hive 显式分区：                                               │
│  ┌──────────────────────────────────────────────┐             │
│  │  用户数据：id, title, author, views          │             │
│  │  分区字段：dt, hour（物理目录）               │             │
│  │                                              │             │
│  │  写入：INSERT ... PARTITION(dt='2026-04-15') │             │
│  │  查询：WHERE dt = '2026-04-15'                │             │
│  │  改分区：重写全部数据                          │             │
│  └──────────────────────────────────────────────┘             │
│                                                                │
│  Iceberg 隐藏分区：                                            │
│  ┌──────────────────────────────────────────────┐             │
│  │  用户数据：id, title, author, created_at,    │             │
│  │           views（created_at 既是数据列又是   │             │
│  │           分区依据）                          │             │
│  │                                              │             │
│  │  写入：INSERT ... VALUES (1, '...', ts)      │             │
│  │  查询：WHERE created_at >= '2026-04-15'      │             │

│  └──────────────────────────────────────────────┘             │
│                                                                │
│  隐藏分区的核心优势：                                           │
│  1. 写入时不需要指定分区值（自动提取）                         │
│  2. 分区字段仍然是普通的数据列                                  │
│  3. 分区策略可以随时变更（分区演进）                            │
│  4. 查询时自动做分区裁剪                                       │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 三、分区变换函数

Iceberg 提供了丰富的分区变换函数，可以从数据列中自动提取分区值。

### 3.1 时间类变换

```sql
-- year(): 按年分区
PARTITIONED BY (year(created_at))
-- 2026 → created_at_year=2026

-- month(): 按月分区
PARTITIONED BY (month(created_at))
-- 2026-04 → created_at_month=2026-04

-- day()/days(): 按天分区
PARTITIONED BY (days(created_at))
-- 2026-04-15 → created_at_day=2026-04-15

-- hour()/hours(): 按小时分区
PARTITIONED BY (hours(created_at))
-- 2026-04-15-10 → created_at_hour=2026-04-15-10
```

```
┌────────────────────────────────────────────────────────────────┐
│                   时间分区变换示意                               │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  原始时间: 2026-04-15 14:30:45                                 │
│                                                                │
│  year(created_at)     →  2026                                 │
│  month(created_at)    →  2026-04                              │
│  days(created_at)     →  2026-04-15                           │
│  hours(created_at)    →  2026-04-15-14                        │
│                                                                │
│  在 HDFS 上的目录结构：                                        │
│                                                                │
│  year:   created_at_year=2026/                                │
│  month:  created_at_month=2026-04/                            │
│  day:    created_at_day=2026-04-15/                            │
│  hour:   created_at_hour=2026-04-15-14/                        │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 3.2 桶分区（Bucket）

```sql
-- bucket(N, column): 按 N 个桶哈希分区
PARTITIONED BY (bucket(8, author))

-- 8 个桶，author 的值通过 hash % 8 分配到不同桶
-- 'browncutie' → hash('browncutie') % 8 → bucket_3
-- 'guest'     → hash('guest') % 8     → bucket_5
```

```
┌────────────────────────────────────────────────────────────────┐
│                    桶分区原理                                    │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  PARTITIONED BY (bucket(4, author))                            │
│                                                                │
│  author 值:         hash 值        桶编号                      │
│  ────────────────────────────────────────────                  │
│  'browncutie'  →  hash(...) % 4  →  bucket_0                  │
│  'alice'       →  hash(...) % 4  →  bucket_2                  │
│  'bob'         →  hash(...) % 4  →  bucket_1                  │
│  'charlie'     →  hash(...) % 4  →  bucket_3                  │
│  'diana'       →  hash(...) % 4  →  bucket_0                  │
│                                                                │
│  桶分区适合：                                                  │
│  - 高基数列（如用户 ID、文章 ID）                              │
│  - 数据均匀分布（避免数据倾斜）                                │
│  - JOIN 优化（关联表的桶策略一致时）                            │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 3.3 截断分区（Truncate）

```sql
-- truncate(N, column): 取列值的前 N 个字符作为分区值
PARTITIONED BY (truncate(10, title))

-- 'Kafka 入门指南'  → 'Kafka 入门指'  (前10个字符)
-- 'Kafka 高级特性'  → 'Kafka 高级特'  (前10个字符)
-- 'Spark 实战教程'  → 'Spark 实战教'  (前10个字符)
```

### 3.4 多级分区

```sql
-- 多级分区：先按月，再按作者桶
PARTITIONED BY (months(created_at), bucket(8, author))

-- HDFS 目录结构：
-- created_at_month=2026-04/author_bucket=0/
-- created_at_month=2026-04/author_bucket=1/
-- created_at_month=2026-05/author_bucket=0/
```

### 3.5 分区选择指南

| 数据特征 | 推荐分区策略 | 示例 |
|---------|------------|------|
| 时间序列数据 | `days()` / `hours()` | 日志、浏览记录 |
| 时间维度查询 | `months()` / `year()` | 月度报表、年度汇总 |
| 高基数列 | `bucket(N, column)` | 用户 ID、文章 ID |
| 低基数分类 | `truncate(N, column)` | 地区前缀、标题前缀 |
| 多维度查询 | 多级分区 | 时间 + 分类 |

## 四、分区演进（Partition Evolution）

**分区演进**是 Iceberg 的杀手级特性：**修改分区策略不需要重写数据**。

### 4.1 添加分区字段

```sql
-- 初始表：不分区
CREATE TABLE iceberg.demo.posts (
    id INT, title STRING, author STRING,
    created_at TIMESTAMP, views INT
) USING iceberg;

-- 添加分区：按天分区
ALTER TABLE iceberg.demo.posts ADD PARTITION FIELD days(created_at);

-- 后续写入的数据会自动按天分区
-- 已有的数据仍然正常可读，Iceberg 会自动处理新旧数据的分区差异
```

### 4.2 替换分区字段

```sql
-- 从按天分区改为按月分区
ALTER TABLE iceberg.demo.posts
REPLACE PARTITION FIELD days(created_at) WITH months(created_at);

-- 从按月分区改为按天+桶分区
ALTER TABLE iceberg.demo.posts
REPLACE PARTITION FIELD months(created_at) WITH days(created_at);

ALTER TABLE iceberg.demo.posts ADD PARTITION FIELD bucket(8, author);
```

### 4.3 删除分区字段

```sql
-- 删除桶分区（保留时间分区）
ALTER TABLE iceberg.demo.posts DROP PARTITION FIELD bucket(8, author);
```

### 4.4 分区演进原理

```
┌────────────────────────────────────────────────────────────────┐
│                   分区演进原理                                   │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  假设表有以下数据文件：                                         │
│                                                                │
│  初始状态（无分区）：                                           │
│  ┌──────────────────────────────────────────┐                 │
│  │ data/00000.parquet (100行, 各种日期)      │                 │
│  │ data/00001.parquet (100行, 各种日期)      │                 │
│  └──────────────────────────────────────────┘                 │
│                                                                │
│  添加分区 days(created_at) 后：                                 │
│  ┌──────────────────────────────────────────┐                 │
│  │ 旧文件（不受影响）：                      │                 │
│  │ data/00000.parquet (100行)               │                 │
│  │ data/00001.parquet (100行)               │                 │
│  │                                          │                 │
│  │ 新文件（按天分区）：                      │                 │
│  │ data/created_at_day=2026-04-15/00002     │                 │
│  │ data/created_at_day=2026-04-16/00003     │                 │
│  └──────────────────────────────────────────┘                 │
│                                                                │
│  Iceberg 在 Manifest File 中记录了每个文件的分区信息：          │
│  - 旧文件：partition=null（无分区）                            │
│  - 新文件：partition=2026-04-15（有分区）                      │
│                                                                │
│  查询时：                                                      │
│  WHERE created_at >= '2026-04-15'                             │
│  → 跳过 created_at_day=2026-04-14（如果有的话）               │
│  → 仍然需要扫描旧的无分区文件（因为无法确定其内容）             │
│  → 这是分区演进的代价：旧文件无法享受分区裁剪                  │
│                                                                │
│  建议：修改分区策略后，用 rewrite_data_files 重写旧数据        │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

```sql
-- 分区演进后重写旧数据，使其享受新分区策略
CALL iceberg.system.rewrite_data_files(
    table => 'iceberg.demo.posts',
    strategy => 'sort',
    sort_order => 'created_at ASC'
);
```

## 五、Schema 演进（Schema Evolution）

Schema 演进是 Iceberg 的另一个核心能力：**修改表结构不需要重写数据**。

### 5.1 添加列

```sql
-- 添加新列
ALTER TABLE iceberg.demo.posts ADD COLUMN tags ARRAY<STRING>;
ALTER TABLE iceberg.demo.posts ADD COLUMN status STRING DEFAULT 'draft';
ALTER TABLE iceberg.demo.posts ADD COLUMN word_count INT AFTER content;

-- 在指定位置添加列（Iceberg 支持列顺序）
ALTER TABLE iceberg.demo.posts ADD COLUMN (
    category STRING AFTER tags,
    priority INT AFTER category
);
```

### 5.2 删除列

```sql
-- 删除列
ALTER TABLE iceberg.demo.posts DROP COLUMN content;
ALTER TABLE iceberg.demo.posts DROP COLUMN priority;
```

### 5.3 重命名列

```sql
-- 重命名列
ALTER TABLE iceberg.demo.posts RENAME COLUMN views TO read_count;
ALTER TABLE iceberg.demo.posts RENAME COLUMN status TO publish_status;
```

### 5.4 修改列类型

```sql
-- 修改列类型（需要兼容）
-- INT → LONG
ALTER TABLE iceberg.demo.posts ALTER COLUMN read_count TYPE BIGINT;

-- STRING → STRING（长度限制）
ALTER TABLE iceberg.demo.posts ALTER COLUMN title TYPE STRING COMMENT '文章标题';

-- 注意：类型变更是有限制的
-- 安全的类型变更：INT→LONG, FLOAT→DOUBLE
-- 不安全的类型变更：LONG→INT（可能丢失精度）
```

### 5.5 列顺序演进

```sql
-- 将列移到第一列
ALTER TABLE iceberg.demo.posts ALTER COLUMN title FIRST;

-- 将列移到另一列之后
ALTER TABLE iceberg.demo.posts ALTER COLUMN author AFTER title;
```

### 5.6 更新列注释

```sql
-- 更新列注释
ALTER TABLE iceberg.demo.posts ALTER COLUMN read_count COMMENT '文章阅读量（累计）';
```

## 六、数据分析项目实战：分区演进

让我们用一个完整的场景来演示分区演进和 Schema 演进的实战价值。

### 场景：博客文章表的演进过程

```sql
-- ===== V1 版本：初始建表（按天分区）=====
DROP TABLE IF EXISTS iceberg.demo.posts;

CREATE TABLE iceberg.demo.posts (
    id          INT,
    title       STRING,
    content     STRING,
    author      STRING,
    created_at  TIMESTAMP,
    views       INT
) USING iceberg
PARTITIONED BY (days(created_at))
TBLPROPERTIES ('format-version' = '2');

-- 写入 V1 数据
INSERT INTO iceberg.demo.posts VALUES
    (1, 'Kafka 入门', 'Kafka 是...', 'browncutie',
     TIMESTAMP '2026-04-15 10:00:00', 150),
    (2, 'Spark 教程', 'Spark 是...', 'browncutie',
     TIMESTAMP '2026-04-15 14:00:00', 230),
    (3, 'HDFS 详解', 'HDFS 是...', 'browncutie',
     TIMESTAMP '2026-04-16 09:00:00', 89);

-- 查看当前分区
SELECT * FROM iceberg.demo.posts.partitions;

-- ===== V2 版本：新增标签和状态列 =====
ALTER TABLE iceberg.demo.posts ADD COLUMN tags ARRAY<STRING>;
ALTER TABLE iceberg.demo.posts ADD COLUMN status STRING DEFAULT 'draft';

-- 更新已有数据的标签和状态
UPDATE iceberg.demo.posts
SET tags = ARRAY('大数据'), status = 'published'
WHERE id IN (1, 2, 3);

-- 写入 V2 格式的新数据
INSERT INTO iceberg.demo.posts VALUES
    (4, 'Flink 实时', 'Flink 是...', 'browncutie',
     ARRAY('flink', '实时'), 'published',
     TIMESTAMP '2026-04-17 10:00:00', 56);

-- 验证：旧数据和新数据都能正常查询
SELECT id, title, tags, status FROM iceberg.demo.posts ORDER BY id;

-- ===== V3 版本：字段重命名 =====
ALTER TABLE iceberg.demo.posts RENAME COLUMN views TO read_count;

-- 验证：旧数据使用旧字段名写入，新数据使用新字段名
-- Iceberg 通过 field_id 映射，两者都能正常读取
SELECT id, title, read_count FROM iceberg.demo.posts ORDER BY id;

-- ===== V4 版本：分区演进（从按天改为按月）=====
-- 业务分析发现大部分查询是月度报表，按天分区太细了
ALTER TABLE iceberg.demo.posts
REPLACE PARTITION FIELD days(created_at) WITH months(created_at);

-- 后续写入的数据会按月分区
INSERT INTO iceberg.demo.posts VALUES
    (5, 'Go 入门', 'Go 是...', 'guest',
     ARRAY('go', '编程'), 'published',
     TIMESTAMP '2026-05-01 10:00:00', 30);

-- 查看分区变化
SELECT * FROM iceberg.demo.posts.partitions;

-- ===== V5 版本：添加桶分区 =====
-- 需要按作者做 JOIN 优化
ALTER TABLE iceberg.demo.posts ADD PARTITION FIELD bucket(4, author);

-- 查看更新后的分区
SELECT * FROM iceberg.demo.posts.partitions;

-- ===== V6 版本：重写旧数据以适配新分区 =====
CALL iceberg.system.rewrite_data_files(
    table => 'iceberg.demo.posts',
    strategy => 'sort',
    sort_order => 'created_at ASC'
);

-- 验证重写后的文件分布
SELECT
    CAST(file_size_in_bytes / 1024 / 1024 AS DECIMAL(10,2)) AS size_mb,
    record_count
FROM iceberg.demo.posts.files;

-- ===== 查看完整的 Schema 演进历史 =====
-- 通过 history 表可以看到所有变更
SELECT * FROM iceberg.demo.posts.history;
```

## 七、面试题

### 面试题 1：Iceberg 的隐藏分区有什么优势？与传统 Hive 分区有什么区别？

**参考答案：**

隐藏分区是 Iceberg 独有的特性，其核心优势在于**将分区逻辑与用户数据完全解耦**。

与传统 Hive 分区相比：

1. **写入透明**：Hive 写入时必须手动指定分区值（`PARTITION (dt='2026-04-15')`），而 Iceberg 用户只需要写入数据，Iceberg 根据分区变换函数自动提取分区值。

2. **分区字段不占用数据列**：Hive 的分区字段在表中是独立存在的物理列，而 Iceberg 的分区字段就是普通的数据列，只是被用于分区计算。

3. **分区演进**：Hive 修改分区策略需要重写所有数据（`INSERT OVERWRITE`），而 Iceberg 只需要一条 `ALTER TABLE` 语句。已有数据不需要重写，仍然可以正常读取。

4. **查询简化**：Hive 查询时需要知道分区字段并在 WHERE 子句中使用，而 Iceberg 用户只需要按数据列查询，Iceberg 自动做分区裁剪。

底层实现上，Iceberg 在 Manifest File 中为每个数据文件记录了分区值，而不是通过目录结构来推断分区。这使得分区演进成为可能，因为新写入的文件可以使用新的分区策略，而旧文件仍然保留原来的分区信息。

### 面试题 2：Schema 演进时，Iceberg 如何处理旧数据文件中不存在的新列？

**参考答案：**

Iceberg 通过 **field_id 机制**来实现 Schema 演进的兼容性。

每个字段在创建时被分配一个唯一的 field_id，即使字段被重命名，field_id 也不会改变。Manifest File 中记录了每个数据文件使用的是哪个 schema-id。

当读取旧数据文件时，Iceberg 会做以下处理：

1. **新增列**：旧数据文件中没有新列的数据，Iceberg 会自动填充 null（或 Schema 中定义的默认值）。

2. **删除列**：旧数据文件中仍然包含已删除列的数据，但 Iceberg 在读取时会忽略这些列，不会返回给用户。

3. **重命名列**：通过 field_id 映射，Iceberg 知道旧文件中的 field_id=3 就是重命名后的列，可以正确读取。

4. **类型变更**：如果类型是兼容的（如 INT→LONG），Iceberg 会自动做类型转换。如果不兼容，读取时会报错。

5. **列顺序变更**：每个 schema 都记录了完整的字段列表和顺序，Iceberg 会按照当前 schema 的顺序返回数据，无论底层文件中的列顺序如何。

### 面试题 3：分区演进后，旧的未分区数据文件会怎样？会影响查询性能吗？

**参考答案：**

分区演进后，旧的数据文件**不会自动重写**，它们仍然保持原来的存储方式。这在 Iceberg 的 Manifest File 中有明确标记。

**对查询性能的影响：**

- 新数据文件会按照新的分区策略存储，查询时可以享受分区裁剪
- 旧数据文件无法享受分区裁剪，查询时需要扫描这些文件中的所有数据
- 随着新数据的积累，旧数据的占比会越来越低，性能影响会逐渐减小

**最佳实践：**

1. 分区演进后，建议在低峰期执行 `rewrite_data_files` 来重写旧数据
2. 如果表数据量很大，可以分批重写，避免一次性消耗过多资源
3. 如果短期内不会有大量查询访问旧数据，可以暂不重写
4. 使用 `expire_snapshots` 和 `remove_orphan_files` 清理重写后产生的旧文件
