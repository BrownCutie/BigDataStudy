# 04 - 时间旅行与快照：穿越数据的历史长河

## 一、什么是时间旅行？

想象一下这个场景：你的数据平台上线后运行了几个月，有一天你发现上周的一个数据同步脚本有 bug，导致部分文章的阅读量被错误覆盖。你需要恢复到上周的正确数据。

如果使用传统的 Hive 表，你只能：
- 从备份恢复整个分区（如果有备份的话）
- 手动重新计算并覆盖错误数据

但如果使用 Iceberg，你可以直接**查询任意历史时间点的数据**，甚至一键**回滚到某个历史快照**。这就是**时间旅行（Time Travel）**。

```
┌────────────────────────────────────────────────────────────────┐
│                     时间旅行概念图                               │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  时间轴 ───────────────────────────────────────────►           │
│                                                                │
│  4月15日         4月16日         4月17日         4月18日         │
│  10:00           09:00           10:00           14:00         │
│    │               │               │               │            │
│    ▼               ▼               ▼               ▼            │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐       │
│  │Snapshot1│   │Snapshot2│   │Snapshot3│   │Snapshot4│       │
│  │INSERT   │   │UPDATE   │   │INSERT   │   │DELETE   │       │
│  │3篇文章  │   │阅读量+50│   │1篇新文  │   │1篇删除  │       │
│  │views:   │   │views:   │   │views:   │   │views:   │       │
│  │150,230,89│  │200,280,139│ │200,280,139│ │200,280,0  │    │
│  └─────────┘   └─────────┘   │0        │   └─────────┘       │
│                              └─────────┘                      │
│                                                                │
│  "我想看 4月16日 10:00 的数据"                                  │
│  → SELECT * FROM posts VERSION AS OF Snapshot2                 │
│  → 结果: 3篇文章，views 分别为 200, 280, 139                   │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 二、快照（Snapshot）基础

在深入时间旅行之前，我们需要先彻底理解快照机制。

### 2.1 快照的核心属性

每个快照包含以下关键信息：

```
┌────────────────────────────────────────────────────────────────┐
│                    快照的核心属性                                │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  Snapshot {                                                    │
│    snapshot_id:     7890123456789       -- 唯一标识符          │
│    parent_id:       4560123456789       -- 父快照 ID           │
│    committed_at:    2026-04-17 10:30:00 -- 提交时间            │
│    operation:       append              -- 操作类型            │
│    summary: {                            -- 操作摘要            │
│      added-data-files:   2                                     │
│      deleted-data-files: 0                                     │
│      added-records:      150                                   │
│      total-records:      450                                   │
│    }                                                          │
│    manifest-list:  snap-7890123456789.avro  -- 清单列表         │
│  }                                                            │
│                                                                │
│  operation 类型：                                              │
│  ├── append:       追加写入（INSERT）                          │
│  ├── overwrite:    覆盖写入（INSERT OVERWRITE）                │
│  ├── replace:      替换（ALTER）                               │
│  └── delete:       删除数据文件（DELETE）                       │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 2.2 查看快照历史

```sql
-- 查看所有快照
SELECT * FROM iceberg.demo.posts.snapshots;

-- 查看详细快照信息（包含操作摘要）
SELECT
    snapshot_id,
    parent_id,
    committed_at,
    operation,
    summary['added-data-files'] AS added_files,
    summary['deleted-data-files'] AS deleted_files,
    summary['added-records'] AS added_records,
    summary['deleted-records'] AS deleted_records,
    summary['total-records'] AS total_records,
    summary['total-data-files'] AS total_files
FROM iceberg.demo.posts.snapshots
ORDER BY committed_at;

-- 只看快照 ID 和时间
SELECT snapshot_id, committed_at, operation
FROM iceberg.demo.posts.snapshots
ORDER BY committed_at DESC;
```

### 2.3 快照之间的关系

```

│                   快照链（Snapshot Chain）                      │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  Snapshot 1 (INSERT)                                           │
│  snapshot_id: 111111                                           │
│  parent_id:  null     ← 第一个快照没有父节点                   │
│  files: [A.parquet, B.parquet]                                 │
│      │                                                        │
│      ▼                                                        │
│  Snapshot 2 (UPDATE)                                           │

│      │                                                        │

│  snapshot_id: 333333                                           │

│      │                                                        │
│      ▼                                                        │

│  snapshot_id: 444444                                           │
│  parent_id:  333333                                           │

```

## 三、时间旅行：按时间查询历史数据

### 3.1 使用 VERSION AS OF 语法

```sql
-- 准备数据：先写入，再更新，再删除
INSERT INTO iceberg.demo.posts VALUES
    (100, '时间旅行测试', '这是一篇用于测试时间旅行的文章...',
     'browncutie', ARRAY('测试'), 'published',
     TIMESTAMP '2026-04-15 10:00:00', TIMESTAMP '2026-04-15 10:00:00', 100, 500);

-- 记录当前快照 ID
SELECT snapshot_id FROM iceberg.demo.posts.snapshots
ORDER BY committed_at DESC LIMIT 1;

-- 假设快照 ID 为 5555555555555

-- 更新数据
UPDATE iceberg.demo.posts SET views = 999 WHERE id = 100;

-- 查询当前数据：views = 999
SELECT id, title, views FROM iceberg.demo.posts WHERE id = 100;

-- 使用时间旅行查询历史数据：views = 100
SELECT id, title, views FROM iceberg.demo.posts VERSION AS OF 5555555555555 WHERE id = 100;
```

### 3.2 使用 TIMESTAMP 语法

```sql
-- 按时间戳查询
SELECT id, title, views
FROM iceberg.demo.posts
VERSION AS OF TIMESTAMP '2026-04-15 10:30:00'
WHERE id = 100;

-- 也可以用 AT 语法（效果相同）
SELECT id, title, views
FROM iceberg.demo.posts
FOR SYSTEM_TIME AS OF TIMESTAMP '2026-04-15 10:30:00'
WHERE id = 100;
```

### 3.3 PySpark 中的时间旅行

```python
spark.sparkContext.setLogLevel("WARN")

# ===== 查看所有快照 =====
snapshots = spark.sql("""
    SELECT snapshot_id, committed_at, operation,
           summary['total-records'] AS total_records
    FROM iceberg.demo.posts.snapshots

""")
snapshots.show(truncate=False)

# ===== 按快照 ID 查询历史数据 =====

historical.show(truncate=False)

# ===== 按时间戳查询历史数据 =====

""")
historical_by_time.show(truncate=False)

# ===== 对比当前数据和历史数据 =====
current = spark.table("iceberg.demo.posts").filter("id = 100").select("views").collect()[0][0]

    SELECT views FROM iceberg.demo.posts VERSION AS OF {snapshot_id} WHERE id = 100

print(f"当前 views: {current}")
print(f"历史 views: {history}")
print(f"变化量: {current - history}")

spark.stop()
```

## 四、回滚到历史快照

有时候你需要不仅仅是查询历史数据，而是要把整个表**回滚**到某个历史状态。

### 4.1 使用 rollback_to_snapshot 存储过程

```sql
-- 查看快照历史，找到要回滚的目标快照
SELECT snapshot_id, committed_at, operation,
    summary['total-records'] AS total_records
FROM iceberg.demo.posts.snapshots
ORDER BY committed_at;

-- 假设我们要回滚到 snapshot_id = 5555555555555

-- 执行回滚（创建一个新快照，指向旧快照的状态）
CALL iceberg.system.rollback_to_snapshot(
    'iceberg.demo.posts',
    5555555555555
);

-- 验证回滚结果
SELECT id, title, views FROM iceberg.demo.posts WHERE id = 100;

-- 查看回滚后的快照历史
SELECT snapshot_id, committed_at, operation
FROM iceberg.demo.posts.snapshots
ORDER BY committed_at DESC;
```

```
┌────────────────────────────────────────────────────────────────┐
│                    回滚操作原理                                   │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  回滚前：                                                      │
│                                                                │
│  Snapshot 1 ──► Snapshot 2 ──► Snapshot 3 (当前)              │
│  (views=100)   (views=999)    (新增了一篇文章)                 │
│                                                                │
│  执行 rollback_to_snapshot(Snapshot 1)：                       │
│                                                                │
│  Snapshot 1 ──► Snapshot 2 ──► Snapshot 3 ──► Snapshot 4     │
│  (views=100)   (views=999)    (当前)        (当前, 回滚后)    │
│                                               │               │
│                                               ▼               │
│                                          指向 Snapshot 1     │
│                                          的数据文件状态       │
│                                          (views=100)          │
│                                                                │
│  注意：                                                        │
│  1. 回滚不是删除快照，而是创建新快照指向旧状态                  │
│  2. 回滚操作本身也是原子性的                                    │
│  3. 所有快照链仍然保留，可以继续回滚                            │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 4.2 使用 rollback_to_timestamp 存储过程

```sql
-- 按时间戳回滚
CALL iceberg.system.rollback_to_timestamp(
    'iceberg.demo.posts',
    TIMESTAMP '2026-04-15 10:30:00'
);
```

## 五、快照过期与清理

时间旅行虽然强大，但有一个代价：**旧快照引用的数据文件会一直占用存储空间**。如果不清理，存储成本会越来越高。

### 5.1 理解快照过期

```
┌────────────────────────────────────────────────────────────────┐
│                    快照过期示意图                                │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  过期前：                                                      │
│  Snapshot1 ──► Snapshot2 ──► Snapshot3 ──► Snapshot4 (当前)   │
│  │               │               │               │            │
│  │ 文件A,B        │ 文件C,D        │ 文件C,D,E      │ 文件C,E  │
│  │               │ (A,B仍存在)    │ (A,B仍存在)    │            │
│  │               │               │               │            │
│  │ 存储占用: A+B+C+D+E = 5个文件                       │      │
│                                                                │
│  expire_snapshots(保留 Snapshot3 和 Snapshot4):                │
│                                                                │
│  Snapshot1 ✗ ─ Snapshot2 ✗ ─ Snapshot3 ──► Snapshot4 (当前)   │
│                              │               │                │
│                              │ 文件C,D,E      │ 文件C,E       │
│                              │               │                │
│                              │ 存储占用: C+D+E = 3个文件       │
│                              │ (A,B 被标记为可删除)             │
│                                                                │
│  注意：                                                        │
│  - 过期的快照被标记为删除，但数据文件还在                       │
│  - 需要额外执行 remove_orphan_files 清理孤立文件               │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 5.2 expire_snapshots 过期旧快照

```sql
-- 保留最近 3 天的快照，更早的过期
CALL iceberg.system.expire_snapshots(
    'iceberg.demo.posts',
    TIMESTAMP '2026-04-12 00:00:00'   -- 早于此时间的快照过期
);

-- 保留最近 10 个快照
CALL iceberg.system.expire_snapshots(
    'iceberg.demo.posts',
    TIMESTAMP '1970-01-01 00:00:00',  -- 不过期（设为很早的时间）
    10                                 -- 保留最近 10 个快照
);

-- 同时指定时间和数量
CALL iceberg.system.expire_snapshots(
    'iceberg.demo.posts',
    TIMESTAMP '2026-04-12 00:00:00',  -- 时间阈值
    100                                -- 最少保留 100 个快照
);

-- 查看过期后的快照列表
SELECT snapshot_id, committed_at, operation
FROM iceberg.demo.posts.snapshots
ORDER BY committed_at;
```

### 5.3 remove_orphan_files 清理孤立文件

过期快照后，有些数据文件不再被任何快照引用，称为"孤立文件"（Orphan Files）。

```sql
-- 清理孤立文件（建议在 expire_snapshots 之后执行）
CALL iceberg.system.remove_orphan_files(
    table => 'iceberg.demo.posts',
    older_than => TIMESTAMP '2026-04-12 00:00:00'   -- 只清理早于此时间的文件
);

-- 注意：older_than 参数很重要！
-- 如果省略，可能会误删正在写入中的文件
-- 建议设置为比当前时间早几个小时
```

## 六、数据分析项目实战：查询博客文章的历史版本

让我们用一个完整的场景来演示时间旅行的实际价值。

### 场景：数据审计 -- 追踪文章阅读量的变化

```sql
-- ===== 第一步：准备测试数据 =====
-- 确保有一篇测试文章
INSERT INTO iceberg.demo.posts VALUES
    (200, '审计测试文章', '用于演示时间旅行的审计功能...',
     'browncutie', ARRAY('审计', '测试'), 'published',
     TIMESTAMP '2026-04-15 10:00:00', TIMESTAMP '2026-04-15 10:00:00', 100, 2000);

-- 记录初始快照
SELECT snapshot_id, committed_at, operation,
    summary['total-records'] AS total
FROM iceberg.demo.posts.snapshots
ORDER BY committed_at DESC LIMIT 1;

-- 假设得到 snapshot_id = 1111111111111

-- ===== 第二步：模拟多次更新 =====

-- 模拟第一天：阅读量增长到 150
UPDATE iceberg.demo.posts SET views = 150, updated_at = TIMESTAMP '2026-04-16 00:00:00' WHERE id = 200;

-- 模拟第二天：阅读量增长到 230
UPDATE iceberg.demo.posts SET views = 230, updated_at = TIMESTAMP '2026-04-17 00:00:00' WHERE id = 200;

-- 模拟第三天：阅读量异常飙升（可能是刷量）
UPDATE iceberg.demo.posts SET views = 9999, updated_at = TIMESTAMP '2026-04-18 00:00:00' WHERE id = 200;

-- ===== 第三步：查看完整的变更历史 =====
SELECT
    snapshot_id,
    committed_at,
    operation,
    summary['added-records'] AS added,
    summary['deleted-records'] AS deleted,
    summary['total-records'] AS total
FROM iceberg.demo.posts.snapshots
ORDER BY committed_at DESC;

-- ===== 第四步：逐个快照查询文章的阅读量 =====
-- 查询初始状态
SELECT id, title, views, updated_at
FROM iceberg.demo.posts VERSION AS OF 1111111111111
WHERE id = 200;

-- 查询第一天更新后的状态
SELECT snapshot_id FROM iceberg.demo.posts.snapshots
WHERE committed_at >= TIMESTAMP '2026-04-16 00:00:00'
ORDER BY committed_at ASC LIMIT 1;

-- 假设得到 snapshot_id = 2222222222222
SELECT id, title, views, updated_at
FROM iceberg.demo.posts VERSION AS OF 2222222222222
WHERE id = 200;

-- ===== 第五步：发现问题并回滚 =====
-- 发现阅读量异常飙升到 9999，需要回滚到异常之前
-- 回滚到第二天正常的状态
CALL iceberg.system.rollback_to_snapshot('iceberg.demo.posts', 2222222222222);

-- 验证回滚结果
SELECT id, title, views, updated_at FROM iceberg.demo.posts WHERE id = 200;
-- 应该显示 views = 150
```

### 场景：数据对比 -- 比较两个时间点的数据差异

```sql
-- 使用 CTE 对比两个快照的数据差异
WITH current_data AS (
    SELECT id, title, views
    FROM iceberg.demo.posts
    WHERE id = 200
),
historical_data AS (
    SELECT id, title, views
    FROM iceberg.demo.posts VERSION AS OF 1111111111111
    WHERE id = 200
)
SELECT
    c.title,
    h.views AS original_views,
    c.views AS current_views,
    c.views - h.views AS views_change
FROM current_data c
JOIN historical_data h ON c.id = h.id;
```

### 场景：增量数据处理

```sql
-- 获取两个快照之间的增量数据
-- 这是 Iceberg 的一个高级特性，可以高效地获取新增或变更的数据

-- 方法一：对比两个快照的文件差异
SELECT
    f.file_path,
    f.record_count,
    f.file_size_in_bytes
FROM iceberg.demo.posts.files f
WHERE f.snapshot_id = (
    SELECT snapshot_id FROM iceberg.demo.posts.snapshots
    ORDER BY committed_at DESC LIMIT 1
)
AND f.file_path NOT IN (
    SELECT file_path FROM iceberg.demo.posts.files
    WHERE snapshot_id = 1111111111111
);
```

## 七、面试题

### 面试题 1：Iceberg 的时间旅行底层是怎么实现的？

**参考答案：**

Iceberg 的时间旅行基于**多版本并发控制（MVCC）**和**不可变快照链**实现。

核心机制：
1. **快照不可变性**：每个快照一旦提交就是不可变的，包含该时间点表的完整数据引用
2. **快照链**：每个快照通过 `parent_id` 指向其父快照，形成完整的变更链
3. **数据文件不删除**：UPDATE/DELETE 操作生成新文件标记旧文件为"删除"，但旧文件在快照过期前不会被物理删除
4. **Manifest List**：每个快照关联一个 Manifest List，其中记录了该快照包含的所有 Manifest File，Manifest File 再记录了所有数据文件

查询流程：
1. 从 `metadata.json` 获取当前快照 ID（或指定历史快照 ID）
2. 通过快照 ID 找到对应的 Manifest List
3. 遍历 Manifest List 中的 Manifest File，获取所有数据文件路径
4. 根据查询条件进行文件级过滤（分区裁剪、列裁剪）
5. 只读取需要的数据文件

### 面试题 2：快照过期时需要注意什么？

**参考答案：**

快照过期是一个不可逆的操作，需要特别注意以下几点：

1. **确认保留策略**：过期前要明确需要保留多长时间的数据。一般建议至少保留 3-7 天的快照。
2. **避免影响正在运行的查询**：如果有长时间运行的查询基于旧快照，过期操作可能会影响它们。建议在低峰期执行。
3. **先过期快照，再清理文件**：`expire_snapshots` 只是标记旧快照为删除，数据文件还在。需要再执行 `remove_orphan_files` 才能真正释放空间。
4. **`remove_orphan_files` 的 `older_than` 参数很关键**：要设置一个合理的时间窗口（建议比当前时间早几个小时），避免误删正在写入中的文件。
5. **生产环境建议使用调度工具**：如 Airflow，定期执行维护任务，而不是手动操作。

### 面试题 3：时间旅行在实际生产中有哪些应用场景？

**参考答案：**

1. **数据审计**：追踪数据在某个时间段内的变化，满足合规要求（如 GDPR、金融审计）。
2. **数据回滚**：发现错误数据后，一键回滚到正确的快照状态，而不是手动修复。
3. **增量 ETL**：只处理两次快照之间的变化数据，而不是每次全量扫描。通过对比快照的 Manifest File 可以高效获取增量。
4. **调试分析**：复现某个时间点的数据状态，排查数据质量问题。
5. **A/B 测试**：对比不同版本的数据效果。
6. **机器学习特征回溯**：用历史数据重新训练模型。
7. **数据备份与灾难恢复**：快照本身就是一种轻量级的备份机制。
