# 04 - DAG 工作流开发：从简单到复杂的工作流设计

## 一、DAG 是什么？

DAG（Directed Acyclic Graph，有向无环图）是 DolphinScheduler 编排任务的核心概念。理解 DAG 是掌握工作流开发的基础。

### 1.1 拆解 DAG 三个关键词

```
┌──────────────────────────────────────────────────────────────┐
│                     DAG = 有向 + 无环 + 图                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  有向（Directed）：任务之间有方向性                           │
│  ├── A → B 表示 "A 完成后才执行 B"                           │
│  └── 箭头的方向就是执行顺序                                   │
│                                                              │
│  无环（Acyclic）：不能形成闭环                                │
│  ├── A → B → C → A ❌ 这是环路，不允许！                     │
│  └── 如果允许环路，任务就会无限循环执行                       │
│                                                              │
│  图（Graph）：由节点和边组成                                 │
│  ├── 节点 = 任务（Shell/SQL/Spark...）                       │
│  └── 边 = 依赖关系（连线）                                   │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 合法 vs 非法的 DAG

```
合法的 DAG：

  场景 1: 线性流水线
  ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐
  │  A   │──→│  B   │──→│  C   │──→│  D   │    ✓ 合法
  └──────┘   └──────┘   └──────┘   └──────┘

  场景 2: 并行汇聚
  ┌──────┐
  │  A   │──┐
  └──────┘  │   ┌──────┐
  ┌──────┐  ├──→│  C   │                ✓ 合法
  │  B   │──┘   └──────┘
  └──────┘

  场景 3: 扇出
            ┌──→┌──────┐
  ┌──────┐  │   │  B   │
  │  A   │──┘   └──────┘               ✓ 合法
  └──────┘
            └──→┌──────┐
                │  C   │
                └──────┘


非法的 DAG（有环）：

  ┌──────┐   ┌──────┐
  │  A   │──→│  B   │
  └──────┘   └──────┘
     ↑            │
     └────────────┘                    ✗ 环路！A 等待 B，B 等待 A
                                       永远不会完成
```

```
│                                                                  │

```

### 2.3 工作流版本管理

DolphinScheduler 支持工作流的版本管理：

```
┌──────────────────────────────────────────────────────────────┐
│                  工作流版本管理                                 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  每次保存工作流都会生成一个新版本                               │
│                                                              │
│  版本历史：                                                   │
│  ├── v3 (2026-04-15) 当前版本  ← 上线运行的就是这个           │

│  ├── 对比版本差异：选择两个版本进行对比                        │

│                                                              │
│  重要提示：                                                   │
│  修改工作流后，如果不上线，运行时用的还是旧版本！               │
│                                                              │

```

## 三、任务参数传递

参数传递是 DAG 工作流开发中最核心的功能之一。上游任务计算的结果，需要传递给下游任务使用。

### 3.1 三种参数类型

```
┌────────────────────────────────────────────────────────────────┐
│                      参数类型总览                                │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  ┌──────────────────────────────────────────────────────┐     │
│  │  1. 系统内置参数                                       │     │
│  │     DolphinScheduler 自动注入的变量                     │     │
│  │     ${system.biz.date}  ${system.datetime}  ...       │     │
│  └──────────────────────────────────────────────────────┘     │
│                                                                │
│  ┌──────────────────────────────────────────────────────┐     │
│  │  2. 本地参数（自定义参数）                               │     │
│  │     在任务节点上自己定义的参数                           │     │
│  │     如：dt=2026-04-15, table=posts               │     │
│  │     可以使用内置参数赋值，如 dt=${system.biz.date}      │     │
│  └──────────────────────────────────────────────────────┘     │
│                                                                │
│  ┌──────────────────────────────────────────────────────┐     │
│  │  3. 上游参数（跨任务传递）                               │     │
│  │     上游任务通过日志输出传递给下游任务                   │     │
│  │     上游：echo '${setValue(task_result=SUCCESS)}'     │     │
│  │     下游：${task_result}  → 得到 "SUCCESS"             │     │
│  └──────────────────────────────────────────────────────┘     │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 3.2 本地参数配置

```
操作步骤：

1. 双击任务节点 → 打开配置面板
2. 找到「自定义参数」区域
3. 点击「+」添加参数

配置示例：
┌────────────────┬─────────────────────────────┐
│     参数名       │          参数值              │
├────────────────┼─────────────────────────────┤
│ dt             │ ${system.biz.date}          │
│ table_name     │ posts                   │
│ output_path    │ /user/hive/warehouse/data/   │
└────────────────┴─────────────────────────────┘

在脚本中使用：
echo "处理日期: ${dt}"
echo "目标表: ${table_name}"
echo "输出路径: ${output_path}"
```

### 3.3 上游参数传递（重点）

上游任务可以通过在日志中输出特定格式来传递参数给下游：

```
┌────────────────────────────────────────────────────────────────┐
│                    上游参数传递流程                               │
│                                                                │
│  上游任务 A（Shell）：                                           │
│  ┌────────────────────────────────────────────────────────┐   │
│  │                                                        │   │
│  │  # 执行一些计算                                         │   │
│  │  ROW_COUNT=$(hadoop fs -count /user/hive/warehouse/ \  │   │
│  │    | awk '{print $2}')                                 │   │
│  │                                                        │   │
│  │  # 将结果传递给下游任务                                 │   │
│  │  echo '${setValue(row_count='${ROW_COUNT}')}'          │   │
│  │  echo '${setValue(process_date='${system.biz.date}')}' │   │
│  │                                                        │   │
│  └────────────────────────────────────────────────────────┘   │
│                           │                                    │
│                     日志解析提取参数                             │
│                           │                                    │
│                           ▼                                    │
│  下游任务 B（SQL）：                                             │
│  ┌────────────────────────────────────────────────────────┐   │
│  │                                                        │   │
│  │  -- 使用上游传递的参数                                  │   │
│  │  SELECT COUNT(*) AS cnt                                │   │
│  │  FROM posts                                       │   │
│  │  WHERE DATE(created_at) = '${process_date}'            │   │
│  │                                                        │   │
│  │  -- 如果上游行数为 0，可以结合条件分支跳过                │   │
│  │                                                        │   │
│  └────────────────────────────────────────────────────────┘   │
│                                                                │
│  注意：下游任务引用上游参数时，使用 ${参数名} 格式               │
│  参数名就是上游 setValue 中设置的 key                           │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 3.4 参数传递的实际例子

```
场景：网站日志行数检查

上游任务 - 统计日志行数（Shell）：
┌────────────────────────────────────────────────────────┐
│  #!/bin/bash                                           │
│                                                        │
│  # 统计当天日志的行数                                    │
│  DATE=${system.biz.date}                               │
│  LOG_PATH=/user/hive/warehouse/demo_access/dt=${DATE}  │
│                                                        │
│  ROW_COUNT=$(hadoop fs -cat ${LOG_PATH}/* \            │
│    | wc -l | tr -d ' ')                                │
│                                                        │
│  echo "日志行数: ${ROW_COUNT}"                          │
│                                                        │
│  # 传递给下游                                           │
│  echo '${setValue(log_rows='${ROW_COUNT}')}'            │
│  echo '${setValue(check_date='${DATE}')}'               │
└────────────────────────────────────────────────────────┘

下游任务 - 数据质量检查（SQL）：
┌────────────────────────────────────────────────────────┐
│  -- 使用上游传递的参数                                  │
│  SELECT                                                 │
│    '${check_date}' AS check_date,                       │
│    '${log_rows}' AS log_row_count,                      │
│    COUNT(*) AS db_row_count,                            │
│    COUNT(*) - ${log_rows} AS diff                       │
│  FROM demo_access                                       │
│  WHERE dt = '${check_date}'                             │
└────────────────────────────────────────────────────────┘
```

## 四、条件分支（Switch 任务）

### 4.1 什么是条件分支？

有时候我们需要根据上游任务的执行结果，选择不同的执行路径。这就是 Switch 任务的作用。

```
┌────────────────────────────────────────────────────────────────┐
│                    条件分支场景                                  │
│                                                                │
│  业务数据质量检查后：                                           │
│  ├── 数据量正常 → 继续执行指标计算                              │
│  ├── 数据量为空 → 发送告警，跳过计算                            │
│  └── 数据量异常 → 发送告警，使用历史数据                        │
│                                                                │
│  如果没有条件分支，我们就需要手动判断，无法实现自动化              │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 4.2 Switch 任务配置

```
操作步骤：

1. 从工具栏拖拽一个「Switch」节点到画布
2. 双击打开配置面板
3. 配置判断条件（支持表达式）

条件表达式示例：
┌──────────────────────────────────────────────────────────┐
│  Switch 配置                                               │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  判断条件：${log_rows} > 0                                 │
│                                                           │
│  分支 1（条件为 true 时执行）：                             │
│  └──→ [计算指标]                                          │
│                                                           │
│  分支 2（条件为 false 时执行，默认分支）：                  │
│  └──→ [发送告警：数据为空]                                 │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

### 4.3 条件分支完整示例

```
┌────────────────────────────────────────────────────────────────┐
│                业务数据质量检查 — 条件分支工作流                  │
│                                                                │
│  ┌──────────────┐                                             │
│  │ 检查数据行数  │                                             │
│  │   (Shell)    │                                             │
│  │              │                                             │
│  │ 输出:        │                                             │
│  │ row_count=   │                                             │
│  │  12345       │                                             │
│  └──────┬───────┘                                             │
│         │                                                     │
│         ▼                                                     │
│  ┌──────────────────┐                                         │
│  │  Switch 任务      │                                         │
│  │                  │                                         │
│  │ 条件: row_count  │                                         │
│  │       > 0        │                                         │
│  └──┬──────────┬────┘                                         │
│     │          │                                              │
│  true│     false│                                              │
│     │          │                                              │
│     ▼          ▼                                              │
│  ┌────────┐ ┌────────────┐                                    │
│  │计算指标  │ │发送告警:    │                                    │
│  │(Spark)  │ │数据为空,    │                                    │
│  │         │ │跳过计算     │                                    │
│  └────────┘ └────────────┘                                    │
│                                                                │

```

Switch 节点的配置详情：

```
┌──────────────────────────────────────────────────────────┐
│  Switch 节点配置                                          │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  判断条件（expr）：                                        │
│  ┌────────────────────────────────────────────────┐      │
│  │ ${log_rows} > 0                                │      │
│  └────────────────────────────────────────────────┘      │
│                                                          │
│  分支列表：                                                │
│  ┌────────────────┬─────────────────────────────┐        │

│                                                          │
│  注意：                                                    │
│  1. 条件表达式支持 JavaScript 语法                         │
│  2. 默认分支在所有条件都不满足时执行                        │
│  3. 多个条件分支可以并列配置                                │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## 五、子工作流

### 5.1 什么时候用子工作流？

当一个工作流变得非常复杂（节点超过 20 个），或者某些处理逻辑在多个工作流中重复使用时，就可以考虑使用子工作流。

```
┌────────────────────────────────────────────────────────────────┐
│                    子工作流的使用场景                             │
│                                                                │
│  场景 1：复用通用逻辑                                          │
│  ├── "数据质量检查"在多个工作流中都需要                        │
│  ├── 抽取为子工作流，其他工作流直接引用                        │
│  └── 修改时只需改一处                                         │
│                                                                │
│  场景 2：简化复杂工作流                                        │
│  ├── 一个工作流 30+ 个节点，画布太拥挤                        │
│  ├── 把相关节点分组为子工作流                                  │
│  └── 主工作流只关注整体流程                                    │
│                                                                │
│  场景 3：权限隔离                                              │
│  ├── 不同团队负责不同环节                                      │
│  ├── 子工作流可以由不同团队维护                                │
│  └── 主工作流只负责编排                                        │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 5.2 子工作流配置

```
操作步骤：

1. 先创建一个普通工作流（如 demo_quality_check），保存并上线
2. 在主工作流中拖拽一个「Sub-Workflow」节点
3. 双击打开配置面板
4. 选择子工作流：demo_quality_check
5. 配置参数传递（可选）
```

```
┌────────────────────────────────────────────────────────────────┐
│                子工作流示例                                      │
│                                                                │
│  主工作流：demo_daily_pipeline                                  │
│                                                                │
│  ┌──────────┐     ┌──────────────┐     ┌──────────┐          │
│  │ 数据采集  │────→│ [子工作流]    │────→│ 指标计算  │          │
│  │          │     │ 数据质量检查  │     │          │          │
│  └──────────┘     │              │     └──────────┘          │
│                   └──────┬───────┘                            │
│                          │                                    │
│                          ▼ 展开                               │
│                   ┌──────────────┐                            │
│                   │ 检查行数      │                            │
│                   │    (SQL)     │                            │
│                   └──────┬───────┘                            │
│                          │                                    │
│                   ┌──────▼───────┐                            │
│                   │ Switch       │                            │
│                   │ 行数 > 0?    │                            │
│                   └──┬───────┬──┘                            │
│                      │       │                                │
│                 是    │       │   否                           │
│                      ▼       ▼                                │
│                   ┌────────┐ ┌────────┐                       │
│                   │ 检查空值│ │ 告警   │                       │
│                   └────────┘ └────────┘                       │
│                                                                │
│  子工作流完成后，主工作流继续执行后续节点                        │
│  子工作流如果失败，主工作流也会失败                             │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 5.3 子工作流参数传递

## 六、并行执行

### 6.1 并行执行的场景

在业务数据处理中，有很多可以并行执行的任务：

```
┌────────────────────────────────────────────────────────────────┐
│                    并行执行场景                                   │
│                                                                │
│  场景：业务数据处理中有多个独立的指标计算                        │
│                                                                │
│  串行执行（慢）：                                              │
│  计算日活(10分钟) → 计算热门文章(15分钟) → 计算热词(8分钟)     │
│  总耗时：33 分钟                                               │
│                                                                │
│  并行执行（快）：                                              │
│  ┌──────────┐                                                  │
│  │ 计算日活  │ 10分钟                                          │
│  └──────────┘                                                  │
│  ┌──────────┐                                                  │
│  │ 热门文章  │ 15分钟                                          │
│  └──────────┘          总耗时：max(10,15,8) = 15 分钟          │
│  ┌──────────┐           节省了 18 分钟！                        │
│  │ 搜索热词  │ 8分钟                                           │
│  └──────────┘                                                  │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 6.2 在编辑器中设置并行

在 DolphinScheduler 中设置并行非常简单——只需要把多个节点连接到同一个上游节点即可：

```
操作步骤：

1. 创建上游节点（如"数据清洗"）
2. 创建多个下游节点（如"计算日活"、"热门文章"、"搜索热词"）
3. 从上游节点分别连线到每个下游节点
4. 所有下游节点会自动并行执行

编辑器中的效果：

       ┌──────────┐
       │ 数据清洗  │
       └────┬─────┘
            │
     ┌──────┼──────┐
     │      │      │
     ▼      ▼      ▼
┌────────┐┌──────┐┌──────┐
│计算日活 ││热门文章││搜索热词│
└────────┘└──────┘└──────┘
     │      │      │
     └──────┼──────┘
            │
            ▼
     ┌──────────┐
     │ 汇总导出  │
     └──────────┘

注意：只有三个并行任务全部完成后，"汇总导出"才会执行
```

### 6.3 并行度控制

DolphinScheduler 可以控制 Worker 的并行执行能力：

```yaml
# application.yaml 中的 Worker 配置
worker:
  # Worker 执行任务的线程数（同时执行的任务数）
  exec-threads: 20

  # Worker 任务队列大小
  exec-task-num: 10
```

```
┌──────────────────────────────────────────────────────────────┐
│                   并行度控制说明                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Standalone 模式下：                                         │
│  ├── 只有 1 个 Worker                                       │
│  ├── exec-threads 决定了这个 Worker 同时执行几个任务           │
│  └── 默认 20 个线程，一般够用                                 │
│                                                              │
│  集群模式下：                                                │
│  ├── 多个 Worker 节点                                        │
│  ├── 任务会被分发到不同 Worker 执行                           │
│  └── 实际并行度 = Worker 数量 × 每个 Worker 的线程数          │
│                                                              │
│  如果任务太多，超过并行度上限怎么办？                          │
│  └── 多余的任务会排队等待                                     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 7.2 各节点详细配置

**节点 1：初始化参数（Shell）**

```bash
#!/bin/bash
# 设置处理日期
echo "初始化参数，处理日期: ${system.biz.date}"

# 传递参数给下游
echo '${setValue(dt=${system.biz.date})}'
echo '${setValue(output_dir=/tmp/demo_report/${system.biz.date})}'
```

**节点 2a：查询日活用户（SQL）**

```sql
-- 数据源：Hive
-- 查询当天活跃用户数

SELECT
    COUNT(DISTINCT user_id) AS dau_count
FROM demo_access
WHERE dt = '${dt}'
  AND user_id IS NOT NULL
  AND user_id != 'anonymous'
```

**节点 2b：查询热门文章（SQL）**

```sql
-- 查询当天浏览量 TOP 10 的文章

SELECT
    post_id,
    title,
    COUNT(*) AS view_count
FROM demo_access
WHERE dt = '${dt}'
  AND post_id IS NOT NULL
GROUP BY post_id, title
ORDER BY view_count DESC
LIMIT 10
```

**节点 2c：查询搜索热词（SQL）**

```sql
-- 查询当天搜索量 TOP 10 的关键词

SELECT
    keyword,
    COUNT(*) AS search_count
FROM demo_search_log
WHERE dt = '${dt}'
  AND keyword IS NOT NULL
  AND keyword != ''
GROUP BY keyword
ORDER BY search_count DESC
LIMIT 10
```

**节点 3：汇总导出（Shell）**

```bash
#!/bin/bash
# 创建输出目录
mkdir -p ${output_dir}

# 生成汇总报告
cat > ${output_dir}/daily_report_${dt}.txt << EOF
===========================================
业务数据日报 - ${dt}
===========================================

日活用户数: ${dau_count}
（来自上游 SQL 查询结果）

热门文章 TOP 10:
${top_posts}

搜索热词 TOP 10:
${hot_keywords}

报告生成时间: $(date '+%Y-%m-%d %H:%M:%S')
EOF

echo "日报已生成: ${output_dir}/daily_report_${dt}.txt"
ls -la ${output_dir}/
```

```

2. 点击「上线」
3. 点击「运行」
```
