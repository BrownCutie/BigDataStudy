# 06 - 生产实战与面试：从开发到面试的全面指南

## 一、生产环境最佳实践

### 1.1 工作流设计规范

```
┌──────────────────────────────────────────────────────────────────┐
│                    工作流设计最佳实践                               │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 单个工作流节点数控制在 20 个以内                              │
│     ├── 超过 20 个节点时，考虑拆分为子工作流                     │
│     └── 太复杂的工作流难以维护和排查问题                         │
│                                                                  │
│  2. 每个任务都要配置合理的重试策略                                │
│     ├── Shell 任务：重试 2-3 次，间隔 1-5 分钟                   │
│     ├── SQL 任务：重试 2 次，间隔 2 分钟                         │
│     ├── Spark 任务：重试 1-2 次，间隔 5 分钟                     │
│     └── HTTP 任务：重试 3 次，间隔 1 分钟                        │
│                                                                  │
│  3. 每个任务都要设置超时告警                                      │
│     ├── 根据历史执行时间设置合理阈值                              │
│     ├── 一般设置为平均执行时间的 2-3 倍                           │
│     └── 避免任务卡住无人发现                                     │
│                                                                  │
│  4. 任务命名要规范                                               │
│     ├── 格式：[模块]_[操作]_[对象]，如 collect_upload_logs       │
│     ├── 避免使用 "test1"、"temp" 等无意义名称                    │
│     └── 命名要能让人一眼看出任务的作用                           │
│                                                                  │
│  5. 使用参数化而不是硬编码                                       │
│     ├── 日期使用 ${system.biz.date} 而不是写死                   │
│     ├── 表名、路径等通过参数传递                                 │
│     └── 方便补数和复用                                           │
│                                                                  │
│  6. 善用前置脚本                                                 │
│     ├── 在任务执行前检查前置条件                                 │
│     ├── 如：检查 HDFS 文件是否存在、检查表是否有数据             │
│     └── 前置条件不满足时直接跳过，避免无意义的执行                │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 三、多租户与权限管理

### 3.1 租户的概念

```
┌────────────────────────────────────────────────────────────────┐
│                    租户与 Linux 用户的关系                       │
│                                                                │
│  DolphinScheduler 的租户对应 Linux 系统用户                    │
│  Worker 执行任务时，会以租户对应的 Linux 用户身份运行           │
│                                                                │
│  例如：                                                        │
│  ├── 租户 hadoop → Linux 用户 hadoop                          │
│  │   ├── 执行 Spark 任务时，以 hadoop 用户提交                 │
│  │   └── 可以访问 HDFS 上 hadoop 有权限的文件                 │
│  │                                                            │
│  ├── 租户 hive → Linux 用户 hive                              │
│  │   ├── 执行 Hive SQL 时，以 hive 用户连接                  │
│  │   └── 可以访问 Hive 的元数据和仓库                        │
│  │                                                            │
│  └── 租户 demo → Linux 用户 demo                             │
│      ├── 执行数据平台相关任务                                     │
│      └── 只能访问 demo 用户有权限的资源                       │
│                                                                │
│  重要：                                                        │
│  创建租户前，需要先在 Linux 系统中创建对应的用户！             │
│  或者开启 tenant-auto-create=true 自动创建                    │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

```

   安全中心 → 用户管理 → 创建用户
```

## 四、资源中心管理

### 4.1 资源中心的作用

```
┌────────────────────────────────────────────────────────────────┐
│                    资源中心                                     │
│                                                                │
│  资源中心用于管理任务执行时需要的文件：                         │
│  ├── Shell 脚本文件 (.sh)                                      │
│  ├── Python 脚本文件 (.py)                                    │
│  ├── Spark jar 包 (.jar)                                      │
│  ├── SQL 文件 (.sql)                                          │
│  ├── 配置文件 (.yaml, .properties, .conf)                    │
│  └── 压缩包 (.tar.gz, .zip)                                  │
│                                                                │
│  为什么要用资源中心？                                          │
│  ├── 脚本文件版本管理（支持上传新版本覆盖旧版本）              │
│  ├── 多个任务可以复用同一个脚本文件                            │
│  ├── 不需要在每台 Worker 机器上手动部署脚本                    │
│  └── Worker 执行任务时自动从资源中心下载文件                   │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 4.2 使用资源中心

```
操作步骤：

1. 上传资源
   资源中心 → 文件管理 → 上传文件
   - 选择文件：/opt/scripts/demo_quality_check.sh
   - 描述：业务数据质量检查脚本

2. 在任务中引用资源
   打开任务配置 → 资源区域 → 选择资源文件
   - 选择：demo_quality_check.sh
   - 引用后在脚本中使用：
     bash demo_quality_check.sh ${system.biz.date}

3. 任务执行时，Worker 会自动：
   ├── 从资源中心下载 demo_quality_check.sh 到本地
   ├── 在本地执行脚本
   └── 执行完成后保留文件（或根据配置清理）
```

## 五、常见问题排查

### 5.1 任务超时

```
┌──────────────────────────────────────────────────────────────┐
│                问题：任务执行时间过长                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  排查步骤：                                                  │
│  1. 查看任务实例的执行日志                                    │
│     ├── 是否卡在某个步骤                                      │
│     ├── 是否在等待外部资源                                    │
│     └── 是否有死循环                                          │
│                                                              │
│  2. 检查 Worker 资源                                         │
│     ├── Worker 所在机器的 CPU/内存是否充足                     │
│     ├── 是否有其他大任务占用了资源                             │
│     └── exec-threads 是否设置合理                              │
│                                                              │
│  3. 检查外部依赖                                             │
│     ├── HiveServer2 是否响应慢                                 │
│     ├── HDFS 是否有瓶颈                                      │
│     └── 数据库连接池是否满了                                   │
│                                                              │
│  解决方案：                                                  │
│  ├── 优化 SQL（加索引、分区裁剪、减少数据量）                 │
│  ├── 增加 Worker 节点或线程数                                 │
│  ├── 增大超时告警时间                                         │
│  └── 将大任务拆分为小任务                                     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 依赖阻塞

```
┌──────────────────────────────────────────────────────────────┐
│              问题：工作流一直在等待状态                        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  现象：工作流实例状态为"等待依赖"，一直不执行                   │
│                                                              │
│  排查步骤：                                                  │
│  1. 检查 Dependent 节点配置                                   │
│     ├── 依赖的工作流名称是否正确                              │
│     ├── 日期表达式是否正确                                    │
│     └── 上游工作流是否成功完成                                │
│                                                              │
│  2. 检查上游工作流实例                                       │
│     ├── 进入工作流实例列表                                    │
│     ├── 查看上游工作流在对应日期的执行记录                     │
│     └── 确认上游状态为"成功"而不是"失败"或"运行中"            │
│                                                              │
│  3. 检查调度时间                                              │
│     ├── 上游工作流是否设置了定时调度                           │
│     ├── 上游的定时时间是否在下游的 Dependent 之前             │
│     └── 手动运行上游后，下游是否自动触发                       │
│                                                              │
│  常见原因：                                                  │
│  ├── 上游工作流今天没有执行（调度被禁用了）                    │
│  ├── 上游工作流执行失败了                                     │
│  ├── Dependent 日期表达式写错了                               │
│  └── 上游和下游不在同一个项目（Dependent 跨项目需注意）       │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 5.3 Worker 负载高

```
┌──────────────────────────────────────────────────────────────┐
│              问题：Worker 执行任务很慢                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  排查步骤：                                                  │
│  1. 查看 Worker 日志                                          │
│     tail -f dolphinscheduler-worker.log                      │
│     ├── 是否有线程池满了的警告                                │
│     ├── 是否有频繁的 GC                                      │
│     └── 是否有任务积压                                        │
│                                                              │
│  2. 检查 Worker 机器资源                                     │
│     top    # 查看 CPU 使用率                                  │
│     free -h # 查看内存使用                                    │
│     df -h  # 查看磁盘使用                                    │
│                                                              │
│  3. 检查并行配置                                             │
│     cat application.yaml | grep exec-threads                 │
│                                                              │
│  解决方案：                                                  │
│  ├── 增加 Worker 节点（集群模式）                             │
│  ├── 调大 exec-threads（注意不要超过机器承载能力）            │
│  ├── 使用 Worker Group 将任务分组到不同 Worker                │
│  └── 错峰调度：将高负载任务分散到不同时间段执行               │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 七、10 道高频面试题

### Q1: DolphinScheduler 的架构是什么样的？为什么说是"去中心化"的？

**参考答案：**

DolphinScheduler 采用 Master-Worker 架构，核心组件包括：

- **Master Server**：负责任务调度、DAG 解析、依赖管理、故障转移
- **Worker Server**：负责实际执行任务（Shell、SQL、Spark 等）
- **API Server**：提供 Web UI 后端和 REST API
- **Alert Server**：处理告警通知
- **Logger Server**：收集和展示任务日志
- **Registry (ZooKeeper)**：服务注册与发现（Standalone 模式用 JDBC）

"去中心化"体现在：
1. **Master 多节点**：多个 Master 通过 ZooKeeper 进行分布式锁和任务分发，任何一个 Master 挂了，其他 Master 自动接管其任务
2. **Worker 多节点**：Worker 是无状态的，任务会被均匀分配到空闲的 Worker 上执行
3. **无单点故障**：不像 Airflow 的 Scheduler 存在单点问题，DolphinScheduler 的 Master 和 Worker 都是高可用的

### Q2: DolphinScheduler 中工作流定义和工作流实例的区别？

**参考答案：**

| 维度 | 工作流定义 (Process Definition) | 工作流实例 (Process Instance) |
|------|-------------------------------|---------------------------|
| 性质 | 模板 | 运行记录 |
| 数量 | 通常只有几个/十几个 | 可能成千上万个 |
| 可变性 | 可以修改、保存新版本 | 一旦生成就不可变 |
| 状态 | 上线/下线 | 成功/失败/运行中/等待中 |
| 版本 | 支持版本管理 | 对应某个版本的一次执行 |

类比：工作流定义就像一个"菜谱"，工作流实例就像按照菜谱做的一顿饭。同一个菜谱可以做无数次饭。

### Q3: DolphinScheduler 支持哪些任务类型？你最常用的是哪些？

**参考答案：**

DolphinScheduler 3.4.1 支持 30+ 种任务类型，包括：

- **Shell**：执行 Shell 脚本
- **SQL**：执行数据库查询（Hive、MySQL、PostgreSQL 等）
- **Spark**：提交 Spark 作业（支持 on YARN、K8s、Standalone）
- **Flink**：提交 Flink 作业
- **Python**：执行 Python 脚本
- **HTTP**：调用 HTTP 接口
- **Sub-Workflow**：子工作流
- **Conditions/Switch**：条件分支
- **Dependent**：跨工作流依赖
- **Procedure**：存储过程
- **DataX/Sqoop**：数据同步
- **MapReduce**：提交 MR 作业

在实际项目中，最常用的是 **Shell、SQL、Spark、HTTP 和 Dependent**。

### Q4: DolphinScheduler 的参数传递机制是什么？怎么实现跨任务传递参数？

**参考答案：**

DolphinScheduler 支持三种参数传递方式：

1. **系统内置参数**：由系统自动注入
   - `${system.biz.date}`：业务日期
   - `${system.datetime}`：系统时间
   - `${schedule.time}`：调度时间

2. **本地参数（自定义参数）**：在任务节点上手动定义
   - 可以引用系统参数赋值，如 `dt=${system.biz.date}`
   - 可以设置默认值

3. **上游参数传递**：通过日志输出传递给下游
   - 上游任务在日志中输出：`echo '${setValue(key=value)}'`
   - 下游任务直接使用：`${key}`

**实现原理**：Master 会解析每个任务实例的日志，提取其中的 `setValue` 输出，作为任务实例的输出参数存储到数据库中。下游任务执行时，会自动注入上游的输出参数。

### Q5: 如何实现工作流间的依赖？Dependent 节点怎么配置？

**参考答案：**

工作流间依赖通过 **Dependent 任务节点**实现：

1. 在下游工作流中拖入一个 Dependent 节点
2. 配置依赖关系：
   - 选择上游工作流所在的项目
   - 选择上游工作流名称
   - 设置日期表达式（如 `${system.biz.date}` 表示同一天）
3. Dependent 节点会检查上游工作流在指定日期是否执行成功
4. 只有上游成功，下游才会继续执行

日期表达式支持：
- `${system.biz.date}`：同一天
- `${system.biz.date} - 1`：前一天
- `${system.biz.date} + 1`：后一天

### Q6: 任务失败了怎么处理？DolphinScheduler 的重试机制是怎样的？

**参考答案：**

DolphinScheduler 提供了完善的重试机制：

1. **任务级重试**：
   - 在任务配置中设置"失败重试次数"和"重试间隔"
   - 任务失败后自动重新执行，不需要人工干预
   - 重试仍然失败后，任务状态变为"失败"，触发告警

2. **工作流级恢复**：
   - 工作流失败后，可以从失败的节点恢复执行
   - 不需要重新执行已成功的节点
   - 在工作流实例页面点击"恢复失败任务"

3. **补数（Backfill）**：
   - 对于历史日期的失败，可以通过补数功能重新执行
   - 支持选择日期范围，批量补数

**注意**：重试次数不宜设置过多（一般 2-3 次），否则可能导致资源浪费。对于持续性故障（如 SQL 语法错误），重试也无法解决。

### Q7: 什么是补数（Backfill）？什么场景下需要补数？

**参考答案：**

补数（Backfill）是指对历史日期的数据重新执行工作流。常见场景：

1. **系统故障恢复**：某天 HDFS 或 Hive 不可用，导致当天工作流全部失败
2. **代码修复后回填**：发现指标计算 SQL 有 bug，修复后需要重算历史数据
3. **新增指标回填**：新增了一个统计指标，需要计算过去 30 天的数据
4. **数据重算**：上游数据源数据修正，需要重新处理

在 DolphinScheduler 中执行补数：
- 选择工作流 → 运行 → 补数
- 设置开始日期和结束日期
- 系统会按照日期顺序依次生成工作流实例
- 补数期间 `${system.biz.date}` 会自动替换为对应的日期

### Q8: DolphinScheduler 怎么保证高可用？Standalone 模式和 Cluster 模式的区别？

**参考答案：**

**高可用机制：**
1. Master 多节点：通过 ZooKeeper 实现分布式锁，故障时其他 Master 自动接管
2. Worker 多节点：无状态设计，任务自动分配到其他 Worker
3. API 多节点：通过 Nginx 负载均衡
4. 数据库：MySQL 主从或集群保证数据可靠性

**Standalone vs Cluster：**

| 维度 | Standalone | Cluster |
|------|-----------|---------|
| 组件部署 | 所有组件在一个 JVM | 组件分布在不同机器 |
| 高可用 | 无 | 有 |
| 扩展性 | 不支持 | 支持 |
| 适用场景 | 开发测试 | 生产环境 |
| 配置复杂度 | 简单 | 需要配置 ZooKeeper |

### Q9: 在生产环境中使用 DolphinScheduler，你会做哪些优化？

**参考答案：**

1. **部署优化**
   - 使用 Cluster 模式，Master 至少 3 个节点，Worker 根据任务量配置
   - 数据库使用 MySQL 主从，定期备份
   - 使用独立 ZooKeeper 集群

2. **工作流优化**
   - 单个工作流节点不超过 20 个，复杂逻辑拆分为子工作流
   - 合理设置重试次数和超时时间
   - 使用并行执行提高效率
   - 使用参数化，方便维护和补数

3. **资源优化**
   - 使用 Worker Group 隔离不同优先级的任务
   - 大任务和小任务分配到不同的 Worker
   - 监控 Worker 的 CPU/内存使用，及时扩容

4. **监控优化**
   - 配置完善的告警策略（钉钉/邮件）
   - 关注首页仪表盘的运行趋势
   - 定期清理历史数据（工作流实例、日志）

5. **安全优化**
   - 修改默认密码
   - 启用多租户，按团队隔离
   - 定期审计用户权限

### Q10: 介绍一下你们数据分析项目的数据流水线是怎么用 DolphinScheduler 编排的？

**参考答案：**

我们的数据平台使用 DolphinScheduler 编排了一套完整的数据处理流水线，由 4 个工作流组成：

1. **demo_data_collect（每日数据采集）**
   - 每天 00:30 定时触发
   - 检查 Nginx 日志 → 上传 HDFS → 加载到 Hive 分区表
   - 使用 Shell 任务操作 HDFS，SQL 任务加载数据

2. **demo_data_quality（数据质量检查）**
   - 依赖工作流 1 完成
   - 行数校验（对比前一天数据量）→ 空值检查 → Switch 条件分支
   - 数据正常则继续，数据异常则告警并跳过

3. **demo_metrics_calc（指标计算）**
   - 依赖工作流 2 通过
   - 并行计算 5 个指标：DAU、PV/UV、热门文章、搜索热词、用户画像
   - 使用 Spark 任务进行分布式计算，结果写入 Iceberg

4. **demo_result_export（结果推送）**
   - 依赖工作流 3 完成
   - 并行执行：同步到 Elasticsearch + 更新报表数据库
   - 生成日报邮件

**技术亮点**：
- 使用 Dependent 节点实现工作流间级联触发，只有第一个工作流需要定时调度
- 使用 Switch 条件分支实现数据质量门禁
- 利用并行执行提高效率（5 个指标同时计算）
- 配置了完善的告警策略（钉钉通知 + 超时告警）
- 支持补数，可以方便地回填历史数据
