# 03 - 核心概念与 Web UI：掌握调度平台的操作界面

## 一、核心概念全景

在正式使用 DolphinScheduler 之前，我们需要先搞清楚几个核心概念。这些概念之间的关系如下：

### 1.1 概念速查表

| 概念 | 说明 | 类比 |
|------|------|------|
| **租户 (Tenant)** | 对应 Linux 用户，用于任务执行时的身份隔离 | 公司的部门 |
| **项目 (Project)** | 工作流和资源的逻辑容器 | 部门下的项目组 |
| **工作流定义 (Process Definition)** | DAG 图模板，定义任务和依赖关系 | 项目中的任务说明书 |
| **任务定义 (Task Definition)** | 工作流中的一个节点 | 任务说明书中的一个步骤 |
| **工作流实例 (Process Instance)** | 工作流的一次实际运行 | 按说明书实际做了一次 |
| **任务实例 (Task Instance)** | 任务的一次实际运行 | 实际执行了其中的一个步骤 |
| **调度 (Schedule)** | 定时触发规则 | 每天固定时间做这件事 |
| **告警组 (Alert Group)** | 告警接收人列表 | 出事了通知谁 |

### 1.2 定义 vs 实例

这是一个初学者容易混淆的点：

```
┌──────────────────────────────────────────────────────────────┐
│                定义 (Definition) vs 实例 (Instance)           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  工作流定义（模板）：                                          │
│  "每天凌晨 2 点，先采集日志，再清洗数据，最后计算指标"           │
│  → 这是一个固定的"计划"                                       │
│                                                              │
│  工作流实例（运行记录）：                                       │
│  2026-04-15 02:00 的那次运行 → 成功                           │
│  2026-04-16 02:00 的那次运行 → 失败（第 2 个任务超时）         │
│  2026-04-17 02:00 的那次运行 → 成功                           │
│  → 这些是实际发生的"记录"                                     │
│                                                              │
│  关系：一个定义可以产生无数个实例                               │
│                                                              │
│  ┌──────────────────┐                                        │
│  │  工作流定义       │ ──→ 产生 ──→ 实例 1 (4/15 02:00)      │
│  │  "每日数据流水线"  │           ──→ 产生 ──→ 实例 2 (4/16 02:00)│
│  │                  │           ──→ 产生 ──→ 实例 3 (4/17 02:00)│
│  └──────────────────┘                                        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 2.1 各任务类型配置要点

**Shell 任务：**

```bash
# 在 Web UI 中配置 Shell 任务时，主要填写：
# 1. 脚本内容（直接写命令）
echo "开始处理业务数据..."
hadoop fs -put /tmp/demo_access.log /user/hive/warehouse/demo_access/
echo "日志上传完成，共处理 $(wc -l < /tmp/demo_access.log) 行"
```

**SQL 任务：**

```sql
-- SQL 任务需要配置数据源（在数据源中心先创建）
-- 数据平台场景：查询当天新增文章数量
SELECT COUNT(*) AS new_posts
FROM default.posts
WHERE DATE(created_at) = '${system.biz.date}'
```

**Spark 任务：**

```bash
# Spark 任务配置
# 主程序参数
--class com.example.DemoMetricCalculator
# 资源
--driver-memory 1g
--executor-memory 2g
--executor-cores 2
--num-executors 2
# 主 jar 包
/opt/demo-jar/metric-calculator.jar
# 程序参数
--date ${system.biz.date}
--output /user/hive/warehouse/demo_metrics/
```

**HTTP 任务：**

```
# HTTP 任务配置
URL: http://localhost:9200/posts/_refresh
Method: POST
Headers: Content-Type: application/json
Body: {}
```

## 三、工作流依赖

### 3.1 任务间依赖

在工作流内部，任务之间的依赖通过**连线**来表示：

```
┌────────────────────────────────────────────────────────────────┐
│                    任务依赖示例                                   │
│                                                                │
│  串行依赖（A 完成后才执行 B）：                                   │
│                                                                │
│  ┌──────────┐         ┌──────────┐         ┌──────────┐        │
│  │  任务 A   │────────→│  任务 B   │────────→│  任务 C   │        │
│  │  采集日志  │         │  清洗数据  │         │  写入 Hive │        │
│  └──────────┘         └──────────┘         └──────────┘        │
│                                                                │
│  并行执行（A 和 B 同时运行，都完成后才执行 C）：                   │
│                                                                │
│  ┌──────────┐                                                 │
│  │  任务 A   │──┐                                              │
│  │  采集日志  │  │     ┌──────────┐                             │
│  └──────────┘  ├──→──→│  任务 C   │                            │
│  ┌──────────┐  │     │  汇总结果  │                             │
│  │  任务 B   │──┘     └──────────┘                             │
│  │  采集业务  │                                                 │
│  └──────────┘                                                 │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 3.2 工作流间依赖

有时候一个工作流需要等待另一个工作流完成后才能开始，这就是**跨工作流依赖**：

```
┌────────────────────────────────────────────────────────────────┐
│                   跨工作流依赖示例                                │
│                                                                │
│  工作流 1: 数据采集                                             │
│  ┌──────────┐     ┌──────────┐                                │
│  │ 采集日志  │────→│ 写入 HDFS │                                │
│  └──────────┘     └──────────┘                                │
│        │                                                      │
│        │ 工作流间依赖（等待工作流 1 成功完成）                    │
│        ▼                                                      │
│  工作流 2: 数据处理                                             │
│  ┌──────────┐     ┌──────────┐                                │
│  │ 数据清洗  │────→│ 指标计算  │                                │
│  └──────────┘     └──────────┘                                │
│                                                                │
│  在 Web UI 中配置：                                              │
│  工作流 2 中添加一个 "Dependent" 任务节点                       │
│  → 选择依赖的工作流 1 和日期表达式                               │
│  → 只有工作流 1 在指定日期成功完成后，工作流 2 才会执行           │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 四、定时调度（Cron 表达式）

### 4.1 Cron 表达式基础

DolphinScheduler 使用 Quartz 的 Cron 表达式格式（7 位）：

```
┌──────────────────────────────────────────────────────────────┐
│              Cron 表达式格式（7 位，DolphinScheduler 用 6 位）  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  字段:    秒   分   时   日   月   周   年                    │
│  位置:    0    1    2    3    4    5    6                    │
│           │    │    │    │    │    │    │                    │
│  范围:  0-59 0-59 0-23 1-31 1-12 1-7  1970-2099             │
│                                                              │
│  注意：DolphinScheduler 的 Cron 表达式通常用 6 位（不含年）    │
│                                                              │
│  特殊字符：                                                   │
│  *    任意值（每秒/每分/每小时...）                             │
│  ?    不指定值（日和周不能同时指定，用 ? 占位）                 │
│  -    范围（如 1-5 表示周一到周五）                            │
│  ,    列表（如 1,3,5 表示 1 月、3 月、5 月）                  │
│  /    步长（如 0/5 表示每 5 秒，0 0/1 * * * ? 表示每小时）      │
│  #    第几个星期几（如 6#3 表示第 3 个星期五）                  │
│  L    最后（如 L 表示月末，6L 表示最后一个星期五）               │
│  W    最近工作日                                               │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 常用 Cron 表达式

| 表达式 | 含义 | 数据平台场景 |
|--------|------|---------|
| `0 0 2 * * ?` | 每天凌晨 2:00 | 每日数据采集 |
| `0 0 */1 * * ?` | 每小时整点 | 实时监控检查 |
| `0 */30 * * * ?` | 每 30 分钟 | Kafka lag 监控 |
| `0 0 2 ? * MON-FRI` | 工作日凌晨 2:00 | 周报表生成 |
| `0 0 0 1 * ?` | 每月 1 日凌晨 | 月度数据归档 |
| `0 0 2 1,15 * ?` | 每月 1 号和 15 号 | 半月数据备份 |

### 4.3 在 Web UI 中设置定时调度

```
操作步骤：

1. 进入工作流定义页面
2. 点击工作流定义列表中的「定时」按钮
3. 在弹出的定时配置页面：
   - 启用定时调度（开关打开）
   - 输入 Cron 表达式：0 0 2 * * ?
   - 选择时区：Asia/Shanghai
   - 点击「创建」
4. 回到工作流定义页面，可以看到定时状态变为"已启用"
```

## 五、告警通知

### 5.1 告警策略

当任务失败或超时时，DolphinScheduler 可以发送告警通知：

```
┌────────────────────────────────────────────────────────────────┐
│                      告警通知流程                               │
│                                                                │
│  任务失败/超时                                                  │
│       │                                                        │
│       ▼                                                        │
│  Alert Server 检测到告警事件                                     │
│       │                                                        │
│       ▼                                                        │
│  查询告警组（该工作流关联了哪个告警组？）                          │
│       │                                                        │
│       ▼                                                        │
│  根据告警组配置发送通知                                          │
│       │                                                        │
│       ├──→ 邮件（SMTP）                                        │
│       ├──→ 钉钉机器人（Webhook）                                │
│       ├──→ 企业社交平台（Webhook）                                  │
│       └──→ 飞书（Webhook）                                     │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 5.2 配置告警组

```
操作步骤：

1. 进入项目 → 左侧菜单「项目设置」→「告警组管理」
2. 点击「创建告警组」
3. 填写：
   - 告警组名称：demo-alert-group
   - 告警组描述：业务数据处理告警
4. 选择告警实例（需要先在「告警实例管理」中配置）
   - 邮件告警：填写 SMTP 服务器和收件人
   - 钉钉告警：填写钉钉机器人 Webhook 地址
5. 保存
```

### 5.3 在工作流中使用告警组

```
操作步骤：

1. 进入工作流编辑器
2. 点击右上角的「设置」图标（齿轮）
3. 在弹出的窗口中：
   - 选择告警组：demo-alert-group
   - 告警策略：任务失败 / 全流程失败 / 成功时也通知
   - 超时告警时间：60 分钟（任务超过 60 分钟未完成则告警）
4. 保存工作流
```

### 5.4 钉钉机器人告警配置示例

```
┌──────────────────────────────────────────────────────────┐
│              钉钉机器人告警配置步骤                         │

│                                                          │

### 6.5 内置系统参数

DolphinScheduler 提供了多个内置的系统参数，可以在任务脚本中直接使用：

| 参数 | 说明 | 示例值 |
|------|------|--------|
| `${system.biz.date}` | 业务日期（调度日期） | `2026-04-15` |
| `${system.biz.curdate}` | 业务日期（精确到秒） | `2026-04-15 00:00:00` |
| `${system.datetime}` | 系统当前时间 | `2026-04-15 10:30:00` |
| `${schedule.time}` | 调度触发时间 | `2026-04-15 02:00:00` |
| `${task.instance.id}` | 任务实例 ID | `12345` |
| `${process.instance.id}` | 工作流实例 ID | `67890` |

**使用示例：**

```bash
# 在 Shell 任务中使用业务日期
echo "处理 ${system.biz.date} 的数据"
hadoop fs -put /tmp/logs/access_${system.biz.date}.log /user/hive/warehouse/

# 在 SQL 任务中使用业务日期
SELECT * FROM posts WHERE DATE(created_at) = '${system.biz.date}'
```

```
┌──────────────────────────────────────────────────────────────┐
│               system.biz.date 的含义                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  这是一个初学者很容易搞混的参数！                               │
│                                                              │
│  假设调度配置为每天凌晨 2:00 执行                              │
│  那么在 2026-04-15 02:00 触发执行时：                          │
│                                                              │
│  system.biz.date       = 2026-04-15      （业务日期 = 当天）  │
│  schedule.time         = 2026-04-15 02:00:00（调度时间）      │
│  system.datetime       = 2026-04-15 02:00:05（实际执行时间）  │
│                                                              │
│  为什么需要 biz.date？                                         │
│  因为 T+1 场景下，凌晨 2 点跑的是"昨天"的数据                   │
│  此时 biz.date 可以通过配置偏移为 T-1                          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 7.2 创建第一个数据平台工作流

```
步骤：

1. 进入 demo-data-pipeline 项目
2. 左侧菜单 → 工作流定义 → 创建工作流 → 在线创建
3. 工作流名称：demo_daily_check
4. 从工具栏拖拽以下节点到画布：

   [Shell: 检查 HDFS] → [SQL: 查询文章数] → [Shell: 发送报告]

5. 配置节点：

   节点 1 - Shell: 检查 HDFS
   脚本：
   ```
   echo "检查 HDFS 中的业务数据..."
   hadoop fs -ls /user/hive/warehouse/demo_access/
   hadoop fs -count /user/hive/warehouse/demo_access/
   ```

   节点 2 - SQL: 查询文章数
   SQL：
   ```sql
   SELECT COUNT(*) AS total_posts
   FROM posts;
   ```

   节点 3 - Shell: 发送报告
   脚本：
   ```
   echo "业务数据检查完成！日期: ${system.biz.date}"
   ```

6. 连线：节点 1 → 节点 2 → 节点 3
7. 保存 → 上线 → 运行
8. 观察执行结果
```
