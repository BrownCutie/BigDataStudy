# 01 - 任务调度概述：让数据流水线自动跑起来

## 一、为什么需要任务调度？

假设你的数据平台已经上线了，每天都会产生大量的数据：

```
┌─────────────────────────────────────────────────────────────┐
│                   数据平台每日数据流水线                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  00:00  日志采集：从 Nginx 收集当天访问日志 → 写入 HDFS       │
│  01:00  数据清洗：清洗日志数据，去除爬虫流量                   │
│  02:00  数仓同步：将 MySQL 业务数据同步到 Hive                │
│  03:00  指标计算：计算日活用户、热门文章、搜索热词              │
│  04:00  结果导出：将计算结果写入 Elasticsearch                │
│  05:00  报表生成：生成数据质量报告，发送邮件通知               │
│                                                             │
│  ⚠️  问题来了：谁来定时触发这些任务？                          │
│     如果中间某个任务失败了怎么办？                             │
│     任务之间有依赖关系，怎么保证执行顺序？                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.1 用 cron 能搞定吗？

你可能会想：Linux 的 crontab 不就能定时执行任务吗？

```bash
# crontab -e
0 0 * * * /opt/scripts/collect_logs.sh
0 1 * * * /opt/scripts/clean_logs.sh
0 2 * * * /opt/scripts/sync_to_hive.sh
0 3 * * * /opt/scripts/calc_metrics.sh
0 4 * * * /opt/scripts/export_to_es.sh
0 5 * * * /opt/scripts/send_report.sh
```

看起来可以工作，但很快你就会遇到一堆问题：

```
┌──────────────────────────────────────────────────────────────┐
│                  cron 调度的致命缺陷                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 没有依赖管理                                              │
│     如果"数据清洗"失败了，"指标计算"还会在 2 点准时跑           │
│     → 结果就是用脏数据算出了错误的报表                         │
│                                                              │
│  2. 没有重试机制                                              │
│     任务因为网络抖动失败了，就只能手动重新执行                   │
│     → 半夜 3 点被电话叫起来手动跑任务，谁受得了？               │
│                                                              │
│  3. 没有监控告警                                              │
│     任务失败了你不知道，直到业务方来催报表                      │
│     → "昨天的数据怎么没更新？"                                 │
│                                                              │
│  4. 没有执行记录                                              │
│     任务跑了什么、跑了多久、有没有报错，全靠翻日志               │
│     → 排查问题像大海捞针                                      │
│                                                              │
│  5. 没有可视化界面                                            │
│     所有任务关系都在脑子里，新人接手一脸懵                      │
│     → "这 50 个 cron 任务之间到底是什么关系？"                 │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 2.1 DolphinScheduler vs Airflow

这两个是最常被拿来比较的，我们来详细说说：

```
┌────────────────────────────────────────────────────────────────┐
│              DolphinScheduler vs Airflow 详细对比               │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  Airflow 的 DAG 定义方式（Python 代码）：                       │
│                                                                │
│  with DAG('data_pipeline', schedule_interval='0 2 * * *'):     │
│      collect = BashOperator(task_id='collect', ...)            │
│      clean   = BashOperator(task_id='clean', ...)              │
│      sync    = HiveOperator(task_id='sync', ...)               │
│      calc    = SparkSubmitOperator(task_id='calc', ...)        │
│                                                                │
│      collect >> clean >> sync >> calc                           │
│                                                                │
│  优点：版本控制友好、灵活性高、适合工程师团队                     │
│  缺点：需要写代码、非技术人员无法操作、代码和配置耦合             │
│                                                                │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  DolphinScheduler 的 DAG 定义方式（Web UI 拖拽）：               │
│                                                                │
│  在浏览器中：                                                   │
│  1. 拖拽一个 Shell 节点到画布                                    │
│  2. 配置脚本内容                                                │
│  3. 再拖一个 SQL 节点                                           │
│  4. 用鼠标连线表示依赖                                          │
│  5. 点击"保存并上线"                                            │
│                                                                │
│  优点：操作简单、可视化直观、非技术人员也能配置                    │
│  缺点：DAG 复杂时代码化管理不如 Airflow 方便                    │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

```
┌──────────────────────────────────────────────────────────┐

```

## 七、面试题

### Q1: DolphinScheduler 和 Airflow 怎么选？各自的优缺点是什么？

**参考答案：**

| 维度 | DolphinScheduler | Airflow |
|------|-----------------|---------|
| 编排方式 | Web UI 可视化拖拽，操作简单 | Python 代码定义，灵活但门槛高 |
| 架构 | 去中心化，Master/Worker 多节点 | 中心化，Scheduler 存在单点风险 |
| 多租户 | 原生支持，租户隔离 | 不原生支持，需要额外配置 |
| 任务类型 | 30+ 种原生支持 | 通过 Operator 扩展，生态丰富 |
| 版本管理 | Web UI 内置版本管理 | Git 管理代码，天然支持 |
| 学习曲线 | 低，非技术人员也能用 | 中高，需要 Python 基础 |
| 社区 | 中文社区活跃 | 国际社区活跃，资源更多 |

**选型建议：**
- 团队有非技术人员、需要多租户隔离 → 选 DolphinScheduler
- 团队以 Python 工程师为主、需要 Git 版本控制 → 选 Airflow

### Q2: 什么是去中心化架构？DolphinScheduler 是怎么实现的？

**参考答案：**

去中心化架构是指系统中没有单点故障（SPOF），任何节点挂掉都不会影响整体服务。

DolphinScheduler 通过以下方式实现：
1. **Master 多节点**：多个 Master 互相协调，通过 ZooKeeper 进行分布式锁和故障转移
2. **Worker 多节点**：任务分发到不同的 Worker 执行，负载均衡
3. **无状态设计**：Master 和 Worker 都是无状态的，元数据存在数据库中
4. **故障自愈**：Master 挂了，其他 Master 自动接管；Worker 挂了，任务自动分配到其他 Worker

### Q3: DolphinScheduler 支持哪些任务类型？

**参考答案：**

DolphinScheduler 3.4.1 支持 30+ 种任务类型，常用的包括：
- **Shell**：执行 Shell 脚本
- **SQL**：执行 Hive/MySQL/Spark SQL
- **Spark**：提交 Spark 作业（on YARN/K8s/Standalone）
- **Flink**：提交 Flink 作业
- **Python**：执行 Python 脚本
- **HTTP**：调用 HTTP 接口
- **Sub-Workflow**：子工作流，实现工作流复用
- **Conditions/Switch**：条件分支，实现 if-else 逻辑
- **Dependent**：跨工作流依赖
- **Procedure**：存储过程调用
- **DataX/ Sqoop**：数据同步
