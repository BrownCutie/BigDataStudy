# 03 - 索引与映射：设计你的数据结构

## 一、索引（Index）的概念

在 Elasticsearch 中，**索引**是最顶层的逻辑容器，你可以把它类比成关系型数据库中的"数据库"或"表"。

```
┌──────────────────────────────────────────────────────────────┐
│              Elasticsearch vs MySQL 概念映射                   │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  MySQL                Elasticsearch                          │
│  ──────────────       ──────────────                         │
│  Database         →   Index（索引）                          │
│  Table             →   Index（索引）或 Type（已废弃）          │
│  Row（行）         →   Document（文档）                       │
│  Column（列）      →   Field（字段）                          │
│  Schema            →   Mapping（映射）                       │
│  SQL               →   Query DSL（JSON）                     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

> 注意：在 ES 7.x 之前，一个 Index 可以有多个 Type，类似 MySQL 的一个 Database 有多个 Table。但从 ES 7.x 开始 Type 被废弃，ES 8.x+ 已完全移除。现在一个 Index 就相当于一张表。

### 1.1 索引的物理结构

一个索引在物理上被分成多个**分片（Shard）**，每个分片是一个独立的 Lucene 索引：

```
┌──────────────────────────────────────────────────────────────────┐
│                    索引的物理结构                                  │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  逻辑视角：                                                       │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │                    demo 索引                               │ │
│  │                                                            │ │
│  │  文档1  文档2  文档3  文档4  文档5  文档6  ...              │ │
│  │                                                            │ │
│  └────────────────────────────────────────────────────────────┘ │
│                                                                  │
│  物理视角（假设 2 主分片 + 1 副本）：                              │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │  demo 索引                                                 │ │
│  │  ┌──────────────────┐  ┌──────────────────┐               │ │
│  │  │  Shard 0 (Primary)│  │ Shard 1 (Primary)│              │ │
│  │  │  文档1, 文档3,    │  │ 文档2, 文档4,    │              │ │
│  │  │  文档5            │  │ 文档6            │              │ │
│  │  └──────────────────┘  └──────────────────┘               │ │
│  │  ┌──────────────────┐  ┌──────────────────┐               │ │
│  │  │Shard 0 (Replica) │  │Shard 1 (Replica) │              │ │
│  │  │  文档1, 文档3,    │  │ 文档2, 文档4,    │              │ │
│  │  │  文档5            │  │ 文档6            │              │ │
│  │  └──────────────────┘  └──────────────────┘               │ │
│  └────────────────────────────────────────────────────────────┘ │
│                                                                  │
│  文档分配到哪个分片？由路由公式决定：                               │
│  shard_num = hash(_routing) % num_primary_shards               │
│  默认 _routing = _id                                            │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 二、Mapping（映射）详解

Mapping 定义了索引中文档的字段类型和属性，类似于 MySQL 的表结构（Schema）。

### 2.1 字段类型

Elasticsearch 支持丰富的字段类型，下面列出最常用的：

```
┌──────────────────────────────────────────────────────────────┐
│                Elasticsearch 常用字段类型                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌────────────────────────────────────────────────────────┐ │
│  │ 字符串类型                                             │ │
│  │  text     → 全文搜索，会分词（如文章内容）               │ │
│  │  keyword  → 精确匹配，不分词（如标签、作者名）          │ │
│  └────────────────────────────────────────────────────────┘ │
│  ┌────────────────────────────────────────────────────────┐ │
│  │ 数值类型                                               │ │
│  │  integer  → 整数（如浏览量、点赞数）                    │ │
│  │  long     → 长整数（如时间戳）                          │ │
│  │  float    → 浮点数                                     │ │
│  │  double   → 双精度浮点数                                │ │
│  │  scaled_float → 缩放浮点数（省空间）                    │ │
│  └────────────────────────────────────────────────────────┘ │
│  ┌────────────────────────────────────────────────────────┐ │
│  │ 日期类型                                               │ │
│  │  date     → 日期时间（如发布时间、更新时间）            │ │
│  │            支持多种格式：yyyy-MM-dd HH:mm:ss            │ │
│  │            或 epoch_millis（毫秒时间戳）                 │ │
│  └────────────────────────────────────────────────────────┘ │
│  ┌────────────────────────────────────────────────────────┐ │
│  │ 布尔类型                                               │ │
│  │  boolean  → true / false（如是否置顶）                  │ │
│  └────────────────────────────────────────────────────────┘ │
│  ┌────────────────────────────────────────────────────────┐ │
│  │ 嵌套类型                                               │ │
│  │  nested   → 嵌套对象数组（如评论列表）                  │ │
│  │  object   → JSON 对象（如作者信息）                     │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 text vs keyword 的区别（重要！）

这是 Elasticsearch 中**最容易混淆**的概念，也是面试中的高频考点。

```
┌──────────────────────────────────────────────────────────────────┐
│                  text vs keyword 的区别                           │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  text（全文搜索类型）                                              │
│  ─────────────────                                               │
│  - 写入时会被分词                                                │
│  - 用于全文搜索（match、multi_match）                             │
│  - 不支持精确匹配（term 查询可能不按预期工作）                     │
│  - 不能用于排序、聚合                                             │
│  - 占用空间较大（倒排索引）                                       │
│                                                                  │
│  keyword（精确匹配类型）                                          │
│  ─────────────────                                               │
│  - 写入时不会分词，整值作为一个词项                               │
│  - 用于精确匹配（term、terms 查询）                               │
│  - 支持排序、聚合、范围查询                                       │
│  - 占用空间较小（doc values）                                     │
│  - 适合枚举值、ID、标签等                                         │
│                                                                  │
│  举例：字段值 "Elasticsearch 入门教程"                             │
│                                                                  │
│  text 分词后：["elasticsearch", "入门", "教程"]                   │
│    → 搜 "入门" 能匹配 ✓                                          │
│    → 搜 "Elasticsearch入门" 也能匹配 ✓                            │
│    → 但 term 查询 "Elasticsearch 入门教程" 不匹配 ✗              │
│      （因为分词后没有这个词项了）                                   │
│                                                                  │
│  keyword 不分词："elasticsearch 入门教程"（完整保留）              │
│    → term 查询 "Elasticsearch 入门教程" 能匹配 ✓                  │
│    → term 查询 "elasticsearch" 不能匹配 ✗                        │
│    → 可以排序、聚合 ✓                                             │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**一个字段同时需要全文搜索和精确匹配怎么办？**

使用 `fields` 参数，对同一个字段定义多种类型：

```json
{
  "title": {
    "type": "text",
    "analyzer": "ik_max_word",
    "fields": {
      "keyword": {
        "type": "keyword"
      }
    }
  }
}
```

这样：
- `title` 字段用于全文搜索（分词）
- `title.keyword` 字段用于精确匹配、排序、聚合（不分词）

```bash
# 全文搜索
curl -X GET "http://localhost:9200/data/_search" -H 'Content-Type: application/json' -d '{
  "query": { "match": { "title": "大数据入门" } }
}'

# 精确匹配（注意用 title.keyword）
curl -X GET "http://localhost:9200/data/_search" -H 'Content-Type: application/json' -d '{
  "query": { "term": { "title.keyword": "大数据入门教程" } }
}'

# 排序（必须用 keyword）
curl -X GET "http://localhost:9200/data/_search" -H 'Content-Type: application/json' -d '{
  "query": { "match_all": {} },
  "sort": [{ "title.keyword": "asc" }]
}'
```

### 2.3 动态映射 vs 显式映射

#### 动态映射（Dynamic Mapping）

Elasticsearch 默认会自动猜测字段类型并创建映射：

```bash
# 直接写入文档，ES 自动推断字段类型
curl -X POST "http://localhost:9200/data/_doc/1" -H 'Content-Type: application/json' -d '{
  "title": "Kafka 入门教程",
  "views": 100,
  "is_published": true,
  "created_at": "2026-04-15"
}'
```

ES 的自动类型推断规则：

| JSON 类型 | ES 推断的类型 |
|-----------|--------------|
| `null` | 不映射 |
| `true` / `false` | `boolean` |
| 整数 | `long` |
| 浮点数 | `double` |
| 字符串（匹配日期格式） | `date` |
| 字符串（匹配数字格式） | `float` 或 `long` |
| 字符串（其他） | `text`，并自动添加 `keyword` 子字段 |
| 对象 | `object` |
| 数组 | 取决于第一个非 null 元素的类型 |

```
┌──────────────────────────────────────────────────────────────┐
│                动态映射的风险                                   │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 类型推断可能不符合预期                                     │
│     "123" 会被推断为 text，但你可能需要 long                   │
│                                                              │
│  2. 字段一旦创建，类型就不能修改                               │
│     如果推断错了，需要删除索引重建                             │
│                                                              │
│  3. 自动添加的字段可能包含不必要的倒排索引                     │
│     浪费存储空间和内存                                         │
│                                                              │
│  结论：生产环境强烈建议使用显式映射！                           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

#### 显式映射（Explicit Mapping）

在创建索引时预先定义好字段类型：

```bash
curl -X PUT "http://localhost:9200/demo" -H 'Content-Type: application/json' -d '{
  "settings": {
    "number_of_shards": 1,
    "number_of_replicas": 0
  },
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "analyzer": "ik_max_word",
        "fields": {
          "keyword": { "type": "keyword" }
        }
      },
      "content": {
        "type": "text",
        "analyzer": "ik_max_word"
      },
      "tags": {
        "type": "keyword"
      },
      "author": {
        "type": "keyword"
      },
      "created_at": {
        "type": "date",
        "format": "yyyy-MM-dd HH:mm:ss||yyyy-MM-dd||epoch_millis"
      },
      "updated_at": {
        "type": "date",
        "format": "yyyy-MM-dd HH:mm:ss||yyyy-MM-dd||epoch_millis"
      },
      "views": {
        "type": "integer"
      },
      "likes": {
        "type": "integer"
      },
      "is_published": {
        "type": "boolean"
      },
      "category": {
        "type": "keyword"
      }
    }
  }
}'
```

#### 控制动态映射的行为

如果不想完全禁用动态映射，但想更严格地控制：

```json
{
  "mappings": {
    "dynamic": "strict",
    "properties": { ... }
  }
}
```

| `dynamic` 值 | 行为 |
|-------------|------|
| `true`（默认） | 自动添加新字段 |
| `false` | 忽略新字段（不索引，但存储在 `_source` 中） |
| `strict` | 遇到新字段直接报错（最严格） |

**推荐：生产环境使用 `strict` 模式**，防止因字段名拼写错误而创建错误的映射。

## 三、索引管理

### 3.1 创建索引

```bash
# 创建索引（同时定义设置和映射）
curl -X PUT "http://localhost:9200/demo" -H 'Content-Type: application/json' -d '{
  "settings": {
    "number_of_shards": 1,
    "number_of_replicas": 0
  },
  "mappings": {
    "properties": {
      "title": { "type": "text", "analyzer": "ik_max_word" },
      "content": { "type": "text", "analyzer": "ik_max_word" }
    }
  }
}'
```

### 3.2 查看索引

```bash
# 查看索引基本信息
curl http://localhost:9200/demo?pretty

# 查看 Mapping
curl http://localhost:9200/data/_mapping?pretty

# 查看 Settings
curl http://localhost:9200/data/_settings?pretty
```

### 3.3 修改索引设置

```bash
# 修改副本数
curl -X PUT "http://localhost:9200/data/_settings" -H 'Content-Type: application/json' -d '{
  "index": {
    "number_of_replicas": 1
  }
}'

# 修改刷新间隔（提高写入性能）
curl -X PUT "http://localhost:9200/data/_settings" -H 'Content-Type: application/json' -d '{
  "index": {
    "refresh_interval": "30s"
  }
}'
```

### 3.4 删除索引

```bash
# 删除索引（数据会全部丢失！）
curl -X DELETE "http://localhost:9200/demo"

# 删除多个索引
curl -X DELETE "http://localhost:9200/demo,demo_v2"

# 删除所有索引（危险操作！）
curl -X DELETE "http://localhost:9200/_all"

# 删除匹配模式的索引
curl -X DELETE "http://localhost:9200/demo*"
```

### 3.5 打开/关闭索引

关闭的索引不占用内存，但无法读写：

```bash
# 关闭索引（释放内存）
curl -X POST "http://localhost:9200/data/_close"

# 打开索引
curl -X POST "http://localhost:9200/data/_open"
```

## 四、分片与副本（Shard & Replica）

### 4.1 分片（Shard）

```
┌──────────────────────────────────────────────────────────────────┐
│                    分片的作用                                      │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  为什么需要分片？                                                 │
│                                                                  │
│  1. 水平扩展                                                     │
│     一个索引的数据量太大，一台机器存不下                           │
│     → 拆成多个分片，分布到多台机器上                               │
│                                                                  │
│  2. 并行处理                                                     │
│     搜索请求可以同时发到多个分片并行执行                           │
│     → 提高搜索吞吐量                                             │
│                                                                  │
│  3. 数据隔离                                                     │
│     一个分片损坏不影响其他分片                                     │
│     → 提高可用性                                                 │
│                                                                  │
│  分片数量：                                                       │
│  - 主分片数在创建索引时设定，创建后不能修改                        │
│  - 副本数可以随时修改                                            │
│  - 主分片数建议根据数据量来定                                     │
│                                                                  │
│  经验法则：                                                       │
│  - 单个分片大小建议 10-50GB                                      │
│  - 如果索引预计 100GB，设 2-5 个主分片                            │
│  - 分片太多会增加管理开销（每个分片都有开销）                      │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 4.2 副本（Replica）

```
┌──────────────────────────────────────────────────────────────────┐
│                    副本的作用                                      │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  副本是主分片的完整复制，有两个作用：                               │
│                                                                  │
│  1. 高可用                                                       │
│     主分片所在节点故障时，副本可以提升为主分片                     │
│     → 数据不丢失，服务不中断                                      │
│                                                                  │
│  2. 负载均衡                                                     │
│     搜索请求可以分发到主分片或副本分片                             │
│     → 提高搜索吞吐量                                             │
│                                                                  │
│  ┌─ Node 1 ─────────────────┐  ┌─ Node 2 ─────────────────┐    │
│  │  Shard 0 (Primary)  ← 读写 │  │  Shard 0 (Replica)  ← 只读 │    │
│  │  Shard 1 (Replica)  ← 只读 │  │  Shard 1 (Primary)  ← 读写 │    │
│  └───────────────────────────┘  └───────────────────────────┘    │
│                                                                  │
│  注意：                                                           │
│  - 写操作（创建/更新/删除）先到主分片，再同步到副本                │
│  - 读操作可以打到主分片或副本                                     │
│  - 同一节点上不能有同一个分片的主和副本                            │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 4.3 分片数设置建议

```
┌──────────────────────────────────────────────────────────────┐
│              分片数设置建议                                     │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  数据量          建议主分片数                                   │
│  ────────────────────────────                               │
│  < 10GB         1                                            │
│  10-50GB        1-2                                          │
│  50-200GB       2-4                                          │
│  200GB-1TB      4-8                                          │
│  > 1TB          8-16（或考虑拆分成多个索引）                   │
│                                                              │
│  黄金法则：                                                     │
│  1. 单个分片大小控制在 10-50GB                                 │
│  2. 分片数不要太多（每个分片都有内存和 CPU 开销）               │
│  3. 主分片数创建后不能修改，请提前规划好                        │
│  4. 副本数可以随时修改                                        │
│  5. 单节点开发环境设 1 主分片 0 副本即可                       │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 五、实战：设计博客文章索引的 Mapping

现在让我们为数据分析项目设计一个完整的索引 Mapping。

### 5.1 需求分析

博客文章需要存储和搜索的字段：

```
┌──────────────────────────────────────────────────────────────┐
│                博客文章字段分析                                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  字段            用途              推荐类型                    │
│  ───────────────────────────────────────────                 │
│  title           标题，需全文搜索     text + keyword 子字段     │
│  content         内容，需全文搜索     text（IK 分词）           │
│  summary         摘要，需全文搜索     text（IK 分词）           │
│  tags            标签，精确匹配/聚合  keyword                   │
│  category        分类，精确匹配/聚合  keyword                   │
│  author          作者，精确匹配       keyword                   │
│  created_at      发布时间，排序/范围  date                      │
│  updated_at      更新时间，排序/范围  date                      │
│  views           浏览量，排序/聚合     integer                   │
│  likes           点赞数，排序/聚合     integer                   │
│  is_published    是否发布，过滤       boolean                   │
│  is_top          是否置顶，过滤/排序  boolean                   │
│  comments        评论数，排序/聚合     integer                   │
│  url             文章 URL，精确匹配  keyword                   │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 创建索引

```bash
curl -X PUT "http://localhost:9200/posts" -H 'Content-Type: application/json' -d '{
  "settings": {
    "number_of_shards": 1,
    "number_of_replicas": 0,
    "analysis": {
      "analyzer": {
        "ik_smart_analyzer": {
          "type": "custom",
          "tokenizer": "ik_smart"
        },
        "ik_max_word_analyzer": {
          "type": "custom",
          "tokenizer": "ik_max_word"
        }
      }
    }
  },
  "mappings": {
    "dynamic": "strict",
    "properties": {
      "title": {
        "type": "text",
        "analyzer": "ik_max_word",
        "search_analyzer": "ik_smart",
        "fields": {
          "keyword": {
            "type": "keyword",
            "ignore_above": 256
          }
        }
      },
      "content": {
        "type": "text",
        "analyzer": "ik_max_word",
        "search_analyzer": "ik_smart"
      },
      "summary": {
        "type": "text",
        "analyzer": "ik_max_word",
        "search_analyzer": "ik_smart"
      },
      "tags": {
        "type": "keyword"
      },
      "category": {
        "type": "keyword"
      },
      "author": {
        "type": "keyword"
      },
      "created_at": {
        "type": "date",
        "format": "yyyy-MM-dd HH:mm:ss||yyyy-MM-dd||epoch_millis"
      },
      "updated_at": {
        "type": "date",
        "format": "yyyy-MM-dd HH:mm:ss||yyyy-MM-dd||epoch_millis"
      },
      "views": {
        "type": "integer"
      },
      "likes": {
        "type": "integer"
      },
      "is_published": {
        "type": "boolean"
      },
      "is_top": {
        "type": "boolean"
      },
      "comments": {
        "type": "integer"
      },
      "url": {
        "type": "keyword"
      }
    }
  }
}'
```

```
┌──────────────────────────────────────────────────────────────┐
│                Mapping 设计要点                                 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. title 用 ik_max_word 分词（索引时细粒度），                 │
│     ik_smart 分词（搜索时粗粒度）                               │
│     → 索引时尽可能多切分，搜索时减少噪音                        │
│                                                              │
│  2. title 添加 keyword 子字段                                  │
│     → 同时支持全文搜索和精确匹配/排序                           │
│                                                              │
│  3. tags 和 category 用 keyword                               │
│     → 标签和分类是枚举值，不需要分词                            │
│     → 支持聚合统计（按标签统计文章数）                          │
│                                                              │
│  4. author 用 keyword                                        │
│     → 作者名不需要分词，精确匹配即可                           │
│                                                              │
│  5. dynamic: strict                                          │
│     → 严格模式，防止拼写错误的字段被自动创建                    │
│                                                              │
│  6. ignore_above: 256                                        │
│     → 超过 256 字符的 keyword 字段不索引，节省空间              │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 六、面试题

### 面试题 1：text 和 keyword 类型有什么区别？什么场景用什么类型？

**参考答案：**

**text 类型：**
- 写入时会经过分析器（Analyzer）处理，被分词成多个词项
- 用于全文搜索场景，如 match、multi_match 查询
- 不支持精确匹配（term 查询），不支持排序和聚合
- 适合文章内容、标题等需要全文搜索的字段

**keyword 类型：**
- 写入时不分词，整值作为一个词项存储
- 用于精确匹配场景，如 term、terms 查询
- 支持排序、聚合、范围查询
- 适合标签、分类、作者、状态等枚举值字段

**实际使用中的最佳实践：**
对同一个字段同时需要全文搜索和精确匹配/排序时，使用 `fields` 参数定义多字段：
```json
{
  "title": {
    "type": "text",
    "fields": {
      "keyword": { "type": "keyword" }
    }
  }
}
```

---

### 面试题 2：Elasticsearch 的主分片数为什么创建后不能修改？

**参考答案：**

因为文档的路由（分配到哪个分片）是基于 `hash(_routing) % num_primary_shards` 计算的。

如果修改了主分片数，路由公式就变了，所有文档的分配位置都会改变，等于需要重新索引所有数据。

副本分片不影响路由公式，所以可以随时修改。

**如果需要修改主分片数怎么办？**
1. 创建一个新索引，设置正确的主分片数
2. 使用 `_reindex` API 将旧索引的数据复制到新索引
3. 给新索引添加别名（让应用无感知切换）
4. 删除旧索引

```bash
# 重建索引示例
curl -X POST "http://localhost:9200/_reindex" -H 'Content-Type: application/json' -d '{
  "source": { "index": "posts_old" },
  "dest": { "index": "posts_new" }
}'
```

### 面试题 3：什么是 Mapping？动态映射有什么风险？

**参考答案：**

Mapping 是 Elasticsearch 中定义索引字段类型和属性的机制，类似于关系型数据库的 Schema。

**动态映射的风险：**
1. **类型推断不准确**：字符串 "123" 会被推断为 text，但你可能需要 long 类型
2. **字段一旦创建不能修改类型**：推断错误后需要删除索引重建
3. **自动添加的字段可能不需要索引**：浪费存储和内存
4. **可能的字段爆炸**：如果应用有 bug，不断写入新字段名，会创建大量无用字段

**建议：**
- 生产环境使用显式映射
- 设置 `dynamic: strict` 防止意外字段
- 在开发环境先用少量数据测试映射是否正确
