# 07 - 中文分词与性能优化：让搜索更精准

## 一、分析器（Analyzer）详解

分析器是 Elasticsearch 处理文本的核心组件。无论是写入文档时的索引阶段，还是搜索时的查询阶段，都需要经过分析器处理。

### 1.1 分析器的组成

```
┌──────────────────────────────────────────────────────────────────┐
│                    分析器的三大组件                                 │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  原始文本: "Kafka 是一个分布式<em>流处理</em>平台"                │
│      │                                                          │
│      ▼                                                          │
│  ┌─────────────────────────────────────┐                        │
│  │  1. Character Filters（字符过滤器）   │                        │
│  │     在分词之前对原始文本进行预处理    │                        │
│  │     - 去除 HTML 标签                 │                        │
│  │     - 转换特殊字符（& → and）         │                        │
│  │     - 可以链式配置多个                │                        │
│  └──────────────┬──────────────────────┘                        │
│                 │                                                │
│                 ▼                                                │
│  ┌─────────────────────────────────────┐                        │
│  │  2. Tokenizer（分词器）              │                        │
│  │     将文本切分成一个个词项（Token）   │                        │
│  │     ⚠️ 只能有一个 Tokenizer！        │                        │
│  │     - standard：按标点空格分词        │                        │
│  │     - ik_smart / ik_max_word：中文    │                        │
│  │     - whitespace：按空格分词          │                        │
│  └──────────────┬──────────────────────┘                        │
│                 │                                                │
│                 ▼                                                │
│  ┌─────────────────────────────────────┐                        │
│  │  3. Token Filters（词项过滤器）      │                        │
│  │     对分词后的词项进行后处理          │                        │
│  │     - 转小写（lowercase）            │                        │
│  │     - 去停用词（stop）              │                        │
│  │     - 同义词（synonym）             │                        │
│  │     - 可以链式配置多个                │                        │
│  └──────────────┬──────────────────────┘                        │
│                 │                                                │
│                 ▼                                                │
│  最终词项: ["kafka", "分布式", "流处理", "平台"]                  │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 二、内置分析器

Elasticsearch 自带的分析器：

| 分析器 | 说明 | 适用场景 |
|--------|------|---------|
| `standard` | 默认分析器，按 Unicode 字符分段，转小写 | 英文通用 |
| `simple` | 按非字母字符分词，转小写 | 简单英文 |
| `whitespace` | 按空格分词 | 空格分隔的语言 |
| `stop` | simple + 去除停用词 | 英文搜索 |
| `keyword` | 不分词，整值作为一个词项 | 精确匹配 |
| `pattern` | 用正则表达式分词 | 特殊格式文本 |
| `english` | 针对 English 语言的分词器 | 英文搜索 |
| `fingerprint` | fingerprint 算法，去重排序 | 指纹匹配 |

```bash
# 对比不同分析器的效果
curl -X POST "http://localhost:9200/_analyze?pretty" -H 'Content-Type: application/json' -d '{
  "analyzer": "whitespace",
  "text": "Kafka is a distributed streaming platform"
}'
# → [kafka, is, a, distributed, streaming, platform]

curl -X POST "http://localhost:9200/_analyze?pretty" -H 'Content-Type: application/json' -d '{
  "analyzer": "stop",
  "text": "Kafka is a distributed streaming platform"
}'
# → [kafka, distributed, streaming, platform]（去除了 is, a）
```


### 3.2 IK 的两种分词模式

```
┌──────────────────────────────────────────────────────────────────┐
│                  ik_smart vs ik_max_word                          │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ik_smart（粗粒度分词）                                           │
│  ────────────────────                                            │
│  尽可能切出最少的词，语义粒度大                                   │
│                                                                  │
│  "Elasticsearch是一个分布式搜索引擎"                              │
│  → [elasticsearch, 是, 一个, 分布式, 搜索引擎]                   │
│                                                                  │
│  适合：搜索时使用（减少噪音，提高搜索精度）                       │
│                                                                  │
│  ik_max_word（细粒度分词）                                        │
│  ────────────────────                                            │
│  尽可能切出最多的词，穷举所有可能的组合                            │
│                                                                  │
│  "Elasticsearch是一个分布式搜索引擎"                              │
│  → [elasticsearch, 是, 一个, 分布, 分布式, 式,                   │
│     搜索, 搜索引擎, 引擎]                                        │
│                                                                  │
│  适合：索引时使用（提高召回率，不漏掉任何匹配）                    │
│                                                                  │
│  最佳实践：                                                      │
│  索引时用 ik_max_word（尽可能多切分，增加匹配机会）               │
│  搜索时用 ik_smart（减少噪音，提高搜索精度）                      │
│  这就是 Mapping 中 analyzer 和 search_analyzer 的区别             │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

```bash
# 测试 ik_max_word
curl -X POST "http://localhost:9200/_analyze?pretty" -H 'Content-Type: application/json' -d '{
  "analyzer": "ik_max_word",
  "text": "Elasticsearch是一个分布式搜索引擎"
}'

# 测试 ik_smart
curl -X POST "http://localhost:9200/_analyze?pretty" -H 'Content-Type: application/json' -d '{
  "analyzer": "ik_smart",
  "text": "Elasticsearch是一个分布式搜索引擎"
}'
```

### 3.3 analyzer vs search_analyzer

```
┌──────────────────────────────────────────────────────────────┐
│        analyzer（索引时分析器） vs search_analyzer（搜索时）    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  analyzer（索引时）：                                          │
│    文档写入时使用的分析器                                       │
│    → 影响倒排索引中存储的词项                                   │
│    → 建议用 ik_max_word（细粒度，穷举所有组合）                 │
│                                                              │
│  search_analyzer（搜索时）：                                   │
│    用户搜索时对关键词使用的分析器                               │
│    → 影响搜索词的切分方式                                       │
│    → 建议用 ik_smart（粗粒度，减少噪音）                       │
│                                                              │
│  为什么索引时细、搜索时粗？                                     │
│                                                              │
│  文档： "分布式流处理平台"                                      │
│  索引用 ik_max_word → 倒排索引中存有：                        │
│    "分布"、"分布式"、"流处理"、"平台"                          │
│                                                              │
│  用户搜 "分布式"（ik_smart → "分布式"）                        │
│    → 倒排索引中有 "分布式" → 匹配 ✓                            │
│                                                              │
│  用户搜 "分布"（ik_smart → "分布"）                            │
│    → 倒排索引中有 "分布" → 匹配 ✓                              │
│    （因为索引时 ik_max_word 也切出了 "分布"）                   │
│                                                              │
│  如果索引用 ik_smart，搜索 "分布" 就匹配不到了                 │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 四、自定义分析器

除了使用内置分析器和 IK，还可以自定义分析器：

```bash
# 创建带自定义分析器的索引
curl -X PUT "http://localhost:9200/demo_custom" -H 'Content-Type: application/json' -d '{
  "settings": {
    "analysis": {
      "char_filter": {
        "html_strip_filter": {
          "type": "html_strip"
        }
      },
      "tokenizer": {
        "ik_max_word_tokenizer": {
          "type": "ik_max_word"
        }
      },
      "filter": {
        "my_stopwords": {
          "type": "stop",
          "stopwords": ["的", "了", "在", "是", "我", "有", "和", "就", "不", "人", "都", "一", "一个"]
        },
        "my_synonym": {
          "type": "synonym",
          "synonyms": [
            "大数据,big data",
            "搜索引擎,search engine",
            "消息队列,message queue"
          ]
        }
      },
      "analyzer": {
        "my_custom_analyzer": {
          "type": "custom",
          "char_filter": ["html_strip_filter"],
          "tokenizer": "ik_max_word_tokenizer",
          "filter": ["lowercase", "my_stopwords", "my_synonym"]
        }
      }
    }
  }
}'
```

```
┌──────────────────────────────────────────────────────────────┐
│              自定义分析器的组件                                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  char_filter（字符过滤器）：                                   │
│  - html_strip：去除 HTML 标签                                │
│  - mapping：字符映射（如 & → and）                            │
│  - pattern_replace：正则替换                                  │
│                                                              │
│  tokenizer（分词器）：                                         │
│  - standard、ik_smart、ik_max_word、whitespace 等              │
│  ⚠️ 每个分析器只能有一个 tokenizer                            │
│                                                              │
│  token_filter（词项过滤器）：                                  │
│  - lowercase：转小写                                         │
│  - stop：去除停用词                                           │
│  - synonym：同义词替换                                        │
│  - stemmer：词干提取（英文 running → run）                    │
│  - ngram：n-gram 切分（用于自动补全）                         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 六、查询性能优化

### 6.1 使用 filter context

在 bool 查询中，不需要参与评分的条件用 `filter`：

```bash
# 优化前（全用 must）
curl -X GET "http://localhost:9200/posts/_search" -H 'Content-Type: application/json' -d '{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "大数据" } },
        { "term": { "is_published": true } },
        { "range": { "views": { "gte": 100 } } }
      ]
    }
  }
}'

# 优化后（不需要评分的用 filter）
curl -X GET "http://localhost:9200/posts/_search" -H 'Content-Type: application/json' -d '{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "大数据" } }
      ],
      "filter": [
        { "term": { "is_published": true } },
        { "range": { "views": { "gte": 100 } } }
      ]
    }
  }
}'
```

```
┌──────────────────────────────────────────────────────────────┐
│              filter context 的优势                              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 不计算相关性得分                                           │
│     → 跳过 BM25 评分计算，减少 CPU 开销                       │
│                                                              │
│  2. 结果缓存（Bitset 缓存）                                   │
│     → filter 的结果会被缓存为 Bitset                          │
│     → 后续查询中如果 filter 条件相同，直接使用缓存             │
│     → 缓存是自动的，无需手动管理                               │
│                                                              │
│  3. 过滤速度更快                                               │
│     → filter 只做是/否判断，不需要复杂的评分计算               │
│                                                              │
│  经验法则：                                                   │
│  - is_xxx、status、category 等精确匹配 → filter               │
│  - 时间范围、数值范围 → filter                                 │
│  - 关键词搜索 → must                                          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 使用 routing 加速查询

如果查询总是按某个字段过滤（如 author），可以使用 routing 将相关数据放在同一个分片上：

```bash
# 写入时指定 routing
curl -X POST "http://localhost:9200/posts/_doc/1?routing=browncutie" -H 'Content-Type: application/json' -d '{
  "title": "Kafka 教程",
  "author": "browncutie",
  "content": "..."
}'

# 查询时指定 routing（只查目标分片，跳过其他分片）
curl -X GET "http://localhost:9200/posts/_search?routing=browncutie" -H 'Content-Type: application/json' -d '{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "Kafka" } }
      ],
      "filter": [
        { "term": { "author": "browncutie" } }
      ]
    }
  }
}'
```

```
┌──────────────────────────────────────────────────────────────┐
│              routing 的原理                                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  默认路由公式：                                               │
│  shard_num = hash(_id) % num_shards                          │
│  → 文档随机分布到各个分片                                     │
│                                                              │
│  自定义路由：                                                 │
│  shard_num = hash(routing) % num_shards                      │
│  → 相同 routing 值的文档在同一个分片                          │
│                                                              │
│  不使用 routing 的查询：                                      │
│  查询 "author=browncutie" → 所有分片都要查一遍                 │
│                                                              │
│  使用 routing 的查询：                                        │
│  查询 "author=browncutie" + routing=browncutie                │
│  → 只查 browncutie 所在的分片，跳过其他分片                   │
│  → 查询速度提升 N 倍（N = 分片数）                            │
│                                                              │
│  注意：routing 可能导致数据倾斜                                │
│  如果某个 routing 值对应的数据量特别大，                      │
│  会导致某个分片过大                                           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 6.3 其他优化技巧

```bash
# 1. 使用 _source 过滤，只返回需要的字段
curl -X GET "http://localhost:9200/posts/_search" -H 'Content-Type: application/json' -d '{
  "_source": ["title", "summary", "url"],
  "query": { "match": { "title": "大数据" } }
}'

# 2. 使用 filter 代替 must 做精确匹配
# 3. 避免使用 wildcard、regexp 等性能差的查询
# 4. 合理设置分片数（不要太多，也不要太少）
# 5. 使用 _seq_no + _id 排序代替 _score 排序（不需要评分时）
# 6. 使用 track_total_hits: false（不需要精确总数时）
curl -X GET "http://localhost:9200/posts/_search" -H 'Content-Type: application/json' -d '{
  "track_total_hits": false,
  "query": { "match": { "title": "大数据" } }
}'
```

## 七、实战：为数据搜索配置中文分词

### 7.1 创建优化后的索引

```bash
# 先删除旧索引（如果存在）
```

### 7.3 Python 完整搜索实现

```python
"""
数据搜索（含中文分词）
"""
from elasticsearch import Elasticsearch

es = Elasticsearch("http://localhost:9200")

def demo_search(keyword, page=1, size=10, category=None):
    """数据搜索"""
    query = {
        "bool": {
            "must": [
                {
                    "multi_match": {
                        "query": keyword,
                        "fields": ["title^3", "summary^2", "content"],
                        "type": "best_fields",
                        "minimum_should_match": "30%"
                    }
                }
            ],
            "filter": [{"term": {"is_published": True}}]
        }
    }

    if category:
        query["bool"]["filter"].append({"term": {"category": category}})

    result = es.search(
        index="posts",
        query=query,
        sort=[{"is_top": {"order": "desc"}}, "_score"],
        from_=(page - 1) * size,
        size=size,
        highlight={
            "pre_tags": ["<strong>"],
            "post_tags": ["</strong>"],
            "fields": {
                "title": {"fragment_size": 0},
                "summary": {"fragment_size": 200, "number_of_fragments": 1}
            }
        },
        source=["title", "summary", "tags", "category", "author",
                "created_at", "views", "likes", "url"]
    )

    hits = []
    for hit in result["hits"]["hits"]:
        doc = {**hit["_source"], "score": hit["_score"]}
        if "highlight" in hit:
            doc["highlight"] = hit["highlight"]
        hits.append(doc)

    return {
        "total": result["hits"]["total"]["value"],
        "page": page,
        "size": size,
        "hits": hits
    }


# 测试
result = demo_search("分布式流处理")
for post in result["hits"]:
    title = post.get("highlight", {}).get("title", [post["title"]])[0]
    print(f"[{post['score']:.2f}] {title}")
    print(f"  URL: {post['url']} | 浏览量: {post['views']}")

es.close()
```

## 八、面试题

### 面试题 1：Elasticsearch 中文分词有哪些方案？IK 分词器的两种模式有什么区别？

**参考答案：**

常见中文分词方案：
1. **IK Analysis**：最流行的中文分词插件，支持自定义词典
2. **jieba**：Python 生态常用的分词器，ES 有社区插件
3. **HanLP**：支持多种分词模式，功能丰富
4. **THULAC**：清华大学开发的中文分词工具

IK 分词器的两种模式：

**ik_smart（粗粒度）：**
- 尽可能切出最少的词
- 例："分布式搜索引擎" → [分布式, 搜索引擎]
- 适合搜索时使用，减少噪音

**ik_max_word（细粒度）：**
- 尽可能穷举所有可能的词
- 例："分布式搜索引擎" → [分布, 分布式, 式, 搜索, 搜索引擎, 引擎]
- 适合索引时使用，提高召回率

最佳实践：索引时用 ik_max_word，搜索时用 ik_smart。

---

### 面试题 2：如何优化 Elasticsearch 的查询性能？

**参考答案：**

1. **使用 filter context**：不需要评分的条件用 filter，结果会被缓存，性能更好

2. **使用 routing**：如果查询总是按某个字段过滤，用 routing 让相关数据在同一分片，查询时只打目标分片

3. **合理设计 Mapping**：
   - 不需要全文搜索的字段用 keyword 或数值类型
   - 不需要索引的字段设置 `"index": false`
   - 不需要 `_source` 的场景可以禁用

4. **控制分片数**：分片不是越多越好，每个分片都有内存和 CPU 开销

5. **使用 _source 过滤**：只返回需要的字段，减少网络传输

6. **避免通配符查询**：wildcard、regexp 等查询性能差，考虑用前缀查询或 n-gram

7. **使用 track_total_hits: false**：不需要精确总数时关闭，减少精确计数开销

8. **预热缓存**：对于高频查询，可以先执行一次查询预热 filter 缓存

9. **硬件层面**：足够的堆内存（但不超过 31GB）、SSD 磁盘、足够的 OS Cache

### 面试题 3：什么是 refresh 机制？对写入和搜索有什么影响？

**参考答案：**

**refresh 机制**是 Elasticsearch 实现近实时搜索的核心：

1. 新写入的文档先存放在内存中的**索引缓冲区（Index Buffer）**
2. 每隔 `refresh_interval`（默认 1 秒），缓冲区的文档被刷新为一个不可变的**段（Segment）**
3. 只有刷新后的段才能被搜索到
4. 刷新不涉及磁盘 I/O（段先写入 OS 文件系统缓存），所以很快

**对写入的影响：**
- refresh 越频繁，写入的文档越快可搜索
- 但每次 refresh 都创建新段，段太多会影响搜索性能
- 批量写入时可以增大 refresh_interval 或设为 -1

**对搜索的影响：**
- 搜索只能看到已 refresh 的数据，最多有 1 秒延迟
- 这是"近实时"而非"实时"的原因

**相关概念：**
- **translog**：事务日志，用于故障恢复。每次 refresh 后 translog 可以清空
- **flush**：将段从 OS 缓存刷到磁盘，同时清空 translog。较慢但保证持久化
