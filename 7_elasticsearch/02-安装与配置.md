# 02 - 安装与配置：从零搭建 Elasticsearch

## 二、下载与安装

### 2.1 下载

Elasticsearch 不在 Apache 镜像站，需要从官方下载。好消息是 ES 原生支持 Apple Silicon：

```bash
cd ~/Programme/claude-study
curl -O https://artifacts.elastic.co/downloads/elasticsearch/elasticsearch-9.3.3-darwin-aarch64.tar.gz
```

### 2.2 解压安装

```bash
mkdir -p /opt/elasticsearch
tar -xzf elasticsearch-9.3.3-darwin-aarch64.tar.gz -C /opt/elasticsearch/
mv /opt/elasticsearch/elasticsearch-9.3.3 /opt/elasticsearch/9.3.3
```


### 2.3 配置环境变量

在 `~/.zshrc` 中添加：

```bash
export ES_HOME=/opt/elasticsearch/9.3.3
export PATH=$PATH:$ES_HOME/bin
```

生效：

```bash
source ~/.zshrc
```

## 三、核心配置

配置文件：`/opt/elasticsearch/9.3.3/config/elasticsearch.yml`

### 3.1 基础配置

```yaml
# ======================== Elasticsearch 配置 ========================

# 集群名称（同一集群的所有节点必须一致）
cluster.name: demo-search-cluster

# 节点名称（每个节点唯一，用于标识）
node.name: BrownCuiedeMini

# 数据和日志存储路径
path.data: /opt/elasticsearch/9.3.3/data
path.logs: /opt/elasticsearch/9.3.3/logs

# 网络配置
network.host: 127.0.0.1
http.port: 9200
transport.port: 9300

# 单节点模式（开发环境）
discovery.type: single-node

# 关闭安全认证（开发环境方便使用）
xpack.security.enabled: false

# 关闭机器学习功能（节省资源）
xpack.ml.enabled: false
```

### 3.2 JVM 调优

JVM 配置文件：`/opt/elasticsearch/9.3.3/config/jvm.options`

Elasticsearch 基于 JVM 运行，堆内存的设置非常关键：

```bash
# JVM 堆内存设置（在 jvm.options 中）
# 规则：设置为物理内存的 50%，且不超过 31GB
# 开发环境 16GB 内存 → 设置 4GB 或 8GB
-Xms4g
-Xmx4g
```

```
┌──────────────────────────────────────────────────────────────────┐
│                    JVM 堆内存设置原则                              │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. Xms 和 Xmx 必须设置成一样大                                   │
│     避免运行时 JVM 动态调整堆大小带来的性能开销                    │
│                                                                  │
│  2. 不要超过物理内存的 50%                                        │
│     ES 大量使用操作系统的文件系统缓存（Lucene 段文件）              │
│     剩下的内存留给 OS Cache                                       │
│                                                                  │
│  3. 不要超过 31GB                                                │
│     超过 31GB 后 JVM 会使用压缩指针（Compressed Oops）             │
│     但如果超过 32GB，压缩指针会失效，反而浪费内存                   │
│                                                                  │
│  4. 本机开发环境示例（16GB 内存）                                  │
│     -Xms4g -Xmx4g  （ES 用 4GB，OS Cache 用 12GB）              │
│                                                                  │
│  内存分配示意：                                                    │
│  ┌─────────────────────────────────────────┐                    │
│  │  物理内存 16GB                           │                    │
│  │  ┌──────────┬──────────────────────────┐│                    │
│  │  │ JVM 堆   │     OS Cache             ││                    │
│  │  │  4GB     │     12GB                 ││                    │
│  │  │ (Xms=Xmx)│  (Lucene 段文件缓存)      ││                    │
│  │  └──────────┴──────────────────────────┘│                    │
│  └─────────────────────────────────────────┘                    │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

> **重要：** 生产环境的 JVM 调优是一个复杂话题，这里只介绍基本原则。详细的生产级调优见 [08-生产与面试.md](08-生产与面试.md)。

## 四、单节点 vs 集群模式

### 4.1 单节点模式

我们当前使用的是单节点模式，适合学习和开发：

```yaml
# 单节点模式配置
discovery.type: single-node
```

```
┌──────────────────────────────────────────────────────────────┐
│                    单节点模式                                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌────────────────────────────────────────────────────┐     │
│  │              单个 Elasticsearch 节点                 │     │
│  │                                                    │     │
│  │  Master eligible + Data + Coordinating             │     │
│  │  （身兼数职，所有角色都在一个进程里）                 │     │
│  │                                                    │     │
│  │  索引分片数：1 主分片 + 1 副本                      │     │
│  │  → 但副本无法分配（只有一个节点），状态为 Yellow      │     │
│  │                                                    │     │
│  └────────────────────────────────────────────────────┘     │
│                                                              │
│  优点：配置简单，资源占用少，适合开发                            │
│  缺点：无高可用，无横向扩展                                    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 集群模式

生产环境使用集群模式，多个节点协同工作：

```
┌──────────────────────────────────────────────────────────────────┐
│                    集群模式（3 节点示例）                          │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│                    ┌──────────────┐                              │
│                    │ Master Node  │  ← 管理集群状态               │
│                    │  (节点 A)     │     分片分配                 │
│                    └──────┬───────┘     索引创建                  │
│                           │                                       │
│              ┌────────────┼────────────┐                        │
│              │            │            │                        │
```

## 六、REST API 基础

Elasticsearch 的所有操作都通过 REST API 完成，这就是它"任何语言都能用"的原因。

### 6.1 API 格式

```
┌──────────────────────────────────────────────────────────────┐
│                 ES REST API 格式                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  <HTTP 方法> <URL>                                          │
│                                                              │
│  URL 格式：http://host:port/<索引名>/<_type>/<_id>           │
│                                                              │
│  常用 HTTP 方法：                                             │
│  GET      → 读取（查询、获取文档）                             │
│  PUT      → 创建/替换（创建索引、全量更新文档）                  │
│  POST     → 创建/操作（创建文档、搜索、批量操作）                │
│  DELETE   → 删除（删除索引、删除文档）                          │
│  HEAD     → 检查是否存在                                       │
│                                                              │
│  请求体和响应体都是 JSON 格式                                  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 常用 API

```bash
# ========== 集群健康状态 ==========
curl http://localhost:9200/_cluster/health?pretty

# 返回示例：
# {
#   "cluster_name" : "demo-search-cluster",
#   "status" : "yellow",          ← green/yellow/red
#   "number_of_nodes" : 1,
#   "active_primary_shards" : 1,
#   "active_shards_percent_as_number" : 50.0
# }

# ========== 节点信息 ==========
curl http://localhost:9200/_cat/nodes?v

# ========== 索引列表 ==========
curl http://localhost:9200/_cat/indices?v

# ========== 别名列表 ==========
curl http://localhost:9200/_cat/aliases?v

# ========== 查看所有索引（另一种方式）==========
curl http://localhost:9200/_aliases?pretty
```

## 九、面试题

### 面试题 1：Elasticsearch 的 JVM 堆内存为什么建议不超过 31GB？

**参考答案：**

这与 JVM 的**压缩指针（Compressed Oops，Ordinary Object Pointers）**机制有关。

在 64 位 JVM 中，对象引用默认占用 8 字节。启用压缩指针后，JVM 可以将引用压缩到 4 字节，但这要求**堆内存不超过 32GB**。

具体来说：
- 堆内存 < 32GB 时：启用压缩指针，对象引用 4 字节，节省内存
- 堆内存 >= 32GB 时：压缩指针自动失效，对象引用回到 8 字节，实际可用内存反而可能更少

所以 31GB 是一个"安全上限"，既能充分利用压缩指针，又能获得较大的堆内存。

另外，ES 还大量依赖操作系统的文件系统缓存来缓存 Lucene 段文件。如果堆内存占太多，留给 OS Cache 的内存就少了，反而影响搜索性能。

---

### 面试题 2：单节点模式下集群状态为什么是 Yellow？

**参考答案：**

Elasticsearch 的索引数据被分成**主分片（Primary Shard）**和**副本分片（Replica Shard）**。

- **主分片**：数据的原始副本，负责读写
- **副本分片**：主分片的复制品，用于高可用和负载均衡

默认情况下，每个索引有 1 个主分片和 1 个副本分片。

在单节点模式下：
- 主分片可以正常分配到唯一的节点上 → 正常
- 副本分片不能分配到同一个节点上（ES 要求主分片和副本分片不能在同一节点） → 无法分配

所以集群状态是 Yellow（所有主分片可用，但副本分片未分配）。

**解决方案：**
- 开发环境：设置 `number_of_replicas: 0`
- 生产环境：增加节点

---

### 面试题 3：Elasticsearch 的近实时（Near Real-Time）是什么意思？为什么不是实时的？

**参考答案：**

Elasticsearch 的数据写入后，不会立刻被搜索到，默认有约 **1 秒**的延迟。这就是"近实时"。

原因在于 Lucene 的**段（Segment）机制**：
1. 新写入的文档先存放在内存中的**索引缓冲区（Index Buffer）**
2. 每隔 1 秒（默认 `refresh_interval`），缓冲区的文档被刷新（refresh）为一个新的**段**
3. 只有刷新后的段才能被搜索到

所以数据写入到可搜索之间，最多有 1 秒的延迟。

**为什么这样设计？**
- 段是不可变的，一旦写入就不能修改，这样读取时不需要加锁
- 每次刷新都创建新段，避免在内存中维护复杂的索引结构
- 如果需要更高的实时性，可以将 `refresh_interval` 调小（但会增加开销）

```bash
# 查看刷新间隔
curl http://localhost:9200/data/_settings?pretty

# 修改刷新间隔（例如设为 500ms）
curl -X PUT "http://localhost:9200/data/_settings" -H 'Content-Type: application/json' -d '{
  "index": {
    "refresh_interval": "500ms"
  }
}'
```
