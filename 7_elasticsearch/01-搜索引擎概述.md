# 01 - 搜索引擎概述：为什么我们需要 Elasticsearch？

## 一、为什么需要搜索引擎？

假设数据平台已经有了几百条博客文章，用户想在里面找到一条关于 "Kafka 消息队列" 的数据。你会怎么做？

最朴素的想法是用数据库的 `LIKE` 查询：

```sql
SELECT * FROM posts WHERE title LIKE '%Kafka%' OR content LIKE '%消息队列%';
```

这在数据量小的时候没问题，但当数据量增长到几十万、上百万条时，问题就来了。

### 1.1 数据库 LIKE 查询的瓶颈

```
┌──────────────────────────────────────────────────────────────┐
│                 数据库 LIKE 查询的执行过程                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  SELECT * FROM posts WHERE content LIKE '%Kafka%';          │
│                                                              │
│  数据库引擎内部：                                              │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐               │
│  │ 第 1 行   │ →  │ 匹配？    │ →  │ 是 → 保留 │              │
│  │ 第 2 行   │ →  │ 匹配？    │ →  │ 否 → 跳过 │              │
│  │ 第 3 行   │ →  │ 匹配？    │ →  │ 否 → 跳过 │              │
│  │ ...      │ →  │ ...      │ →  │ ...      │               │
│  │ 第 N 行   │ →  │ 匹配？    │ →  │ 是 → 保留 │              │
│  └──────────┘    └──────────┘    └──────────┘               │
│                                                              │
│  这叫"全表扫描"（Full Table Scan）                             │
│  N 越大，速度越慢！                                            │
│                                                              │
│  100 万行 × 平均 5KB/行 = 5GB 数据要逐一扫描                   │
│  → 耗时可能几十秒甚至更久                                      │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**LIKE 查询的致命缺陷：**

| 问题 | 说明 |
|------|------|
| **全表扫描** | `%关键词%` 无法使用 B+ 树索引，必须逐行扫描 |
| **速度慢** | 数据量越大越慢，线性增长 |
| **无相关性排序** | 找到的结果不能按相关度排序 |
| **不支持分词** | 搜 "消息队列" 匹配不到 "消息中间件" |
| **不支持模糊搜索** | 搜 "Kaffa"（拼写错误）找不到 "Kafka" |
| **不支持高亮** | 无法自动高亮匹配的关键词 |

### 1.2 搜索引擎的解决方案

搜索引擎的核心武器是**倒排索引（Inverted Index）**，它把 "逐行扫描" 变成了 "直接定位"，查询速度从秒级降到毫秒级。

```
┌──────────────────────────────────────────────────────────────┐
│               搜索引擎 vs 数据库查询对比                        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  数据库 LIKE：                                                │
│  "给我所有文章，我来逐个检查是否包含 Kafka"                     │
│  → O(N)，N = 文章总数                                         │
│                                                              │
│  搜索引擎：                                                   │
│  "Kafka 这个词出现在哪些文章里？我早就记好了"                   │
│  → O(K)，K = 包含关键词的文章数（远小于 N）                     │
│                                                              │
│  举例：100 万篇文章，搜 "Kafka"                                │
│  数据库：扫描 100 万行 → 几秒                                  │
│  搜索引擎：查倒排索引 → 几毫秒                                 │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 二、三种搜索类型

在介绍 Elasticsearch 之前，我们先搞清楚搜索的三种类型：

```
┌──────────────────────────────────────────────────────────────────┐
│                         搜索的三种类型                             │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 结构化搜索（Structured Search）                                │
│     查找精确匹配的数据                                            │
│     例：查找作者为 "browncutie" 的所有文章                         │
│     类比：SQL 的 WHERE author = 'browncutie'                     │
│     工具：关系型数据库就够了                                       │
│                                                                  │
│  2. 全文搜索（Full-text Search）                                  │
│     在大量文本中查找相关内容，支持模糊匹配、相关性排序               │
│     例：搜索包含 "大数据入门" 的文章，按相关度排序                  │
│     特点：支持分词、同义词、拼写纠错、高亮                          │
│     工具：Elasticsearch、Solr、Meilisearch                        │
│                                                                  │
│  3. 向量搜索（Vector Search / 语义搜索）                           │
│     通过 AI 模型将文本转为向量，计算语义相似度                      │
│     例：搜 "分布式消息系统" 能找到讲 Kafka 的文章（虽然没提到 Kafka）│
│     特点：理解语义，而不仅仅是关键词匹配                            │
│     工具：Milvus、Elasticsearch 8.x+（kNN 搜索）                  │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**Elasticsearch 的强大之处在于：同时支持以上三种搜索！**

```
┌──────────────────────────────────────────────┐
│           Elasticsearch 的能力矩阵             │
├──────────────────────────────────────────────┤
│                                              │
│   结构化搜索  ──→  term、range 查询           │
│   全文搜索    ──→  match、multi_match 查询     │
│   向量搜索    ──→  kNN、script_score 查询      │
│   聚合分析    ──→  terms、date_histogram 聚合  │
│   地理搜索    ──→  geo_distance、geo_polygon  │
│                                              │
│   一个引擎，搞定所有搜索需求                    │
│                                              │
└──────────────────────────────────────────────┘
```

## 三、倒排索引原理详解

倒排索引（Inverted Index）是 Elasticsearch 的灵魂，也是它快的根本原因。

### 3.1 正排索引 vs 倒排索引

```
┌──────────────────────────────────────────────────────────────────┐
│                    正排索引（Forward Index）                       │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  从文档出发，记录每个文档包含哪些词                                  │
│                                                                  │
│  文档 1 → [Kafka, 是, 一个, 分布式, 流处理, 平台]                │
│  文档 2 → [Spark, 是, 一个, 快速, 大数据, 处理, 引擎]            │
│  文档 3 → [Kafka, 高级, 指南, 分布式, 系统]                      │
│                                                                  │
│  查 "Kafka" → 需要遍历所有文档的词列表                             │
│  这就是数据库 LIKE 查询干的事情                                     │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────┐
│                    倒排索引（Inverted Index）                      │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  从词出发，记录每个词出现在哪些文档中                                │
│                                                                  │
│  词项（Term）    →  文档列表（Posting List）                      │
│  ─────────────────────────────────────────                       │
│  Kafka         →  [文档 1, 文档 3]                               │
│  Spark         →  [文档 2]                                       │
│  分布式         →  [文档 1, 文档 3]                               │
│  流处理         →  [文档 1]                                       │
│  大数据         →  [文档 2]                                       │
│  指南          →  [文档 3]                                       │
│  ...           →  ...                                            │
│                                                                  │
│  查 "Kafka" → 直接看 Kafka 这一行 → [文档 1, 文档 3]              │
│  一步到位！                                                       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 3.2 倒排索引的完整结构

实际的倒排索引比上面更复杂，它还包含词频、位置等信息：

```
┌──────────────────────────────────────────────────────────────────┐
│                  倒排索引的完整结构                                 │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌──────────┐  ┌───────────┐  ┌────────┐  ┌──────────────────┐ │
│  │ 词典     │  │ Posting   │  │ 词频   │  │ 位置              │ │
│  │ Dictionary│ │ List      │  │ (TF)   │  │ (Position)       │ │
│  ├──────────┤  ├───────────┤  ├────────┤  ├──────────────────┤ │
│  │ Kafka    │→ │ [doc1,    │  │ doc1:2 │  │ doc1: [0, 4]    │ │
│  │          │  │  doc3]    │  │ doc3:1 │  │ doc3: [0]       │ │
│  │          │  │           │  │        │  │                  │ │
│  │ 分布式   │→ │ [doc1,    │  │ doc1:1 │  │ doc1: [3]       │ │
│  │          │  │  doc3]    │  │ doc3:1 │  │ doc3: [3]       │ │
│  │          │  │           │  │        │  │                  │ │
│  │ 流处理   │→ │ [doc1]    │  │ doc1:1 │  │ doc1: [4, 5]    │ │
│  └──────────┘  └───────────┘  └────────┘  └──────────────────┘ │
│                                                                  │
│  词典（Dictionary）：所有唯一词项的集合，通常存储在内存中             │
│  Posting List：包含该词的文档 ID 列表，用 FST 压缩                  │
│  词频（TF）：词在文档中出现的次数，用于计算相关性得分                  │
│  位置（Position）：词在文档中的位置，用于短语查询和高亮               │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 3.3 分词（Analysis）——倒排索引的基石

在建立倒排索引之前，文本需要经过**分词**处理：

```
┌──────────────────────────────────────────────────────────────────┐
│                     分词过程                                      │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  原始文本："Kafka 是一个分布式流处理平台"                          │
│      │                                                          │
│      ▼                                                          │
│  ┌─────────────────────────┐                                    │
│  │  Character Filters      │  字符过滤（去除 HTML 标签等）         │
│  └────────────┬────────────┘                                    │
│               │                                                  │
│               ▼                                                  │
│  ┌─────────────────────────┐                                    │
│  │  Tokenizer              │  分词器（切分成词）                   │
│  │                         │                                    │
│  │  "Kafka/是/一个/分布式/   │                                    │
│  │   流处理/平台"           │                                    │
│  └────────────┬────────────┘                                    │
│               │                                                  │
│               ▼                                                  │
│  ┌─────────────────────────┐                                    │
│  │  Token Filters          │  词项过滤（转小写、去停用词等）        │
│  │                         │                                    │
│  │  去除"是"、"一个"等      │                                    │
│  │  → [kafka, 分布式,      │                                    │
│  │     流处理, 平台]        │                                    │
│  └────────────┬────────────┘                                    │
│               │                                                  │
│               ▼                                                  │
│  写入倒排索引                                                   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

> 分词的详细讲解见 [07-中文分词与优化.md](07-中文分词与优化.md)

### 3.4 相关性评分（Relevance Scoring）

搜索引擎不仅要找到匹配的文档，还要按**相关度**排序。Elasticsearch 使用 **BM25 算法**（改进的 TF-IDF）来计算相关性得分：

```
┌──────────────────────────────────────────────────────────────────┐
│                    BM25 评分的核心因素                              │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. TF（Term Frequency）—— 词频                                  │
│     关键词在文档中出现越多 → 得分越高                               │
│     但有饱和度：第 5 次出现带来的加分 < 第 1 次出现的加分            │
│                                                                  │
│  2. IDF（Inverse Document Frequency）—— 逆文档频率                 │
│     关键词越稀有 → 得分越高                                       │
│     "Kafka" 出现在 10 篇文章中 → IDF 高                            │
│     "的" 出现在 10000 篇文章中 → IDF 低（几乎不加分）               │
│                                                                  │
│  3. Field Length —— 字段长度                                      │
│     字段越短，关键词的权重越高                                     │
│     标题中出现 "Kafka" 比正文中出现 "Kafka" 更重要                 │
│                                                                  │
│  公式（简化版）：                                                  │
│  score = TF × IDF × fieldLengthNorm                              │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 四、Elasticsearch 是什么？

### 4.1 一句话介绍

**Elasticsearch** 是一个基于 Apache Lucene 构建的**分布式全文搜索和分析引擎**，通过 RESTful API 提供近实时的搜索能力。

```
┌──────────────────────────────────────────────────────────────┐
│                   Elasticsearch 的定位                        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   "如果你有大量数据，需要在毫秒级别搜索它们，                     │
│    那就用 Elasticsearch。"                                     │
│                                                              │
│   不是数据库（虽然可以当文档数据库用）                           │
│   不是数仓（虽然能做聚合分析）                                  │
│   是一个专注于搜索和分析的引擎                                  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 Elasticsearch 核心特性

| 特性 | 说明 |
|------|------|
| **全文检索** | 基于 Lucene，支持分词、同义词、模糊搜索 |
| **近实时** | 数据写入后约 1 秒内可被搜索（refresh 机制） |
| **分布式** | 自动分片、副本、负载均衡、故障转移 |
| **RESTful API** | 通过 HTTP + JSON 操作，任何语言都能用 |
| **聚合分析** | 内置强大的聚合框架，类似 SQL 的 GROUP BY |
| **Schema Free** | 支持动态映射，也可以显式定义字段类型 |
| **高可用** | 副本机制保证数据不丢失，节点故障自动恢复 |

## 五、Elasticsearch 在大数据生态中的位置

在你的数据分析项目大数据管道中，Elasticsearch 扮演着**搜索层**的角色：

```
┌──────────────────────────────────────────────────────────────────┐
│                    数据分析项目大数据管道                               │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  用户操作（发布文章 / 评论 / 浏览）                                │
│      │                                                          │
│      ▼                                                          │
│  Kafka（事件流）── 消息缓冲和解耦                                  │
│      │                                                          │
│      ├──→ Flink（实时计算）── 实时访问量统计                      │
│      │                                                          │
│      └──→ Spark（批量 ETL）── 数据清洗 → Iceberg/Hive             │
│                      │                                          │
│                      ▼                                          │
│              Elasticsearch（搜索与分析）                           │
│              - 数据全文搜索                                       │
│              - 文章聚合分析                                       │
│              - 搜索推荐                                           │
│                                                                  │
│  DolphinScheduler（定时调度以上流程）                               │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**Elasticsearch 与其他组件的协作关系：**

| 组件 | 与 ES 的关系 |
|------|-------------|
| **Kafka** | 实时数据源，通过 Flink 消费 Kafka 消息写入 ES |
| **Flink** | 实时 ETL，将处理后的数据写入 ES |
| **Spark** | 批量 ETL，定时将 Hive/Iceberg 中的历史数据同步到 ES |
| **Hive/Iceberg** | 历史数据的"源"，ES 是搜索的"副本" |
| **Milvus** | 向量搜索（语义搜索），ES 负责关键词搜索 |

## 七、数据分析项目场景

### 7.1 数据全文搜索

用户在数据平台首页的搜索框输入关键词，系统需要在所有文章的标题、内容、标签中搜索，并按相关度排序返回结果。

```
┌──────────────────────────────────────────────────────────────┐
│                  数据搜索场景                                   │
```

## 八、面试题

### 面试题 1：请解释倒排索引的工作原理

**参考答案：**

倒排索引（Inverted Index）是搜索引擎的核心数据结构。与传统的正排索引（记录每个文档包含哪些词）不同，倒排索引记录的是**每个词出现在哪些文档中**。

工作原理分为两步：

**1. 建立索引（写入时）**
- 对文档内容进行分词（Analysis），提取词项（Term）
- 对每个词项，记录它出现在哪些文档中（Posting List）
- 同时记录词频（TF）、位置（Position）等信息

**2. 搜索（查询时）**
- 对用户查询的关键词进行同样的分词
- 在词典中查找每个词项，获取对应的 Posting List
- 对多个词项的 Posting List 求交集（AND）或并集（OR）
- 根据 BM25 算法计算相关性得分，按得分排序返回

**举例：**
- 文档 1："Kafka 是一个分布式流处理平台"
- 文档 2："Spark 是一个快速的大数据处理引擎"

倒排索引中：
- "kafka" → [文档 1]
- "分布式" → [文档 1]
- "spark" → [文档 2]
- "大数据" → [文档 2]

搜索 "分布式平台" 时，分别查 "分布式" 和 "平台"，取交集得到文档 1。

**优势：** 查询时间复杂度从 O(N) 降到 O(K)，K 是匹配文档数，远小于文档总数 N。

---

### 面试题 2：Elasticsearch 和关系型数据库有什么区别？能替代数据库吗？

**参考答案：**

核心区别：
- **设计目标不同**：数据库擅长事务（ACID）和结构化查询；ES 擅长全文搜索和聚合分析
- **数据模型不同**：数据库是关系型（行列表）；ES 是文档型（JSON）
- **查询方式不同**：数据库用 SQL；ES 用 JSON DSL
- **实时性不同**：数据库是强一致性的；ES 是近实时的（有 ~1s 延迟）
- **事务支持不同**：数据库支持完整的 ACID 事务；ES 不支持跨文档事务

**ES 不能替代数据库的场景：**
- 需要强事务保证的场景（如金融转账）
- 需要复杂 JOIN 的场景
- 需要外键约束的场景
- 数据频繁更新的场景

**正确用法：** ES 通常作为数据库的"搜索副本"，数据库负责存储和事务，ES 负责搜索。数据通过 ETL 工具从数据库同步到 ES。

---

### 面试题 3：ES 为什么搜索这么快？

**参考答案：**

1. **倒排索引**：核心原因。通过 "词 → 文档" 的映射，直接定位到包含关键词的文档，避免全表扫描
2. **分片并行**：索引被分成多个分片，搜索时多个分片并行执行，结果在协调节点合并
3. **操作系统缓存**：Lucene 大量利用操作系统的文件系统缓存，热点数据常驻内存
4. **段（Segment）不可变**：索引数据写入后不可修改，不需要加锁，读取无竞争
5. **FST 压缩**：词典使用 FST（Finite State Transducer）数据结构，占用内存小，查找快
6. **BM25 评分**：评分算法在写入时预计算部分数据，搜索时只需增量计算
