## 二、集群规划

### 2.1 节点角色

```
┌──────────────────────────────────────────────────────────────────┐
│                    ES 节点角色                                    │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. Master-eligible Node（主节点候选）                             │
│     - 参与集群管理：创建/删除索引、分配分片、节点发现              │
│     - 不存储数据（建议），不处理搜索请求                          │
│     - 集群中至少 3 个，保证高可用（奇数个）                       │
│     - 配置：node.roles: [ master ]                               │
│                                                                  │
│  2. Data Node（数据节点）                                        │
│     - 存储索引数据                                               │
│     - 执行 CRUD、搜索、聚合操作                                   │
│     - 最消耗 CPU、内存、磁盘                                      │
│     - 配置：node.roles: [ data ]                                 │
│                                                                  │
│  3. Coordinating Node（协调节点）                                 │
│     - 接收客户端请求，分发到数据节点，合并结果                     │
│     - 不存储数据，不参与集群管理                                  │
│     - 减轻数据节点的负担                                         │
│     - 配置：node.roles: [ ]（空列表，即只做协调）                │
│     - 小集群可以省略，大集群建议独立部署                          │
│                                                                  │
│  4. Ingest Node（摄入节点）                                      │
│     - 在数据写入前进行预处理（管道处理）                          │
│     - 配置：node.roles: [ ingest ]                               │
│                                                                  │
│  5. Machine Learning Node（ML 节点）                             │
│     - 运行机器学习任务                                           │
│     - 配置：node.roles: [ ml ]                                   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 三、索引生命周期管理（ILM）

### 3.1 ILM 概述

索引生命周期管理（Index Lifecycle Management, ILM）是 ES 内置的索引自动化管理工具。

```
┌──────────────────────────────────────────────────────────────────┐
│                    ILM 的四个阶段                                  │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐    │
│  │  Hot     │ → │  Warm    │ → │  Cold    │ → │  Delete  │    │
│  │  热阶段   │   │  温阶段   │   │  冷阶段   │   │  删除阶段  │    │
│  └──────────┘   └──────────┘   └──────────┘   └──────────┘    │
│                                                                  │
│  Hot（热阶段）：                                                  │
│  - 索引正在被活跃写入和搜索                                      │
│  - 数据存储在 SSD 上                                             │
│  - 可以配置 Rollover（索引滚动：大小/文档数/时间超阈值自动建新索引）│
│                                                                  │
│  Warm（温阶段）：                                                 │
│  - 索引不再写入，但偶尔被搜索                                    │
│  - 可以缩小副本数、强制合并段                                    │
│  - 数据可以迁移到 HDD                                            │
│                                                                  │
│  Cold（冷阶段）：                                                 │
│  - 索引很少被搜索                                                │
│  - 数据可以迁移到更便宜的存储                                    │
│  - 最小化资源占用                                                │
│                                                                  │
│  Delete（删除阶段）：                                             │
│  - 索引不再需要，自动删除                                        │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 3.2 ILM 配置示例

```bash
# 创建 ILM 策略
curl -X PUT "http://localhost:9200/_ilm/policy/demo_ilm_policy" -H 'Content-Type: application/json' -d '{
  "policy": {
    "phases": {
      "hot": {
        "min_age": "0ms",
        "actions": {
          "rollover": {
            "max_size": "30gb",
            "max_age": "30d",
            "max_docs": 1000000
          },
          "set_priority": {
            "priority": 100
          }
        }
      },
      "warm": {
        "min_age": "30d",
        "actions": {
          "forcemerge": {
            "max_num_segments": 1
          },
          "shrink": {
            "number_of_shards": 1
          },
          "set_priority": {
            "priority": 50
          }
        }
      },
      "delete": {
        "min_age": "90d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}'
```

## 四、数据备份与恢复（Snapshot）

### 4.1 注册快照仓库

```bash
# 注册本地文件系统仓库
curl -X PUT "http://localhost:9200/_snapshot/demo_backup" -H 'Content-Type: application/json' -d '{
  "type": "fs",
  "settings": {
    "location": "/opt/elasticsearch/backup"
  }
}'
```

> 备份目录需要在 `elasticsearch.yml` 中配置 `path.repo: ["/opt/elasticsearch/backup"]`

### 4.2 创建快照

```bash
# 备份所有索引
curl -X PUT "http://localhost:9200/_snapshot/demo_backup/snapshot_20260415?pretty"

# 备份指定索引
curl -X PUT "http://localhost:9200/_snapshot/demo_backup/snapshot_demo?pretty" -H 'Content-Type: application/json' -d '{
  "indices": "posts",
  "ignore_unavailable": true,
  "include_global_state": false
}'
```

### 4.3 查看和恢复快照

```bash
# 查看所有快照
curl http://localhost:9200/_snapshot/demo_backup/_all?pretty

# 查看快照状态
curl http://localhost:9200/_snapshot/demo_backup/snapshot_20260415/_status?pretty

# 从快照恢复
curl -X POST "http://localhost:9200/_snapshot/demo_backup/snapshot_20260415/_restore?pretty" -H 'Content-Type: application/json' -d '{
  "indices": "posts",
  "include_global_state": false
}'

# 删除快照
curl -X DELETE "http://localhost:9200/_snapshot/demo_backup/snapshot_20260415"
```

```
┌──────────────────────────────────────────────────────────────┐
│              Snapshot 注意事项                                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. Snapshot 是增量的                                          │
│     第二次备份只备份新增和修改的数据，速度快                    │
│                                                              │
│  2. 备份过程中索引可以正常读写                                  │
│     ES 使用 Lucene 的段文件快照，不影响在线服务                │
│                                                              │
│  3. 恢复前建议关闭索引                                        │
│     或者恢复到不同的索引名（避免覆盖当前数据）                 │
│                                                              │
│  4. 定期备份 + 异地备份                                       │
│     本地文件系统仓库只适合开发环境                             │
│     生产环境建议使用 S3、HDFS 等远程仓库                      │
│                                                              │
│  5. 可以用定时任务自动备份（配合 Cron 或 DolphinScheduler）    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 5.4 OOM（内存溢出）

```
┌──────────────────────────────────────────────────────────────┐
│              OOM 排查与解决                                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  原因分析：                                                   │
│  1. JVM 堆内存设置过大（超过 31GB，压缩指针失效）              │
│  2. 堆内存设置过小（大量数据查询时不够用）                      │
│  3. 查询请求过大（返回太多字段、聚合嵌套太深）                 │
│  4. 分片数过多（每个分片都有内存开销）                         │
│  5. 字段类型不当（text 字段太多，倒排索引占用内存大）          │
│                                                              │
│  解决方案：                                                   │
│  1. 调整 JVM 堆内存到合理大小（物理内存 50%，<=31GB）          │
│  2. 减少 _source 返回的字段                                   │
│  3. 使用 search_after 替代大 from + size                      │
│  4. 减少分片数量（合并索引或重建索引）                        │
│  5. 不需要搜索的字段禁用索引（"index": false）                 │
│  6. 使用 doc_values 替代 fielddata（keyword 数值类型默认）     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 六、ES 与 Spark/Flink 的集成

### 6.1 Spark 写入 Elasticsearch

```python
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("ES_Write") \
    .config("spark.jars.packages", "org.elasticsearch:elasticsearch-spark-30_2.12:9.0.0") \
    .getOrCreate()

# 从 Hive 读取文章数据
df = spark.sql("""
    SELECT id, title, content, tags, author, created_at, views
    FROM demo.posts
    WHERE is_published = true
""")

# 写入 Elasticsearch
df.write \
    .format("org.elasticsearch.spark.sql") \
    .option("es.nodes", "localhost") \
    .option("es.port", "9200") \
    .option("es.resource", "posts") \
    .option("es.mapping.id", "id") \
    .mode("overwrite") \
    .save()

spark.stop()
```

### 6.2 Flink 实时写入 Elasticsearch

```python
from pyflink.datastream import StreamExecutionEnvironment
from pyflink.table import StreamTableEnvironment

env = StreamExecutionEnvironment.get_execution_environment()
t_env = StreamTableEnvironment.create(env)

# 创建 Kafka 源表
t_env.execute_sql("""
    CREATE TABLE demo_events (
        event_type STRING,
        post_id INT,
        title STRING,
        content STRING,
        tags ARRAY<STRING>,
        author STRING,
        created_at STRING,
        views INT,
        event_time TIMESTAMP(3),
        WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND
    ) WITH (
        'connector' = 'kafka',
        'topic' = 'blog-events',
        'properties.bootstrap.servers' = 'localhost:9092',
        'properties.group.id' = 'flink-es-sink',
        'format' = 'json',
        'scan.startup.mode' = 'latest-offset'
    )
""")

# 创建 ES 目标表
t_env.execute_sql("""
    CREATE TABLE posts_es (
        _id STRING,
        title STRING,
        content STRING,
        tags ARRAY<STRING>,
        author STRING,
        created_at STRING,
        views INT,
        is_published BOOLEAN
    ) WITH (
        'connector' = 'elasticsearch-9',
        'hosts' = 'http://localhost:9200',
        'index' = 'posts'
    )
""")

# 写入 ES
t_env.execute_sql("""
    INSERT INTO posts_es
    SELECT
        CAST(post_id AS STRING) as _id,
        title,
        content,
        tags,
        author,
        created_at,
        views,
        true as is_published
    FROM demo_events
    WHERE event_type = 'PUBLISH'
""")

env.execute("Flink to Elasticsearch")
```

## 七、15 道高频面试题

### 面试题 1：倒排索引的工作原理是什么？

**参考答案：**

倒排索引是搜索引擎的核心数据结构。与传统正排索引（文档 → 词）不同，倒排索引记录的是"每个词出现在哪些文档中"（词 → 文档）。

建立过程：
1. 文档写入时，通过分析器（Analyzer）对文本进行分词
2. 提取出词项（Term），记录每个词项对应的文档 ID 列表（Posting List）
3. 同时记录词频（TF）、位置（Position）等信息

搜索过程：
1. 用户查询关键词经过同样的分词处理
2. 在词典中查找每个词项的 Posting List
3. 对多个词项的 Posting List 求交集/并集
4. 根据 BM25 算法计算相关性得分
5. 按得分排序返回结果

优势：查询时间复杂度从 O(N) 降到 O(K)，K 是匹配文档数。

### 面试题 2：Elasticsearch 写入数据的流程是什么？

**参考答案：**

1. 客户端发送写请求到协调节点（Coordinating Node）
2. 协调节点根据路由公式 `hash(_id) % num_shards` 找到目标分片
3. 将请求转发到目标分片所在的主节点
4. 主节点执行写入操作：
   - 文档写入内存中的索引缓冲区（Index Buffer）
   - 同时写入事务日志（Translog，持久化到磁盘）
5. 返回写入成功给客户端
6. 每隔 `refresh_interval`（默认 1 秒），缓冲区数据被刷新为新的段（Segment）
7. 段被写入 OS 文件系统缓存（可搜索但不持久化）
8. 每隔一段时间（或 Translog 达到阈值），执行 flush 操作：
   - 将段刷到磁盘（持久化）
   - 清空 Translog
9. 主节点将数据同步到副本分片

### 面试题 3：Elasticsearch 搜索数据的流程是什么？

**参考答案：**

以搜索请求为例：

1. 客户端发送搜索请求到协调节点
2. 协调节点将请求广播到索引的所有分片（主分片和副本分片）
3. 每个分片在本地执行查询：
   - 先检查 query cache（查询缓存）
   - 执行 bool 查询中的 filter 子句（使用 Bitset 缓存）
   - 执行 must/should 子句，计算相关性得分
4. 每个分片返回本地 Top N 结果（默认 `from + size` 条）给协调节点
5. 协调节点合并所有分片的结果：
   - 对所有结果重新排序
   - 取最终的 Top N
6. 如果有聚合，协调节点合并各分片的聚合结果
7. 返回最终结果给客户端

注意：协调节点需要处理的数据量 = 分片数 × (from + size)，所以深度分页时开销很大。

### 面试题 4：什么是 Segment？它在 ES 中扮演什么角色？

**参考答案：**

Segment（段）是 Lucene 中索引数据存储的最小单位，是一个不可变的文件。

特点：
1. **不可变**：段一旦创建就不能修改。新数据写入新段，更新和删除通过标记实现
2. **自包含**：每个段包含完整的倒排索引、存储字段、词项字典等
3. **独立可搜索**：搜索时可以并行查询多个段

生命周期：
1. 新文档写入内存中的 Index Buffer
2. refresh 操作将 Buffer 中的数据写入新段（写入 OS Cache，可搜索）
3. flush 操作将段刷到磁盘（持久化）
4. 多个小段通过 Merge 操作合并成大段（减少段数量，提高搜索性能）

优势：不可变的设计使得读取无需加锁，多线程并发安全。

### 面试题 5：Elasticsearch 如何保证高可用？

**参考答案：**

1. **分片 + 副本机制**：每个主分片有多个副本分片，分布在不同节点上。主分片故障时，副本自动提升为主分片
2. **集群发现与主节点选举**：节点加入或离开集群时自动发现，主节点故障时自动选举新主节点
3. **Zen Discovery / 7.x+ 的选举机制**：保证集群中只有一个可用的主节点
4. **脑裂防护**：通过 quorum 机制（`minimum_master_nodes` 或自动配置）防止脑裂
5. **分片自动重分配**：节点故障后，分片会自动分配到其他健康节点

```
节点 A（Shard 0 主）故障：
  → 副本所在节点 B 上的 Shard 0 副本自动提升为主分片
  → 如果有足够节点，自动创建新的副本分片
  → 整个过程对应用透明
```

### 面试题 6：什么是 Mapping？为什么字段类型不能修改？

**参考答案：**

Mapping 定义了索引中文档的字段类型和属性，类似数据库的 Schema。

字段类型不能修改的原因：
1. 字段类型决定了数据在磁盘上的存储格式（倒排索引结构不同）
2. 修改类型意味着要重建所有数据的存储结构
3. 如果允许修改，可能导致查询结果不一致

如果确实需要修改字段类型：
1. 创建新索引（使用新的 Mapping）
2. 使用 `_reindex` API 将数据从旧索引复制到新索引
3. 给新索引添加别名（让应用无感知切换）
4. 确认无误后删除旧索引

```bash
curl -X POST "http://localhost:9200/_reindex" -H 'Content-Type: application/json' -d '{
  "source": { "index": "old_index" },
  "dest": { "index": "new_index" }
}'
```

### 面试题 7：如何优化 Elasticsearch 的写入性能？

**参考答案：**

1. **批量写入**：使用 `_bulk` API，每批 1000-5000 条
2. **调大 refresh_interval**：写入期间设为 30s 或 -1，减少段刷新频率
3. **减少副本数**：写入期间设为 0，写入完成后恢复
4. **使用 SSD 磁盘**：SSD 的随机写入性能远好于 HDD
5. **合理设置分片数**：分片太多增加管理开销，太少无法并行
6. **自动生成 ID**：POST `_doc` 比 PUT `_doc/id` 更快（不需要查重）
7. **关闭不必要的索引**：不活跃的索引 close 掉，释放内存
8. **调整 translog**：`index.translog.durability: async`（牺牲安全性换性能）
9. **增加 thread_pool.write.queue_size**：增大写入线程池队列

### 面试题 8：如何优化 Elasticsearch 的搜索性能？

**参考答案：**

1. **使用 filter context**：不需要评分的条件用 filter，结果可被缓存
2. **合理设计 Mapping**：不需要全文搜索的字段用 keyword，不需要索引的字段设 `"index": false`
3. **使用 routing**：按业务维度将数据路由到固定分片，减少查询的分片数
4. **控制分片数**：分片越多，每次搜索需要合并的开销越大
5. **使用 _source 过滤**：只返回需要的字段
6. **避免通配符查询**：wildcard、regexp 性能差，用前缀查询或 n-gram 替代
7. **预热缓存**：对高频查询预先执行，建立 filter 缓存
8. **使用 search_after**：替代大 from + size 的深度分页
9. **合并段**：定期执行 `_forcemerge`，减少段数量
10. **硬件层面**：足够的堆内存 + SSD + 充足的 OS Cache

### 面试题 9：Elasticsearch 的 Master 选举机制是什么？

**参考答案：**

ES 使用基于 Raft 协议变体的选举机制（7.x+）：

1. 每个节点都有一个 `current_state`，值为 `candidate`、`follower` 或 `leader`
2. 当主节点不可用时，候选节点发起选举
3. 候选节点向其他节点请求投票
4. 获得大多数节点（quorum = master_eligible_nodes / 2 + 1）投票的候选节点成为新主节点
5. 选举完成后，新主节点负责集群管理（分片分配、索引创建等）

关键配置：
- `discovery.seed_hosts`：集群节点的初始列表
- `cluster.initial_master_nodes`：首次启动时的主节点候选列表
- ES 7.x+ 自动管理 `minimum_master_nodes`，无需手动配置

### 面试题 10：什么是脑裂（Split Brain）？如何防止？

**参考答案：**

**脑裂**是指集群中出现了两个主节点，导致集群分裂成两个独立的集群。这会导致数据不一致。

发生场景：
- 网络分区导致节点间无法通信
- 节点负载过高，心跳超时
- 被误判为故障

防止脑裂：
- ES 7.x 之前：配置 `minimum_master_nodes = (master_eligible_nodes / 2) + 1`
  - 例如 3 个 master 节点，设为 2
  - 只有获得 2 票以上才能成为主节点
- ES 7.x+：自动管理，基于 quorum 机制
- 确保至少 3 个 Master 节点

```
脑裂场景：
  节点 A（主）    节点 B    节点 C
       |           |          |
  ─────网络故障──────          |
       |                      |
  A 自认为是主    B、C 选举 C 为主
  → 两个集群各自为政，数据不一致！

配置 quorum=2 后：
  网络分区后，只有 B 和 C 之间能通信（2 个节点）
  A 只有 1 票，无法获得 quorum → 降级
  B 和 C 选出 C 作为主节点 → 集群只有一个主
```

### 面试题 11：Elasticsearch 如何实现近实时搜索？

**参考答案：**

ES 的近实时搜索依赖 **refresh 机制**：

1. 新文档写入内存中的 Index Buffer（同时写入 Translog）
2. 每隔 `refresh_interval`（默认 1 秒），执行 refresh 操作
3. refresh 将 Index Buffer 中的数据生成一个新的不可变段（Segment）
4. 新段写入 OS 文件系统缓存（不刷磁盘，但可被搜索）
5. refresh 后新数据就可以被搜索到了

从数据写入到可搜索，最多有 1 秒延迟，这就是"近实时"。

如果需要更实时：
- 减小 `refresh_interval`（但会增加段数量，影响性能）
- 手动调用 `_refresh` API

如果需要写入后立即搜索：
- 写入后立即调用 `POST /index/_refresh`

### 面试题 12：Elasticsearch 和 Solr 有什么区别？

**参考答案：**

| 对比项 | Elasticsearch | Solr |
|--------|---------------|------|
| **底层** | Apache Lucene | Apache Lucene |
| **分布式** | 原生分布式 | 需要 ZooKeeper |
| **实时性** | 近实时（~1s） | 准实时 |
| **配置** | Schema Free | 需要 Schema 配置 |
| **社区** | 更活跃，生态更丰富 | 成熟稳定 |
| **学习曲线** | 较低（REST API） | 较高 |
| **聚合** | 非常强大 | 强大 |
| **全文搜索** | 优秀 | 优秀 |

选择建议：
- 日志分析、实时搜索、需要分布式 → Elasticsearch
- 传统企业搜索、已有 Solr 基础设施 → Solr

### 面试题 13：如何设计一个日均千万级文档的 ES 搜索系统？

**参考答案：**

**索引设计：**
1. 按时间分索引（如 demo_2026-04），结合 ILM 自动管理
2. 每个索引 3-5 个主分片，1 个副本
3. 使用显式 Mapping + `dynamic: strict`
4. text 字段配置合适的分析器（中文用 IK）
5. 不需要搜索的字段禁用索引

**集群规划：**
1. 3 个 Master 节点（2C 4G）
2. 5-10 个 Data 节点（8C 32G SSD）
3. 2 个 Coordinating 节点

**写入优化：**
1. 使用 _bulk 批量写入
2. Kafka → Flink → ES 实时写入
3. Spark 批量补偿（兜底）
4. 调大 refresh_interval

**查询优化：**
1. 用 filter context 处理不需要评分的条件
2. 使用 routing 优化常见查询
3. 使用 search_after 替代深度分页
4. 合理设置分页上限

**运维保障：**
1. 定时 Snapshot 备份
2. 监控集群健康、JVM、磁盘、分片状态
3. ILM 策略自动滚动索引和清理过期数据

### 面试题 14：Elasticsearch 的 fielddata 和 doc_values 有什么区别？

**参考答案：**

两者都是用于排序和聚合的数据结构，但有本质区别：

**fielddata：**
- 存储在 JVM 堆内存中
- 用于 text 字段的排序和聚合
- 构建成本高（需要读取倒排索引并重建）
- 可能导致 OOM（如果 text 字段值很多）
- 可以设置 `fielddata` 的断路器（circuit breaker）限制内存使用

**doc_values：**
- 存储在磁盘上（列式存储）
- 用于 keyword、数值、日期等非 text 字段
- 构建成本低，在段创建时就生成
- 不会导致 OOM（因为数据在磁盘上）
- 通过 OS Cache 加速访问

**最佳实践：**
- keyword 和数值类型字段：使用 doc_values（默认开启）
- text 字段的排序和聚合：尽量避免（使用 keyword 子字段代替）
- 如果必须对 text 字段排序/聚合：开启 fielddata，注意设置断路器

### 面试题 15：如何监控 Elasticsearch 集群？

**参考答案：**

**ES 内置 API：**
```bash
# 集群健康状态
curl localhost:9200/_cluster/health?pretty

# 节点状态
curl localhost:9200/_cat/nodes?v

# 索引状态
curl localhost:9200/_cat/indices?v

# 分片状态
curl localhost:9200/_cat/shards?v

# 分片分配
curl localhost:9200/_cat/allocation?v

# 线程池
curl localhost:9200/_cat/thread_pool?v

# JVM 状态
curl localhost:9200/_nodes/stats/jvm?pretty

# 磁盘使用
curl localhost:9200/_cat/allocation?v

# 待处理任务
curl localhost:9200/_cluster/pending_tasks?pretty
```

**关键监控指标：**
1. 集群状态（Green/Yellow/Red）
2. JVM 堆内存使用率（不超过 75%）
3. 磁盘使用率（不超过 85%）
4. 分片数量（单节点不超过 20 个）
5. 搜索延迟（P99 < 200ms）
6. 索引速率（docs/s）
7. 搜索 QPS
8. 段数量（单分片不超过几十个）

**外部监控：**
1. **Kibana Stack Monitoring**：ES 自带的监控方案
2. **Prometheus + Grafana**：通过 elasticsearch_exporter 采集指标
3. **ELK（Elasticsearch + Logstash + Kibana）**：收集和分析 ES 自身日志
