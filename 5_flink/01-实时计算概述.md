# 01 - 实时计算概述：从批处理到流处理的演进

## 一、为什么需要实时计算？

在学习 Flink 之前，我们先想一个问题：**你的数据平台每天有多少人访问？**

你可能会说："查一下 Nginx 日志就知道了。"

没错，如果只是想知道"昨天"的访问量，传统的批处理完全够用：

```
传统批处理模式：

  用户访问 → Nginx 日志 → 每天凌晨跑 Spark 作业 → 生成报表
                                           ↑
                                    等到第二天才能看到结果
```

但如果你需要知道**此时此刻**有多少人在看你的数据平台？哪篇文章正在走红？某个用户是不是在恶意刷接口？

```
实时处理模式：

  用户访问 → Kafka → Flink 实时计算 → 实时 Dashboard
                                     ↑
                              秒级甚至毫秒级看到结果
```

这就是实时计算的核心价值：**让数据的价值在产生的瞬间就被释放。**

### 1.1 批处理 vs 实时处理的区别

```
┌─────────────────────────────────────────────────────────────────┐
│                   批处理 vs 实时处理 对比                         │
├──────────────────┬──────────────────┬───────────────────────────┤
│                  │   批处理（Batch）  │   实时处理（Streaming）    │
├──────────────────┼──────────────────┼───────────────────────────┤
│  数据范围         │   有界数据集      │   无界数据流              │
│  处理方式         │   收集齐再处理    │   来一条处理一条           │
│  延迟             │   分钟~小时级     │   毫秒~秒级               │
│  代表框架         │   MapReduce/Spark│   Flink/Storm             │
│  典型场景         │   离线报表       │   实时监控、推荐           │
│  数据量           │   TB~PB 级       │   取决于数据产生速度       │
├──────────────────┼──────────────────┼───────────────────────────┤
│  比喻             │   "看完书再写   │   "边看边做笔记"           │
│                  │    读笔记"       │                           │
└──────────────────┴──────────────────┴───────────────────────────┘
```

```
│  - 没有精确一次语义                                          │
│  - 没有高级窗口 API                                          │
│  - 不支持事件时间处理                                         │
│  - 社区活跃度下降                                             │
│                                                             │
│  状态：已逐渐被 Flink 取代                                    │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 Spark Streaming

```
┌─────────────────────────────────────────────────────────────┐
│  Spark Streaming（2013 年）                                  │
│                                                             │
│  核心思想：微批处理（Micro-batch）                            │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │                                                     │    │
│  │  数据流： ───●──●──●──●──●──●──●──●──●──●──→       │    │

│  │               ↓    ↓    ↓    ↓    ↓                 │    │
│  │             [批1] [批2] [批3] [批4] [批5]           │    │
│  │               ↓    ↓    ↓    ↓    ↓                 │    │
│  │             处理  处理  处理  处理  处理              │    │

│  │  本质：把流切成小批次，用批处理引擎处理                  │    │
│  │                                                     │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
│  优点：                                                      │
│  - 与 Spark 生态无缝集成                                     │
│  - 吞吐量高                                                 │
│                                                             │
│  缺点：                                                      │
│  - 微批模型导致延迟较高（秒级）                                │
│  - 事件时间支持不够原生                                       │

│                                                             │

└─────────────────────────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────────────────────────┐

│  └─────────────────────────────────────────────────────┘    │
│                                                             │
│  优点：                                                      │
│  - 真正的流式处理，毫秒级延迟                                 │
│  - 内置强大的状态管理                                        │
│  - 原生支持事件时间和 Watermark                               │
│  - 精确一次（Exactly-Once）语义                               │
│  - 批流一体：同一套 API                                       │
│  - 丰富的窗口和连接器                                         │
│                                                             │
```

### 3.4 三大框架全面对比

| 对比项 | Storm | Spark Streaming | Flink |
|--------|-------|-----------------|-------|
| **处理模型** | 真正的流 | 微批处理 | 真正的流 |
| **延迟** | 亚毫秒级 | 秒级 | 毫秒级 |
| **状态管理** | 需自己实现 | 有，但有限 | 内置，非常强大 |
| **精确一次** | 不支持 | 支持 | 支持 |
| **事件时间** | 不支持 | 支持（有限） | 原生支持 |
| **窗口** | 基础窗口 | 较丰富 | 非常丰富 |
| **SQL** | 不支持 | 支持 | 支持 |
| **批处理** | 不支持 | 原生支持 | 支持（批流一体） |
| **社区活跃度** | 低 | 高 | 高 |
| **适用场景** | 极低延迟需求 | 批处理为主 | 实时计算首选 |

## 四、Flink 的核心优势

### 4.1 真正的流式处理

Flink 从底层设计就是面向流处理的，而不是像 Spark Streaming 那样把流切成批来处理。

```
┌─────────────────────────────────────────────────────────────┐
│                Flink 的数据处理模型                            │
│                                                             │
│  无界数据流（Unbounded Stream）                              │
│  ──●──●──●──●──●──●──●──●──●──●──●──●──●──→               │
│                                                             │
│  有界数据流（Bounded Stream）                                │
│  ──●──●──●──●──●──●──●──●──●──●──●──●──●──|               │
│                                              ↑               │
│                                          有明确的结束         │
│                                                             │
│  Flink 用同一套 API 同时处理这两种情况                         │
│  - 流模式：持续处理无界数据                                   │
│  - 批模式：一次性处理有界数据（其实就是特殊的流）                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 精确一次语义（Exactly-Once）

这是流处理中最难、也最重要的保证之一。什么意思呢？

```
┌─────────────────────────────────────────────────────────────┐
│                   三种消息语义                                │
│                                                             │
│  最多一次（At-Most-Once）：                                  │
│  - 数据可能丢失，但不会重复                                   │
│  - 处理最简单，但不够可靠                                     │
│  - 适合：可以容忍少量数据丢失的场景（如日志采集）               │
│                                                             │
│  至少一次（At-Least-Once）：                                 │
│  - 数据不会丢失，但可能重复处理                               │
│  - 需要下游系统能去重                                         │
│  - 适合：不能丢数据，但可以容忍重复的场景（如消息推送）          │
│                                                             │
│  精确一次（Exactly-Once）：                                  │
│  - 数据既不会丢失，也不会重复处理                             │
│  - 实现最复杂，但最可靠                                      │
│  - 需要 Checkpoint + 两阶段提交（2PC）配合                    │
│  - 适合：金融交易、计费系统等对数据准确性要求极高的场景          │
│                                                             │
│  Flink 通过 Checkpoint 机制实现了端到端的 Exactly-Once        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 强大的状态管理

```
┌─────────────────────────────────────────────────────────────┐
│                   Flink 的状态管理                            │
│                                                             │
│  什么是状态？简单说就是算子需要"记住"的数据。                    │
│                                                             │
│  例如：统计每篇文章的浏览量                                    │
│  - 第 1 条数据：文章A 被浏览 → 计数 = 1                       │
│  - 第 2 条数据：文章B 被浏览 → 文章B计数 = 1                   │
│  - 第 3 条数据：文章A 被浏览 → 文章A计数 = 2  ← 需要记住上次的值│
│                                                             │
│  这个"记住上次的值"就是状态（State）。                         │
│                                                             │
│  Flink 内置了多种状态类型：                                    │
│  ┌──────────────────────────────────────────────────┐       │
│  │  ValueState<T>        保存单个值                  │       │
│  │  ListState<T>         保存一个列表                │       │
│  │  MapState<K, V>       保存键值对                  │       │
│  │  ReducingState<T>     自动聚合的值                │       │
│  │  AggregatingState<I,O> 自动聚合（更灵活）         │       │
│  └──────────────────────────────────────────────────┘       │
│                                                             │
│  而且状态可以持久化到磁盘（RocksDB），支持 TB 级别！            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.4 事件时间处理

```
┌─────────────────────────────────────────────────────────────┐
│                   三种时间语义                                │
│                                                             │
│  Event Time（事件时间）：                                     │
│  - 数据实际发生的时间                                         │
│  - 最准确，但可能有延迟（数据可能乱序到达）                     │
│  - Flink 通过 Watermark 机制处理乱序数据                      │
│                                                             │
│  Ingestion Time（摄入时间）：                                │
│  - 数据进入 Flink 的时间                                     │
│  - 比事件时间稍晚，但不会有乱序                                │
│                                                             │
│  Processing Time（处理时间）：                                │
│  - 数据被 Flink 算子处理的时间                               │
│  - 最简单，但结果不确定（相同数据重跑可能结果不同）             │
│                                                             │
│  ─────────────────────────────────────────────              │
│                                                             │
│  举例：用户在北京 10:00 浏览了一篇文章                          │
│  - Event Time = 10:00（用户实际浏览的时间）                    │
│  - 网络延迟，数据 10:01 到达 Kafka                           │
│  - Ingestion Time = 10:01                                   │
│  - Flink 10:02 才处理到这条数据                               │
│  - Processing Time = 10:02                                   │
│                                                             │
│  Flink 推荐使用 Event Time，保证结果的可重现性                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

| **实时 PV** | 每分钟页面浏览量 | 滚动窗口 + 计数 |

| **热门文章 Top 10** | 最近 5 分钟浏览量最高的文章 | 滑动窗口 + 排序 |
| **实时搜索词** | 用户正在搜索的关键词 | 窗口聚合 |

```
│                                                             │
```

**参考答案：**

Lambda 架构由 Nathan Marz 提出，包含批处理层和速度层。批处理层负责全量数据的准确计算，速度层负责实时增量计算，最后在服务层合并结果。缺点是需要维护两套代码。

Kappa 架构由 Jay Kreps（Kafka 作者）提出，去掉了批处理层，只用一套流处理引擎处理所有数据。核心思想是：如果需要重新计算历史数据，就启动一个新的流处理作业从头消费 Kafka 中的数据。Flink 的批流一体能力使得 Kappa 架构成为现实。

**在实际生产中，由于 Flink 的批流一体能力，越来越多的公司选择 Kappa 架构或其变体。**

### Q2：Flink 和 Spark Streaming 的本质区别是什么？

**参考答案：**

本质区别在于**处理模型**不同。Spark Streaming 是微批处理（Micro-batch），它将数据流按时间切分成小批次，然后用 Spark 引擎以批处理的方式处理。而 Flink 是真正的流式处理（True Streaming），每条数据到达后立即处理，不需要等待批次积累。

这导致了几个关键差异：
1. **延迟**：Flink 是毫秒级，Spark Streaming 是秒级
2. **状态管理**：Flink 内置了完善的状态管理（Keyed State、Operator State），Spark Streaming 的状态管理相对有限
3. **事件时间**：Flink 原生支持事件时间和 Watermark 机制，Spark Streaming 虽然也支持但实现较复杂
4. **窗口**：Flink 的窗口机制更灵活，支持会话窗口、自定义窗口触发器等

### Q3：什么是端到端的精确一次语义？Flink 是如何实现的？

**参考答案：**

端到端的精确一次语义（End-to-End Exactly-Once）意味着从数据源读取、经过 Flink 处理、到写入外部系统，整个过程每条数据恰好被处理一次。

Flink 通过以下机制实现：
1. **Source**：Kafka Consumer 将 offset 保存到 Checkpoint 中
2. **Flink 内部**：通过 Checkpoint（基于 Chandy-Lamport 算法）保证算子状态的精确一次
3. **Sink**：通过两阶段提交（Two-Phase Commit, 2PC）协议保证写入外部系统的精确一次。Flink 的 TwoPhaseCommitSinkFunction 接口支持与支持事务的外部系统（如 Kafka、文件系统）实现端到端精确一次

## 八、总结

本篇我们了解了实时计算的必要性、主流框架的对比，以及 Flink 的核心优势。关键要点：

1. **实时计算的核心价值**：让数据在产生瞬间就产生价值
2. **Flink vs Spark Streaming**：Flink 是真正的流处理，Spark Streaming 是微批处理
3. **Flink 三大核心能力**：状态管理、精确一次语义、事件时间处理
4. **架构模式**：Lambda 架构（批流分离）→ Kappa 架构（批流一体）
