# 05 - 窗口与时间：把无界流切成有界块来计算

## 一、为什么需要窗口？

上一篇文章中，我们学习了状态管理——Flink 可以"记住"之前的数据。但如果状态一直累积，会有什么问题？

```
┌─────────────────────────────────────────────────────────────┐
│                没有窗口的问题                                 │
│                                                             │
│  统计数据平台的总浏览量：                                         │
│  ──●(view)──●(view)──●(view)──●(view)── ... ──→           │
│                                                             │
│  状态会无限增长：viewCount = 1 → 2 → 3 → 4 → ... → ∞       │
│                                                             │
│  问题：                                                      │
│  1. 内存会爆（状态无限增长）                                  │
│  2. 结果没有意义（"总浏览量 1 亿"不如"每小时 PV 1000"有用）    │
│  3. 无法回答"最近 5 分钟哪篇文章最火"这种问题                 │
│                                                             │
│  解决方案：**窗口（Window）**                                  │
│  把无界的数据流切分成有界的块，每个块独立计算                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、时间语义

在讲解窗口之前，必须先理解三种时间语义，因为这直接决定了窗口怎么切分。

```
┌─────────────────────────────────────────────────────────────────┐
│                       三种时间语义                               │
│                                                                 │
│  1. Event Time（事件时间）                                       │
│     ── 数据实际发生的时间                                        │
│     ── 最准确，但数据可能乱序到达                                  │
│     ── Flink 推荐！                                             │
│                                                                 │
│  2. Processing Time（处理时间）                                  │
│     ── 数据被 Flink 算子处理的时间                               │
│     ── 最简单，延迟最低                                          │
│     ── 但结果不确定（相同数据重跑结果不同）                        │
│                                                                 │
│  3. Ingestion Time（摄入时间）                                  │
│     ── 数据进入 Flink Source 的时间                             │
│     ── 折中方案，比 Event Time 延迟低，比 Processing Time 准确    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────────────────────────┐
│               三种时间的关系（举例说明）                        │
│                                                             │
│  用户在北京 10:00 浏览了一篇博客文章                            │
│  事件产生 ──→ 网络传输 ──→ 到达 Kafka ──→ Flink 处理         │
│     │                          │                │            │
│     ▼                          ▼                ▼            │
│  Event Time=10:00      Ingestion Time    Processing Time    │
│                           =10:01           =10:02            │
│                                                             │
│  如果网络延迟很大呢？                                          │
│  用户 10:00 浏览，但数据 10:05 才到达 Kafka                    │
│                                                             │
│  如果有多条数据乱序呢？                                        │
│  事件顺序：10:00 → 10:02 → 10:01 → 10:03                    │
│  到达顺序：10:00 → 10:02 → 10:01 → 10:03                    │
│                          ↑                                   │
│                     10:01 的事件比 10:02 晚到！               │
│                                                             │
│  Event Time 可以正确处理这种乱序情况                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 三、水位线（Watermark）

### 3.1 什么是 Watermark？

```
┌─────────────────────────────────────────────────────────────────┐
│                   Watermark 原理                                │
│                                                                 │
│  核心问题：使用 Event Time 时，Flink 怎么知道一个时间窗口的数据   │
│  已经全部到达了？（因为数据可能乱序到达）                          │
│                                                                 │
│  Watermark = 当前最大事件时间 - 允许的最大延迟                    │
│                                                                 │
│  举例：允许 5 秒延迟的 Watermark                                │
│                                                                 │
│  数据到达顺序（事件时间）：                                      │
│  ──(10:00)──(10:03)──(10:01)──(10:05)──(10:02)──(10:08)──→   │
│                                                                 │
│  Watermark 变化：                                               │
│  ── W(09:55)──W(09:58)──W(09:58)──W(10:00)──W(10:00)──W(10:03)│
│                                                                 │
│  解释：                                                          │
│  - 收到 10:00 → Watermark = 10:00 - 5s = 09:55                │
│  - 收到 10:03 → Watermark = 10:03 - 5s = 09:58                │
│  - 收到 10:01 → Watermark = max(10:03,10:01) - 5s = 09:58     │
│  - 收到 10:05 → Watermark = max(10:03,10:05) - 5s = 10:00     │
│  - ...                                                          │
│  - 收到 10:08 → Watermark = 10:08 - 5s = 10:03                │
│                                                                 │
│  当 Watermark >= 窗口结束时间时，窗口触发计算！                   │
│  例如：[10:00, 10:05) 窗口在 Watermark 到达 10:05 时触发        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────────────────────────┐
│              Watermark 与窗口触发                              │
│                                                             │
│  窗口 [10:00, 10:05)（统计 10:00-10:05 的浏览量）            │
│                                                             │
│  数据：──(10:01)──(10:04)──(10:02)──(10:03)──(10:06)──→    │
│                                              ↑              │
│  Watermark（允许延迟 2s）：                                  │
│  ── W(09:59)──W(10:02)──W(10:02)──W(10:02)──W(10:04)──→   │
│                                                             │
│  W(10:02) < 10:05 → 窗口不触发（可能还有数据没到）           │
│  W(10:04) < 10:05 → 窗口不触发                              │
│  ... 当收到事件时间 10:07 的数据时：                          │
│  W(10:05) >= 10:05 → 窗口触发！计算 [10:00, 10:05) 的结果   │
│                                                             │
│  注意：10:06 的数据会被分配到 [10:05, 10:10) 窗口             │
│        10:02 的数据虽然比 10:06 晚到，但仍然在正确窗口内        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 Watermark 策略

```python
from pyflink.datastream import WatermarkStrategy
from pyflink.common import Duration
import json

# 有界乱序 Watermark（推荐）
# 允许最多 5 秒的延迟
watermark_strategy = WatermarkStrategy \
    .for_bounded_out_of_orderness(Duration.of_seconds(5)) \
    .with_timestamp_assigner(
        # 从事件中提取事件时间戳
        lambda event, timestamp: event["timestamp"]
    )

# 使用 Watermark 策略
stream = env.from_source(
    kafka_source,
    watermark_strategy=watermark_strategy,
    source_name="Kafka-BlogEvents"
)
```

## 四、窗口类型

### 4.1 滚动窗口（Tumbling Window）

```
┌─────────────────────────────────────────────────────────────┐
│                 滚动窗口（Tumbling Window）                   │
│                                                             │
│  特点：窗口大小固定，不重叠                                    │
│                                                             │
│  数据流：──●──●──●──●──●──●──●──●──●──●──●──●──●──●──→    │
│                                                             │
│  窗口大小 = 5 分钟：                                         │
│  ┌───────────┬───────────┬───────────┬───────────┐         │
│  │  [0, 5)   │  [5, 10)  │ [10, 15)  │ [15, 20)  │         │
│  │  5条数据   │  4条数据   │  3条数据   │  3条数据   │         │
│  └───────────┴───────────┴───────────┴───────────┘         │
│                                                             │
│  每条数据只属于一个窗口，窗口之间没有重叠                      │
│                                                             │
│  适用场景：                                                    │
│  - 每小时的 PV 统计                                           │
│  - 每天的注册用户数                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 滑动窗口（Sliding Window）

```
┌─────────────────────────────────────────────────────────────┐
│                 滑动窗口（Sliding Window）                    │
│                                                             │
│  特点：窗口大小固定，按滑动步长移动，窗口之间有重叠            │
│                                                             │
│  数据流：──●──●──●──●──●──●──●──●──●──●──●──●──●──●──→    │
│                                                             │
│  窗口大小 = 10 分钟，滑动步长 = 5 分钟：                      │
│  ┌─────────────┐                                             │
│  │   [0, 10)   │                                             │
│  └──────┬──────┘                                             │
│         │ 5分钟                                                │
│         ▼                                                    │
│       ┌─────────────┐                                        │
│       │   [5, 15)   │ ← 与上一个窗口重叠了 [5, 10)           │
│       └──────┬──────┘                                        │
│              │ 5分钟                                           │
│              ▼                                               │
│            ┌─────────────┐                                   │
│            │  [10, 20)   │                                   │
│            └─────────────┘                                   │
│                                                             │
│  适用场景：                                                    │
│  - 每 1 分钟统计最近 5 分钟的热门文章                         │
│  - 每 10 秒统计最近 1 分钟的流量                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 会话窗口（Session Window）

```
┌─────────────────────────────────────────────────────────────┐
│                 会话窗口（Session Window）                    │
│                                                             │
│  特点：没有固定大小，按活动间隙划分                            │
│                                                             │
│  数据流：──●──●──●─── ── ── ──●──●─── ── ── ── ──●──→      │
│            活跃期    间隙   活跃期      大间隙    活跃期      │
│                                                             │
│  会话间隔（Gap）= 30 分钟：                                   │
│  ┌──────────┐   ┌──────────┐              ┌──┐             │
│  │ Session1 │   │ Session2 │              │S3│             │
│  │ 3条数据   │   │ 2条数据   │              │1条│             │
│  └──────────┘   └──────────┘              └──┘             │
│                                                             │
│  如果两条数据之间的间隔超过 Gap，就划分为两个会话              │
│                                                             │
│  适用场景：                                                    │
│  - 分析用户的访问会话（一次连续浏览行为）                      │
│  - 计算每个会话的停留时长                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.4 三种窗口对比

| 特性 | 滚动窗口 | 滑动窗口 | 会话窗口 |
|------|---------|---------|---------|
| **窗口大小** | 固定 | 固定 | 不固定 |
| **是否重叠** | 不重叠 | 可重叠 | 不重叠 |
| **触发方式** | 时间到达 | 时间到达 | 间隙超时 |
| **参数** | 窗口大小 | 窗口大小 + 滑动步长 | 会话间隔 |
| **典型场景** | 每小时统计 | 最近 N 分钟统计 | 用户会话分析 |

## 五、窗口函数

窗口切好之后，怎么对窗口内的数据做计算？这就是窗口函数的作用。

### 5.1 增量聚合函数

```
┌─────────────────────────────────────────────────────────────┐
│                 增量聚合 vs 全窗口聚合                        │
│                                                             │
│  增量聚合（Reduce / Aggregate）：                             │
│  ── 每来一条数据就更新中间结果，窗口结束时输出最终结果          │
│                                                             │
│  ┌──────────────────────────────────────────────────┐      │
│  │  窗口 [0, 5):                                    │      │
│  │                                                  │      │
│  │  ●(value=3) → result=3                           │      │
│  │  ●(value=5) → result=8  (3+5)                   │      │
│  │  ●(value=2) → result=10 (8+2)                   │      │
│  │  窗口结束 → 输出 result=10                       │      │
│  │                                                  │      │
│  │  内存占用：O(1) — 只需要保存中间结果               │      │
│  └──────────────────────────────────────────────────┘      │
│                                                             │
│  全窗口聚合（ProcessWindowFunction）：                       │
│  ── 收集窗口内所有数据，窗口结束时一起处理                    │
│                                                             │
│  ┌──────────────────────────────────────────────────┐      │
│  │  窗口 [0, 5):                                    │      │
│  │                                                  │      │
│  │  ●(value=3) → buffer=[3]                        │      │
│  │  ●(value=5) → buffer=[3, 5]                     │      │
│  │  ●(value=2) → buffer=[3, 5, 2]                  │      │
│  │  窗口结束 → 对 [3, 5, 2] 做计算                  │      │
│  │                                                  │      │
│  │  内存占用：O(N) — 需要保存所有数据                 │      │
│  └──────────────────────────────────────────────────┘      │
│                                                             │
│  建议：能增量就增量，需要全量数据时才用全窗口聚合              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 代码示例

```python
from pyflink.datastream import StreamExecutionEnvironment, Window
from pyflink.datastream.functions import WindowFunction
from pyflink.common import Duration
from pyflink.datastream import WatermarkStrategy

env = StreamExecutionEnvironment.get_execution_environment()

# 模拟数据平台浏览事件
events = [
    {"postId": 1, "timestamp": 1713200000000},  # 10:00:00
    {"postId": 2, "timestamp": 1713200001000},  # 10:00:01
    {"postId": 1, "timestamp": 1713200002000},  # 10:00:02
    {"postId": 1, "timestamp": 1713200003000},  # 10:00:03
    {"postId": 3, "timestamp": 1713200004000},  # 10:00:04
    {"postId": 1, "timestamp": 1713200030000},  # 10:00:30
    {"postId": 2, "timestamp": 1713200031000},  # 10:00:31
]

# 设置 Watermark（允许 5 秒延迟）
watermark_strategy = WatermarkStrategy \
    .for_bounded_out_of_orderness(Duration.of_seconds(5)) \
    .with_timestamp_assigner(lambda e, ts: e["timestamp"])

stream = env.from_collection(events) \
    .assign_timestamps_and_watermarks(watermark_strategy)

# ─── 增量聚合：统计每个窗口内每篇文章的浏览量 ───
tumbling_result = stream \
    .map(lambda e: (e["postId"], 1)) \
    .key_by(lambda x: x[0]) \
    .window(TumblingEventTimeWindows.of(Duration.of_minutes(1))) \
    .sum(1)

tumbling_result.print("滚动窗口(1分钟)浏览量:")
```

### 5.3 获取窗口信息

```python
from pyflink.datastream.functions import ProcessWindowFunction
from pyflink.datastream import Window

class WindowInfoPrinter(ProcessWindowFunction):
    """全窗口聚合函数：可以访问窗口的元信息"""

    def process(self, key, context, elements):
        # 获取窗口信息
        window = context.window()

        # 输出：窗口开始时间、结束时间、窗口内的数据
        yield (
            key,
            window.start,
            window.end,
            len(elements),
            sum(1 for _ in elements)
        )

# 使用
result = stream \
    .key_by(lambda e: e["postId"]) \
    .window(TumblingEventTimeWindows.of(Duration.of_minutes(1))) \
    .process(WindowInfoPrinter())
```

## 六、迟到数据处理

### 6.1 问题：数据迟到了怎么办？

```
┌─────────────────────────────────────────────────────────────┐
│                 迟到数据问题                                  │
│                                                             │
│  窗口 [10:00, 10:05) 已经触发计算完毕                        │
│                                                             │
│  此时来了一条事件时间为 10:03 的数据（迟到数据）                │
│                                                             │
│  默认情况下：这条数据会被直接丢弃！                            │
│                                                             │
│  解决方案：                                                  │
│  1. 增大 Watermark 延迟（但会增加整体延迟）                    │
│  2. 使用 allowedLateness 允许窗口继续接收迟到数据              │
│  3. 使用侧输出（Side Output）收集迟到数据                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 allowedLateness

```python
from pyflink.datastream import StreamExecutionEnvironment
from pyflink.common import Duration

# 允许迟到 1 分钟
result = stream \
    .key_by(lambda e: e["postId"]) \
    .window(TumblingEventTimeWindows.of(Duration.of_minutes(5))) \
    .allowed_lateness(Duration.of_minutes(1))  # 窗口结束后再等 1 分钟
    .sum(1)
```

```
┌─────────────────────────────────────────────────────────────┐
│              allowedLateness 工作原理                          │
│                                                             │
│  Watermark 到达 10:05 → 窗口 [10:00, 10:05) 首次触发         │
│                                                             │
│  但窗口不会立即清除！                                        │
│  在 Watermark 到达 10:06 之前（即 1 分钟内），                │
│  如果有事件时间在 [10:00, 10:05) 的迟到数据到达，             │
│  窗口会重新计算并输出更新后的结果。                             │
│                                                             │
│  Watermark 到达 10:06 → 窗口真正清除，不再接收迟到数据         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 侧输出（Side Output）

```python
from pyflink.datastream import StreamExecutionEnvironment
from pyflink.datastream import OutputTag, ProcessWindowFunction
from pyflink.common import Duration

# 定义侧输出标签
late_data_tag = OutputTag("late-events", dict)

# 处理主流程，迟到数据发送到侧输出
result = stream \
    .key_by(lambda e: e["postId"]) \
    .window(TumblingEventTimeWindows.of(Duration.of_minutes(5))) \
    .allowed_lateness(Duration.of_minutes(1)) \
    .side_output_late_data(late_data_tag)  # 超过 allowedLateness 的数据
    .sum(1)

# 获取迟到数据流
late_stream = result.get_side_output(late_data_tag)

# 迟到数据可以写入单独的 Topic 或文件
late_stream.print("迟到数据:")
```

```
┌─────────────────────────────────────────────────────────────┐
│                 迟到数据的完整处理链路                         │
│                                                             │
│  数据到达                                                     │
│    │                                                        │
│    ├── 事件时间在当前窗口内 → 正常处理                         │
│    │                                                        │
│    ├── 事件时间在已关闭窗口内，但在 allowedLateness 内         │
│    │   → 重新触发窗口计算                                     │
│    │                                                        │
│    └── 事件时间超过 allowedLateness                           │
│        → 发送到侧输出（Side Output）                          │
│        → 可以写入单独的 Kafka Topic 或文件                    │
│        → 后续可以批处理修正                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 七、数据分析项目：每 5 分钟统计热门文章 Top 10

```python
# demo_hot_articles.py
"""
每 5 分钟统计热门文章 Top 10
使用滑动窗口：窗口大小 5 分钟，滑动步长 1 分钟
"""
import json
from pyflink.datastream import StreamExecutionEnvironment, Window
from pyflink.datastream import WatermarkStrategy, Duration
from pyflink.datastream.connectors import KafkaSource
from pyflink.common.serialization import SimpleStringSchema
from pyflink.datastream.functions import ProcessWindowFunction, KeyedProcessFunction
from pyflink.datastream.state import ValueStateDescriptor


def main():
    env = StreamExecutionEnvironment.get_execution_environment()
    env.enable_checkpointing(30000)

    # 创建 Kafka Source
    kafka_source = KafkaSource.builder() \
        .set_bootstrap_servers("localhost:9092") \
        .set_topics("blog-events") \
        .set_group_id("flink-hot-articles") \
        .set_value_deserialization_schema(SimpleStringSchema()) \
        .build()

    # Watermark 策略：允许 10 秒延迟
    watermark_strategy = WatermarkStrategy \
        .for_bounded_out_of_orderness(Duration.of_seconds(10)) \
        .with_timestamp_assigner(
            lambda event, ts: json.loads(event).get("timestamp", ts)
        )

    # 从 Kafka 读取
    stream = env.from_source(
        kafka_source,
        watermark_strategy=watermark_strategy,
        source_name="Kafka-BlogEvents"
    )

    # 解析 JSON，过滤浏览事件
    view_events = stream \
        .map(lambda v: json.loads(v)) \
        .filter(lambda e: e.get("type") == "view")

    # 分配时间戳和 Watermark
    timed_events = view_events.assign_timestamps_and_watermarks(
        WatermarkStrategy.for_bounded_out_of_orderness(Duration.of_seconds(10))
            .with_timestamp_assigner(lambda e, ts: e.get("timestamp", ts))
    )

    # 按 postId 分组，使用滑动窗口统计浏览量
    # 窗口大小 5 分钟，滑动步长 1 分钟
    article_views = timed_events \
        .map(lambda e: (e["postId"], 1)) \
        .key_by(lambda x: x[0]) \
        .window(SlidingEventTimeWindows.of(
            Duration.of_minutes(5),
            Duration.of_minutes(1)
        )) \
        .sum(1)

    article_views.print("文章浏览量(滑动窗口):")

    env.execute("热门文章 Top 10")


if __name__ == "__main__":
    main()
```

## 八、面试题

### Q1：Watermark 是如何处理乱序数据的？

**参考答案：**

Watermark 通过引入"允许的最大延迟"来处理乱序数据。Watermark 是一个时间戳，等于当前已接收到的最大事件时间减去允许的延迟。

工作原理：
1. 当 Watermark 的时间戳超过某个窗口的结束时间时，该窗口触发计算
2. 如果有事件时间小于当前 Watermark 的数据到达，会被视为迟到数据
3. 迟到数据默认被丢弃，但可以通过 `allowedLateness` 设置额外等待时间

关键点：
- Watermark 是单调递增的（取所有已接收数据中最大的事件时间减去延迟）
- Watermark 延迟越大，结果越准确，但整体延迟也越高
- 需要根据实际业务的数据乱序程度来设置合理的延迟值

### Q2：滚动窗口和滑动窗口的区别？什么场景用什么窗口？

**参考答案：**

**滚动窗口**：窗口大小固定，不重叠。参数只有一个（窗口大小）。适合固定周期的统计，如每小时 PV、每天的注册用户数。

**滑动窗口**：窗口大小固定，按步长滑动，有重叠。参数有两个（窗口大小和滑动步长）。适合需要平滑统计的场景，如"每 1 分钟统计最近 5 分钟的流量"。

选择建议：
- 需要精确的周期统计 → 滚动窗口
- 需要平滑的实时监控 → 滑动窗口（步长小，窗口大）
- 需要分析用户行为序列 → 会话窗口

### Q3：如果窗口内数据量太大（数据倾斜），怎么优化？

**参考答案：**

数据倾斜是窗口计算中常见的问题，某些 Key 的数据量远大于其他 Key。

优化方案：
1. **两阶段聚合**：先在局部聚合（Local-Global），减少数据传输量。第一阶段按 Key 局部聚合，第二阶段全局聚合。
2. **增加并行度**：提高窗口算子的并行度，让数据更均匀分布
3. **预聚合**：在 keyBy 之前先做一层局部聚合，减少下游数据量
4. **使用增量聚合函数**：避免使用 ProcessWindowFunction（需要缓存所有数据），优先使用 reduce/aggregate

## 九、总结

本篇我们学习了窗口和时间语义，关键要点：

1. **时间语义**：Event Time（最准确）> Ingestion Time > Processing Time
2. **Watermark**：告诉 Flink "不会再有比这个时间更早的数据了"，触发窗口计算
3. **窗口类型**：滚动窗口（不重叠）、滑动窗口（重叠）、会话窗口（按间隙划分）
4. **窗口函数**：增量聚合（内存 O(1)）vs 全窗口聚合（内存 O(N)）
5. **迟到数据**：allowedLateness（重新触发）+ Side Output（收集丢弃数据）
