# 01 - OLAP 与列式存储概述：为什么需要 ClickHouse？

## 一、从一个真实问题说起

假设你的数据平台已经运营了一年多，积累了大量的访问日志。老板（其实就是你自己）想看看数据报表：

- 每天的 PV/UV 趋势是什么？
- 哪些文章最受欢迎？
- 用户主要来自哪些城市？
- 移动端和 PC 端的比例是多少？

你本能地打开 MySQL，写了一条查询：

```sql
SELECT DATE(created_at) AS date, COUNT(*) AS pv, COUNT(DISTINCT user_id) AS uv
FROM access_logs
WHERE created_at >= '2026-01-01'
GROUP BY DATE(created_at)
ORDER BY date;
```

几百条日志时，秒回。几十万条时，等了 5 秒。几百万条时... 咖啡都凉了还没出结果。

**为什么 MySQL 查分析这么慢？** 这就是 OLTP 和 OLAP 的根本区别。

## 二、OLTP vs OLAP：两种完全不同的场景

### 2.1 核心区别

```
┌──────────────────────────────────────────────────────────────────┐
│                    OLTP vs OLAP 对比                              │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  OLTP（联机事务处理）              OLAP（联机分析处理）             │
│  On-Line Transaction Processing     On-Line Analytical Processing │
│  ─────────────────────────         ─────────────────────────     │
│                                                                  │
│  ┌──────────────┐                  ┌──────────────┐              │
│  │ 操作：增删改查 │                  │ 操作：聚合分析 │              │
│  │ 特点：高频小额 │                  │ 特点：低频大量 │              │
│  │ 用户：终端用户 │                  │ 用户：分析师/  │              │
│  │              │                  │       决策者  │              │
│  │ 代表：MySQL   │                  │ 代表：ClickHouse│             │
│  │       PostgreSQL                │       Doris   │              │
│  └──────────────┘                  └──────────────┘              │
│                                                                  │
│  数据平台中的 OLTP 场景：               数据平台中的 OLAP 场景：            │
│  - 用户注册/登录                    - 每日 PV/UV 统计              │
│  - 发布文章                         - 热门文章排行                 │
│  - 写评论                           - 用户地域分布                 │
│  - 点赞/收藏                        - 流量趋势分析                 │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 2.2 用一张表总结

| 维度 | OLTP | OLAP |
|------|------|------|
| **核心操作** | 增删改查（CRUD） | 聚合分析（GROUP BY） |
| **数据量** | 单次操作少量数据 | 一次扫描海量数据 |
| **响应时间** | 毫秒级 | 秒级~分钟级 |
| **并发** | 高并发（几千~几万 QPS） | 低并发（几十个查询） |
| **数据规模** | GB 级 | TB~PB 级 |
| **事务** | 强 ACID 事务 | 不需要事务 |
| **索引** | B+ 树索引 | 列存储 + 稀疏索引 |
| **代表产品** | MySQL、PostgreSQL、Oracle | ClickHouse、Doris、Druid |

## 三、行存储 vs 列存储：为什么分析场景需要列式？

这是理解 ClickHouse 的**最核心概念**。

### 3.1 行存储（Row-oriented）

传统数据库（MySQL、PostgreSQL）使用行存储，每条记录的各列连续存储：

```
┌──────────────────────────────────────────────────────────────────┐
│                    行存储结构（Row-oriented）                      │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  磁盘上的物理布局：                                               │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ [id=1][user=张三][city=北京][device=iOS][time=08:30][action=view] │
│  │ [id=2][user=李四][city=上海][device=Android][time=08:31][action=view]│
│  │ [id=3][user=王五][city=广州][device=iOS][time=08:32][action=like]│
│  │ [id=4][user=赵六][city=北京][device=PC][time=08:33][action=view]│
│  │ ...                                                      │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  适合 OLTP：                                                     │
│  SELECT * FROM users WHERE id = 1                                │
│  → 找到第 1 行，读取整行数据，一次 IO 搞定                         │
│                                                                  │
│  不适合 OLAP：                                                   │
│  SELECT city, COUNT(*) FROM logs GROUP BY city                   │
│  → 需要读取每一行的 city 列，跳过其他所有列                        │
│  → 大量无效 IO！                                                 │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 3.2 列存储（Column-oriented）

列式数据库（ClickHouse、Doris）将每一列单独存储：

```
┌──────────────────────────────────────────────────────────────────┐
│                    列存储结构（Column-oriented）                    │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  磁盘上的物理布局：                                               │
│                                                                  │
│  id 列:     [1, 2, 3, 4, 5, 6, 7, ...]                          │
│  user 列:   [张三, 李四, 王五, 赵六, ...]                         │
│  city 列:   [北京, 上海, 广州, 北京, ...]                         │
│  device 列: [iOS, Android, iOS, PC, ...]                         │
│  time 列:   [08:30, 08:31, 08:32, 08:33, ...]                   │
│  action 列: [view, view, like, view, ...]                        │
│                                                                  │
│  适合 OLAP：                                                     │
│  SELECT city, COUNT(*) FROM logs GROUP BY city                   │
│  → 只读取 city 列，其他列完全不用碰                               │
│  → IO 量减少到 1/N！（N = 列数）                                 │
│                                                                  │
│  不适合 OLTP：                                                   │
│  SELECT * FROM users WHERE id = 1                                │
│  → 需要从每个列文件中分别读取第 1 个值                             │
│  → 多次随机 IO，反而更慢                                          │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 3.3 直观对比

```
┌──────────────────────────────────────────────────────────────────┐
│                  行存储 vs 列存储：IO 对比                         │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  假设：1000 万条日志，每条 6 个字段，每条记录 200 字节             │
│  查询：SELECT city, COUNT(*) FROM logs GROUP BY city              │
│                                                                  │
│  行存储需要读取的数据量：                                          │
│  ┌──────────────────────────────────────────────────────┐       │
│  │ 1000 万行 × 200 字节 = 2 GB                          │       │
│  │ 其中 city 列只有 ~100 MB，其余 1.9 GB 都是无效读取     │       │
│  │ 有效读取率：5%                                        │       │
│  └──────────────────────────────────────────────────────┘       │
│                                                                  │
│  列存储需要读取的数据量：                                          │
│  ┌──────────────────────────────────────────────────────┐       │
│  │ 只读取 city 列：~100 MB                               │       │
│  │ IO 减少了 95%！                                       │       │
│  │ 有效读取率：100%                                       │       │
│  └──────────────────────────────────────────────────────┘       │
│                                                                  │
│  结论：分析查询场景下，列存储的 IO 效率碾压行存储                   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 3.4 列存储的额外优势：压缩率极高

同一列的数据类型相同、取值范围相近，压缩效果远好于行存储：

```
┌──────────────────────────────────────────────────────────────────┐
│                    列存储的高压缩率                                 │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  city 列：[北京, 上海, 广州, 北京, 深圳, 上海, 北京, ...]         │
│                                                                  │
│  使用字典编码（Dictionary Encoding）：                              │
│  ┌────────────────────────────────────────┐                      │
│  │ 字典表：                                 │                      │
│  │   0 → 北京                               │                      │
│  │   1 → 上海                               │                      │
│  │   2 → 广州                               │                      │
│  │   3 → 深圳                               │                      │
│  │                                          │                      │
│  │ 原始数据（压缩后）：                        │                      │
│  │   [0, 1, 2, 0, 3, 1, 0, ...]             │                      │
│  │   每个值只需 1~2 个字节                     │                      │
│  │   而不是存储完整的字符串                     │                      │
│  └────────────────────────────────────────┘                      │
│                                                                  │
│  实际压缩效果：                                                   │
│  - 行存储：1 TB 原始数据 → 压缩后 ~400 GB（3-5x）                 │
│  - 列存储：1 TB 原始数据 → 压缩后 ~50-100 GB（10-20x）            │
│  → 磁盘空间省了，读取速度也更快！                                   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 五、ClickHouse 的核心优势

### 5.1 向量化执行（Vectorized Execution）

传统数据库逐行处理数据，ClickHouse 使用 SIMD 指令**批量处理数据**：

```
┌──────────────────────────────────────────────────────────────────┐
│                    逐行执行 vs 向量化执行                           │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  传统逐行执行：                                                   │
│  ┌──────────────────────────────────────────────────────┐       │
│  │ for row in table:                                    │       │
│  │     result += row.city == "北京"  # 每次只处理一行     │       │
│  │                                                      │       │
│  │ CPU 寄存器利用率低，大量时间花在循环控制上              │       │
│  └──────────────────────────────────────────────────────┘       │
│                                                                  │
│  ClickHouse 向量化执行：                                          │
│  ┌──────────────────────────────────────────────────────┐       │
│  │ # 一次加载 64/256 个值到 CPU 向量寄存器                 │       │
│  │ batch = [北京, 上海, 广州, 北京, 深圳, 上海, ...]     │       │
│  │ result = batch == "北京"  # CPU 一条指令处理一批数据    │       │
│  │                                                      │       │
│  │ CPU 寄存器利用率高，吞吐量提升 10-100 倍                │       │
│  └──────────────────────────────────────────────────────┘       │
│                                                                  │
│  类比：                                                          │
│  逐行执行 = 一个人搬砖，一次搬一块                                 │
│  向量化执行 = 开叉车，一次搬一车                                   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 5.2 暴力扫描（Brute Force）

ClickHouse 不追求精细的索引优化，而是通过**并行 + 向量化 + 压缩**，直接暴力扫描全表：

```
┌──────────────────────────────────────────────────────────────────┐
│                  ClickHouse 的"暴力美学"                           │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  MySQL 的思路：建索引，通过索引精准定位数据                        │
│  - 需要精心设计索引                                               │
│  - 索引多了写入变慢                                               │
│  - 复杂查询索引可能失效                                           │
│                                                                  │
│  ClickHouse 的思路：暴力扫描，用速度弥补                            │
│  - 不依赖复杂索引                                                 │
│  - 利用多核 CPU 并行扫描                                           │
│  - 利用 SIMD 指令向量化处理                                       │
│  - 利用列存储减少 IO                                             │
│  - 利用压缩减少数据量                                             │
│  → 综合下来，暴力扫描反而比索引查找更快！                           │
│                                                                  │
│  打个比方：                                                      │
│  MySQL = 精心整理的书架，按分类找书                                │
│  ClickHouse = 把所有书铺在操场上，100 个人同时翻                   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 数据平台中的数据分析需求

| 分析需求 | 查询类型 | ClickHouse 优势 |
|---------|---------|----------------|
| 每日 PV/UV 趋势 | `GROUP BY date` + `count()` + `uniq()` | 聚合函数极快 |
| 热门文章排行 | `GROUP BY post_id` + `ORDER BY views DESC` | 排序速度快 |
| 用户地域分布 | `GROUP BY city` | city 列压缩率高 |
| 设备类型分布 | `GROUP BY device` | device 基数低，压缩极佳 |
| 实时流量监控 | `WHERE timestamp > now() - 1h` | 分区裁剪 + 暴力扫描 |

## 七、面试题

### 面试题 1：行存储和列存储的区别是什么？分别适用于什么场景？

**参考答案：**

行存储（Row-oriented）将每条记录的所有列连续存储在磁盘上，代表典型有 MySQL、PostgreSQL。它的优势在于写入一行数据只需要一次连续 IO，适合 OLTP 场景，比如用户注册、发布文章、写评论等操作和写操作。

列存储（Column-oriented）将同一列的所有值连续存储，代表产品有 ClickHouse、Doris、Parquet 文件格式。它的优势在于分析查询时只需要读取涉及的列，大幅减少 IO 量。同时，同类型数据连续存储有利于压缩（字典编码、RLE 等），压缩率通常比行存储高 5-10 倍。适合 OLAP 场景，比如统计报表、趋势分析、多维聚合等。

**核心取舍**：
- 行存储：写快、点查快，分析慢
- 列存储：分析快、压缩率高，点查慢、写入需要攒批

---

### 面试题 2：ClickHouse 为什么查询这么快？

**参考答案：**

ClickHouse 的查询速度来自几个层面的优化：

**存储层**：
1. **列式存储**：分析查询只读取需要的列，IO 量大幅减少
2. **高压缩率**：同类型数据连续存储，配合 LZ4/ZSTD 压缩算法，压缩率可达 10-20 倍
3. **稀疏索引**：MergeTree 的主键索引是稀疏的，占用空间小，查找效率高
4. **分区裁剪**：按时间分区，查询时自动跳过不相关的分区

**计算层**：
5. **向量化执行**：利用 CPU SIMD 指令，一次处理一批数据，而不是逐行处理
6. **多线程并行**：自动利用所有 CPU 核心，并行扫描和处理数据
7. **暴力扫描哲学**：不依赖复杂索引，通过并行 + 向量化 + 压缩直接扫描全表

**工程层**：
8. **C++ 编写**：贴近硬件，无 GC 开销
9. **内存映射**：利用操作系统 Page Cache，热数据自动缓存在内存
10. **JIT 编译**：对部分查询进行即时编译优化

---

### 面试题 3：ClickHouse 和 Apache Doris 如何选择？

**参考答案：**

两者都是优秀的 OLAP 数据库，选择取决于具体场景：

**选 ClickHouse**：
- 单表聚合查询为主（如日志分析、大宽表扫描）
- 对查询延迟要求极高
- 数据压缩率要求高
- 单表数据量大（百亿级）
- 不需要复杂的 JOIN 操作

**选 Apache Doris**：
- 多表 JOIN 查询多（如星型/雪花模型）
- 需要实时高频写入（Doris 的流式导入更成熟）
- 需要高并发查询（Doris 的 MPP 架构更适合）
- 团队熟悉 MySQL 协议（Doris 兼容 MySQL 协议）

**一句话总结**：大宽表扫描选 ClickHouse，多表关联选 Doris。
