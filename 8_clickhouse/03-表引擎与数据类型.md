# 03 - 表引擎与数据类型：ClickHouse 的数据管理核心

## 一、表引擎是什么？

在 MySQL 里，你可能听说过 InnoDB、MyISAM 这些"存储引擎"。ClickHouse 的"表引擎"概念类似，但更加丰富和灵活。

**表引擎决定了：**
- 数据如何存储在磁盘上
- 数据如何读取
- 是否支持索引
- 是否支持副本
- 是否支持分区
- 并发访问能力

```
┌──────────────────────────────────────────────────────────────────┐
│                    表引擎的重要性                                   │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  建表时不指定引擎？报错！                                         │
│                                                                  │
│  CREATE TABLE test (id UInt64, name String);                     │
│  → Code: 62. DB::Exception: Missing engine.                      │
│                                                                  │
│  ClickHouse 要求每张表必须显式指定引擎：                           │
│                                                                  │
│  CREATE TABLE test (id UInt64, name String)                      │
│  ENGINE = MergeTree()     ← 必须指定！                           │
│  ORDER BY id;                                                     │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 三、MergeTree 引擎详解

MergeTree 是 ClickHouse 最核心的表引擎，也是绝大多数场景的首选。

### 3.1 数据存储结构

```
┌──────────────────────────────────────────────────────────────────┐
│                    MergeTree 数据存储结构                           │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  磁盘上的目录结构：                                               │
│                                                                  │
│  demo/access_logs/                                               │
│  ├── 202604_1_1_0/          ← 分区目录（2026年4月，版本1）        │
│  │   ├── timestamp.bin      ← timestamp 列的数据文件            │
│  │   ├── user_id.bin        ← user_id 列的数据文件              │
│  │   ├── city.bin           ← city 列的数据文件                 │
│  │   ├── ...                                                   │
│  │   ├── primary.idx        ← 主键索引文件（稀疏索引）           │
│  │   ├── checksums.txt      ← 校验和文件                        │
│  │   └── columns.txt        ← 列信息文件                        │
│  │                                                              │
│  ├── 202605_2_2_0/          ← 分区目录（2026年5月）              │
│  │   ├── ...                                                   │
│  │                                                              │
│  └── 202604_3_3_0/          ← 后台合并后生成的新分区              │
│      ├── ...                                                   │
│                                                              │
│  关键概念：                                                      │
│  - Partition（分区）：按 PARTITION BY 拆分数据                    │
│  - Part（数据块）：每次 INSERT 生成一个 Part                      │
│  - 后台线程会自动合并小 Part 为大 Part                            │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 3.2 分区（PARTITION BY）

分区是将数据按某个规则拆分到不同的目录中：

```
┌──────────────────────────────────────────────────────────────────┐
│                    分区示意图                                       │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  PARTITION BY toYYYYMMDD(timestamp)                              │
│                                                                  │
│  写入数据后，ClickHouse 自动按日期分区：                           │
│                                                                  │
│  20260401_1_1_0/        ← 4月1日的数据                           │
│  ├── timestamp.bin                                              │
│  ├── user_id.bin                                                │
│  └── ...                                                       │
│                                                                  │
│  20260402_2_2_0/        ← 4月2日的数据                           │
│  ├── timestamp.bin                                              │
│  ├── user_id.bin                                                │
│  └── ...                                                       │
│                                                                  │
│  20260403_3_3_0/        ← 4月3日的数据                           │
│  ├── ...                                                       │
│                                                                  │
│  查询 "WHERE timestamp >= '2026-04-02'" 时：                     │
│  → ClickHouse 自动跳过 20260401 分区，只扫描 20260402 和之后     │
│  → 这叫"分区裁剪"（Partition Pruning）                           │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**分区策略建议：**

| 分区策略 | 适用场景 | 示例 |
|---------|---------|------|
| 按天分区 | 日志数据、时序数据 | `PARTITION BY toYYYYMMDD(timestamp)` |
| 按月分区 | 数据量不是特别大 | `PARTITION BY toYYYYMM(timestamp)` |
| 按列表分区 | 枚举值（如地区） | `PARTITION BY city` |
| 不分区 | 小数据量（<1亿） | 不写 PARTITION BY |

**注意**：分区粒度不要太细（比如按小时），否则会产生大量小文件，影响性能。

### 3.3 排序（ORDER BY）—— 最核心的配置

```
┌──────────────────────────────────────────────────────────────────┐
│                    ORDER BY 的重要性                               │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ORDER BY 是 MergeTree 最核心的配置！                             │
│  它决定了：                                                      │
│  1. 数据在磁盘上的物理排序方式                                     │
│  2. 主键索引的构建方式                                            │
│  3. 查询时数据裁剪的效率                                          │
│                                                                  │
│  ┌──────────────────────────────────────────────────────┐       │
│  │ ORDER BY (timestamp, user_id, post_id)               │       │
│  │                                                      │       │
│  │ 数据按 timestamp 排序，相同 timestamp 按 user_id 排序  │       │
│  │ 相同 user_id 按 post_id 排序                          │       │
│  │                                                      │       │
│  │ 稀疏索引记录的是每个排序键的边界值                      │       │
│  │ 查询时利用索引快速定位到数据范围                        │       │
│  └──────────────────────────────────────────────────────┘       │
│                                                                  │
│  排序键设计原则：                                                 │
│  1. 查询条件中最常用的字段放前面（如时间范围）                     │
│  2. 经常 GROUP BY 的字段放后面                                   │
│  3. 基数太高的字段（如 user_id）不建议单独放在前面                  │
│  4. 排序键的顺序很重要！                                          │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 3.4 稀疏索引（Sparse Index）

MergeTree 的索引不是 B+ 树，而是一种**稀疏索引**：

```
┌──────────────────────────────────────────────────────────────────┐
│                    稀疏索引工作原理                                 │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  数据文件（按 ORDER BY 排序后）：                                  │
│  ┌──────┬────────┬───────┬──────┬────────┐                     │
│  │ Row 1│ 08:30  │ 北京  │ 1001 │ ...    │                     │
│  │ Row 2│ 08:31  │ 上海  │ 1002 │ ...    │                     │
│  │ ...  │ ...    │ ...   │ ...  │ ...    │                     │
│  │ Row 8192 │ 10:15 │ 北京 │ 1050 │ ... │ ← 索引粒度边界       │
│  │ Row 8193 │ 10:16 │ 上海 │ 1051 │ ... │                     │
│  │ ...  │ ...    │ ...   │ ...  │ ...    │                     │
│  │ Row 16384│ 12:00 │ 广州 │ 1100 │ ... │ ← 索引粒度边界       │
│  └──────┴────────┴───────┴──────┴────────┘                     │
│                                                                  │
│  稀引索引（primary.idx）：                                       │
│  ┌──────────────────────────────────┐                           │
│  │ [08:30, 1001, 1]  → offset=0    │ ← 指向 Row 1              │
│  │ [10:15, 1050, 5]  → offset=8192 │ ← 指向 Row 8192          │
│  │ [12:00, 1100, 8]  → offset=16384│ ← 指向 Row 16384         │
│  └──────────────────────────────────┘                           │
│                                                                  │
│  默认 index_granularity = 8192（每 8192 行记录一个索引项）         │
│                                                                  │
│  查询 "WHERE timestamp >= '10:00'"：                             │
│  1. 二分查找索引，找到 [10:15] 这个索引项                         │
│  2. 从 Row 8192 开始扫描                                         │
│  3. 不需要从头开始！这就是稀疏索引的作用                           │
│                                                                  │
│  注意：稀疏索引不能精确到每一行，找到索引项后还需要在数据块内扫描   │
│  这就是 ClickHouse "暴力扫描" 的一部分——但配合向量化执行，很快！   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 四、MergeTree 家族详解

### 4.1 MergeTree（基础引擎）

最基本的引擎，只负责数据存储和排序，不做任何额外处理。

```sql
CREATE TABLE demo.access_logs (
    log_id     UInt64,
    user_id    UInt32,
    post_id    UInt32,
    action     String,
    city       String,
    device     String,
    timestamp  DateTime
)
ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(timestamp)
ORDER BY (timestamp, user_id)
SETTINGS index_granularity = 8192;
```

**特点：**
- 保留所有写入的数据，包括重复的
- 后台自动合并小 Part 为大 Part
- 适合作为明细数据存储

### 4.2 ReplacingMergeTree（去重引擎）

自动去除排序键相同的重复数据，保留最后写入的版本：

```
┌──────────────────────────────────────────────────────────────────┐
│                    ReplacingMergeTree 去重原理                      │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ENGINE = ReplacingMergeTree([ver])                              │
│  ORDER BY (user_id, post_id)                                     │
│                                                                  │
│  写入数据（可能有重复）：                                          │
│  ┌────────┬────────┬────────┐                                     │
│  │user_id │post_id │action  │                                     │
│  ├────────┼────────┼────────┤                                     │
│  │ 1001   │ 1      │ view   │ ← 第1次写入                        │
│  │ 1001   │ 1      │ like   │ ← 第2次写入（更新）                │
│  │ 1001   │ 1      │ share  │ ← 第3次写入（再次更新）            │
│  │ 1002   │ 2      │ view   │                                     │
│  └────────┴────────┴────────┘                                     │
│                                                                  │
│  后台合并后（相同排序键只保留最后一条）：                            │
│  ┌────────┬────────┬────────┐                                     │
│  │user_id │post_id │action  │                                     │
│  ├────────┼────────┼────────┤                                     │
│  │ 1001   │ 1      │ share  │ ← 只保留了最后写入的版本             │
│  │ 1002   │ 2      │ view   │                                     │
│  └────────┴────────┴────────┘                                     │
│                                                                  │
│  注意：去重发生在后台合并时，不是写入时！                           │
│  查询时加上 FINAL 关键字可以立即去重：                             │
│  SELECT * FROM table FINAL                                       │
│                                                                  │
│  但 FINAL 会降低查询性能，建议在后台合并后再查询                    │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

```sql
-- 博客文章最终状态表（每篇文章只保留最新版本）
CREATE TABLE demo.posts_latest (
    post_id    UInt32,
    title      String,
    author     String,
    updated_at DateTime
)
ENGINE = ReplacingMergeTree(updated_at)  -- 按更新时间判断版本
ORDER BY post_id;

-- 查询时使用 FINAL 确保去重
SELECT * FROM demo.posts_latest FINAL WHERE post_id = 1;
```

### 4.3 SummingMergeTree（预聚合引擎）

数值列自动求和，非数值列保留最新值：

```
┌──────────────────────────────────────────────────────────────────┐
│                    SummingMergeTree 聚合原理                       │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ENGINE = SummingMergeTree()                                     │
│  ORDER BY (date, city, device)                                   │
│                                                                  │
│  写入数据（同一排序键的多条记录）：                                │
│  ┌───────────┬──────┬────────┬────┬────┐                        │
│  │ date      │ city │ device │ pv │ uv │                        │
│  ├───────────┼──────┼────────┼────┼────┤                        │
│  │2026-04-01 │ 北京 │ PC     │ 50 │ 30 │                        │
│  │2026-04-01 │ 北京 │ PC     │ 20 │ 15 │                        │
│  │2026-04-01 │ 北京 │ PC     │ 10 │ 8  │                        │
│  └───────────┴──────┴────────┴────┴────┘                        │
│                                                                  │
│  后台合并后（数值列自动求和）：                                    │
│  ┌───────────┬──────┬────────┬────┬────┐                        │
│  │ date      │ city │ device │ pv │ uv │                        │
│  ├───────────┼──────┼────────┼────┼────┤                        │
│  │2026-04-01 │ 北京 │ PC     │ 80 │ 53 │ ← pv=50+20+10, uv=30+15+8│
│  └───────────┴──────┴────────┴────┴────┘                        │
│                                                                  │
│  适合场景：                                                      │
│  - 实时统计指标的预聚合                                           │
│  - 配合物化视图，自动聚合明细数据                                  │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

```sql
-- 每日访问统计表
CREATE TABLE demo.daily_stats (
    date        Date,
    city        String,
    device      String,
    pv          UInt64,       -- 数值列：自动求和
    uv          UInt64,       -- 数值列：自动求和
    comment_cnt UInt64        -- 数值列：自动求和
)
ENGINE = SummingMergeTree()
PARTITION BY toYYYYMM(date)
ORDER BY (date, city, device);
```

### 4.4 AggregatingMergeTree（复杂聚合引擎）

支持任意聚合函数（avg、uniq、sum 等），通常配合 `AggregateFunction` 类型使用：

```sql
-- 使用 AggregateFunction 类型的聚合表
CREATE TABLE demo.hourly_stats (
    date     Date,
    hour     UInt8,
    city     String,
    pv       SimpleAggregateFunction(sum, UInt64),
    uv       SimpleAggregateFunction(sum, UInt64),
    users    AggregateFunction(uniq, UInt64)  -- 去重用户数
)
ENGINE = AggregatingMergeTree()
PARTITION BY toYYYYMM(date)
ORDER BY (date, hour, city);
```

### 4.5 引擎对比总结

| 引擎 | 用途 | 数值处理 | 去重 | 典型场景 |
|------|------|---------|------|---------|
| MergeTree | 明细存储 | 不处理 | 不去重 | 原始日志表 |
| ReplacingMergeTree | 去重 | 不处理 | 保留最新 | 文章最新状态 |
| SummingMergeTree | 预聚合 | 自动求和 | 不去重 | 统计指标表 |
| AggregatingMergeTree | 复杂聚合 | 自定义函数 | 不去重 | 复杂聚合指标 |

## 五、数据类型详解

### 5.1 数值类型

```
┌──────────────────────────────────────────────────────────────────┐
│                    数值类型                                        │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  整数类型（范围越大占用的空间越大）：                               │
│  ┌──────────┬────────────┬──────────┐                           │
│  │ 类型     │ 范围        │ 字节数   │                           │
│  ├──────────┼────────────┼──────────┤                           │
│  │ UInt8    │ 0 ~ 255     │ 1        │                           │
│  │ UInt16   │ 0 ~ 65535   │ 2        │                           │
│  │ UInt32   │ 0 ~ 42亿    │ 4        │  ← 最常用                 │
│  │ UInt64   │ 0 ~ 2^64-1  │ 8        │  ← ID 用这个              │
│  │ Int8     │ -128 ~ 127  │ 1        │                           │
│  │ Int16    │ -32768~32767│ 2        │                           │
│  │ Int32    │ -21亿~21亿  │ 4        │  ← 计数用这个              │
│  │ Int64    │ -2^63~2^63  │ 8        │                           │
│  └──────────┴────────────┴──────────┘                           │
│                                                                  │
│  浮点类型：                                                      │
│  Float32  →  4 字节（类似 C 的 float）                           │
│  Float64  →  8 字节（类似 C 的 double）                          │
│                                                                  │
│  注意：ClickHouse 不支持 DECIMAL 精度问题                         │
│  需要精确计算时用 Decimal(P, S)                                   │
│                                                                  │
│  布尔值：ClickHouse 没有 Boolean 类型！                            │
│  用 UInt8 代替（0 = false, 1 = true）                            │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 5.2 字符串类型

```sql
-- String：可变长度字符串，不限长度
CREATE TABLE t1 (name String);

-- FixedString(N)：固定长度字符串，不足补 \0，超出截断
CREATE TABLE t2 (md5_hash FixedString(32));

-- LowCardinality(String)：低基数字符串（强烈推荐！）
-- 当一个字符串列的取值种类很少时（如城市、设备类型），用这个类型
-- 内部使用字典编码，大幅减少存储空间和内存占用
CREATE TABLE t3 (
    city LowCardinality(String),    -- 可能只有几十个城市
    device LowCardinality(String)   -- 只有 desktop/mobile/tablet
);
```

```
┌──────────────────────────────────────────────────────────────────┐
│                    LowCardinality 的魔法                            │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  city 列有 100 万行，但只有 30 个不同的城市值：                    │
│                                                                  │
│  普通 String：                                                    │
│  每行存储完整的字符串 "北京市" = 9 字节                            │
│  100 万行 × 9 字节 = 9 MB                                        │
│                                                                  │
│  LowCardinality(String)：                                        │
│  字典表：{0: "北京市", 1: "上海市", 2: "广州市", ...}  30个值     │
│  数据列：[0, 1, 0, 2, 0, 1, ...]  每行只需 1-2 字节              │
│  100 万行 × 1.5 字节 = 1.5 MB                                   │
│                                                                  │
│  存储节省 83%！而且查询更快（字典缓存常驻内存）                     │
│                                                                  │
│  建议：基数值 < 10000 的字符串列都使用 LowCardinality              │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 5.3 时间类型

```sql
-- DateTime：秒级精度，4 字节
-- 范围：1970-01-01 ~ 2106-02-07
CREATE TABLE t4 (event_time DateTime);

-- DateTime64：毫秒/微秒/纳秒精度，8 字节
CREATE TABLE t5 (
    event_time DateTime64(3),       -- 毫秒精度
    created_at DateTime64(6, 'Asia/Shanghai')  -- 微秒精度+时区
);

-- Date：天级精度，2 字节
-- 范围：1970-01-01 ~ 2149-06-06
CREATE TABLE t6 (event_date Date);
```

### 5.4 Nullable 类型

```sql
-- Nullable(T)：允许为 NULL 的类型
-- 注意：Nullable 会降低查询性能，尽量避免使用！
CREATE TABLE t7 (
    user_id    UInt32,
    nickname   Nullable(String),     -- 可能为 NULL
    age        Nullable(UInt8)       -- 可能为 NULL
);

-- 更好的做法：用默认值代替 NULL
CREATE TABLE t7_better (
    user_id    UInt32,
    nickname   String DEFAULT '',    -- 空字符串代替 NULL
    age        UInt8 DEFAULT 0       -- 0 代替 NULL
);
```

### 5.5 复合类型

```sql
-- Array：数组类型
CREATE TABLE t8 (
    tags Array(String),          -- 字符串数组：['bigdata', 'kafka']
    scores Array(UInt16)         -- 数值数组：[90, 85, 92]
);

-- Tuple：元组类型（不同类型的组合）
CREATE TABLE t9 (
    location Tuple(Float64, Float64),  -- 经纬度：(116.40, 39.90)
    info Tuple(String, UInt32)          -- 名称和数量：('click', 5)
);

-- Enum：枚举类型（用整数存储，显示字符串）
CREATE TABLE t10 (
    action Enum8(
        'view' = 1,
        'click' = 2,
        'comment' = 3,
        'like' = 4,
        'share' = 5
    )
);
-- Enum8 用 1 字节存储（最多 256 个值）
-- Enum16 用 2 字节存储（最多 65536 个值）
-- 存储效率高，查询时显示可读的字符串
```

### 5.6 类型选择建议

| 场景 | 推荐类型 | 原因 |
|------|---------|------|
| ID、计数 | UInt32 / UInt64 | 无负数，节省空间 |
| 用户行为类型 | Enum8 | 存储为数字，显示为字符串 |
| 城市、设备 | LowCardinality(String) | 基数低，压缩效果好 |
| 时间戳 | DateTime | 秒级精度，4 字节 |
| 日期 | Date | 2 字节，按天分区 |
| 标签列表 | Array(String) | 支持多值 |
| 可能缺失的值 | 默认值（避免 Nullable） | 性能更好 |

## 六、数据分析项目：设计访问日志表

结合前面学到的知识，我们来设计数据平台的访问日志表：

```sql
-- 数据平台访问日志表（最终版）
CREATE TABLE demo.access_logs (
    log_id     UInt64,            -- 日志ID，全局唯一
    user_id    UInt32,            -- 用户ID（0=匿名访客）
    post_id    UInt32,            -- 文章ID（0=非文章页面）
    action     Enum8(             -- 用户行为类型
        'view' = 1,
        'click' = 2,
        'comment' = 3,
        'like' = 4,
        'share' = 5,
        'search' = 6
    ),
    ip         String,            -- IP地址
    city       LowCardinality(String),   -- 城市（低基数用LowCardinality）
    device     LowCardinality(String),   -- 设备类型（低基数）
    timestamp  DateTime           -- 访问时间
)
ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(timestamp)     -- 按天分区
ORDER BY (timestamp, user_id, post_id) -- 排序键：时间+用户+文章
SETTINGS index_granularity = 8192;     -- 索引粒度
```

**设计决策解释：**

```
┌──────────────────────────────────────────────────────────────────┐
│                    设计决策分析                                     │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Q1：为什么 log_id 用 UInt64 而不是 UInt32？                       │

│                                                                  │
│  Q2：为什么 action 用 Enum8 而不是 String？                       │
│  A：Enum8 只占 1 字节，String 每行要存完整字符串。                  │
│     'view' 用 String 存 = 4 字节，用 Enum8 = 1 字节，省 75%。     │
│                                                                  │
│  Q3：为什么 city 和 device 用 LowCardinality？                     │
│  A：城市通常几十个，设备类型只有 3 种。                             │
│     LowCardinality 用字典编码，大幅节省存储。                       │
│                                                                  │

│                                                                  │

```

## 七、面试题

### 面试题 1：MergeTree 引擎的工作原理是什么？

**参考答案：**

MergeTree 是 ClickHouse 最核心的表引擎，其工作原理如下：

**数据写入**：每次 INSERT 会生成一个新的数据块（Part），数据按照 ORDER BY 指定的排序键排序后写入磁盘。如果配置了 PARTITION BY，数据会按分区规则存放到不同的目录中。

**后台合并**：后台线程会不断检查并合并小的 Part。当多个 Part 的排序键范围相邻时，会被合并成一个更大的 Part。合并过程是异步的，不影响读写。

**稀疏索引**：每个 Part 都有一个主键索引文件（primary.idx），默认每 8192 行记录一个索引项。索引存储排序键的边界值和对应的数据偏移量。查询时通过二分查找索引，快速定位到需要扫描的数据范围，然后在这个范围内进行暴力扫描。

**分区裁剪**：如果查询条件包含分区键，ClickHouse 会自动跳过不相关的分区目录，减少需要扫描的数据量。

**排序键的重要性**：ORDER BY 不仅决定数据排序方式，还决定了索引结构和查询效率。好的排序键设计能让大部分查询命中索引，大幅减少扫描量。

---

### 面试题 2：ReplacingMergeTree 和 SummingMergeTree 有什么区别？

**参考答案：**

**ReplacingMergeTree**：当多条数据的排序键相同时，后台合并时只保留最后写入的那条。适合需要去重的场景，比如存储每篇文章的最新状态。需要注意，去重不是实时的，而是在后台合并时发生。如果需要立即看到去重结果，可以在查询时加 `FINAL` 关键字，但这会降低查询性能。

**SummingMergeTree**：当多条数据的排序键相同时，后台合并时将数值列自动求和，非数值列保留最后写入的值。适合预聚合场景，比如统计每天每个城市的 PV/UV。通常配合物化视图使用，将明细数据自动聚合到统计表。

两者的共同点：
- 都是在后台合并时才生效，不是写入时
- 都依赖 ORDER BY 作为判断"相同"的依据
- 查询时加 FINAL 可以立即生效，但性能差

---

### 面试题 3：ClickHouse 的 LowCardinality 类型有什么好处？

**参考答案：**

LowCardinality 是 ClickHouse 特有的类型修饰符，用于基数值较小的列（如城市、设备类型、性别等）。

**工作原理**：内部维护一个字典（Dictionary），将每个不同的字符串值映射为一个整数 ID。数据列中实际存储的是这些整数 ID，而不是原始字符串。

**三大好处**：
1. **存储节省**：每个值从可能几十字节（完整字符串）压缩到 1-2 字节（整数 ID）
2. **查询加速**：字典缓存在内存中，比较操作直接比较整数，比字符串比较快
3. **聚合加速**：GROUP BY、DISTINCT 等操作可以利用整数 ID 加速

**使用建议**：基数值（唯一值数量）小于 10000 的字符串列都建议使用 LowCardinality。如果基数太大（比如 user_id），使用 LowCardinality 反而会增加开销，不建议使用。
