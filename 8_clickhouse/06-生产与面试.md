# 06 - 生产与面试：最佳实践与高频面试题

## 一、生产环境最佳实践

### 1.1 表设计原则

#### 排序键选择

排序键（ORDER BY）是 ClickHouse 表设计中最关键的决定，直接影响查询性能。

```
┌──────────────────────────────────────────────────────────────────┐
│                    排序键设计原则                                   │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  原则 1：查询条件中最常用的字段放前面                              │
│                                                                  │
│  好的设计：                                                      │
│  ORDER BY (timestamp, city, device)                              │
│  → WHERE timestamp BETWEEN ... AND ... 能命中索引                 │
│  → WHERE timestamp ... AND city = '北京' 能进一步裁剪             │
│                                                                  │
│  坏的设计：                                                      │
│  ORDER BY (city, timestamp)                                      │
│  → WHERE timestamp BETWEEN ... AND ... 无法利用索引               │
│  → 必须扫描所有城市的 timestamp                                   │
│                                                                  │
│  ─────────────────────────────────────────────                    │
│                                                                  │
│  原则 2：基数适中的字段更适合做排序键                               │
│                                                                  │
│  基数太高（如 user_id，几百万个不同值）：                          │
│  → 索引效果一般，每个值只有几条数据                               │
│                                                                  │
│  基数太低（如 device，只有 3 个值）：                              │
│  → 索引粒度太大，一个索引项覆盖太多数据                           │
│                                                                  │
│  基数适中（如 timestamp 按天，每天几万条）：                       │
│  → 索引效果最好，每个索引项覆盖适量数据                           │
│                                                                  │
│  ─────────────────────────────────────────────                    │
│                                                                  │
│  原则 3：排序键的列数不宜过多                                      │
│                                                                  │
│  推荐 2-4 列，超过 4 列要仔细评估                                 │
│  列数越多，索引占用的空间越大                                     │
│  而且要求查询条件必须包含前面的列才能利用后面的列                   │
│                                                                  │
│  ─────────────────────────────────────────────                    │
│                                                                  │
│  原则 4：排序键不能修改！                                          │
│                                                                  │
│  如果需要修改排序键：                                             │
│  1. 创建新表（新排序键）                                          │
│  2. INSERT INTO new_table SELECT * FROM old_table                 │
│  3. RENAME TABLE old TO old_bak, new TO old                       │
│  4. DROP TABLE old_bak                                           │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

#### 分区策略

```
┌──────────────────────────────────────────────────────────────────┐
│                    分区策略选择                                     │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  按天分区（推荐日志场景）：                                        │
│  PARTITION BY toYYYYMMDD(timestamp)                              │
│  → 每天一个分区，查询某天只需扫描对应分区                          │
│  → 适合每天数据量 > 1000 万的场景                                 │
│                                                                  │
│  按月分区（数据量不大时）：                                        │
│  PARTITION BY toYYYYMM(timestamp)                                │
│  → 每月一个分区，减少分区目录数量                                 │
│  → 适合每天数据量 < 1000 万的场景                                 │
│                                                                  │
│  按列表分区（维度数据）：                                         │
│  PARTITION BY city                                               │
│  → 按城市分区，查询特定城市时裁剪其他城市                         │
│  → 适合按城市独立分析的场景                                       │
│                                                                  │
│  不分区（小数据量）：                                              │
│  → 数据量 < 1 亿时可以不分区                                     │
│  → 减少管理复杂度                                                │
│                                                                  │
│  注意事项：                                                      │
│  - 分区粒度不要太细（如按小时），会产生大量小文件                  │
│  - 建议每个分区 1GB-50GB                                          │
│  - 分区后可以通过 ALTER TABLE DROP PARTITION 清理旧数据            │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

#### 数据类型选择

| 场景 | 推荐 | 避免 | 原因 |
|------|------|------|------|
| ID | UInt64 | String | 数值比较更快 |
| 状态/类型 | Enum8 | String | 节省存储，比较快 |
| 低基数文本 | LowCardinality(String) | String | 压缩率高 |
| 时间 | DateTime | String | 支持时间函数 |
| 金额 | Decimal(10,2) | Float64 | 精确计算 |
| 可选字段 | 默认值 | Nullable | Nullable 降低性能 |

### 1.2 写入最佳实践

```
┌──────────────────────────────────────────────────────────────────┐
│                    写入最佳实践                                     │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 批量写入，不要单条写入                                        │
│     每批 10000 - 100000 条                                       │
│     单条写入比批量写入慢 100-1000 倍                              │
│                                                                  │
│  2. 写入频率不要太高                                              │
│     建议 1-10 秒写入一次，而不是每条都立即写入                     │
│     高频写入会产生大量小 Part，后台合并压力大                      │
│                                                                  │
│  3. 避免并发写入同一个分区                                        │
│     ClickHouse 支持并发写入，但同一分区的并发写入会增加锁竞争      │
│     可以通过轮询分区或增加分区粒度来缓解                           │
│                                                                  │
│  4. 不要 UPDATE / DELETE                                         │
│     ClickHouse 的 UPDATE/DELETE 是"重写"语义，性能极差            │
│     需要修改数据时：                                              │
│     - 用 ReplacingMergeTree 去重                                  │
│     - 用 CollapsingMergeTree 折叠                                │
│     - 用 ALTER TABLE UPDATE（ Mutation，后台异步执行）             │
│                                                                  │
│  5. 写入前考虑分区键                                              │
│     确保每批数据尽量落入同一个分区                                 │
│     跨分区的批量写入会产生多个 Part                                │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 二、性能调优

### 2.1 索引粒度调优

```sql
-- 查看当前索引粒度
SELECT name, metadata_modification_time, engine
FROM system.tables
WHERE database = 'demo' AND name = 'access_logs';

-- 创建表时指定索引粒度（默认 8192）
CREATE TABLE ... SETTINGS index_granularity = 8192;

-- 常见粒度选择：
-- index_granularity = 4096   → 索引更密，点查更快，但索引更大
-- index_granularity = 8192   → 默认值，平衡选择
-- index_granularity = 16384  → 索引更稀，范围扫描快，但跳过的精度低
-- index_granularity = 32768  → 大粒度，适合大范围扫描场景
```

```
┌──────────────────────────────────────────────────────────────────┐
│                    索引粒度选择指南                                 │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  查询模式              推荐粒度        原因                        │
│  ──────────────────   ──────────    ──────────                  │
│  点查为主（按ID查）     4096         索引更密，减少扫描范围        │
│  混合查询（推荐）       8192         默认值，平衡性能             │
│  大范围扫描（报表）     16384+       减少索引大小，降低内存占用    │
│                                                                  │
│  大部分场景用默认值 8192 就好，不需要调                          │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 三、常见问题排查

### 3.1 ZooKeeper 压力

```
┌──────────────────────────────────────────────────────────────────┐
│                    ZooKeeper 压力问题                              │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  现象：                                                          │
│  - ClickHouse 查询偶尔超时                                        │
│  - 日志中有 "ZooKeeper session expired" 错误                     │
│  - ZooKeeper CPU 使用率高                                         │

│  1. 单节点不需要副本时，用普通 MergeTree，不用 Replicated          │
│  2. 减少小 Part 数量（批量写入，调整 merge 策略）                  │
│  3. 升级 ZooKeeper 到 3.8+，增加 JVM 堆内存                      │
│  4. 考虑用 ClickHouse Keeper 替代 ZooKeeper（21.8+ 版本）        │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 3.4 查询慢排查

```sql
-- 1. 查看当前正在执行的查询

    query_id,
    query,
    elapsed,
    read_rows,
    memory_usage

    query,
    event_time,
    query_duration_ms,
```

## 六、15 道高频面试题

### Q1：ClickHouse 的 MergeTree 引擎工作原理？

**参考答案：**

MergeTree 是 ClickHouse 最核心的表引擎，采用 LSM-Tree（Log-Structured Merge-Tree）架构。

**写入过程**：每次 INSERT 操作生成一个新的数据块（Part），数据按照 ORDER BY 指定的排序键排序后写入磁盘。如果配置了 PARTITION BY，数据按分区规则存入不同目录。

**后台合并**：后台线程不断检查 Part，当多个 Part 的排序键范围相邻时会自动合并为一个更大的 Part。合并过程异步进行，不影响读写。

**索引结构**：每个 Part 维护一个稀疏索引（primary.idx），默认每 8192 行记录一个索引条目，存储排序键的边界值。查询时通过二分查找索引定位到需要扫描的 Granule（数据块），然后在块内暴力扫描。

**分区裁剪**：查询条件包含分区键时，ClickHouse 自动跳过不相关分区目录。

MergeTree 不保证数据去重（需要 ReplacingMergeTree），不支持事务（需要 ReplicatedMergeTree），但提供了极高的写入吞吐和查询性能。

### Q2：ClickHouse 为什么不支持 UPDATE/DELETE？

**参考答案：**

ClickHouse 的设计目标是 OLAP 分析场景，而非 OLTP 事务场景。不支持传统意义的 UPDATE/DELETE 是有意为之：

**原因 1：列式存储的物理限制**。列存储中，一列的数据是连续存储的。修改一行数据需要在所有列文件中找到对应位置并修改，涉及大量随机 IO，性能极差。

**原因 2：ClickHouse 使用 Mutation（异步变异）来实现修改**。`ALTER TABLE UPDATE` 和 `ALTER TABLE DELETE` 实际上是重写整个 Part，后台异步执行，不是即时生效。

**原因 3：分析场景通常不需要修改**。日志数据、统计数据通常只追加不修改。需要修改时，推荐使用：
- **ReplacingMergeTree**：相同排序键只保留最新版本（实现"更新"效果）
- **CollapsingMergeTree**：通过 sign 列实现增删
- **版本化**：保留所有历史版本，查询时取最新版本

---

### Q3：ClickHouse 的物化视图和 MySQL 的物化视图有什么区别？

**参考答案：**

**MySQL 的物化视图**：MySQL 原生不支持物化视图，需要通过触发器或定时刷新来实现。刷新策略通常是定时全量刷新，不支持增量刷新。

**ClickHouse 的物化视图**：是真正的增量物化视图。当源表有新数据写入时，ClickHouse 自动触发物化视图的 SELECT 查询，将新数据的聚合结果追加到目标表中。整个过程是实时的、增量的。

**关键区别**：
1. ClickHouse 物化视图是增量触发，MySQL 需要定时全量刷新
2. ClickHouse 物化视图写入时立即触发，延迟在毫秒级
3. ClickHouse 的物化视图不能修改（DROP 重建），MySQL 可以修改定义
4. ClickHouse 物化视图不处理历史数据（创建前的数据不触发）

---

### Q4：ClickHouse 的数据压缩原理是什么？

**参考答案：**

ClickHouse 的压缩分为两个层面：

**列级压缩**：由于列存储中同一列的数据类型相同、取值范围相近，可以采用高效的压缩算法：
- **LZ4**：默认算法，压缩速度极快，解压速度也快，压缩比中等
- **ZSTD（Zstandard）**：压缩比更高（比 LZ4 高 20-50%），但压缩/解压速度稍慢
- 可以按列指定不同的压缩算法

**编码优化**：除了通用压缩，ClickHouse 还使用列级编码：
- **LowCardinality**：字典编码，将字符串映射为整数 ID
- **Delta 编码**：存储相邻值的差值，适合单调递增的列（如时间戳）
- **RLE（Run-Length Encoding）**：连续相同值只存一次 + 重复次数

综合效果：通常能达到 10-20 倍的压缩比，远高于行存储的 3-5 倍。

### Q5：ClickHouse 适合什么场景？不适合什么场景？

**参考答案：**

**适合的场景**：
1. **日志分析**：Nginx 日志、应用日志、审计日志
2. **实时大屏**：实时 PV/UV、流量监控
3. **BI 报表**：多维分析、趋势分析、排行榜
4. **用户行为分析**：漏斗分析、留存分析
5. **时序数据**：监控指标、IoT 数据

**不适合的场景**：
1. **OLTP 事务**：高频的点查、UPDATE、DELETE
2. **精确去重**：需要实时精确去重的场景（count(DISTINCT) 大数据量下用近似算法）
3. **复杂 JOIN**：多表关联查询（不如 Doris/StarRocks）
4. **频繁修改**：数据需要频繁更新的场景
5. **Key-Value 存储**：单行点查场景（用 Redis 更合适）

---

### Q6：ClickHouse 如何保证数据一致性？

**参考答案：**

ClickHouse 的数据一致性保证分为几个层面：

**单节点**：
- 写入是原子的：一次 INSERT 要么全部成功，要么全部失败
- Part 文件是不可变的：一旦写入就不会被修改（合并时生成新文件）
- 通过 checksum 校验数据完整性

**集群（ReplicatedMergeTree）**：
- 依赖 ZooKeeper/Keeper 进行副本协调
- 写入时先写 ZooKeeper 日志，再写本地，最后同步到其他副本
- 支持多副本读取（quorum read）保证读到最新数据
- 使用 Mutation 实现 UPDATE/DELETE，后台异步重写

**限制**：
- 不支持跨表事务
- ReplicatedMergeTree 的副本同步是异步的（默认）
- 如果需要强一致性，需要在应用层处理

### Q7：ClickHouse 的小 Part 问题怎么解决？

**参考答案：**

小 Part 问题是指表中有大量行数很少的 Part，导致查询时需要打开大量文件，性能下降。

**产生原因**：
- 高频小批量 INSERT（每秒写入几条数据）
- 并发写入产生碎片
- 后台合并速度跟不上写入速度

**解决方案**：
1. **批量写入**：攒批到 10000-100000 条再写入，这是最有效的方法
2. **调整合并参数**：增加 `background_pool_size`（后台合并线程数），加快合并速度
3. **手动触发合并**：`OPTIMIZE TABLE table FINAL`
4. **调整写入频率**：使用 Buffer 引擎暂存数据，定期批量写入
5. **监控 Part 数量**：通过 `system.parts` 表监控，设置告警

**判断标准**：活跃 Part 数量超过 1000 就需要注意，超过 10000 就有严重问题了。

---

### Q8：ClickHouse 的查询优化技巧有哪些？

**参考答案：**

1. **利用排序键**：查询条件尽量包含 ORDER BY 的前缀列，利用稀疏索引裁剪
2. **利用分区裁剪**：查询条件包含分区键，自动跳过不相关分区
3. **使用物化视图**：将聚合查询预计算，查询时直接读预聚合结果
4. **使用 LowCardinality**：低基数列使用此类型，减少 IO 和内存
5. **使用 Enum**：枚举类型替代 String，节省存储
6. **避免 SELECT ***：只查需要的列，列存储的 IO 优势才明显
7. **使用 Approximate 函数**：大数据量下用 `uniq()` 代替 `count(DISTINCT)`
8. **使用 xxxIf 函数**：用 `countIf` 代替 `CASE WHEN`，性能更好
9. **合理设置 max_threads**：单查询的并行线程数
10. **使用 PREWHERE**：先过滤再读取列，减少 IO（`SELECT ... PREWHERE condition`）

### Q9：ClickHouse 的分布式查询是怎么工作的？

**参考答案：**

ClickHouse 的分布式查询通过 Distributed 引擎和本地表实现：

**架构**：每个节点存储部分数据（分片），Distributed 引擎是一个虚拟表，不存储数据，只负责协调查询。

**查询流程**：
1. 客户端发送查询到任意节点（协调节点）
2. 协调节点将查询分发到所有分片节点
3. 各分片节点在本地数据上并行执行查询
4. 各分片返回部分结果给协调节点
5. 协调节点合并结果返回给客户端

**写入流程**：
1. 客户端发送写入请求到协调节点
2. 协调节点根据分片规则将数据路由到对应节点
3. 各节点本地写入

**注意事项**：
- 跨节点查询需要网络传输，注意数据量
- 使用 GLOBAL JOIN 避免数据重复
- 分片键选择影响数据分布均匀度

---

### Q10：ClickHouse 和 Elasticsearch 能互相替代吗？

**参考答案：**

不能互相替代，它们解决不同的问题：

**ClickHouse**：OLAP 分析引擎
- 强项：聚合分析（GROUP BY）、大范围扫描、数值计算
- 弱项：全文搜索、模糊匹配、相关性排序
- 典型场景：数据报表、统计分析、趋势分析

**Elasticsearch**：搜索引擎
- 强项：全文搜索、模糊匹配、相关性排序、聚合分析
- 弱项：大数据量下的聚合性能不如 ClickHouse
- 典型场景：站内搜索、日志搜索、应用监控

**实际中经常配合使用**：
- ES 负责搜索（用户搜文章）
- ClickHouse 负责分析（PV/UV 统计、趋势报表）
- 数据源相同（如 Kafka），分别消费到 ES 和 ClickHouse

---

### Q11：ClickHouse 如何处理高并发查询？

**参考答案：**

ClickHouse 单节点的并发查询能力不如专门的 MPP 数据库（如 Doris），但可以通过以下方式提升：

1. **连接池**：应用层使用连接池，复用 TCP 连接
2. **查询队列**：限制最大并发查询数（`max_concurrent_queries`）
3. **内存限制**：限制单查询内存（`max_memory_usage`），防止单查询占满资源
4. **缓存层**：对热点查询结果加缓存（Redis / 应用层缓存）
5. **物化视图**：预聚合热点查询，减少计算量
6. **读副本**：集群模式下，读请求分散到多个副本
7. **限流**：通过 `query_thread` 配置限制用户/查询类型的资源

---

### Q12：ClickHouse 的 Buffer 引擎是什么？怎么用？

**参考答案：**

Buffer 引擎是一个内存缓冲引擎，用于暂存数据，定期批量写入目标表。

**工作原理**：
- 数据先写入内存缓冲区
- 当缓冲区达到阈值（行数、大小、时间）时，批量写入目标表
- 如果查询时缓冲区有数据，会和目标表数据合并返回

**使用场景**：
- 解决高频小批量写入产生的小 Part 问题
- 写入缓冲，降低目标表的写入压力

```sql
CREATE TABLE demo.access_logs_buffer (
    -- 和目标表相同的结构
    log_id UInt64, user_id UInt32, ...
) ENGINE = Buffer(
    demo,                    -- 目标数据库
    access_logs,             -- 目标表
    16,                      -- 分区数
    10,                      -- 最小缓冲行数
    100,                     -- 最大缓冲行数
    10000,                   -- 最小缓冲字节数
    100000000                -- 最大缓冲字节数（100MB）
);
```

### Q13：ClickHouse 中 count(DISTINCT) 为什么慢？怎么优化？

**参考答案：**

`count(DISTINCT x)` 在 ClickHouse 中等价于 `uniqExact(x)`，使用精确的去重算法，需要将所有值存入哈希表，内存消耗大，大数据量下性能差。

**优化方案**：

1. **使用近似算法**（推荐）：
   - `uniq(x)`：基于 HyperLogLog，误差 ~2%，速度快
   - `uniqCombined(x)`：更好算法，误差 ~0.5%，推荐使用
   - `uniqCombined64(x)`：64位精度，误差更小

2. **使用物化视图预聚合**：将 `count(DISTINCT)` 的结果预先计算好

3. **使用 BitMap**：如果基数不大（< 几百万），可以用 AggregateFunction(groupBitmap, UInt64) 类型

4. **分区后再去重**：先按分区 GROUP BY，再在应用层汇总

### Q14：ClickHouse 如何做数据备份？

**参考答案：**

ClickHouse 没有内置的备份工具，常见的备份方案：

**方案一：文件级备份**
- 直接复制数据目录（`/var/lib/clickhouse/data/`）
- 优点：简单
- 缺点：需要停写（或使用 LVM 快照），占用大量磁盘

**方案二：SQL 导出**
- `SELECT * FROM table FORMAT CSVWithNames > backup.csv`
- 优点：通用，可跨版本
- 缺点：大数据量导出慢，CSV 占用空间大

**方案三：clickhouse-backup 工具（推荐）**
- 开源工具 Altinity/clickhouse-backup
- 支持全量/增量备份
- 支持上传到 S3/本地/NFS
- 通过 freeze 分区实现一致性备份

**方案四：副本备份**
- 集群模式下，多个副本本身就是备份
- 但不能防误删（DROP TABLE 会同步到所有副本）

### Q15：在你们的项目中，ClickHouse 是怎么用的？

**参考答案（结合数据分析项目）：**

在我们的个人数据平台中，ClickHouse 承担着**数据分析引擎**的角色：

**架构**：Web 服务产生的访问日志通过 Kafka 消息队列，实时写入 ClickHouse。同时 Flink 定时将 MySQL 中的业务数据同步到 ClickHouse。

**表设计**：
- `access_logs`（MergeTree）：存储所有访问日志明细，按天分区，按 (timestamp, user_id, post_id) 排序
- `daily_stats`（SummingMergeTree）：通过物化视图自动聚合的每日统计表
- `post_stats`（SummingMergeTree）：文章热度统计表

**查询场景**：
- 数据大屏：实时 PV/UV、热门文章排行、地域分布
- 运营报表：每日/每周/每月趋势分析
- 用户分析：活跃用户排行、留存分析

**遇到的问题和解决方案**：
- 小 Part 问题：通过批量写入解决（每 5 秒一批，每批 5000 条）
- 内存溢出：GROUP BY 数据量大时使用 `max_bytes_before_external_group_by`
- 查询慢：通过物化视图预聚合，热点查询从秒级降到毫秒级
