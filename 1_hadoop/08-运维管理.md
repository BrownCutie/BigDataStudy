# 08 - HDFS 运维

## 一、前言

在生产环境中，HDFS 的运维是一项非常重要的工作。本篇文档将介绍 HDFS 运维中最常用的工具和操作，包括数据均衡、健康检查、快照、配额管理、数据迁移和故障排查。

## 二、数据均衡（Balancer）

### 2.1 为什么需要数据均衡？

当集群运行一段时间后，各 DataNode 的磁盘使用率可能会出现不均衡：

```
┌──────────────────────────────────────────────────────────────┐
│              数据不均衡的例子                                   │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  加入新节点前:                                               │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐                     │
│  │  DN-1   │  │  DN-2   │  │  DN-3   │                     │
│  │  85% 满 │  │  90% 满 │  │  88% 满 │                     │
│  └─────────┘  └─────────┘  └─────────┘                     │
│                                                              │
│  加入新节点 DN-4 后:                                         │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐       │
│  │  DN-1   │  │  DN-2   │  │  DN-3   │  │  DN-4   │       │
│  │  85% 满 │  │  90% 满 │  │  88% 满 │  │  0% 满   │       │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘       │
│                                                              │
│  新节点完全是空的，但旧节点已经很满了。                      │
│  → 新写入的数据会倾向于分配到 DN-4                           │
│  → 但已有数据不会自动迁移                                    │
│  → 需要手动执行 Balancer                                     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 Balancer 原理

```
┌────────────────────────────────────────────────────────────────┐
│              Balancer 工作原理                                  │
│                                                                │
│  1. Balancer 从 NameNode 获取所有 DataNode 的磁盘使用信息     │
│                                                                │
│  2. 计算集群平均使用率:                                        │
│     平均 = (DN1使用率 + DN2使用率 + ... + DNn使用率) / n       │
│                                                                │
│  3. 设定均衡阈值（默认 10%）：                                 │
│     如果某个 DN 的使用率 > 平均 + 阈值 → "过载"               │
│     如果某个 DN 的使用率 < 平均 - 阈值 → "欠载"               │
│                                                                │
│  4. 从过载节点选择 Block 移动到欠载节点                        │
│     移动时遵循以下原则:                                       │
│     - 不能破坏副本放置策略（同一 Block 的副本不能移到同一节点）│
│     - 移动不会导致节点变为过载                                 │
│     - 尽量减少跨机架的传输                                    │
│                                                                │
│  5. 重复步骤 2-4 直到所有节点使用率在阈值范围内                │
│                                                                │
│  ┌────────────────────────────────────────────────────┐     │
│  │  均衡前:                         均衡后:            │     │
│  │  DN-1: 85%   →→→→              DN-1: 65%          │     │
│  │  DN-2: 90%   →→→→              DN-2: 70%          │     │
│  │  DN-3: 88%   →→→→              DN-3: 68%          │     │
│  │  DN-4: 0%    ←←←←              DN-4: 60%          │     │
│  │                                                    │     │
│  │  差异: 90% vs 0%                 差异: 70% vs 60%  │     │
│  └────────────────────────────────────────────────────┘     │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 2.3 使用 Balancer

```bash
# 启动均衡（默认阈值 10%）
hdfs balancer

# 指定阈值 5%（更精确的均衡，但耗时更长）
hdfs balancer -threshold 5

# 指定阈值 20%（更快的均衡，但精确度较低）
hdfs balancer -threshold 20

# 均衡完成后输出:
# Time Stamp               Iteration#  Bytes Already Moved  Bytes Left To Move  Bytes Being Moved
# 2026-04-15 10:00:00      0           0 MB                  50 GB                0 MB
# 2026-04-15 10:05:00      1           10 GB                 40 GB                2 GB
# 2026-04-15 10:10:00      2           25 GB                 25 GB                2 GB
# ...
# 2026-04-15 10:30:00      10          50 GB                 0 MB                 0 MB
# The cluster is balanced. Exiting...
```

### 2.4 Balancer 带宽限制

Balancer 在移动数据时会占用网络带宽。可以通过配置限制带宽：

```xml
<!-- hdfs-site.xml -->
<!-- Balancer 使用的最大带宽（字节/秒），默认 1048576 (1MB/s) -->
<property>
    <name>dfs.datanode.balance.bandwidthPerSec</name>
    <value>1048576</value>
</property>
```

```bash
# 运行时动态修改带宽（不需要重启）
hdfs dfsadmin -setBalancerBandwidth 10485760
# 设置为 10MB/s
```

## 三、fsck 检查文件系统健康

### 3.1 什么是 fsck？

**fsck**（File System Check）是 HDFS 的文件系统检查工具，类似于 Linux 的 `fsck` 命令。它可以检查文件的一致性、副本数量和 Block 的健康状态。

### 3.2 常用 fsck 命令

```bash
# 检查整个文件系统
hdfs fsck /

# 检查特定文件
hdfs fsck /user/你的用户名/data/data.csv

# 显示 Block 信息
hdfs fsck /user/你的用户名/data/data.csv -files -blocks

# 显示 Block 的位置信息
hdfs fsck /user/你的用户名/data/data.csv -files -blocks -locations

# 显示 Block 的副本信息
hdfs fsck /user/你的用户名/data/data.csv -files -blocks -racks

# 列出所有损坏的文件
hdfs fsck / -list-corruptfileblocks

# 删除损坏的文件
hdfs fsck / -delete
```

## 四、快照（Snapshot）功能

### 4.1 什么是快照？

快照是某个目录在某一时刻的**只读副本**。它不是数据的物理复制，而是记录了数据变更的元数据差异，创建速度极快。

```
┌────────────────────────────────────────────────────────────────┐
│              快照原理                                            │
│                                                                │
│  创建快照前的目录结构:                                          │
│  /data/                                                        │
│  ├── data/                                                     │
│  │   ├── access.log                                           │
│  │   └── error.log                                            │
│  └── config/                                                   │
│      └── app.conf                                             │
│                                                                │
│  执行: hdfs dfsadmin -allowSnapshot /data                      │
│  执行: hdfs dfs -createSnapshot /data snapshot_20260415       │
│                                                                │
│  创建快照后:                                                   │
│  /data/                                                        │
│  ├── data/                                                     │
│  │   ├── access.log                                           │
│  │   └── error.log                                            │
│  ├── config/                                                   │
│  │   └── app.conf                                             │
│  └── .snapshot/                     ← 快照目录（隐藏）        │
│      └── snapshot_20260415/                                    │
│          ├── data/                   ← 快照时的完整目录结构    │
│          │   ├── access.log                                       │
│          │   └── error.log                                        │
│          └── config/                                               │
│              └── app.conf                                         │
│                                                                │
│  快照不是复制数据！而是创建了一个指向当前数据的指针。          │
│  即使后续修改了原文件，快照仍然保留原始版本。                  │
│                                                                │
│  存储开销: 只有被修改/删除的文件才会额外占用空间（Copy-on-Write）│
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 4.2 快照操作命令

```bash
# 步骤 1: 允许目录创建快照（每个目录只需执行一次）
hdfs dfsadmin -allowSnapshot /data

# 禁止目录创建快照
hdfs dfsadmin -disallowSnapshot /data

# 步骤 2: 创建快照
hdfs dfs -createSnapshot /data snapshot_20260415
# 快照名称可选，不指定则自动生成时间戳

# 创建命名快照
hdfs dfs -createSnapshot /data before_upgrade

# 步骤 3: 列出快照
hdfs dfs -listSnapshot /data

# 步骤 4: 访问快照中的文件
hdfs dfs -cat /data/.snapshot/snapshot_20260415/data/access.log

# 步骤 5: 从快照恢复文件
# 复制快照中的文件到原目录
hdfs dfs -cp /data/.snapshot/snapshot_20260415/data/access.log /data/data/

# 步骤 6: 删除快照（释放空间）
hdfs dfs -deleteSnapshot /data snapshot_20260415

# 比较快照差异
hdfs snapshotDiff /data snapshot_before snapshot_after
```

### 4.3 快照使用场景

```
┌──────────────────────────────────────────────────────────────┐
│              快照的典型使用场景                                 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 数据备份                                                 │
│     定期创建快照，作为数据的"时间机器"                       │
│     误删文件后可以快速恢复                                   │
│                                                              │
│  2. 灾难恢复                                                 │

│                                                              │
│  3. 数据审计                                                 │
│     查看某个时间点的数据状态                                 │
│     用于合规审计                                             │
│                                                              │
│  4. 机器学习实验                                             │
│     在不同版本的数据集上做实验对比                           │
│                                                              │
│  5. ETL 回滚                                                 │
│     ETL 处理前创建快照，处理失败后回滚                       │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 五、HDFS 配额管理

### 5.1 配额类型

HDFS 支持两种配额：

| 配额类型 | 说明 | 配置参数 |
|----------|------|----------|
| **名称配额** | 目录下的文件和子目录总数上限 | `dfs.quota.max` |
| **空间配额** | 目录下的文件总大小上限 | `dfs.quota.max-bytes` |

### 5.2 配额操作命令

```bash
# 设置名称配额（最多 1000 个文件/目录）
hdfs dfsadmin -setQuota 1000 /data/data

# 设置空间配额（最多 10GB）
hdfs dfsadmin -setSpaceQuota 10g /data/data

# 同时设置两种配额
hdfs dfsadmin -setQuota 1000 /data/data
hdfs dfsadmin -setSpaceQuota 10g /data/data

# 查看配额
hdfs dfs -count -q /data/data
# output: 1000      10000    10737418240      5        8        134217728 /data/data
#          ↑ 名称配额 ↑ 空间配额    ↑ 实际文件数 ↑ 实际大小

# 清除名称配额
hdfs dfsadmin -clrQuota /data/data

# 清除空间配额
hdfs dfsadmin -clrSpaceQuota /data/data

# 清除所有配额
hdfs dfsadmin -clrQuota /data/data
hdfs dfsadmin -clrSpaceQuota /data/data
```

### 5.3 配额使用场景

```
┌──────────────────────────────────────────────────────────────┐
│              配额使用场景                                       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  场景 1: 防止用户占用过多空间                                │
│  hdfs dfsadmin -setSpaceQuota 100g /user/zhangsan            │
│  → 张三最多使用 100GB                                        │
│                                                              │
│  场景 2: 防止小文件过多（保护 NameNode 内存）                │
│  hdfs dfsadmin -setQuota 10000 /user/data/raw               │
│  → 最多 10000 个文件                                         │
│                                                              │
│  场景 3: 临时目录限制                                        │
│  hdfs dfsadmin -setSpaceQuota 50g /tmp                      │
│  → 临时目录最多 50GB                                        │
│                                                              │
│  超出配额时的行为:                                           │
│  - 名称配额: 创建新文件/目录时报错                           │
│  - 空间配额: 写入数据时磁盘满报错                            │
│  - 删除文件可以释放配额                                      │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 六、数据迁移（DistCp）

### 6.1 什么是 DistCp？

**DistCp**（Distributed Copy）是 Hadoop 自带的分布式数据复制工具。它利用 MapReduce 实现大规模数据的并行复制，比普通的 `hdfs dfs -cp` 快得多。

```
┌────────────────────────────────────────────────────────────────┐
│              DistCp 工作原理                                     │
│                                                                │
│  hdfs dfs -cp (普通复制):                                     │
│  ┌──────────┐         ┌──────────┐                            │
│  │ 源目录   │────────→│ 目标目录 │                            │
│  │ 单线程   │ 串行复制│ 单线程   │                            │
│  └──────────┘         └──────────┘                            │
│  100GB 数据 → 约 30 分钟                                       │
│                                                                │
│  hadoop distcp (分布式复制):                                   │
│  ┌──────────┐                                               │
│  │ 源目录   │                                               │
│  │          │  Map Task 1: 复制文件 1-N                     │
│  │          │  Map Task 2: 复制文件 N+1-M                    │
│  │          │  Map Task 3: 复制文件 M+1-K                    │
│  │          │  Map Task 4: 复制文件 K+1-L                    │
│  └────┬─────┘                                               │
│       │                                                      │
│       ▼                                                      │
│  ┌──────────┐         ┌──────────┐                            │
│  │ 多线程   │────────→│ 目标目录 │                            │
│  │ 并行复制 │         │ 多线程   │                            │
│  └──────────┘         └──────────┘                            │
│  100GB 数据 → 约 5 分钟（取决于集群规模）                      │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 6.2 DistCp 常用命令

```bash
# 基本用法: 复制 HDFS 目录到另一个 HDFS 目录
hadoop distcp /data/data /backup/data_data

# 复制到远程集群
hadoop distcp hdfs://cluster1/data/data hdfs://cluster2/backup/data_data

# 覆盖已有文件
hadoop distcp -update /data/data /backup/data_data

# 只追加新文件（不覆盖已有文件）
hadoop distcp -update -skipcrccheck /data/data /backup/data_data

# 忽略复制失败的文件
hadoop distcp -i /data/data /backup/data_data

# 限制每个 Map Task 复制的文件数
hadoop distcp -m 50 /data/data /backup/data_data

# 复制时删除目标端多余的文件
hadoop distcp -delete /data/data /backup/data_data

# 设置带宽限制（每个 Map Task 的带宽，单位 MB/s）
hadoop distcp -bandwidth 10 /data/data /backup/data_data

# 只复制特定文件（使用正则表达式过滤）
hadoop distcp -f /tmp/filelist /backup/
# filelist 文件中每行一个源文件路径
```

### 6.3 DistCp 使用注意事项

```
┌──────────────────────────────────────────────────────────────┐
│              DistCp 注意事项                                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 目标目录不能是源目录的子目录                             │
│     ✗ distcp /data /data/backup  (backup 是 data 的子目录)  │
│     ✓ distcp /data /backup/data                              │
│                                                              │
│  2. 使用 -update 时，只复制新增或修改的文件                   │
│     适合增量备份                                             │
│                                                              │
│  3. 大规模复制时，建议在低峰期执行                           │
│     避免影响正常业务                                         │
│                                                              │
│  4. 复制完成后，用 fsck 验证数据完整性                       │
│     hdfs fsck /backup/data_data                              │
│                                                              │
│  5. 使用 -p 保留文件的权限、所有者、块大小等信息             │
│     hadoop distcp -p /data/data /backup/data_data            │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 7.2 DataNode 无法启动

```bash
# 查看 DataNode 日志
tail -100 $HADOOP_LOG_DIR/hadoop-xxx-datanode-xxx.log

# 常见原因:
# 1. clusterID 不匹配（重复格式化了 NameNode）
#    解决: rm -rf ${hadoop.tmp.dir}/dfs/data && 重启

# 2. 端口被占用
#    lsof -i :9866

# 3. 数据目录权限问题
#    ls -la ${hadoop.tmp.dir}/dfs/data
#    chmod 755 ${hadoop.tmp.dir}/dfs/data

# 4. NameNode 未启动
#    先启动 NameNode，再启动 DataNode
```

### 7.3 HDFS 读写异常

```bash
# 问题 1: 写入文件时报错 "Could only replicate to 0 nodes"
# 原因: 没有可用的 DataNode
# 排查:
hdfs dfsadmin -report    # 查看 Live DataNodes 数量
# 如果为 0，说明 DataNode 全部挂了

# 问题 2: 读取文件报错 "Checksum error"
# 原因: Block 数据损坏
# 排查:
hdfs fsck /path/to/file -files -blocks -locations
# 修复: HDFS 会自动从其他副本恢复损坏的 Block

# 问题 3: "NameNode is in safe mode"
# 排查:
hdfs dfsadmin -safemode get
# 解决: 等待自动退出，或
hdfs dfsadmin -safemode leave  # 强制退出（谨慎使用！）

# 问题 4: "No space left on device"
# 排查:
hdfs dfs -df -h
# 解决: 清理不需要的文件或增加磁盘
```

### 7.4 YARN 作业失败

```bash
# 查看 YARN 应用日志
yarn logs -applicationId application_xxx

# 常见失败原因:
# 1. Container 被 OOM Kill
#    日志中看到: "Container killed by YARN for exceeding memory limits"
#    解决: 增大 mapreduce.map.memory.mb 或 mapreduce.map.java.opts

# 2. 任务重试次数耗尽
#    日志中看到: "Task failed" + "attempt x of y failed"
#    解决: 检查任务代码逻辑，增加 maxattempts

# 3. Shuffle 错误
#    日志中看到: "Connection refused" 或 "Timeout"
#    解决: 检查 mapreduce.shuffle 端口配置

# 4. 作业卡在 ACCEPTED 状态
#    原因: 集群资源不足，AM 无法启动
#    解决: 等待其他作业释放资源，或减少 AM 内存需求
```

### 8.4 定时监控脚本

```bash
#!/bin/bash
# hadoop-health-check.sh - Hadoop 集群健康检查脚本

echo "========================================="
echo "  Hadoop 集群健康检查"
echo "  $(date)"
echo "========================================="

# 检查 NameNode
echo -n "NameNode: "
if jps | grep -q NameNode; then
    echo "RUNNING"
else
    echo "DOWN!"
fi

# 检查 DataNode
echo -n "DataNode: "
DN_COUNT=$(jps | grep -c DataNode)
echo "RUNNING ($DN_COUNT)"

# 检查 ResourceManager
echo -n "ResourceManager: "
if jps | grep -q ResourceManager; then
    echo "RUNNING"
else
    echo "DOWN!"
fi

# 检查 NodeManager
echo -n "NodeManager: "
NM_COUNT=$(jps | grep -c NodeManager)
echo "RUNNING ($NM_COUNT)"

# 检查 HDFS 安全模式
echo -n "HDFS Safe Mode: "
hdfs dfsadmin -safemode get 2>/dev/null

# 检查 HDFS 磁盘使用
echo ""
echo "HDFS 磁盘使用:"
hdfs dfs -df -h

# 检查损坏的 Block
echo ""
echo "损坏的 Block:"
CORRUPT=$(hdfs fsck / 2>/dev/null | grep -c "CORRUPT" || true)
if [ "$CORRUPT" -eq 0 ]; then
    echo "无损坏 Block"
else
    echo "发现 $CORRUPT 个损坏 Block!"
fi

echo ""
echo "检查完成"
```
