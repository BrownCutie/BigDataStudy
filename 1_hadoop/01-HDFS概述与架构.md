# 01 - HDFS 概述：分布式文件系统的基石

## 一、为什么需要分布式文件系统？

在学习 HDFS 之前，我们先想一个问题：**一台机器能存多少数据？**

假设你有一台普通的 Linux 服务器，挂载了一块 4TB 的硬盘。听起来很多，但如果你的公司每天产生 500GB 的日志数据，那么不到 8 天这块硬盘就满了。

你可能会说："多加几块硬盘不就行了？"

没错，但问题远不止于此：

```
┌──────────────────────────────────────────────────────────┐
│                   单机存储面临的挑战                        │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  1. 容量瓶颈：单块硬盘上限通常 10-20TB                    │
│  2. 读写速度：单块硬盘顺序读写约 200MB/s                  │
│  3. 可靠性：硬盘会坏，数据会丢                            │
│  4. 成本：企业级存储设备极其昂贵                          │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

所以，我们需要一种方案：**把数据分散存储在多台机器上，但对用户来说就像使用一台机器一样简单。** 这就是分布式文件系统（Distributed File System）的核心思想。

### 分布式文件系统要解决的核心问题

| 问题 | 说明 |
|------|------|
| **海量存储** | 横向扩展，加机器就能加容量 |
| **高吞吐** | 多台机器并行读写，速度成倍提升 |
| **高可靠** | 数据多副本，一台机器坏了数据不丢 |
| **容错性** | 自动检测故障、自动恢复 |
| **易用性** | 对用户提供统一的文件系统接口 |

## 二、HDFS 是什么？

**HDFS**（Hadoop Distributed File System）是 Hadoop 项目的核心组件之一，是一个分布式、可扩展、高容错的文件系统，专为存储海量数据而设计。

```
┌─────────────────────────────────────────────────────────────┐
│                         HDFS 的设计目标                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   "一次写入，多次读取"（Write Once, Read Many）              │
│                                                             │
│   - 数据写入后很少修改                                       │
│   - 适合大规模数据分析场景                                    │
│   - 不适合低延迟的随机访问                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### HDFS 的核心设计哲学

HDFS 的设计借鉴了 Google 发表的 **GFS（Google File System）** 论文，有几个非常关键的设计决策：

**1. 流式数据访问**

HDFS 假设数据被整体处理而不是随机访问。这意味着它优化的是**顺序读取**的速度，而不是低延迟的随机读取。

```
传统文件系统：  [随机读] → 找到第N个字节 → 读取（毫秒级延迟）
HDFS：         [顺序读] → 从头读到尾 → 吞吐量最大化（可达 GB/s）
```

**2. 大文件存储**

HDFS 适合存储 MB 到 TB 甚至 PB 级别的大文件。在 HDFS 中，一个文件可能被拆分成多个 Block 分散存储在集群中。

**3. 硬件容错**

HDFS 的设计假设**硬件故障是常态**，而不是异常。它通过数据副本机制来应对硬件故障。

### 3.1 NameNode — HDFS 的大脑

NameNode 是 HDFS 的主节点，可以理解为**文件系统的"目录管理员"**。

**NameNode 存储的数据：**

```
┌─────────────────────────────────────────┐
│          NameNode 元数据（内存中）         │
├─────────────────────────────────────────┤
│                                         │
│  文件 → 数据块映射表                      │
│  ┌──────────────────────────────────┐   │
│  │ /user/data/access.log            │   │
│  │   ├── Block 1: [DN1, DN3, DN5]  │   │
│  │   ├── Block 2: [DN2, DN4, DN6]  │   │
│  │   └── Block 3: [DN1, DN3, DN7]  │   │
│  │                                  │   │
│  │ /user/data/error.log             │   │
│  │   ├── Block 4: [DN2, DN5, DN8]  │   │
│  │   └── Block 5: [DN1, DN4, DN6]  │   │
│  └──────────────────────────────────┘   │
│                                         │
│  文件系统的目录树结构                     │
│  文件的权限信息                           │
│  文件的创建/修改时间                      │
│                                         │
└─────────────────────────────────────────┘
```

**关键点：**
- NameNode 的所有元数据都保存在**内存**中，因此响应速度极快
- 元数据同时会持久化到磁盘上的两个文件：**fsimage**（文件系统镜像）和 **edits**（操作日志）
- NameNode 是**单点**，如果 NameNode 挂了，整个 HDFS 将不可用（Hadoop 3.x 引入了 HDFS Federation 解决这个问题）

### 3.2 DataNode — 数据的仓库

DataNode 是 HDFS 的工作节点，负责**实际存储数据**。

```
┌─────────────────────────────────────────────────┐
│              DataNode 磁盘上的数据组织             │
├─────────────────────────────────────────────────┤
│                                                 │
│  ${hadoop.tmp.dir}/dfs/data/                    │
│  └── current/                                   │
│      ├── BP-xxx-xxx/                            │
│      │   ├── current/                           │
│      │   │   ├── blk_1073741824                 │
│      │   │   │   └── (数据块的实际内容)          │
│      │   │   ├── blk_1073741824_1001.meta       │
│      │   │   │   └── (数据块的校验和等元数据)     │
│      │   │   ├── blk_1073741825                 │
│      │   │   │   └── (另一个数据块)              │
│      │   │   └── blk_1073741825_1002.meta       │
│      │   └── ...                                │
│                                                 │
└─────────────────────────────────────────────────┘
```

**DataNode 的核心职责：**

| 职责 | 说明 |
|------|------|
| **存储数据块** | 将数据以 Block 的形式存储在本地磁盘 |
| **读写请求** | 响应客户端和其他 DataNode 的读写请求 |
| **心跳** | 定期向 NameNode 发送心跳，报告存活状态 |
| **块报告** | 定期向 NameNode 发送所存储的所有数据块列表 |
| **数据校验** | 定期对存储的数据块进行校验和检查 |

### 3.3 SecondaryNameNode — NameNode 的助手

**这是一个非常容易被误解的组件！**

SecondaryNameNode **不是** NameNode 的热备份。它的真正作用是**帮助 NameNode 合并 edits 日志和 fsimage**，防止 edits 文件过大。

```
┌────────────────────────────────────────────────────────────────┐
│              SecondaryNameNode 合并流程                          │
│                                                                │
│  步骤1: SecondaryNameNode 请求 NameNode 滚动 edits              │
│         NameNode 停止写入当前 edits，创建新的 edits              │
│                                                                │
│  步骤2: SecondaryNameNode 下载 fsimage 和旧的 edits             │
│         ┌──────────┐    fsimage + edits    ┌─────────────────┐ │
│         │NameNode  │ ─────────────────────→ │SecondaryNameNode│ │
│         └──────────┘                       └────────┬────────┘ │
│                                                        │          │
│  步骤3: SecondaryNameNode 在内存中合并                   ▼          │
│                                                        │          │
│                                              ┌─────────────────┐ │
│                                              │ 合并生成新的     │ │
│                                              │ fsimage.ckpt    │ │
│                                              └────────┬────────┘ │
│                                                       │           │
│  步骤4: 将新的 fsimage 上传回 NameNode                 │           │
│         ┌──────────┐    新的 fsimage.ckpt  ┌──────────▼───────┐ │
│         │NameNode  │ ←──────────────────── │SecondaryNameNode│ │
│         └──────────┘                       └─────────────────┘ │
│                                                                │
│  结果: NameNode 的 edits 文件被截断，启动速度大大加快             │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 四、Block（数据块）和 Replica（副本）机制

### 4.1 Block — 数据的分割单位

HDFS 中的文件被切分成固定大小的**数据块（Block）**进行存储。

```
┌─────────────────────────────────────────────────────────────┐
│                    Block 切分示意图                           │
│                                                             │
│  一个 300MB 的文件（假设 Block 大小为 128MB）                 │
│                                                             │
│  ┌────────────────────────────────────────────┐             │
│  │              原始文件 (300MB)               │             │
│  │  ┌──────────┬──────────┬────────────────┐  │             │
│  │  │ Block 1  │ Block 2  │    Block 3     │  │             │
│  │  │  128MB   │  128MB   │     44MB       │  │             │
│  │  └──────────┴──────────┴────────────────┘  │             │
│  └────────────────────────────────────────────┘             │
│                                                             │
│  注意：Block 3 虽然只有 44MB，但在 HDFS 中仍然                │
│        占用 128MB 的逻辑空间（不占用实际磁盘空间）             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**为什么要有 Block？**

```
┌─────────────────────────────────────────────────────────────┐
│                   Block 的三大优势                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 大文件处理                                               │
│     一个 10TB 的文件可以被拆分成约 8 万个 128MB 的 Block      │
│     分布在几百台机器上存储，任何一台机器都不需要存完整文件      │
│                                                             │
│  2. 简化存储系统                                             │
│     不需要关心文件大小，统一按 Block 管理                     │
│     磁盘分配、容错恢复都以 Block 为单位                      │
│                                                             │
│  3. 适合容错和数据负载均衡                                   │
│     Block 可以在不同机器间复制、迁移                          │
│     读取时可以从多个 Block 并行读取，提高吞吐量               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Hadoop 3.4.2 中 Block 的默认大小是 128MB**（早期版本默认 64MB）。

### 4.2 Replica — 数据的保险箱

HDFS 通过**副本（Replica）**机制来保证数据的可靠性。默认情况下，每个 Block 会有 3 个副本，分布存储在不同的机器上。

```
┌────────────────────────────────────────────────────────────────┐
│                    副本分布策略（默认 3 副本）                    │
│                                                                │
│  假设集群有多个机架（Rack），每个机架有多台服务器                  │
│                                                                │
│  ┌─ Rack 1 ─────────────────────────────────────────────┐      │
│  │                                                      │      │
│  │   ┌──────────┐                                       │      │
│  │   │ DataNode1│ ◄── 副本 1（同一台机器）               │      │
│  │   │ [Block 1]│                                       │      │
│  │   └──────────┘                                       │      │
│  │                                                      │      │
│  │   ┌──────────┐                                       │      │
│  │   │ DataNode2│ ◄── 副本 2（同一机架不同机器）         │      │
│  │   │ [Block 1]│                                       │      │
│  │   └──────────┘                                       │      │
│  │                                                      │      │
│  └──────────────────────────────────────────────────────┘      │
│                                                                │
│  ┌─ Rack 2 ─────────────────────────────────────────────┐      │
│  │                                                      │      │
│  │   ┌──────────┐                                       │      │
│  │   │ DataNode3│ ◄── 副本 3（不同机架）                 │      │
│  │   │ [Block 1]│                                       │      │
│  │   └──────────┘                                       │      │
│  │                                                      │      │
│  └──────────────────────────────────────────────────────┘      │
│                                                                │
│  策略总结：                                                     │
│  - 第 1 个副本：写入请求所在的 DataNode                          │
│  - 第 2 个副本：同一机架的另一个 DataNode（减少网络传输）        │
│  - 第 3 个副本：不同机架的 DataNode（防止机架故障）             │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

**副本数量可以在 `hdfs-site.xml` 中配置：**

```xml
<!-- 默认副本数量，一般设为 3 -->
<property>
    <name>dfs.replication</name>
    <value>3</value>
</property>
```

## 五、HDFS 与传统文件系统的对比

| 特性 | 本地文件系统（ext4/NTFS） | HDFS |
|------|-------------------------|------|
| **存储规模** | 单机 TB 级 | 集群 PB~EB 级 |
| **文件大小** | 受磁盘大小限制 | 无理论上限 |
| **读写模式** | 随机读写 + 顺序读写 | 顺序写、顺序读 |
| **数据可靠性** | RAID/备份 | 多副本自动复制 |
| **故障处理** | 手动干预 | 自动检测、自动恢复 |
| **访问接口** | POSIX 标准接口 | HDFS API / Shell |
| **修改文件** | 支持任意位置修改 | 只支持追加（Append） |
| **适用场景** | 通用场景 | 大数据分析、日志存储 |
| **成本** | 企业级存储昂贵 | 普通服务器即可 |

```
┌──────────────────────────────────────────────────────────┐
│               HDFS 不能做什么？                            │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  ✗ 不支持低延迟数据访问                                   │
│    HDFS 优化的是高吞吐，不是低延迟                         │
│    如果需要毫秒级响应，考虑 HBase                          │
│                                                          │
│  ✗ 不支持大量小文件                                       │
│    小文件过多会耗尽 NameNode 的内存                        │
│    因为每个文件/目录/Block 都会在 NameNode 内存中占用空间   │
│                                                          │
│  ✗ 不支持文件的随机修改                                    │
│    HDFS 支持"追加写"，但不支持在文件中间修改内容            │
│                                                          │
│  ✗ 不支持多用户并发写入同一文件                             │
│    一个文件同一时间只能有一个 Writer                        │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## 六、HDFS 的适用场景

### 适合的场景

```
┌──────────────────────────────────────────────────────────┐
│                   HDFS 的典型应用场景                      │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  1. 大规模日志存储与分析                                   │
│     例如：网站访问日志、应用运行日志、审计日志              │
│     → 每天产生 TB 级日志，需要长期保存和批量分析           │
│                                                          │
│  2. 数据仓库                                             │
│     例如：博客交易数据、用户行为数据                        │
│     → 历史数据需要长期保存，定期进行全量分析                │
│                                                          │
│  3. 机器学习训练数据集                                     │
│     例如：图像数据集、文本语料库                            │
│     → 大量数据需要顺序读取进行模型训练                    │
│                                                          │
│  4. 备份和归档                                           │
│     例如：数据库备份、业务数据归档                          │
│     → 数据写入后基本不修改，偶尔需要读取                   │
│                                                          │
│  5. ETL 数据处理                                         │
│     例如：从多个数据源抽取数据，转换后加载到 HDFS           │
│     → 作为数据湖的底层存储                                 │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 不适合的场景

```
┌──────────────────────────────────────────────────────────┐
│                 不适合使用 HDFS 的场景                      │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  1. 在线事务处理（OLTP）                                   │
│     需要低延迟的随机读写，应使用关系型数据库                │
│                                                          │
│  2. 存储大量小文件                                        │
│     比如图片缩略图、用户头像等，应使用对象存储（如 S3）      │
│                                                          │
│  3. 需要频繁修改的数据                                    │
│     比如配置文件、状态数据，应使用数据库或键值存储          │
│                                                          │
│  4. 低延迟实时查询                                        │
│     比如实时推荐、实时风控，应使用 HBase 或 Elasticsearch  │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## 七、HDFS 的重要概念汇总

```
┌────────────────────────────────────────────────────────────────┐
│                    HDFS 核心概念速查表                          │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  概念              说明                                       │
│  ─────────────────────────────────────────────────────         │
│  NameNode         主节点，管理元数据，单点                     │
│  DataNode         数据节点，存储实际数据                       │
│  SecondaryNameNode 辅助节点，合并 edits 和 fsimage            │
│  Block            数据块，默认 128MB                          │
│  Replica          副本，默认 3 个                             │
│  Rack Awareness   机架感知，优化网络传输和容灾                 │
│  fsimage          文件系统镜像，NameNode 元数据的快照          │
│  edits            操作日志，记录所有文件系统操作               │
│  Heartbeat        心跳机制，DataNode 向 NameNode 报告状态     │
│  Block Report     块报告，DataNode 向 NameNode 汇报数据块     │
│  Safe Mode        安全模式，NameNode 启动时的特殊状态          │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 八、总结

HDFS 是大数据生态系统的基石，它用简单的架构解决了海量数据存储的问题。它的核心思想可以概括为：

1. **分而治之**：将大文件切分成 Block，分散存储
2. **多副本冗余**：通过副本机制保证数据可靠性
3. **主从架构**：NameNode 管理元数据，DataNode 存储数据，职责清晰
4. **流式访问**：优化顺序读写，追求高吞吐而非低延迟

在接下来的文档中，我们将一步步搭建 HDFS 环境，学习 HDFS Shell 操作，深入理解读写流程，并最终掌握 YARN 和 MapReduce 编程。
