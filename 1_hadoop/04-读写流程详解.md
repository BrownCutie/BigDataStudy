# 04 - HDFS 读写流程深入

## 一、前言

理解 HDFS 的读写流程是掌握 HDFS 的核心。这篇文档将详细拆解文件写入和读取的每一步，帮助你真正理解 HDFS 是如何工作的。

### 2.2 逐步详解

**第一步：客户端请求创建文件**

客户端调用 `FileSystem.create(path)` 方法（Java API），或者执行 `hdfs dfs -put` 命令。客户端会通过 RPC 连接 NameNode，发送创建文件请求。

**第二步：NameNode 执行检查**

NameNode 收到请求后，会进行以下检查：
- 文件是否已经存在（不允许覆盖，除非显式指定）
- 客户端是否有写入权限
- 父目录是否存在
- 文件系统是否处于安全模式

**第三步：NameNode 创建文件记录**

检查通过后，NameNode 在元数据中创建一条文件记录（此时文件还没有任何 Block），返回操作令牌给客户端。

**第四步：客户端请求分配数据块**

客户端开始写入数据。数据被切分成 Packet（默认 64KB），当第一个 Packet 填满后，客户端向 NameNode 请求分配 DataNode 来存储第一个 Block。

**第五步：NameNode 选择 DataNode**

NameNode 根据副本策略选择合适的 DataNode 列表（详见第四节）。

**第六步：建立数据管道（Pipeline）**

客户端按照 NameNode 返回的 DataNode 列表顺序建立数据管道。

```
┌──────────────────────────────────────────────────────────────┐
│                   数据管道（Pipeline）                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Client  ───→  DN1  ───→  DN2  ───→  DN3                   │
│  (发送)        (接收+转发)  (接收+转发)  (接收)              │
│                                                              │
│  数据流向: Client → DN1 → DN2 → DN3                          │
│  ACK 流向: Client ← DN1 ← DN2 ← DN3                         │
│                                                              │
│  这种"流水线"方式的好处:                                      │
│  - 数据通过一次网络传输就完成了 3 个副本的写入                 │
│  - 不需要客户端分别连接每个 DataNode                          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**第七步：数据传输**

客户端将 Packet 通过管道发送到 DN1，DN1 收到后转发给 DN2，DN2 收到后转发给 DN3。每个 DN 收到数据后会写入本地磁盘，并发送 ACK 确认。

**第八步：Block 完成与下一个 Block**

当一个 Block（128MB）写完后，客户端关闭当前管道，向 NameNode 请求下一个 Block 的 DataNode 分配，重复上述过程。

**第九步：文件写入完成**

所有 Block 写完后，客户端通知 NameNode 文件写入完成。NameNode 将文件状态从"写入中"改为"已完成"。

### 3.2 读取流程详解

**第一步：客户端请求文件信息**

客户端通过 RPC 调用 NameNode 的 `getBlockLocations()` 方法，获取文件的 Block 列表以及每个 Block 所在的 DataNode 列表。

**第二步：NameNode 返回元数据**

NameNode 返回每个 Block 的 ID、大小、以及存储该 Block 副本的所有 DataNode 地址。

**第三步：客户端选择最优 DataNode**

客户端根据**网络距离**选择离自己最近的 DataNode 来读取每个 Block。网络距离排序如下：

```
┌──────────────────────────────────────────────────────────────┐
│                  网络距离排序（从近到远）                       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  距离 = 0  同一台机器上的 DataNode                            │
│  距离 = 2  同一机架上的不同 DataNode                          │
│  距离 = 4  不同机架上的 DataNode                              │
│                                                              │
│  选择策略:                                                   │
│  1. 优先选择同一节点上的副本（距离=0）                       │
│  2. 其次选择同一机架的副本（距离=2）                          │
│  3. 最后选择不同机架的副本（距离=4）                          │
│  4. 如果同一机架有多个副本，随机选择一个（负载均衡）          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**第四步：读取数据**

客户端直接与选定的 DataNode 建立连接，读取 Block 数据。

**第五步：容错处理**

如果读取过程中 DataNode 不可用或数据校验失败：
1. 客户端自动切换到下一个最近的副本
2. 向 NameNode 报告损坏的 Block
3. NameNode 会安排重新复制该 Block

## 四、副本选择策略

NameNode 在分配 Block 时，需要选择在哪些 DataNode 上存储副本。这个选择策略叫做**副本放置策略（Replica Placement Policy）**。

```
┌────────────────────────────────────────────────────────────────┐
│                    副本选择策略（3 副本）                        │
│                                                                │
│  ┌─ Rack 1 ──────────────────────────────────────┐            │
│  │                                                │            │
│  │   ┌─────────┐  ┌─────────┐  ┌─────────┐     │            │
│  │   │  DN-1   │  │  DN-2   │  │  DN-3   │     │            │
│  │   │ 30% 空  │  │ 70% 空  │  │ 50% 空  │     │            │
│  │   └─────────┘  └─────────┘  └─────────┘     │            │
│  │                                                │            │
│  └────────────────────────────────────────────────┘            │
│                                                                │
│  ┌─ Rack 2 ──────────────────────────────────────┐            │
│  │                                                │            │
│  │   ┌─────────┐  ┌─────────┐  ┌─────────┐     │            │
│  │   │  DN-4   │  │  DN-5   │  │  DN-6   │     │            │
│  │   │ 80% 空  │  │ 20% 空  │  │ 60% 空  │     │            │
│  │   └─────────┘  └─────────┘  └─────────┘     │            │
│  │                                                │            │
│  └────────────────────────────────────────────────┘            │
│                                                                │
│  选择过程:                                                     │
│                                                                │
│  第 1 个副本:                                                  │
│  - 如果客户端在集群中 → 选客户端所在节点（DN-1）               │
│  - 如果客户端不在集群中 → 随机选一个，但倾向于磁盘空间多的      │
│    → 选择 DN-4（Rack 2，空间最多）                              │
│                                                                │
│  第 2 个副本:                                                  │
│  - 在第 1 个副本的不同机架上选一个                              │
│    → 在 Rack 2 上选 DN-4（如果第1副本在 Rack1）               │
│    → 或在 Rack 1 上选 DN-2（如果第1副本在 Rack2）             │
│                                                                │
│  第 3 个副本:                                                  │
│  - 与第 2 个副本同一机架，不同节点                             │
│    → 在 Rack 1 上选 DN-2（如果第2副本在 Rack2 的 DN-4）       │
│                                                                │
│  设计原则:                                                     │
│  1. 减少跨机架网络传输（1/3 副本在同一机架）                   │
│  2. 防止单机架故障（至少有一个副本在不同机架）                  │
│  3. 均衡负载（倾向于选择磁盘空间多的节点）                     │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 五、心跳机制

心跳机制是 HDFS 保证集群健康的核心手段。

### 5.1 DataNode → NameNode 心跳

```
┌────────────────────────────────────────────────────────────────┐
│              DataNode 心跳和块报告                               │
│                                                                │
│  DataNode                         NameNode                    │
│    │                                │                          │
│    │──── 心跳 (每 3 秒) ──────────→│                          │
│    │    包含:                       │                          │
│    │    - DataNode 标识             │                          │
│    │    - 总容量                    │                          │
│    │    - 已用容量                  │                          │
│    │    - 正在处理的 Block 数量      │                          │
│    │                                │                          │
│    │←── 心跳响应 ─────────────────│                          │
│    │    包含:                       │                          │
│    │    - 操作指令                  │                          │
│    │    - 复制/删除 Block 的命令    │                          │
│    │    - NameNode 的时间戳         │                          │
│    │                                │                          │
│    │──── 块报告 (每 6 小时) ─────→│                          │
│    │    包含:                       │                          │
│    │    - 该 DN 上所有 Block 的列表  │                          │
│    │                                │                          │
│    │                                │ 对比块报告和元数据         │
│    │                                │ 发现:                     │
│    │                                │ - 多余的副本 → 通知删除   │
│    │                                │ - 不足的副本 → 通知复制   │
│    │                                │                          │
│    │←── 复制/删除指令 ────────────│                          │
│    │                                │                          │
└────────────────────────────────────────────────────────────────┘
```

**关键配置参数：**

```xml
<!-- DataNode 心跳间隔（秒），默认 3 秒 -->
<property>
    <name>dfs.heartbeat.interval</name>
    <value>3</value>
</property>

<!-- NameNode 判定 DataNode 死亡的超时时间 -->
<!-- 计算公式: dfs.namenode.heartbeat.recheck.interval + dfs.heartbeat.interval * 10 -->
<!-- 默认: 300000ms + 3000ms * 10 = 330000ms = 5.5 分钟 -->
<property>
    <name>dfs.namenode.heartbeat.recheck.interval</name>
    <value>300000</value>  <!-- 5 分钟 -->
</property>

<!-- 块报告间隔（秒），默认 21600 秒 = 6 小时 -->
<property>
    <name>dfs.blockreport.intervalMsec</name>
    <value>21600000</value>
</property>
```

### 5.2 NameNode → DataNode 指令

NameNode 通过心跳响应向 DataNode 下发指令：

| 指令 | 说明 |
|------|------|
| **复制 Block** | 某个 Block 的副本数不足，需要复制到其他 DN |
| **删除 Block** | 某个 Block 的副本数过多，需要删除多余副本 |
| **重新复制** | Block 数据损坏，需要从其他副本重新复制 |
| **立即发送块报告** | NameNode 需要立即了解 DataNode 的状态 |

## 六、安全模式（Safe Mode）

### 6.1 什么是安全模式？

NameNode 启动时会进入**安全模式**，此时 HDFS 只允许读取，不允许写入和删除。

```
┌────────────────────────────────────────────────────────────────┐
│                    安全模式原理                                  │
│                                                                │
│  NameNode 启动过程:                                            │
│                                                                │
│  1. NameNode 从磁盘加载 fsimage 到内存                        │
│                                                                │
│  2. NameNode 回放 edits 日志                                  │
│                                                                │
│  3. 此时 NameNode 内存中有所有文件的元数据，                   │
│     但还不知道哪些 Block 真正存在（需要等待 DataNode 的块报告） │
│                                                                │
│  4. 进入安全模式，等待 DataNode 报告 Block 状态                │
│                                                                │
│  ┌─────────────────────────────────────────────────────┐      │
│  │                                                      │      │
│  │  安全模式期间:                                        │      │
│  │  ✅ 允许: 查看文件列表、读取文件内容                   │      │
│  │  ❌ 禁止: 创建文件、删除文件、上传文件                 │      │
│  │  ❌ 禁止: 修改副本数量、修改文件元数据                 │      │
│  │                                                      │      │
│  └─────────────────────────────────────────────────────┘      │
│                                                                │
│  5. DataNode 陆续启动，发送块报告                              │
│                                                                │
│  6. NameNode 统计可用 Block 数量                               │
│                                                                │
│  7. 当满足退出条件时，自动退出安全模式                         │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 6.2 安全模式的进入和退出条件

**进入条件（自动）：**
- NameNode 启动时自动进入
- 系统可用 Block 比例低于阈值

**退出条件（同时满足）：**
1. 可用的 Block 数量 / 总 Block 数量 >= `dfs.namenode.safemode.threshold.pct`（默认 0.999，即 99.9%）
2. 存活的 DataNode 数量 >= `dfs.namenode.replication.min`（默认 1）
3. 满足条件后等待 `dfs.namenode.safemode.extension`（默认 30000ms = 30 秒）

```xml
<!-- 安全模式退出阈值（0-1），默认 0.999 -->
<property>
    <name>dfs.namenode.safemode.threshold.pct</name>
    <value>0.999</value>
</property>

<!-- 满足阈值后额外等待时间（毫秒），默认 30000 -->
<property>
    <name>dfs.namenode.safemode.extension</name>
    <value>30000</value>
</property>
```

### 6.3 安全模式操作命令

```bash
# 查看安全模式状态
hdfs dfsadmin -safemode get
# 输出: Safe mode is ON/OFF

# 等待安全模式退出（阻塞直到退出）
hdfs dfsadmin -safemode wait

# 手动进入安全模式（维护时使用）
hdfs dfsadmin -safemode enter

# 手动退出安全模式（生产环境谨慎使用！）
hdfs dfsadmin -safemode leave
```

## 七、数据完整性（校验和）

### 7.1 校验和机制

HDFS 通过**校验和（Checksum）**来保证数据的完整性。每个 Block 都有一个对应的 `.meta` 文件，存储了校验和信息。

```
┌────────────────────────────────────────────────────────────────┐
│                    校验和机制                                    │
│                                                                │
│  数据被切分成 Chunk（默认 512 字节），每个 Chunk 计算一个 CRC32 │
│                                                                │
│  Block 文件 (blk_xxx):                                         │
│  ┌────────┬────────┬────────┬────────┬────────┬────────┐      │
│  │Chunk 1 │Chunk 2 │Chunk 3 │Chunk 4 │Chunk 5 │ ...    │      │
│  │512B    │512B    │512B    │512B    │512B    │        │      │
│  └────────┴────────┴────────┴────────┴────────┴────────┘      │
│                                                                │
│  Meta 文件 (blk_xxx.meta):                                     │
│  ┌────────┬────────┬────────┬────────┬────────┬────────┐      │
│  │CRC32_1 │CRC32_2 │CRC32_3 │CRC32_4 │CRC32_5 │ ...    │      │
│  │4B      │4B      │4B      │4B      │4B      │        │      │
│  └────────┴────────┴────────┴────────┴────────┴────────┘      │
│                                                                │
│  校验时机:                                                     │
│  ┌─────────────────────────────────────────────────────┐      │
│  │  1. 写入时: 客户端计算校验和，DN 存储到 .meta 文件  │      │
│  │  2. 读取时: DN 计算校验和，与 .meta 文件对比        │      │
│  │  3. 后台: DN 定期扫描所有 Block，验证数据完整性     │      │
│  └─────────────────────────────────────────────────────┘      │
│                                                                │
│  发现数据损坏时:                                               │
│  1. 客户端读取 → 校验失败 → 切换到其他副本                    │
│  2. 客户端通知 NameNode 该 Block 损坏                         │
│  3. NameNode 标记该 Block 为损坏                               │
│  4. NameNode 安排从其他副本重新复制                            │
│  5. DataNode 删除损坏的 Block                                  │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 7.2 校验和相关配置

```xml
<!-- 每个 Chunk 的大小（字节），默认 512 -->
<property>
    <name>io.bytes.per.checksum</name>
    <value>512</value>
</property>

<!-- DataNode 后台扫描 Block 的间隔（秒），默认 518400 秒 = 6 天 -->
<property>
    <name>dfs.datanode.scan.period.hours</name>
    <value>504</value>
</property>
```

### 7.3 手动验证数据完整性

```bash
# 使用 fsck 检查文件的健康状态
hdfs fsck /user/你的用户名/data/data.csv -files -blocks -locations

# 输出示例:
# /user/你的用户名/data/data.csv 134217728 bytes, 1 block(s): OK
# 0. BP-xxx blk_1073741824_1001 len=134217728 repl=1 [DatanodeInfoWithStorage(xxx:9866, ...)]
# Status: HEALTHY
```

