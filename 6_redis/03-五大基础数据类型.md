# 03 - 五大基础数据类型：Redis 的万能瑞士军刀

## 一、为什么 Redis 要设计多种数据类型？

传统的键值存储（如 Memcached）只支持 String 类型，你要存一个用户对象，只能把整个对象序列化成 JSON 字符串：

```
┌──────────────────────────────────────────────────────────────┐
│                  Memcached 的局限性                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  只能用 String：                                              │
│  SET user:1001 '{"name":"browncutie","age":25,"city":"shanghai"}'│
│                                                              │
│  想修改年龄？                                                │
│  → 要把整个 JSON 取出来 → 反序列化 → 修改 → 序列化 → 写回去    │
│  → 3 次网络请求，CPU 开销大                                   │
│                                                              │
│  想做排行榜？                                                │
│  → 没有内置的有序结构，得自己实现                               │
│  → 维护成本高，性能差                                         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

Redis 的做法是：**在键值对的基础上，为 Value 提供多种数据结构**。这样你不需要每次都"全量读写"，而是可以精确操作数据的某一部分。

```
┌──────────────────────────────────────────────────────────────────┐
│                    Redis 的五大基础数据类型                         │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Key（键）         Value（值）的类型                               │
│  ─────────────    ─────────────────────────────                  │
│  任意字符串  →  String   │ Hash    │ List   │ Set   │ ZSet       │
│               (字符串)   │(哈希)   │(列表)  │(集合) │(有序集合)  │
│                                                                  │
│  同一个 Key 只能对应一种类型的 Value                              │
│  一旦设置了类型，就不能再用其他类型的命令操作                        │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 二、String（字符串）

String 是 Redis 最基本的数据类型，能存文本、数字、甚至二进制数据（图片、视频等，最大 512MB）。

### 2.1 常用命令

```bash
# ==================== 基本操作 ====================

# 设置值
SET key value
SET name "browncutie"

# 获取值
GET name
# "browncutie"

# 设置值并指定过期时间（秒）
SET key value EX seconds
SET session:abc123 "user_data" EX 3600    # 1 小时后过期

# 设置值并指定过期时间（毫秒）
SET key value PX milliseconds
SET captcha:13800138000 "5678" PX 300000  # 5 分钟后过期

# ==================== 批量操作 ====================

# 同时设置多个键值对
MSET k1 "v1" k2 "v2" k3 "v3"

# 同时获取多个值
MGET k1 k2 k3
# 1) "v1"
# 2) "v2"
# 3) "v3"

# ==================== 数值操作 ====================

# 自增 +1
SET views:1001 0
INCR views:1001
# (integer) 1
INCR views:1001
# (integer) 2

# 自增指定值
INCRBY views:1001 10
# (integer) 12

# 自减 -1
DECR views:1001
# (integer) 11

# 自减指定值
DECRBY views:1001 5
# (integer) 6

# ==================== 其他操作 ====================

# 追加字符串
SET greeting "hello"
APPEND greeting " world"
GET greeting
# "hello world"

# 获取字符串长度
STRLEN greeting
# (integer) 11

# 设置新值并返回旧值
GETSET counter 100
# 返回旧值：(nil)
GET counter
# "100"

# 如果 key 不存在才设置（分布式锁常用）
SETNX lock:publish "1"
# (integer) 1    ← 设置成功
SETNX lock:publish "2"
# (integer) 0    ← 设置失败（key 已存在）

# ==================== 范围操作 ====================

# 设置多个字符的值（从 offset 开始）
SET msg "Hello World"
SETRANGE msg 6 "Redis"
GET msg
# "Hello Redis"

# 获取子字符串（类似 Python 的切片）
GETRANGE msg 0 4
# "Hello"
```

### 2.2 数据平台场景：文章缓存

```bash
# 缓存文章详情（JSON 格式）
SET demo:article:1001 '{"id":1001,"title":"Redis入门教程","author":"browncutie","content":"Redis是一个内存数据库...","views":0,"created_at":"2024-03-15"}'

# 设置缓存过期时间（1 小时）
EXPIRE demo:article:1001 3600

# 读取缓存
GET demo:article:1001
```

### 2.3 数据平台场景：计数器

```bash
# 文章浏览量计数
SET demo:views:1001 0
INCR demo:views:1001          # 浏览量 +1
INCR demo:views:1001          # 浏览量 +1
GET demo:views:1001           # 查看当前浏览量

# 文章点赞数
SET demo:likes:1001 42
INCR demo:likes:1001          # 点赞 +1
DECR demo:likes:1001          # 取消点赞 -1

# 分布式 ID 生成器
SET demo:next_id 10000
INCR demo:next_id             # 获取下一个 ID: 10001
INCR demo:next_id             # 获取下一个 ID: 10002
```

### 2.4 底层实现：SDS

```
┌──────────────────────────────────────────────────────────────┐
│                 String 的底层实现：SDS                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Redis 没有直接使用 C 语言的字符串（char*），                   │
│  而是自己实现了 SDS（Simple Dynamic String）：                  │
│                                                              │
│  ┌──────┬──────┬──────┬──────┬──────┬──────┐                 │
│  │ len  │ free │ buf  │  R   │  e   │  d   │  i   │  s   │  \0│
│  │(已用) │(空闲) │(标志) │      │      │      │      │      │   │
│  └──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘   │
│                                                              │
│  SDS 相比 C 字符串的优势：                                     │
│  1. O(1) 获取长度（C 字符串需要遍历到 \0，O(N)）              │
│  2. 杜绝缓冲区溢出（自动扩容）                                  │
│  3. 二进制安全（可以存图片等二进制数据）                          │
│  4. 减少内存分配次数（空间预分配 + 惰性释放）                    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 三、Hash（哈希）

Hash 是一个键值对集合，类似于 Python 的字典或 Java 的 HashMap。特别适合存储**对象**。

```
┌──────────────────────────────────────────────────────────────┐
│                  Hash 结构示意图                                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Redis 键         Hash (field → value)                       │
│  ───────────     ─────────────────────                      │
│  user:1001  →   ┌──────────┬──────────────┐                  │
│                 │  name    │ browncutie    │                  │
│                 │  age     │ 25            │                  │
│                 │  email   │ b@example.com │                  │
│                 │  city    │ shanghai      │                  │
│                 └──────────┴──────────────┘                  │
│                                                              │
│  优势：可以只读写对象的某个字段，不需要序列化/反序列化            │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 3.1 常用命令

```bash
# ==================== 基本操作 ====================

# 设置单个字段
HSET user:1001 name "browncutie"
HSET user:1001 age 25
HSET user:1001 email "b@example.com"

# 同时设置多个字段
HMSET user:1001 name "browncutie" age 25 email "b@example.com" city "shanghai"

# 获取单个字段
HGET user:1001 name
# "browncutie"

# 同时获取多个字段
HMGET user:1001 name age email
# 1) "browncutie"
# 2) "25"
# 3) "b@example.com"

# 获取所有字段和值
HGETALL user:1001
# 1) "name"
# 2) "browncutie"
# 3) "age"
# 4) "25"
# ...

# 获取所有字段名
HKEYS user:1001
# 1) "name"
# 2) "age"
# 3) "email"

# 获取所有值
HVALS user:1001
# 1) "browncutie"
# 2) "25"
# 3) "b@example.com"

# ==================== 检查与删除 ====================

# 字段是否存在
HEXISTS user:1001 name
# (integer) 1    ← 存在

# 删除字段
HDEL user:1001 email

# 获取字段数量
HLEN user:1001
# (integer) 3

# ==================== 数值操作 ====================

# 字段值自增
HINCRBY user:1001 age 1
HINCRBY user:1001 login_count 1

# 浮点数自增
HINCRBYFLOAT user:1001 score 0.5

# ==================== 仅当字段不存在时设置 ====================

HSETNX user:1001 name "new_name"
# (integer) 0    ← name 已存在，设置失败
```

### 3.2 数据平台场景：用户信息

```bash
# 存储用户基本信息
HSET user:1001 username "browncutie"
HSET user:1001 nickname "大数据学习者"
HSET user:1001 avatar "https://example.com/avatar.png"
HSET user:1001 bio "热爱大数据技术"
HSET user:1001 register_date "2024-01-01"
HSET user:1001 article_count 15
HSET user:1001 follower_count 120

# 修改单个字段（无需读取整个对象）
HSET user:1001 bio "热爱大数据和云计算"

# 用户发了一篇文章，文章数 +1
HINCRBY user:1001 article_count 1

# 获取用户信息展示
HGET user:1001 username
HGET user:1001 bio
HGETALL user:1001
```

### 3.3 数据平台场景：文章详情

```bash
# 存储文章详情（比 String 更灵活，可以单独修改某个字段）
HSET article:1001 title "Redis入门教程"
HSET article:1001 author "browncutie"
HSET article:1001 content "Redis是一个..."
HSET article:1001 category "大数据"
HSET article:1001 tags "Redis,缓存,内存数据库"
HSET article:1001 status "published"
HSET article:1001 views 0
HSET article:1001 likes 0
HSET article:1001 created_at "2024-03-15 10:30:00"

# 浏览量 +1
HINCRBY article:1001 views 1

# 点赞 +1
HINCRBY article:1001 likes 1

# 只获取标题和作者（不需要取全部字段）
HMGET article:1001 title author
```

### 3.4 Hash vs String 存对象

```
┌──────────────────────────────────────────────────────────────────┐
│               Hash vs String 存储对象对比                          │
├──────────────┬────────────────────┬──────────────────────────────┤
│   操作        │    String          │    Hash                      │
├──────────────┼────────────────────┼──────────────────────────────┤
│ 存储方式      │ 整个 JSON 序列化    │ 每个字段单独存储               │
│ 读取单个字段  │ 取出全部 → 反序列化  │ HGET 直接读取                 │
│ 修改单个字段  │ 取出 → 改 → 写回    │ HSET 直接修改                 │
│ 内存占用      │ 略低（无字段冗余）   │ 略高（每个字段独立存储）        │
│ 适用场景      │ 数据经常整体读写    │ 数据经常部分读写               │
├──────────────┼────────────────────┼──────────────────────────────┤
│ 建议          │ 缓存整个对象用 String │ 存储对象详情用 Hash           │
└──────────────┴────────────────────┴──────────────────────────────┘
```

## 四、List（列表）

List 是一个有序的字符串列表，按照插入顺序排序。底层实现是**双向链表**或**快速列表（quicklist）**。

```
┌──────────────────────────────────────────────────────────────┐
│                  List 结构示意图                                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  demo:latest_posts →  [文章5] ←→ [文章4] ←→ [文章3] ←→ ... │
│  (从左到右 = 从新到旧)                                        │
│                                                              │
│  LPUSH（从左边推入）        RPUSH（从右边推入）                │
│  LPOP（从左边弹出）         RPOP（从右边弹出）                 │
│                                                              │
│  可以当做：栈、队列、阻塞队列、最新列表                         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 4.1 常用命令

```bash
# ==================== 基本操作 ====================

# 从左边推入（头部）
LPUSH mylist "a" "b" "c"
# mylist: [c, b, a]

# 从右边推入（尾部）
RPUSH mylist "d" "e"
# mylist: [c, b, a, d, e]

# 从左边弹出
LPOP mylist
# "c"

# 从右边弹出
RPOP mylist
# "e"

# ==================== 查看操作 ====================

# 获取列表长度
LLEN mylist
# (integer) 3

# 获取指定范围的元素（索引从 0 开始）
LRANGE mylist 0 -1    # 获取全部元素
LRANGE mylist 0 1     # 获取前 2 个元素

# 获取指定索引的元素
LINDEX mylist 0
# "b"

# ==================== 阻塞操作 ====================

# 阻塞式弹出（队列常用）
# 如果列表为空，会阻塞等待，直到有数据或超时
BLPOP queue 30       # 从左边弹出，最多等 30 秒
BRPOP queue 30       # 从右边弹出，最多等 30 秒

# ==================== 删除操作 ====================

# 删除指定值的元素（从左到右删除 count 个）
LREM mylist 2 "b"    # 从左到右删除 2 个 "b"
LREM mylist -1 "b"   # 从右到左删除 1 个 "b"
LREM mylist 0 "b"    # 删除所有 "b"

# 保留指定范围的元素（删除其他）
LTRIM mylist 0 9     # 只保留前 10 个元素
```

### 4.2 数据平台场景：最新文章列表

```bash
# 发布新文章，加入最新文章列表（最多保留 50 篇）
LPUSH demo:latest_posts "1001:Redis入门教程"
LPUSH demo:latest_posts "1002:Spark实战指南"
LPUSH demo:latest_posts "1003:Kafka深度解析"

# 只保留最新 50 篇
LTRIM demo:latest_posts 0 49

# 获取首页最新 10 篇文章
LRANGE demo:latest_posts 0 9
# 1) "1003:Kafka深度解析"
# 2) "1002:Spark实战指南"
# 3) "1001:Redis入门教程"
```

### 4.3 数据平台场景：消息队列

```bash
# 简单的消息队列（生产者-消费者模式）

# 生产者：发送消息
LPUSH demo:email_queue '{"to":"user@example.com","subject":"欢迎注册","body":"..."}'

# 消费者：消费消息（阻塞等待）
BRPOP demo:email_queue 30
# 如果队列有消息，立即返回
# 如果队列为空，阻塞等待最多 30 秒

# 查看队列中有多少待处理消息
LLEN demo:email_queue
```

│                                                              │
│  生产者（Web 服务）                消费者（邮件服务）             │
│  ┌──────────┐                   ┌──────────┐                │

## 五、Set（集合）

Set 是一个无序的、不重复的字符串集合。支持集合间的交、并、差运算。

```
┌──────────────────────────────────────────────────────────────┐
│                  Set 结构示意图                                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  user:1001:tags →  { Redis, 大数据, 缓存, Python }           │
│  user:1002:tags →  { Redis, Kafka, 大数据, Java }            │
│                                                              │
│  交集：SINTER → { Redis, 大数据 }                             │
│  并集：SUNION → { Redis, 大数据, 缓存, Python, Kafka, Java }  │
│  差集：SDIFF  → { 缓存, Python }                             │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 5.1 常用命令

```bash
# ==================== 基本操作 ====================

# 添加元素（自动去重）
SADD tags:1001 "Redis" "大数据" "缓存" "Python" "Redis"
# (integer) 4    ← "Redis" 重复了，只算 1 个

# 获取所有元素
SMEMBERS tags:1001
# 1) "Python"
# 2) "Redis"
# 3) "大数据"
# 4) "缓存"

# 删除元素
SREM tags:1001 "Python"

# 判断元素是否存在
SISMEMBER tags:1001 "Redis"
# (integer) 1    ← 存在

# 获取集合元素数量
SCARD tags:1001
# (integer) 3

# 随机获取一个/多个元素
SRANDMEMBER tags:1001
SRANDMEMBER tags:1001 2    # 随机获取 2 个

# 随机弹出元素（弹出后从集合中移除）
SPOP tags:1001

# ==================== 集合运算 ====================

# 交集：两个集合都有的元素
SINTER tags:1001 tags:1002

# 并集：两个集合的所有元素（去重）
SUNION tags:1001 tags:1002

# 差集：在第一个集合中但不在第二个集合中的元素
SDIFF tags:1001 tags:1002

# 将集合运算结果存储到新集合
SINTERSTORE common:tags tags:1001 tags:1002
SUNIONSTORE all:tags tags:1001 tags:1002
SDIFFSTORE diff:tags tags:1001 tags:1002
```

### 5.2 数据平台场景：文章标签

```bash
# 给文章添加标签
SADD article:1001:tags "Redis" "缓存" "内存数据库" "NoSQL"
SADD article:1002:tags "Spark" "大数据" "分布式计算" "NoSQL"

# 查看文章的所有标签
SMEMBERS article:1001:tags

# 查看某个标签是否属于某篇文章
SISMEMBER article:1001:tags "Redis"
# (integer) 1    ← 是

# 统计文章有多少个标签
SCARD article:1001:tags
# (integer) 4
```

### 5.3 数据平台场景：共同关注

```bash
# 用户 1001 关注的人
SADD user:1001:following "2001" "2002" "2003" "2004"

# 用户 1002 关注的人
SADD user:1002:following "2002" "2003" "2005" "2006"

# 共同关注（交集）
SINTER user:1001:following user:1002:following
# 1) "2002"
# 2) "2003"

# 我关注的人中，他没有关注的（差集）
SDIFF user:1001:following user:1002:following
# 1) "2001"
# 2) "2004"

# 可能认识的人（共同好友的好友 - 已关注的 - 自己）
# 这个需要多步操作，这里只演示交集部分
```

### 5.4 数据平台场景：点赞

```bash
# 用户给文章点赞
SADD article:1001:likers "user:1001" "user:1002" "user:1003"

# 判断用户是否已点赞
SISMEMBER article:1001:likers "user:1001"
# (integer) 1    ← 已点赞

# 取消点赞
SREM article:1001:likers "user:1001"

# 获取点赞总数
SCARD article:1001:likers
# (integer) 2

# 获取所有点赞用户（分页展示）
SRANDMEMBER article:1001:likers 10
```

## 六、ZSet（有序集合，Sorted Set）

ZSet 是 Redis 最强大的数据类型之一。它在 Set 的基础上给每个元素关联了一个**分数（score）**，通过分数来自动排序。

```
┌──────────────────────────────────────────────────────────────┐
│                  ZSet 结构示意图                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  hot_articles（热门文章）                                      │
│  ┌──────────────────────────────────────┐                    │
│  │  分数 (score)     成员 (member)       │                    │
│  │  ───────────      ──────────────     │                    │
│  │    9520        →  "文章1001"          │                    │
│  │    8300        →  "文章1003"          │                    │
│  │    6100        →  "文章1002"          │                    │
│  │    4200        →  "文章1005"          │                    │
│  │    2100        →  "文章1004"          │                    │
│  └──────────────────────────────────────┘                    │
│                                                              │
│  自动按分数从低到高排序                                        │
│  REVRANGE 可以反向（从高到低）                                  │
│                                                              │
│  底层实现：跳表（Skip List）+ 字典（Hash Table）               │
│  - 跳表：支持范围查询，O(log N)                                │
│  - 字典：支持 O(1) 查找某个成员的分数                           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 6.1 常用命令

```bash
# ==================== 基本操作 ====================

# 添加元素（指定分数）
ZADD hot_articles 100 "文章1001"
ZADD hot_articles 200 "文章1002" 150 "文章1003"

# 获取某个元素的分数
ZSCORE hot_articles "文章1001"
# "100"

# 获取元素排名（从 0 开始，分数从低到高）
ZRANK hot_articles "文章1002"
# (integer) 1

# 获取元素排名（分数从高到低）
ZREVRANK hot_articles "文章1002"
# (integer) 1

# ==================== 范围查询 ====================

# 按分数从低到高获取（指定索引范围）
ZRANGE hot_articles 0 -1
# 1) "文章1001"
# 2) "文章1003"
# 3) "文章1002"

# 按分数从高到低获取（排行榜常用）
ZREVRANGE hot_articles 0 -1
# 1) "文章1002"
# 2) "文章1003"
# 3) "文章1001"

# 带分数一起返回
ZREVRANGE hot_articles 0 -1 WITHSCORES
# 1) "文章1002"
# 2) "200"
# 3) "文章1003"
# 4) "150"
# 5) "文章1001"
# 6) "100"

# 按分数范围查询
ZRANGEBYSCORE hot_articles 100 150    # 分数在 100~150 之间
ZREVRANGEBYSCORE hot_articles 200 100  # 分数在 100~200 之间（降序）

# ==================== 分数操作 ====================

# 增加分数
ZINCRBY hot_articles 10 "文章1001"    # 分数 +10

# ==================== 删除操作 ====================

# 删除元素
ZREM hot_articles "文章1001"

# 按排名删除
ZREMRANGEBYRANK hot_articles 0 2      # 删除排名 0~2 的元素

# 按分数删除
ZREMRANGEBYSCORE hot_articles 0 100   # 删除分数 0~100 的元素

# ==================== 统计操作 ====================

# 获取元素数量
ZCARD hot_articles

# 获取指定分数范围内的元素数量
ZCOUNT hot_articles 100 200

# ==================== 集合运算 ====================

# 并集（多个 ZSet 合并，分数可以相加或取最大值）
ZUNIONSTORE result 2 zset1 zset2 WEIGHTS 1 2    # zset1 分数 x1, zset2 分数 x2

# 交集
ZINTERSTORE result 2 zset1 zset2
```

## 八、面试题

### Q1：Redis 的五大数据类型分别适用于什么场景？

**参考答案：**

| 类型 | 场景 | 命令示例 |
|------|------|---------|
| **String** | 缓存、计数器、分布式锁 | `SET/GET/INCR/SETNX` |
| **Hash** | 用户信息、对象属性（需要部分读写） | `HSET/HGET/HINCRBY` |
| **List** | 最新文章列表、简单消息队列、时间线 | `LPUSH/LRANGE/BRPOP` |
| **Set** | 标签、共同关注、点赞去重 | `SADD/SISMEMBER/SINTER` |
| **ZSet** | 排行榜、热度排序、滑动窗口限流 | `ZADD/ZINCRBY/ZREVRANGE` |

核心原则：**根据业务需求选择最匹配的数据结构，而不是什么场景都用 String。**

---

### Q2：ZSet 的底层实现是什么？为什么用跳表而不是平衡树或红黑树？

**参考答案：**

ZSet 的底层实现取决于元素数量和元素大小：
- 元素较少且较小时：使用 ziplist（压缩列表），节省内存
- 元素较多或较大时：使用 **skiplist（跳表）+ dict（字典）**

**为什么选择跳表而不是红黑树？**

1. **实现简单**：跳表的实现比红黑树简单得多，代码更易维护
2. **范围查询更高效**：跳表只需要定位到起始节点，然后顺序遍历即可；红黑树需要中序遍历
3. **内存效率**：跳表通过调整层级来平衡空间和时间，更灵活
4. **并发友好**：跳表的插入和删除只需要修改相邻节点的指针，更容易实现并发版本

```
┌──────────────────────────────────────────────────────────────┐
│                  跳表（Skip List）示意图                       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Level 3:  [头] ──────────────────────→ [6] ──────→ [尾]     │
│  Level 2:  [头] ──────→ [3] ────────────→ [6] ──────→ [尾]   │
│  Level 1:  [头] ─→ [1] → [3] ──→ [4] ──→ [6] → [9] → [尾]  │
│  Level 0:  [头] → 1  →  3  →  4  →  5  →  6  →  9  → [尾]  │
│           (底层链表，包含所有元素)                               │
│                                                              │
│  查找 6：从最高层开始，比较后决定向右还是向下                      │
│  Level 3: 头 → 6（找到！）                                    │
│  时间复杂度：O(log N)                                         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Q3：Redis 的 Hash 和 String 都能存对象，怎么选择？

**参考答案：**

| 对比维度 | String | Hash |
|---------|--------|------|
| 存储方式 | 整个对象序列化后存储 | 每个字段独立存储 |
| 读取部分字段 | 需要取出全部 → 反序列化 | `HGET` 直接获取 |
| 修改部分字段 | 需要取出 → 改 → 写回 | `HSET` 直接修改 |
| 内存占用 | 略低 | 略高（字段名有冗余） |
| 序列化开销 | 每次读写都需要 | 无需序列化 |

**选择建议：**
- 如果经常需要**整体读写**（如缓存完整的文章 JSON）→ 用 String
- 如果经常需要**部分读写**（如只修改用户头像）→ 用 Hash
- 如果对象字段很少（< 10 个）→ 两者差异不大，看个人偏好
- 如果字段很多且经常修改部分字段 → 用 Hash
