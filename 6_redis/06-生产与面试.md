### 1.2 配置最佳实践

```bash
# redis.conf 生产环境推荐配置

# ========== 网络配置 ==========
bind 10.0.0.1                    # 绑定内网 IP，不绑定 0.0.0.0
port 16379                       # 不用默认端口
protected-mode yes               # 开启保护模式
requirepass "你的强密码_至少16位"   # 必须设置密码
tcp-keepalive 300                # TCP 保活

# ========== 内存配置 ==========
maxmemory 4gb                    # 限制最大内存（根据服务器配置调整）
maxmemory-policy allkeys-lru     # 淘汰最近最少使用的键
maxmemory-samples 10             # LRU 采样数（增大更精确，但更慢）

# ========== 持久化配置 ==========
save 900 1                       # RDB 快照
save 300 10
save 60 10000
appendonly yes                   # 开启 AOF
appendfsync everysec             # 每秒同步
auto-aof-rewrite-percentage 100  # AOF 文件增长 100% 时重写
auto-aof-rewrite-min-size 64mb   # AOF 文件最小 64MB 才重写

# ========== 安全配置 ==========
rename-command FLUSHALL ""        # 禁用危险命令
rename-command FLUSHDB ""
rename-command CONFIG ""
rename-command DEBUG ""

# ========== 日志配置 ==========
loglevel notice                   # 日志级别
logfile /var/log/redis/redis.log # 日志文件

# ========== 客户端配置 ==========
timeout 300                      # 客户端空闲 300 秒后断开
maxclients 10000                 # 最大客户端连接数
```

### 1.3 连接池配置

```python
import redis
from redis.connection import ConnectionPool

# 推荐的连接池配置
pool = ConnectionPool(
    host='127.0.0.1',
    port=6379,
    password='your_password',
    db=0,
    max_connections=50,         # 最大连接数
    retry_on_timeout=True,      # 超时自动重试
    socket_timeout=5,           # Socket 超时 5 秒
    socket_connect_timeout=5,   # 连接超时 5 秒
    health_check_interval=30,   # 每 30 秒检查连接健康
)

r = redis.Redis(connection_pool=pool)

# 使用管道（Pipeline）批量操作，减少网络往返
pipe = r.pipeline()
pipe.set('key1', 'value1')
pipe.set('key2', 'value2')
pipe.get('key1')
pipe.get('key2')
results = pipe.execute()
```

### 2.3 内存优化技巧

```bash
# 1. 使用 Hash 存储对象（减少 key 的数量和前缀冗余）
# 不推荐：多个 String key
SET user:1001:name "browncutie"
SET user:1001:age "25"
SET user:1001:email "b@example.com"
# 每个键都有 "user:1001:" 的前缀开销

# 推荐：一个 Hash
HSET user:1001 name "browncutie" age "25" email "b@example.com"
# 前缀只存一次

# 2. 合理使用编码（小数据量自动使用 ziplist，更省内存）
# Hash 元素 < 512 且所有值 < 64 字节 → 使用 ziplist
hash-max-ziplist-entries 512
hash-max-ziplist-value 64

# 3. 使用 Pipeline 批量操作，减少网络开销
# 不推荐：多次单独调用
for i in range(1000):
    r.set(f"key:{i}", f"value:{i}")

# 推荐：Pipeline 批量
pipe = r.pipeline()
for i in range(1000):
    pipe.set(f"key:{i}", f"value:{i}")
pipe.execute()

# 4. 设置合理的过期时间，让不用的数据自动释放
# 5. 使用 BITMAP 做签到等场景，极度节省空间
#    1 亿用户的签到数据只需 ~12MB
```

## 三、常见问题排查

### 3.1 大 Key 问题

```
┌──────────────────────────────────────────────────────────────┐
│                  什么是大 Key？                                 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  大 Key 的判断标准（仅供参考）：                                │
│  - String 类型 > 10KB                                        │
│  - Hash/List/Set/ZSet 元素 > 5000 个                         │
│                                                              │
│  大 Key 的危害：                                               │
│  1. 内存不均：集群模式下某个节点内存过大                        │
│  2. 阻塞：删除大 Key 耗时长，阻塞主线程                        │
│  3. 网络拥塞：读取大 Key 占用大量带宽                          │
│  4. 慢查询：操作大 Key 的命令耗时变长                          │
│                                                              │
│  数据分析项目中的大 Key 示例：                                      │
│  - 文章内容缓存（String）：如果文章很长（> 10KB）就是大 Key    │
│  - 最新文章列表（List）：如果存了 10 万篇就是大 Key            │
│  - 所有用户标签（Set）：如果存了所有用户的标签就是大 Key        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

```bash
# 发现大 Key
# 1. 使用 redis-cli --bigkeys 扫描
redis-cli --bigkeys

# 2. 使用 SCAN 命令逐个检查
redis-cli --scan | head -100 | while read key; do
    size=$(redis-cli DEBUG OBJECT $key | grep "serializedlength" | awk '{print $3}')
    echo "$key: $size bytes"
done

# 解决大 Key
# 1. 拆分：将大 Hash 拆分成多个小 Hash
# 2. 压缩：使用压缩算法（如 GZIP）减小 Value 大小
# 3. 异步删除：使用 UNLINK 代替 DEL（后台线程异步删除）
UNLINK big_key    # 非阻塞删除
```

### 3.2 热点 Key 问题

```
┌──────────────────────────────────────────────────────────────┐
│                  什么是热点 Key？                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  热点 Key：某个 Key 被大量并发请求集中访问                       │
│                                                              │
│  数据平台场景示例：                                                │
│  - 一篇爆款文章的缓存（被疯狂刷新）                              │
│  - 首页推荐列表（所有用户都在访问）                              │
│                                                              │
│  危害：                                                      │
│  - 单节点 Redis 压力过大                                      │
│  - 网络带宽打满                                               │
│  - 本地缓存可能被击穿                                          │
│                                                              │
│  解决方案：                                                    │
│  1. 本地缓存 + Redis：热点数据同时存在应用本地内存和 Redis      │
│  2. Key 分散：将热点 Key 复制成多个，分散请求                    │
│     例如：hot_article:1001_0, hot_article:1001_1, ...         │
│     客户端随机选择一个副本读取                                  │
│  3. 读写分离：增加从节点分担读压力                              │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 四、Redis 集群模式（Cluster）

### 4.1 Cluster 核心概念

```
┌──────────────────────────────────────────────────────────────┐
│                  Redis Cluster 架构                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  核心设计：数据分片（Sharding）                                 │
│                                                              │
│  Redis 将所有数据分为 16384 个 Hash Slot（哈希槽）              │
│  每个节点负责一部分 Slot                                       │
│                                                              │
│  key → CRC16(key) % 16384 → Slot 编号 → 负责的节点             │
│                                                              │
│  ┌────────────────────────────────────────────────────┐     │
│  │  16384 个 Hash Slot                                │     │
│  │                                                    │     │
│  │  0 ─────────── 5460 ─────────── 10922 ────── 16383 │     │
│  │  │                  │                  │          │     │
│  │  │                  │                  │          │     │
│  │  ▼                  ▼                  ▼          │     │
│  │  Master 0         Master 1         Master 2     │     │
│  │  (6381)           (6382)           (6383)        │     │
│  │     │                 │                 │        │     │
│  │     ▼                 ▼                 ▼        │     │
│  │  Slave 0          Slave 1          Slave 2      │     │
│  │  (6384)           (6385)           (6386)        │     │
│  │                                                    │     │
│  │  最小集群：3 Master + 3 Slave = 6 节点               │     │
│  └────────────────────────────────────────────────────┘     │
│                                                              │
│  特点：                                                      │
│  - 自动数据分片                                                │
│  - 高可用：Master 故障时 Slave 自动接管                        │
│  - 水平扩展：增加节点即可扩容                                  │
│  - 部分可用：某个节点故障不影响其他节点的数据                    │
│  - 不支持多 Key 操作（除非所有 key 在同一个 Slot）              │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 Hash Tag

在 Cluster 模式下，如果多个 key 需要一起操作（如 MGET），可以使用 **Hash Tag** 确保它们落在同一个 Slot：

```bash
# Hash Tag 语法：{tag}
# Redis 只对 {} 内的内容计算 Slot

# 这两个 key 会被分配到同一个 Slot
SET {user:1001}:name "browncutie"
SET {user:1001}:age "25"

# MGET 操作可以正常执行（因为它们在同一个 Slot）
MGET {user:1001}:name {user:1001}:age

# 没有 Hash Tag 的多 key 操作可能报错
# 因为不同 key 可能在不同节点上
```

## 六、15 道高频面试题

### Q1：Redis 为什么这么快？

**参考答案：**

1. **纯内存操作**：数据存在内存中，读写延迟约 100ns，远快于磁盘的 ms 级
2. **单线程模型**：避免多线程的锁竞争、上下文切换开销，每个操作天然原子性
3. **I/O 多路复用**：使用 epoll/kqueue，单个线程高效处理大量并发连接
4. **高效的数据结构**：SDS（O(1) 获取长度）、跳表（O(log N) 操作）、ziplist（省内存）
5. **优化的协议**：RESP 协议简单高效，解析开销小
6. **合理的事件循环**：将网络 I/O 和命令处理放在同一个事件循环中

注意：Redis 6.0 引入了多线程 I/O（网络读写），但命令执行仍然是单线程。

---

### Q2：Redis 的数据类型有哪些？分别用于什么场景？

**参考答案：**

五大基础类型：String、Hash、List、Set、ZSet

| 类型 | 底层实现 | 典型场景 |
|------|---------|---------|
| String | SDS/Int | 缓存、计数器、分布式锁、Session |
| Hash | ziplist/hashtable | 用户信息、对象属性（部分读写） |
| List | quicklist/linkedlist | 最新列表、消息队列、时间线 |
| Set | intset/hashtable | 标签、去重、共同关注、点赞 |
| ZSet | ziplist/skiplist+hashtable | 排行榜、热度排序、限流 |

高级类型：Bitmap（位图）、HyperLogLog（基数统计）、GEO（地理位置）、Stream（消息流）

---

### Q3：Redis 持久化 RDB 和 AOF 怎么选？

**参考答案：**

- **RDB**：定时全量快照，文件小、恢复快，但可能丢失几分钟数据
- **AOF**：记录每条写命令，数据安全性高（everysec 最多丢 1 秒），但文件大、恢复慢
- **推荐**：生产环境同时开启 RDB + AOF，Redis 4.0+ 使用混合持久化（aof-use-rdb-preamble yes）

---

### Q4：缓存穿透、击穿、雪崩的区别和解决方案？

**参考答案：**

| 问题 | 定义 | 解决方案 |
|------|------|---------|
| **穿透** | 查询不存在的数据，每次都打到数据库 | 缓存空值、布隆过滤器、参数校验 |
| **击穿** | 热点 key 过期瞬间，大量并发打到数据库 | 互斥锁、逻辑过期、热点 key 永不过期 |
| **雪崩** | 大量 key 同时过期或 Redis 宕机 | 随机 TTL、多级缓存、高可用、限流降级 |

---

### Q5：Redis 和 MySQL 双写一致性怎么保证？

**参考答案：**

推荐方案：**Cache Aside + 延迟双删**

1. **读**：先读 Redis，未命中读 MySQL，回写 Redis
2. **写**：先更新 MySQL，再删除 Redis 缓存
3. **延迟双删**：写操作时，先删缓存 → 更新数据库 → 延迟几百毫秒再删一次缓存

极端情况下仍有短暂不一致，但概率极低。如果要求强一致性，可以使用分布式读写锁。

---

### Q6：Redis 的过期策略是什么？

**参考答案：**

Redis 使用**惰性删除 + 定期删除**的组合策略：

1. **惰性删除**：访问 key 时检查是否过期，过期则删除并返回 nil
2. **定期删除**：每隔一段时间随机抽查一批设置了过期时间的 key，过期的删除

两者互补，平衡了 CPU 使用率和内存利用率。

---

### Q7：Redis 的内存淘汰策略有哪些？

**参考答案：**

- **noeviction**：不淘汰，写入报错（默认）
- **allkeys-lru**：从所有 key 中淘汰最近最少使用的（推荐用于缓存）
- **allkeys-lfu**：从所有 key 中淘汰使用频率最低的
- **volatile-lru**：从设置了过期时间的 key 中淘汰最近最少使用的
- **volatile-lfu**：从设置了过期时间的 key 中淘汰使用频率最低的
- **allkeys-random**：随机淘汰
- **volatile-random**：从有过期时间的 key 中随机淘汰
- **volatile-ttl**：淘汰 TTL 最小的

### Q8：Redis 的分布式锁怎么实现？有什么问题？

**参考答案：**

基本实现：`SET key value NX EX seconds`

```bash
# 加锁（原子操作）
SET lock:resource unique_token NX EX 30

# 释放锁（Lua 脚本保证原子性）
EVAL "if redis.call('GET', KEYS[1]) == ARGV[1] then return redis.call('DEL', KEYS[1]) else return 0 end" 1 lock:resource unique_token
```

**存在的问题：**

1. **锁超时**：业务没执行完锁就过期了 → 解决：看门狗（Watchdog）自动续期
2. **不可重入**：同一线程不能多次获取同一把锁 → 解决：记录加锁次数（Redisson）
3. **集群主从切换**：Master 加锁后还没同步到 Slave 就宕机了 → 解决：RedLock 算法
4. **GC 停顿**：JVM GC 时可能导致看门狗续期失败 → 这是所有分布式锁的通用问题

生产环境推荐使用 Redisson 框架，它解决了以上大部分问题。

### Q9：Redis 的跳表是什么？为什么用跳表而不是红黑树？

**参考答案：**

跳表（Skip List）是一种基于有序链表的概率性数据结构，通过多级索引实现 O(log N) 的查找、插入、删除。

**选择跳表而不是红黑树的原因：**

1. **实现简单**：跳表代码量约为红黑树的 1/3
2. **范围查询高效**：跳表找到起始节点后顺序遍历即可，红黑树需要中序遍历
3. **并发友好**：跳表插入删除只需修改相邻节点指针，更容易实现无锁并发
4. **内存可调**：通过调整概率参数来平衡空间和时间

---

### Q10：Redis Cluster 的工作原理是什么？

**参考答案：**

Redis Cluster 将数据分为 16384 个 Hash Slot，通过 `CRC16(key) % 16384` 确定每个 key 属于哪个 Slot，每个 Master 负责一部分 Slot。

- **数据分片**：每个节点负责一部分 Slot，客户端可以连接任意节点
- **请求转发**：如果请求的 key 不在当前节点，会返回 MOVED 重定向
- **高可用**：每个 Master 有至少一个 Slave，Master 故障时 Slave 自动提升
- **故障检测**：节点间通过 Gossip 协议互相通信，超过半数节点标记某节点为 PFAIL → FAIL
- **故障转移**：类似哨兵的选举过程，选出新的 Master

---

### Q11：如何保证 Redis 和 MySQL 的数据最终一致性？

**参考答案：**

1. **Cache Aside 模式**：先更新数据库，再删除缓存
2. **延迟双删**：写操作时先删缓存 → 更新数据库 → 延迟再删缓存（补偿机制）
3. **消息队列重试**：更新数据库后发消息到 MQ，消费者异步删除缓存，失败可重试
4. **Canal 监听 Binlog**：通过 Canal 监听 MySQL Binlog，自动同步到 Redis
5. **定期全量同步**：定时任务全量对比 Redis 和 MySQL 数据

---

### Q12：Redis 的 Pipeline 有什么用？

**参考答案：**

Pipeline（管道）将多个命令打包，一次性发送给 Redis 服务端，服务端依次执行后一次性返回结果。

**优势：**
- 减少 TCP 往返次数：N 个命令从 N 次 RTT 降到 1 次 RTT
- 大幅提升吞吐量：批量操作场景性能提升 5-10 倍

**注意：**
- Pipeline 中的命令不是原子执行的（不是事务）
- Pipeline 不宜过大，否则会占用过多内存
- 适合批量读写场景，不适合需要实时返回结果的单条查询

---

### Q13：什么是 Redis 的 Hot Key 和 Big Key？如何解决？

**参考答案：**

**Big Key（大 Key）：** Value 占用内存过大的 Key
- 发现：`redis-cli --bigkeys`
- 危害：删除阻塞、内存不均、网络拥塞
- 解决：拆分数据、压缩 Value、使用 UNLINK 异步删除

**Hot Key（热点 Key）：** 被大量并发访问的 Key
- 发现：监控 QPS 异常高的 Key
- 危害：单节点压力过大
- 解决：本地缓存 + Redis、Key 分散（复制多份）、增加从节点

---

### Q14：Redis 的 Sentinel 和 Cluster 有什么区别？

**参考答案：**

| 对比项 | Sentinel | Cluster |
|-------|----------|---------|
| **功能** | 高可用（故障转移） | 高可用 + 数据分片 |
| **数据分布** | 所有节点数据相同 | 数据分片到不同节点 |
| **容量** | 受限于单节点内存 | 水平扩展，容量无限 |
| **最小节点** | 1 Master + 1 Slave + 1 Sentinel | 3 Master + 3 Slave |
| **客户端** | 连接任意节点 | 需要感知 Cluster |
| **多 Key 操作** | 支持 | 仅支持同一 Slot 的 Key |
| **适用场景** | 数据量不大，需要高可用 | 数据量大，需要分片 + 高可用 |

### Q15：Redis 如何做限流？

**参考答案：**

三种常见方案：

**1. 固定窗口计数器**
```python
# 每分钟最多 100 次请求
key = f"rate:{user_id}:{minute}"
count = r.incr(key)
if count == 1:
    r.expire(key, 60)
return count <= 100
```
问题：窗口边界可能突发 2 倍流量。

**2. 滑动窗口计数器（ZSet）**
```python
# 60 秒内最多 100 次请求
now = time.time()
r.zadd(f"rate:{user_id}", {f"{now}": now})
r.zremrangebyscore(f"rate:{user_id}", 0, now - 60)
count = r.zcard(f"rate:{user_id}")
return count <= 100
```

**3. 令牌桶 / 漏桶（Lua 脚本实现）**
更精确的限流算法，推荐使用 Redis + Lua 脚本实现，或使用专门的限流库（如 ratelimit）。
