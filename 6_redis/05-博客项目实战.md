## 二、文章详情缓存（Cache Aside 模式）

Cache Aside 是最经典的缓存模式：**读时回填缓存，写时删除缓存**。

```
┌──────────────────────────────────────────────────────────────┐
│              Cache Aside 模式流程                              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  读操作：                                                    │
│  ┌────────┐    ┌───────┐    ┌───────┐                        │
│  │ 客户端  │───→│ Redis │───→│ MySQL │                        │
│  │ 请求   │    │(缓存)  │    │(数据库)│                        │
│  └────────┘    └───┬───┘    └───┬───┘                        │
│       │            │            │                             │
│       │     ┌──────▼──────┐    │                             │
│       │     │ 缓存命中？  │    │                             │
│       │     └──┬───────┬──┘    │                             │
│       │      是│       │否     │                             │
│       │     ┌──▼──┐  ┌──▼──────▼──┐                         │
│       │     │直接  │  │查数据库    │                         │
│       │     │返回  │  │回写缓存    │                         │
│       │     └──┬──┘  └──┬─────────┘                         │
│       │        └────┬───┘                                   │
│       │             ▼                                       │
│       │         返回数据                                     │
│                                                              │
│  写操作：                                                    │
│  1. 先更新数据库                                             │
│  2. 再删除缓存（不是更新缓存！）                               │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 三、实时浏览量计数

```
┌──────────────────────────────────────────────────────────────┐
│                 浏览量计数方案对比                              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  方案一：直接更新数据库（不推荐）                               │
│  UPDATE posts SET views = views + 1 WHERE id = 1001          │
│  → 每次浏览都要写数据库，高并发时数据库扛不住                    │
│  → 行锁竞争严重                                               │
│                                                              │
│  方案二：Redis 计数 + 定时同步（推荐）                          │
│  INCR demo:views:1001  → Redis 实时计数                      │
│  定时任务（每 5 分钟）→ 批量同步到 MySQL                       │
│  → 写 Redis 极快（内存操作，无锁竞争）                         │
│  → 数据库压力大幅降低                                          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 四、文章热度排行榜

```
┌──────────────────────────────────────────────────────────────┐
│                 热度排行榜设计                                 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  热度计算规则：                                                │
│  - 浏览 +1 分                                                 │
│  - 点赞 +5 分                                                 │
│  - 评论 +10 分                                                │
│  - 收藏 +8 分                                                 │
│                                                              │
│  使用 ZSet 存储：                                              │
│  key = demo:hot_articles                                      │
│  score = 热度分数                                             │
│  member = 文章 ID                                             │
│                                                              │
│  操作：                                                      │
│  ZINCRBY demo:hot_articles 1 "1001"    → 浏览 +1             │
│  ZINCRBY demo:hot_articles 5 "1001"    → 点赞 +5             │
│  ZREVRANGE demo:hot_articles 0 9      → Top 10              │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Python 实现

```python
HOT_ARTICLES_KEY = "demo:hot_articles"

# 热度权重配置
SCORE_WEIGHTS = {
    "view": 1,
    "like": 5,
    "comment": 10,
    "collect": 8,
}


def increase_heat(article_id, action="view"):
    """
    增加文章热度
    :param article_id: 文章 ID
    :param action: 动作类型 (view/like/comment/collect)
    """
    score = SCORE_WEIGHTS.get(action, 1)
    r.zincrby(HOT_ARTICLES_KEY, score, article_id)
    print(f"[热度] 文章 {article_id} {action} +{score}")


def get_hot_articles(top_n=10):
    """
    获取热门文章 Top N
    :return: [(article_id, score), ...]
    """
    results = r.zrevrange(
        HOT_ARTICLES_KEY, 0, top_n - 1, withscores=True
    )
    return [(member, int(score)) for member, score in results]


def get_article_rank(article_id):
    """
    获取文章的排名
    :return: 排名（从 1 开始），None 表示不在排行榜中
    """
    rank = r.zrevrank(HOT_ARTICLES_KEY, article_id)
    if rank is not None:
        return rank + 1  # 排名从 1 开始
    return None


def get_article_heat(article_id):
    """
    获取文章的热度分数
    """
    score = r.zscore(HOT_ARTICLES_KEY, article_id)
    if score is not None:
        return int(score)
    return 0


# ==================== 测试 ====================
if __name__ == "__main__":
    print("=== 热度排行榜测试 ===\n")

    # 模拟用户行为
    import random

    articles = ["1001", "1002", "1003", "1004", "1005"]
    actions = ["view", "like", "comment", "collect"]

    for _ in range(200):
        article_id = random.choice(articles)
        # 浏览的概率更高
        action = random.choices(
            actions, weights=[70, 15, 10, 5], k=1
        )[0]
        increase_heat(article_id, action)

    # 获取 Top 5
    print("\n--- 热门文章 Top 5 ---")
    top5 = get_hot_articles(5)
    for rank, (aid, score) in enumerate(top5, 1):
        print(f"  第 {rank} 名: 文章 {aid} (热度: {score})")

    # 查询某篇文章的排名和热度
    print("\n--- 文章 1001 详情 ---")
    rank = get_article_rank("1001")
    heat = get_article_heat("1001")
    print(f"  排名: 第 {rank} 名")
    print(f"  热度: {heat}")
```

## 五、用户 Session 管理

```
┌──────────────────────────────────────────────────────────────┐
│              Session 管理方案对比                              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  方案一：服务器本地 Session（不推荐）                           │
│  - Session 存在服务器内存中                                    │
│  - 多台服务器无法共享 Session                                  │
│  - 用户请求打到不同服务器需要重新登录                           │
│                                                              │
│  方案二：Redis 集中存储 Session（推荐）                         │
│  - Session 存在 Redis 中                                      │
│  - 所有服务器共享同一个 Redis                                  │
│  - 用户无感知切换服务器                                       │
│  - 设置过期时间自动清理                                        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Python 实现

```python
import uuid


def create_session(user_id, user_info, expire_seconds=7200):
    """
    创建用户 Session
    :param user_id: 用户 ID
    :param user_info: 用户信息字典
    :param expire_seconds: 过期时间（秒），默认 2 小时
    :return: session_token
    """
    # 生成唯一的 session token
    session_token = str(uuid.uuid4())
    session_key = f"demo:session:{session_token}"

    # 将用户信息存入 Redis（Hash 类型）
    user_info["user_id"] = user_id
    user_info["login_time"] = time.strftime("%Y-%m-%d %H:%M:%S")

    r.hset(session_key, mapping=user_info)
    r.expire(session_key, expire_seconds)

    print(f"[Session] 创建会话: {session_token}")
    return session_token


def get_session(session_token):
    """
    获取 Session 信息
    :return: 用户信息字典，None 表示 Session 无效
    """
    session_key = f"demo:session:{session_token}"
    user_info = r.hgetall(session_key)

    if not user_info:
        return None

    # 每次访问刷新过期时间（滑动过期）
    r.expire(session_key, 7200)
    return user_info


def destroy_session(session_token):
    """
    销毁 Session（用户登出）
    """
    session_key = f"demo:session:{session_token}"
    r.delete(session_key)
    print(f"[Session] 销毁会话: {session_token}")


# ==================== 测试 ====================
if __name__ == "__main__":
    print("=== Session 管理测试 ===\n")

    # 用户登录
    token = create_session(
        "1001",
        {"username": "browncutie", "nickname": "大数据学习者"},
    )

    # 验证 Session
    user_info = get_session(token)
    print(f"当前用户: {user_info['username']}")
    print(f"登录时间: {user_info['login_time']}")

    # 查看过期时间
    ttl = r.ttl(f"demo:session:{token}")
    print(f"Session 剩余时间: {ttl} 秒")

    # 用户登出
    destroy_session(token)

    # 验证 Session 已失效
    user_info = get_session(token)
    print(f"登出后: {user_info}")  # None
```

## 六、分布式锁

```
┌──────────────────────────────────────────────────────────────┐
│                  分布式锁的必要性                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  场景：数据平台部署了 2 台服务器                                │
│  用户快速点击"发布"按钮，请求可能同时到达两台服务器              │
│                                                              │
│  服务器 A                    服务器 B                         │
│  ┌──────────┐              ┌──────────┐                     │
│  │ 检查文章  │              │ 检查文章  │                     │
│  │ 不存在    │              │ 不存在    │                     │
│  │ 创建文章  │              │ 创建文章  │ ← 重复创建！         │
│  └──────────┘              └──────────┘                     │
│                                                              │
│  解决方案：分布式锁                                            │
│  - 先获取锁，获取成功才能执行操作                               │
│  - 执行完成后释放锁                                            │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Python 实现

```python
import uuid
import time


def acquire_lock(lock_name, expire_seconds=30):
    """
    获取分布式锁
    :param lock_name: 锁的名称
    :param expire_seconds: 锁的过期时间（秒）
    :return: 锁标识符（用于释放锁），None 表示获取失败
    """
    lock_key = f"demo:lock:{lock_name}"
    # 生成唯一标识，确保只释放自己加的锁
    lock_token = str(uuid.uuid4())

    # SET key value NX EX seconds
    # NX = 只在 key 不存在时设置（原子操作）
    # EX = 设置过期时间
    acquired = r.set(lock_key, lock_token, nx=True, ex=expire_seconds)

    if acquired:
        print(f"[锁] 获取锁成功: {lock_name} (token: {lock_token[:8]}...)")
        return lock_token
    else:
        print(f"[锁] 获取锁失败: {lock_name}（已被占用）")
        return None


def release_lock(lock_name, lock_token):
    """
    释放分布式锁
    使用 Lua 脚本保证原子性：只有持有锁的线程才能释放
    """
    lua_script = """
    if redis.call('GET', KEYS[1]) == ARGV[1] then
        return redis.call('DEL', KEYS[1])
    else
        return 0
    end
    """
    lock_key = f"demo:lock:{lock_name}"
    result = r.eval(lua_script, 1, lock_key, lock_token)
    if result:
        print(f"[锁] 释放锁成功: {lock_name}")
    else:
        print(f"[锁] 释放锁失败: {lock_name}（锁已过期或被其他人持有）")
    return bool(result)


def publish_article_with_lock(article_id, article_data):
    """
    带分布式锁的文章发布
    """
    lock_name = f"publish:{article_id}"
    lock_token = acquire_lock(lock_name, expire_seconds=30)

    if lock_token is None:
        print(f"文章 {article_id} 正在发布中，请稍后再试")
        return False

    try:
        # 模拟发布操作
        print(f"开始发布文章 {article_id}...")
        time.sleep(2)  # 模拟耗时操作
        print(f"文章 {article_id} 发布成功！")
        return True
    finally:
        # 无论成功还是失败，都要释放锁
        release_lock(lock_name, lock_token)


# ==================== 测试 ====================
if __name__ == "__main__":
    print("=== 分布式锁测试 ===\n")

    # 测试 1：正常获取和释放锁
    publish_article_with_lock("1001", {"title": "新文章"})

    # 测试 2：模拟并发（锁已被占用）
    print("\n--- 模拟并发场景 ---")

    # 手动加锁模拟另一个进程
    lock_token = acquire_lock("publish:1002", expire_seconds=30)

    # 尝试获取同一个锁（应该失败）
    token2 = acquire_lock("publish:1002", expire_seconds=30)

    # 释放锁后再获取
    release_lock("publish:1002", lock_token)
    token3 = acquire_lock("publish:1002", expire_seconds=30)
    release_lock("publish:1002", token3)
```

```
│  → MySQL 也没有                                               │
│  → 每次都要查 MySQL，缓存形同虚设                               │
│  → 恶意攻击：大量请求不存在的 key，压垮数据库                    │
│                                                              │
│  ┌────────┐    ┌───────┐    ┌───────┐                        │
│  │ 攻击者  │───→│ Redis │───→│ MySQL │ ← 每次都穿透！          │
│  │请求99999│    │  miss │    │  miss │                        │
│  └────────┘    └───────┘    └───────┘                        │
│                                                              │
│  解决方案：                                                    │
│  1. 缓存空值：查询不到的也缓存一个空值，设短过期时间             │
│  2. 布隆过滤器：在缓存前加一层过滤，不存在的 key 直接拦截        │
│  3. 参数校验：在入口处拦截非法请求（如 ID 为负数）              │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

```python
def get_article_anti_penetration(article_id):
    """
    获取文章（防缓存穿透）
    """
    # 参数校验
    if not article_id.isdigit():
        return None

    cache_key = f"demo:article:{article_id}"

    # 查缓存
    cached = r.get(cache_key)
    if cached is not None:
        if cached == "NULL":
            # 缓存了空值，直接返回 None
            return None
        return json.loads(cached)

    # 查数据库
    article_json = db.get_article(article_id)
    if article_json is None:
        # 缓存空值，防止穿透（过期时间设短一点，比如 5 分钟）
        r.setex(cache_key, 300, "NULL")
        print(f"[防穿透] 文章 {article_id} 不存在，缓存空值 5 分钟")
        return None

    # 回写缓存
    r.setex(cache_key, 3600, article_json)
    return json.loads(article_json)
```

### 7.2 缓存击穿

```
┌──────────────────────────────────────────────────────────────┐
│                  缓存击穿问题                                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  问题：某个热点 key 过期的瞬间，大量并发请求同时打到数据库       │
│                                                              │
│  正常情况：                                                   │
│  ┌───────┐  命中  ┌───────┐                                  │
│  │ 请求1 │──────→│ Redis │                                  │
│  │ 请求2 │──────→│       │                                  │
│  └───────┘       └───────┘                                  │
│                                                              │
│  key 过期瞬间：                                                │
│  ┌───────┐  miss  ┌───────┐  全部打到  ┌───────┐            │
│  │ 请求1 │──────→│ Redis │──────────→ │ MySQL │ ← 瞬间高并发│
│  │ 请求2 │──────→│  miss │──────────→ │       │            │
│  │ 请求3 │──────→│       │──────────→ │       │            │
│  │ ...   │       └───────┘           └───────┘            │
│  └───────┘                                                  │
│                                                              │
│  解决方案：                                                    │
│  1. 互斥锁：只允许一个线程去查数据库并重建缓存                   │
│  2. 逻辑过期：缓存不设过期时间，在 value 中存逻辑过期时间        │
│     过期了返回旧数据，同时异步更新缓存                          │
│  3. 热点 key 永不过期                                          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

```python
def get_article_anti_stamping(article_id):
    """
    获取文章（防缓存击穿 - 互斥锁方案）
    """
    cache_key = f"demo:article:{article_id}"
    lock_key = f"demo:lock:rebuild:{article_id}"

    # 查缓存
    cached = r.get(cache_key)
    if cached:
        return json.loads(cached)

    # 缓存未命中，尝试获取互斥锁
    lock_token = str(uuid.uuid4())
    acquired = r.set(lock_key, lock_token, nx=True, ex=10)

    if acquired:
        # 获取到锁，去查数据库
        try:
            print(f"[重建缓存] 文章 {article_id} 缓存过期，重建中...")
            article_json = db.get_article(article_id)
            if article_json:
                r.setex(cache_key, 3600, article_json)
                return json.loads(article_json)
            return None
        finally:
            # 释放锁（Lua 脚本保证原子性）
            lua_script = """
            if redis.call('GET', KEYS[1]) == ARGV[1] then
                return redis.call('DEL', KEYS[1])
            end
            return 0
            """
            r.eval(lua_script, 1, lock_key, lock_token)
    else:
        # 没有获取到锁，短暂等待后重试
        time.sleep(0.1)
        return get_article_anti_stamping(article_id)
```

### 7.3 缓存雪崩

```
┌──────────────────────────────────────────────────────────────┐
│                  缓存雪崩问题                                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  问题：大量 key 同时过期，或者 Redis 宕机                      │
│  → 所有请求同时打到数据库                                       │
│  → 数据库瞬间崩溃                                             │
│                                                              │
│  ┌───────┐  miss  ┌───────┐                                  │
│  │ 请求1 │──────→│ Redis │ ← 大量 key 同时过期                │
│  │ 请求2 │──────→│ 全部  │                                  │
│  │ 请求3 │──────→│ miss  │──────→ ┌───────┐                  │
│  │ ...   │       └───────┘        │ MySQL │ ← 数据库崩溃      │
│  │ 请求N │                         │ 崩溃！ │                  │
│  └───────┘                         └───────┘                  │
│                                                              │
│  解决方案：                                                    │
│  1. 过期时间加随机值：避免大量 key 同时过期                     │
│     原本设 3600 秒 → 改为 3600 + random(0, 600)              │
│  2. 多级缓存：本地缓存（如 LRU Cache）+ Redis                 │
│  3. Redis 高可用：主从 + 哨兵，防止单点故障                     │
│  4. 限流降级：数据库压力过大时，部分请求降级返回                 │
│  5. 缓存预热：系统启动时提前加载热点数据到 Redis                │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

```python
import random

def cache_with_random_ttl(cache_key, value, base_ttl=3600, jitter=600):
    """
    带随机 TTL 的缓存设置
    防止大量 key 同时过期导致缓存雪崩
    """
    ttl = base_ttl + random.randint(0, jitter)
    r.setex(cache_key, ttl, value)
    print(f"[缓存] key={cache_key}, TTL={ttl}秒")
    return ttl


def get_article_anti_avalanche(article_id):
    """
    获取文章（防缓存雪崩）
    """
    cache_key = f"demo:article:{article_id}"

    # 查缓存
    cached = r.get(cache_key)
    if cached:
        return json.loads(cached)

    # 查数据库
    article_json = db.get_article(article_id)
    if article_json is None:
        return None

    # 回写缓存（带随机 TTL，防止雪崩）
    cache_with_random_ttl(cache_key, article_json, base_ttl=3600, jitter=600)
    return json.loads(article_json)
```

## 八、面试题

### Q1：什么是缓存穿透？怎么解决？

**参考答案：**

**缓存穿透**是指请求的数据在 Redis 和数据库中都不存在，每次请求都会穿透缓存直接打到数据库。

**解决方案：**

1. **缓存空值**：查询不到的 key 也缓存一个空值（如 "NULL"），设置较短的过期时间（如 5 分钟）
2. **布隆过滤器**：在 Redis 前面加一层布隆过滤器，将所有可能存在的 key 映射到位图中。请求先过布隆过滤器，不在位图中的直接返回
3. **参数校验**：在接口入口做基本参数校验（如 ID 必须为正整数）
4. **限流**：对频繁请求不存在的 key 的 IP 进行限流

**布隆过滤器的特点：**
- 判断"不存在"则一定不存在（不会误判）
- 判断"存在"则可能不存在（有误判率）
- 适合大数据量下的"快速判断是否存在"

---

### Q2：什么是缓存击穿？怎么解决？

**参考答案：**

**缓存击穿**是指某个热点 key 过期的瞬间，大量并发请求同时穿透到数据库。

**解决方案：**

1. **互斥锁（Mutex）**：缓存未命中时，只允许一个线程去查数据库并重建缓存，其他线程等待后重试
2. **逻辑过期**：不设物理过期时间，在 value 中存逻辑过期时间。发现过期后返回旧数据，同时异步更新缓存
3. **热点 key 永不过期**：对确认的热点 key 不设置过期时间，通过后台任务定期刷新

---

### Q3：什么是缓存雪崩？怎么解决？

**参考答案：**

**缓存雪崩**是指大量 key 在同一时间过期，或者 Redis 宕机，导致大量请求同时打到数据库。

**解决方案：**

1. **过期时间加随机值**：给每个 key 的 TTL 加一个随机偏移量，避免同时过期
2. **Redis 高可用**：主从复制 + 哨兵/Cluster，防止单点故障
3. **多级缓存**：本地缓存（如 Caffeine/Guava Cache）+ Redis
4. **限流降级**：数据库压力大时，部分请求降级返回默认值
5. **缓存预热**：系统启动或大促前，提前加载热点数据
