# 04 - 高级特性：持久化、事务、发布订阅与高可用

## 一、持久化：让内存数据不丢失

Redis 的数据存在内存里，一旦服务器重启或宕机，数据就没了。**持久化（Persistence）** 就是把内存数据保存到磁盘上，重启后可以恢复。

Redis 提供两种持久化方案：**RDB** 和 **AOF**。

```
┌──────────────────────────────────────────────────────────────┐
│                    Redis 持久化方案概览                        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────────┐              ┌──────────────┐              │
│  │   RDB 快照   │              │  AOF 日志    │              │
│  │              │              │              │              │
│  │ 定时拍照     │              │ 记录每次操作  │              │
│  │ 类似"快照"   │              │ 类似"日记"   │              │
│  │              │              │              │              │
│  │ 文件小       │              │ 文件大       │              │
│  │ 恢复快       │              │ 恢复慢       │              │
│  │ 可能丢数据   │              │ 数据更安全   │              │
│  └──────────────┘              └──────────────┘              │
│                                                              │
│  两者可以同时开启，互为补充                                    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 1.1 RDB（Redis Database）-- 快照持久化

RDB 持久化就是在某个时间点，把 Redis 内存中的**全部数据**拍一张"快照"，保存到磁盘上。

```
┌──────────────────────────────────────────────────────────────┐
│                  RDB 快照工作原理                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Redis 内存                           磁盘                    │
│  ┌──────────────────────┐            ┌──────────────┐        │
│  │ k1=v1  k2=v2  k3=v3 │ ────→      │ dump.rdb    │        │
│  │ k4=v4  k5=v5  ...    │  定时拍照   │              │        │
│  └──────────────────────┘            │ (二进制文件)  │        │
│                                      └──────────────┘        │
│                                                              │
│  触发条件（redis.conf 配置）：                                 │
│  save 900 1      → 900 秒（15 分钟）内有 1 次修改就拍照        │
│  save 300 10     → 300 秒（5 分钟）内有 10 次修改就拍照        │
│  save 60 10000   → 60 秒（1 分钟）内有 10000 次修改就拍照      │
│                                                              │
│  也可以手动触发：                                              │
│  redis-cli SAVE      → 阻塞式保存（不推荐，会卡住 Redis）      │
│  redis-cli BGSAVE    → 后台保存（fork 子进程，推荐）           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**BGSAVE 的执行过程：**

```
┌──────────────────────────────────────────────────────────────┐
│                  BGSAVE 执行过程                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  主进程（继续处理客户端请求）                                   │
│  ┌──────────────────────────────────────────┐               │
│  │  1. fork() 创建子进程                     │               │
│  │  2. 子进程拷贝父进程的内存页表              │               │
│  │  3. 子进程将内存数据写入 RDB 文件           │               │
│  │  4. 写入完成后替换旧的 RDB 文件             │               │
│  └──────────────────────────────────────────┘               │
│                                                              │
│  子进程（负责写磁盘）                                          │
│  ┌──────────────────────────────────────────┐               │
│  │  Copy-on-Write（写时复制）                │               │
│  │  - fork 后，父子进程共享内存页             │               │
│  │  - 父进程修改某页时，才复制该页            │               │
│  │  - 大部分情况下不需要复制                 │               │
│  └──────────────────────────────────────────┘               │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 AOF（Append Only File）-- 追加日志持久化

AOF 持久化把 Redis 执行的每一条**写命令**都记录到日志文件中。恢复时，重新执行这些命令就能恢复数据。

```
┌──────────────────────────────────────────────────────────────┐
│                  AOF 日志工作原理                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  客户端操作                AOF 日志文件                        │
│  ──────────               ──────────────                      │
│  SET k1 v1          →    *3                                  │
│  SET k2 v2          →    $3                                  │
│  INCR k1            →    SET                                  │
│  ...                →    $2                                  │
│                             k1                                │
│                             $2                                │
│                             v1                                │
│                             ...                               │
│                                                              │
│  恢复时：逐条重新执行 AOF 中的命令，还原数据                     │
│                                                              │
│  AOF 同步策略（appendfsync）：                                  │
│  ┌────────────┬──────────────────────────────────────────┐   │
│  │  always    │ 每次写操作都同步到磁盘 → 最安全，最慢      │   │
│  │  everysec  │ 每秒同步一次 → 折中方案（推荐）           │   │
│  │  no        │ 由操作系统决定何时同步 → 最快，最不安全    │   │
│  └────────────┴──────────────────────────────────────────┘   │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**AOF 重写（AOF Rewrite）：**

AOF 文件会越来越大（每次操作都追加），需要定期"瘦身"：

```
┌──────────────────────────────────────────────────────────────┐
│                  AOF 重写示例                                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  重写前（冗余命令多）：                                        │
│  SET count 1                                                │
│  SET count 2                                                │
│  SET count 3                                                │
│  SET count 4                                                │
│  SET count 5                                                │
│  DEL other_key                                             │
│  SET other_key "new_value"                                  │
│                                                              │
│  重写后（只保留最终结果）：                                     │
│  SET count 5                                                │
│  SET other_key "new_value"                                  │
│                                                              │
│  触发方式：                                                   │
│  - 自动：当 AOF 文件大小是上次重写后的 2 倍且 > 64MB           │
│  - 手动：BGREWRITEAOF 命令                                   │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 1.3 RDB vs AOF 对比

```
┌──────────────────────────────────────────────────────────────────┐
│                    RDB vs AOF 完整对比                            │
├──────────────┬────────────────────┬──────────────────────────────┤
│   对比维度    │      RDB           │       AOF                    │
├──────────────┼────────────────────┼──────────────────────────────┤
│ 持久化方式    │ 定时全量快照        │ 记录每条写命令                │
│ 数据安全性    │ 可能丢失几分钟数据  │ 最多丢失 1 秒（everysec）    │
│ 文件大小      │ 小（二进制压缩）    │ 大（文本格式，可重写压缩）     │
│ 恢复速度      │ 快（直接加载）      │ 慢（逐条重放命令）            │
│ 对性能影响    │ fork 时可能有短暂   │ always 模式影响大             │
│              │ 延迟                │ everysec 模式影响小           │
│ 适用场景      │ 备份、灾难恢复      │ 数据安全要求高的场景           │
│ 配置开关      │ 默认开启            │ 默认关闭（需手动开启）         │
└──────────────┴────────────────────┴──────────────────────────────┘
```

### 1.4 如何选择持久化方案？

```
┌──────────────────────────────────────────────────────────────┐
│                 持久化方案选择指南                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  方案一：只用 RDB（推荐用于缓存场景）                           │
│  ├── 数据可以接受少量丢失                                      │
│  ├── 恢复速度快                                               │
│  └── 配置：save 900 1 / save 300 10 / save 60 10000          │
│                                                              │
│  方案二：只用 AOF（推荐用于数据安全场景）                       │
│  ├── 数据不能丢                                               │
│  ├── 最多丢失 1 秒数据                                        │
│  └── 配置：appendonly yes / appendfsync everysec              │
│                                                              │
│  方案三：RDB + AOF（推荐用于生产环境）                          │
│  ├── RDB 用于定期备份和快速恢复                                │
│  ├── AOF 用于保证数据安全                                      │
│  └── Redis 4.0+ 支持混合持久化（RDB + 增量 AOF）              │
│                                                              │
│  数据分析项目推荐：方案三（RDB + AOF）                              │
│  - 文章浏览量等计数数据，丢失少量可以接受                        │
│  - 用户 Session 等数据，丢失会影响用户体验                      │
│  - 综合考虑，RDB + AOF 兼顾安全和性能                           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 二、事务：MULTI / EXEC / DISCARD / WATCH

Redis 的事务和数据库的事务不同。Redis 的事务**保证一组命令按顺序执行，且不被其他客户端打断**，但**不支持回滚**。

### 2.1 基本事务操作

```bash
# ==================== 事务基本流程 ====================

# 开启事务
MULTI
# 输出：OK

# 在事务中输入命令（不会立即执行，而是进入队列）
SET user:1001:name "browncutie"
SET user:1001:age 25
INCR user:1001:login_count
# 每条命令都返回：QUEUED（已入队）

# 执行事务（所有命令按顺序执行）
EXEC
# 1) OK
# 2) OK
# 3) (integer) 1

# ==================== 取消事务 ====================

MULTI
SET k1 "v1"
SET k2 "v2"
# 改变主意了，取消事务
DISCARD
# 输出：OK（所有入队的命令被丢弃）

# ==================== 事务中的错误 ====================

# 情况 1：命令语法错误 → 整个事务不执行
MULTI
SET k1 "v1"
SETTT k2 "v2"          # 语法错误（命令不存在）
SET k3 "v3"
EXEC
# 输出：(error) ERR unknown command 'SETTT'
# k1 和 k3 也不会执行！

# 情况 2：命令类型错误 → 其他命令继续执行（不支持回滚！）
MULTI
SET k1 "v1"
INCR k1                  # k1 是字符串，不能 INCR → 类型错误
SET k3 "v3"
EXEC
# 1) OK              ← k1 设置成功
# 2) (error) ...     ← INCR 失败
# 3) OK              ← k3 设置成功（没有回滚！）
```

```
┌──────────────────────────────────────────────────────────────┐
│                 Redis 事务 vs 数据库事务                       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│                    数据库事务 (ACID)                           │
│  ┌──────────────────────────────────────────┐               │
│  │  Atomicity（原子性）  → 支持              │               │
│  │  Consistency（一致性）→ 支持              │               │
│  │  Isolation（隔离性） → 支持              │               │
│  │  Durability（持久性） → 支持              │               │
│  └──────────────────────────────────────────┘               │
│                                                              │
│                    Redis 事务                                 │
│  ┌──────────────────────────────────────────┐               │
│  │  Atomicity（原子性）  → 部分支持（语法错误时整体不执行）    │
│  │  Consistency（一致性）→ 不保证（类型错误时部分命令执行）   │
│  │  Isolation（隔离性） → 支持（事务执行期间不被打断）        │
│  │  Durability（持久性） → 取决于持久化配置                   │
│  └──────────────────────────────────────────┘               │
│                                                              │
│  核心区别：Redis 事务不支持回滚！                               │
│  设计理念：错误应该在开发阶段发现，而不是靠运行时回滚            │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 乐观锁：WATCH

`WATCH` 命令可以在事务执行前监控一个或多个键。如果在事务执行期间，这些键被其他客户端修改了，整个事务会被取消。

```bash
# ==================== WATCH 乐观锁示例 ====================

# 场景：用户余额扣款

# 客户端 A
SET balance:1001 1000          # 初始余额 1000
WATCH balance:1001             # 监控余额
balance = GET balance:1001      # balance = 1000
MULTI
SET balance:1001 900            # 扣款 100
# （此时客户端 A 还没有 EXEC）

# 客户端 B（在 A 执行 EXEC 之前）
SET balance:1001 800            # 另一笔扣款

# 客户端 A
EXEC
# 输出：(nil)    ← 事务被取消！因为 balance:1001 被修改了

# 客户端 A 需要重试：
WATCH balance:1001
balance = GET balance:1001      # balance = 800（最新的）
MULTI
SET balance:1001 700            # 扣款 100
EXEC
# 输出：OK    ← 事务成功
```

```
┌──────────────────────────────────────────────────────────────┐
│                  WATCH 乐观锁流程                              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  客户端 A                       客户端 B                     │
│  ──────────                     ──────────                   │
│  WATCH balance                  │                            │
│  balance = GET balance          │                            │
│  (得到 1000)                    │                            │
│  MULTI                          │                            │
│  SET balance 900                │                            │
│                                 │  SET balance 800          │
│                                 │  (修改了被 WATCH 的键)      │
│  EXEC                           │                            │
│  → 返回 nil                     │                            │
│  → 事务被取消！                  │                            │
│                                 │                            │
│  (重新开始)                      │                            │
│  WATCH balance                  │                            │
│  balance = GET balance          │                            │
│  (得到 800)                     │                            │
│  MULTI                          │                            │
│  SET balance 700                │                            │
│  EXEC                           │                            │
│  → OK                           │                            │
│  → 事务成功！                    │                            │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 2.3 数据平台场景：文章发布事务

```bash
# 发布文章需要同时做以下操作：
# 1. 保存文章详情
# 2. 更新用户文章计数
# 3. 加入最新文章列表
# 4. 初始化浏览量
```

## 三、发布订阅（Pub/Sub）

Redis 提供了轻量级的消息发布订阅功能。发布者向频道发送消息，所有订阅该频道的订阅者都能收到消息。

```
┌──────────────────────────────────────────────────────────────┐
│                  发布订阅模型                                   │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  发布者                   Redis                  订阅者       │
│  ──────                  ──────                 ──────        │
│  ┌──────────┐           ┌──────────┐          ┌──────────┐   │
│  │ PUBLISH  │──消息──→  │ channel  │──消息──→ │ 订阅者 1 │   │
│  │ channel  │           │  (频道)  │──消息──→ │ 订阅者 2 │   │
│  │  msg     │           │          │──消息──→ │ 订阅者 3 │   │
│  └──────────┘           └──────────┘          └──────────┘   │
│                                                              │
│  特点：                                                      │
│  - 一条消息可以被所有订阅者收到（广播）                          │
│  - 订阅者只能收到订阅之后的消息（不存储历史消息）                 │
│  - 没有持久化，Redis 重启后消息丢失                             │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 3.1 常用命令

```bash
# ==================== 订阅频道 ====================

# 订阅一个频道（会进入阻塞状态，等待消息）
SUBSCRIBE demo:notifications
# 输出：
# Reading messages... (press Ctrl-C to quit)
# 1) "subscribe"
# 2) "demo:notifications"
# 3) (integer) 1

# 订阅多个频道
SUBSCRIBE demo:notifications demo:comments

# 订阅模式（通配符）
PSUBSCRIBE demo:*    # 订阅所有 demo: 开头的频道

# ==================== 发布消息 ====================

# 向频道发布消息
PUBLISH demo:notifications "新文章发布：Redis入门教程"
# 输出：(integer) 1    ← 1 个订阅者收到了消息

PUBLISH demo:notifications "系统维护通知：今晚 23:00-24:00"
# 输出：(integer) 1

# ==================== 查看频道信息 ====================

# 查看当前活跃的频道
PUBSUB CHANNELS
PUBSUB CHANNELS demo:*    # 按模式过滤

# 查看某个频道的订阅者数量
PUBSUB NUMSUB demo:notifications

# 查看某个模式的订阅者数量
PUBSUB NUMPAT
```

### 3.2 数据平台场景：实时通知

```bash
# 终端 1：用户 1001 订阅通知
SUBSCRIBE user:1001:notifications

# 终端 2：系统发布通知
PUBLISH user:1001:notifications '{"type":"comment","content":"你的文章收到了新评论"}'
PUBLISH user:1001:notifications '{"type":"like","content":"用户X赞了你的文章"}'
PUBLISH user:1001:notifications '{"type":"follow","content":"用户Y关注了你"}'

# 终端 1 会实时收到消息
```

```
┌──────────────────────────────────────────────────────────────┐
│              Pub/Sub 的局限性                                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 没有持久化：消息发完就丢了，离线的订阅者收不到               │
│  2. 没有消息确认：不知道订阅者是否成功处理了消息                 │
│  3. 没有消息积压：如果订阅者处理慢，消息会直接丢失               │
│                                                              │
│  → 生产环境的消息通知推荐用 Kafka 或 Redis Stream               │
│  → Pub/Sub 适合简单的实时通知场景（如聊天室、实时提醒）          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 四、过期策略与内存淘汰

### 4.1 键的过期时间

```bash
# 设置过期时间
SET session:abc "user_data" EX 3600     # 3600 秒后过期
SET temp_key "value" PX 60000           # 60000 毫秒后过期

# 单独设置过期时间
EXPIRE key 3600                          # 秒
PEXPIRE key 60000                        # 毫秒

# 查看剩余过期时间
TTL key                                  # 秒（-1 = 永不过期，-2 = 已过期）
PTTL key                                 # 毫秒

# 取消过期时间（变为永久）
PERSIST key
```

### 4.2 过期策略

Redis 什么时候删除过期的键？答案不是"到期立刻删除"，而是结合了**惰性删除**和**定期删除**两种策略：

```
┌──────────────────────────────────────────────────────────────┐
│                 Redis 过期键删除策略                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  策略一：惰性删除（Lazy Deletion）                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  访问 key 时，先检查是否过期                            │   │
│  │  过期了 → 删除，返回 nil                               │   │
│  │  没过期 → 返回值                                       │   │
│  │                                                       │   │
│  │  优点：不浪费 CPU                                      │   │
│  │  缺点：过期键可能长期占用内存                            │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  策略二：定期删除（Periodic Deletion）                         │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  Redis 每隔一段时间，随机抽取一批 key 检查              │   │
│  │  过期了 → 删除                                         │   │
│  │  抽取比例和间隔通过算法动态调整                          │   │
│  │                                                       │   │
│  │  优点：平衡 CPU 和内存                                  │   │
│  │  缺点：还是有部分过期键没被及时清理                       │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  两者结合：                                                  │
│  - 大部分过期键通过定期删除清理                               │
│  - 漏网的通过惰性删除在访问时清理                             │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 4.3 内存淘汰策略

当 Redis 内存使用达到 `maxmemory` 限制时，就需要淘汰一些数据腾出空间。

```
┌──────────────────────────────────────────────────────────────────┐
│                    八种内存淘汰策略                                 │
├────────────────────┬──────────────────────────────────────────────┤
│  策略              │  说明                                         │
├────────────────────┼──────────────────────────────────────────────┤
│ noeviction        │ 不淘汰任何数据，写入直接报错（默认）             │
│ allkeys-lru       │ 从所有键中淘汰最近最少使用的（推荐）             │
│ allkeys-lfu       │ 从所有键中淘汰使用频率最低的（Redis 4.0+）      │
│ allkeys-random    │ 从所有键中随机淘汰                              │
│ volatile-lru      │ 从设置了过期时间的键中淘汰最近最少使用的          │
│ volatile-lfu      │ 从设置了过期时间的键中淘汰使用频率最低的          │
│ volatile-random   │ 从设置了过期时间的键中随机淘汰                   │
│ volatile-ttl      │ 淘汰即将过期的键（TTL 最小的）                  │
└────────────────────┴──────────────────────────────────────────────┘
```

**LRU vs LFU：**

```
┌──────────────────────────────────────────────────────────────┐
│                  LRU vs LFU 对比                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  LRU（Least Recently Used）最近最少使用                        │
│  ├── 淘汰最长时间没被访问的键                                  │
│  ├── 适合热点数据明显变化的场景                                 │
│  └── 问题：偶尔被访问一次的键不会被淘汰                         │
│                                                              │
│  LFU（Least Frequently Used）最不经常使用                      │
│  ├── 淘汰访问频率最低的键                                      │
│  ├── 适合区分"冷热"数据的场景                                  │
│  └── 适合缓存大部分数据都比较热的场景                           │
│                                                              │
│  数据分析项目推荐：allkeys-lru                                     │
│  - 文章缓存有明确的冷热区分                                     │
│  - 最近被访问的文章应该保留                                     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 五、Lua 脚本：原子操作

有时候你需要执行一组操作，要求它们**整体是原子的**（要么全成功，要么全失败）。Redis 的事务不支持回滚，而 Lua 脚本是更好的选择。

### 5.1 为什么用 Lua 脚本？

```
┌──────────────────────────────────────────────────────────────┐
│              Lua 脚本 vs 事务 对比                             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  问题：读取计数器，判断是否超过限制，如果没超过则 +1            │
│  这需要"读-判断-写"三步，普通方式无法保证原子性                  │
│                                                              │
│  方式一：事务（不行！）                                       │
│  MULTI                                                       │
│  GET rate_limit:user:1001          ← 这个值在 EXEC 前拿不到    │
│  ...无法判断...                                                │
│  EXEC                                                         │
│                                                              │
│  方式二：Lua 脚本（完美！）                                    │
│  EVAL "local count = redis.call('GET', KEYS[1]);              │
│        if tonumber(count) < 100 then                           │
│          redis.call('INCR', KEYS[1]);                         │
│          return 1;                                            │
│        else                                                   │
│          return 0;                                            │
│        end" 1 rate_limit:user:1001                             │
│                                                              │
│  Lua 脚本在 Redis 中是原子执行的：                              │
│  - 执行期间不会被其他命令打断                                   │
│  - 整个脚本要么全执行，要么全不执行                              │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 常用命令

```bash
# ==================== EVAL（直接执行） ====================

# EVAL script numkeys key [key ...] arg [arg ...]
EVAL "return redis.call('SET', KEYS[1], ARGV[1])" 1 mykey myvalue

# KEYS[]：传入的键，通过索引访问（KEYS[1], KEYS[2], ...）
# ARGV[]：传入的参数，通过索引访问（ARGV[1], ARGV[2], ...）
# numkeys：KEYS[] 的个数

# ==================== EVALSHA（通过 SHA1 执行，推荐） ====================

# 先加载脚本，获取 SHA1 摘要
SCRIPT LOAD "return redis.call('GET', KEYS[1])"
# 输出：a5e8d4c3b2a1...（SHA1 摘要）

# 通过摘要执行（节省网络带宽）
EVALSHA a5e8d4c3b2a1... 1 mykey
```

### 5.3 数据平台场景：限流器

```bash
# 令牌桶限流：每个用户每分钟最多 10 次请求
EVAL "
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local current = tonumber(redis.call('GET', key) or '0')

if current < limit then
    redis.call('INCR', key)
    if current == 0 then
        redis.call('EXPIRE', key, window)
    end
    return 1
else
    return 0
end
" 1 rate_limit:user:1001 10 60
# 返回 1 = 允许，返回 0 = 被限流
```

### 5.4 数据平台场景：分布式锁（更安全的实现）

```bash
# 带唯一标识的分布式锁（防止误删别人的锁）
EVAL "
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else
    return 0
end
" 1 lock:publish:1001 unique_token
```

## 六、主从复制与哨兵（Sentinel）

### 6.1 主从复制

```
┌──────────────────────────────────────────────────────────────┐
│                  主从复制架构                                   │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────────┐                                           │
│  │   Master     │  ← 写操作都发到这里                         │
│  │  (主节点)    │                                           │
│  └──┬───────┬───┘                                           │
│     │       │  数据同步（异步）                                │
│     │       │                                               │
│  ┌──▼───┐ ┌▼──────┐                                         │
│  │Slave1│ │Slave2 │  ← 读操作可以分发到这里                  │
│  │(从)  │ │(从)   │                                          │
│  └──────┘ └──────┘                                          │
│                                                              │
│  特点：                                                      │
│  - 读写分离：Master 写，Slave 读                              │
│  - 数据冗余：Slave 是 Master 的副本                           │
│  - 故障恢复：Master 宕机后需要手动切换                         │
│  - 注意：复制是异步的，可能有短暂的数据不一致                    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 哨兵（Sentinel）

```
┌──────────────────────────────────────────────────────────────┐
│                  哨兵架构                                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────────┐      ┌──────────────┐                      │
│  │  Sentinel 1  │      │  Sentinel 2  │                      │
│  │  (哨兵)      │◄────►│  (哨兵)      │                      │
│  └──────┬───────┘      └──────┬───────┘                      │
│         │ 互相监控               │                             │
│  ┌──────▼───────┐      ┌──────▼───────┐                      │
│  │  Sentinel 3  │      │              │                      │
│  │  (哨兵)      │◄────►│              │                      │
│  └──────┬───────┘      │              │                      │
│         │ 监控          │              │                      │
│         ▼               ▼              │                      │
│  ┌──────────────┐      ┌──────────────┐                      │
│  │   Master     │─────→│   Slave      │                      │
│  │  (主节点)    │同步  │  (从节点)    │                      │
│  └──────────────┘      └──────────────┘                      │
│                                                              │
│  哨兵的职责：                                                  │
│  1. 监控：持续检查 Master 和 Slave 是否正常                   │
│  2. 通知：当故障发生时，通知管理员                              │
│  3. 自动故障转移：                                              │
│     - Master 宕机 → 哨兵协商选举新 Master                      │
│     - 通知客户端新的 Master 地址                               │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**故障转移过程：**

```
┌──────────────────────────────────────────────────────────────┐
│                 哨兵故障转移过程                                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  步骤 1：主观下线（SDOWN）                                     │
│  Sentinel A 发现 Master 无响应 → 标记为"主观下线"              │
│                                                              │
│  步骤 2：客观下线（ODOWN）                                     │
│  Sentinel A 询问其他 Sentinel                                  │
│  超过半数 Sentinel 认为 Master 下线 → 标记为"客观下线"          │
│                                                              │
│  步骤 3：Sentinel 选举                                         │
│  多个 Sentinel 协商，选出一个"领导者"来执行故障转移              │
│                                                              │
│  步骤 4：选择新 Master                                         │
│  根据优先级、复制偏移量、RunID 选择一个 Slave 升级为 Master     │
│                                                              │
│  步骤 5：修改配置                                              │
│  其他 Slave 改为复制新的 Master                                │
│  通知客户端新的 Master 地址                                    │

└──────────────────────────────────────────────────────────────┘
```

## 七、面试题

### Q1：Redis 持久化方案怎么选？RDB 和 AOF 的区别？

**参考答案：**

**RDB** 是全量快照，定时把整个内存数据写入磁盘。优点是文件小、恢复快；缺点是两次快照之间的数据可能丢失。

**AOF** 是增量日志，记录每条写命令。优点是数据安全性高（everysec 模式最多丢 1 秒）；缺点是文件大、恢复慢。

**推荐方案：**

| 场景 | 推荐方案 |
|------|---------|
| 纯缓存（数据可以重建） | 只用 RDB |
| 数据安全要求高 | 只用 AOF（everysec） |
| 生产环境（推荐） | RDB + AOF 混合 |
| Redis 4.0+ | 混合持久化（aof-use-rdb-preamble yes） |

混合持久化的好处：RDB 的快照速度 + AOF 的数据安全。AOF 文件前半部分是 RDB 格式的二进制数据，后半部分是 AOF 格式的增量命令。

---

### Q2：Redis 过期键的删除策略是什么？

**参考答案：**

Redis 使用**惰性删除 + 定期删除**的组合策略：

1. **惰性删除**：访问 key 时检查是否过期，过期则删除。不访问就不管，节省 CPU。
2. **定期删除**：每隔一段时间（默认 100ms），随机抽查一批设置了过期时间的 key，删除过期的。抽查比例和间隔根据过期 key 的比例动态调整。

两种策略互补：
- 定期删除清理大部分过期键，防止内存浪费
- 惰性删除兜底，处理漏网的过期键
- 平衡了 CPU 使用率和内存利用率

---

### Q3：缓存和数据库双写一致性怎么保证？

**参考答案：**

常见的四种策略：

```
┌──────────────────────────────────────────────────────────────────┐
│                    缓存一致性策略                                   │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  策略 1：先更新缓存，再更新数据库                                   │
│  → 不推荐！两个操作非原子，可能缓存更新成功但数据库失败              │
│                                                                  │
│  策略 2：先更新数据库，再更新缓存                                   │
│  → 不推荐！并发时可能导致缓存和数据库数据不一致                     │
│  → 线程 A 更新数据库（v1 → v2）                                   │
│  → 线程 B 更新数据库（v2 → v3）                                   │
│  → 线程 B 更新缓存（v3）                                          │
│  → 线程 A 更新缓存（v2）← 脏数据！                                │
│                                                                  │
│  策略 3：先删除缓存，再更新数据库（Cache Aside Pattern）             │
│  → 推荐方案                                                      │
│  → 并发问题：线程 A 删除缓存，线程 B 读缓存 miss 读数据库旧值       │
│  → 线程 A 更新数据库 → 线程 B 把旧值写入缓存 ← 短暂不一致           │
│  → 解决方案：延迟双删（删除缓存 → 更新数据库 → 延迟再删一次）        │
│                                                                  │
│  策略 4：先更新数据库，再删除缓存                                   │
│  → 推荐方案（最终一致性）                                         │
│  → 读：先读缓存，miss 读数据库，回写缓存                           │
│  → 写：先更新数据库，再删除缓存                                    │
│  → 极端情况可能短暂不一致，但概率很低                               │
│                                                                  │
│  最终结论：                                                      │
│  - 如果能容忍短暂不一致 → "先更新数据库，再删除缓存"                │
│  - 如果不能容忍 → 使用分布式读写锁 + 消息队列保证一致性              │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```
