# 数据仓库基础概念

> Hive 4.0.1 学习系列 - 第 1 篇
> 适用环境：macOS Apple Silicon / JDK 21 / Hive 4.0.1

## 一、什么是数据仓库

### 1.1 数据仓库的定义

数据仓库（Data Warehouse, DW）是由 Bill Inmon 在 1990 年提出的概念，其核心定义是：

> **面向主题的（Subject-Oriented）、集成的（Integrated）、相对稳定的（Non-Volatile）、反映历史变化的（Time-Variant）数据集合，用于支持管理决策。**

与操作型数据库（OLTP）相比，数据仓库服务于分析型处理（OLAP），两者的核心区别如下：

```
+-------------------+--------------------+--------------------+
|      特性         |    数据库 (OLTP)    |   数据仓库 (OLAP)  |
+-------------------+--------------------+--------------------+
| 核心目的         | 日常业务操作        | 分析与决策支持     |
| 数据特征         | 当前最新数据        | 历史数据累积       |
| 操作类型         | 增删改查 (CRUD)     | 大批量查询为主     |
| 查询复杂度       | 简单（单表/少量联表）| 复杂（多表关联/聚合）|
| 数据量级         | GB 级别             | TB ~ PB 级别       |
| 响应时间要求     | 毫秒级              | 秒 ~ 分钟级        |
| 典型场景         | 用户发布文章、转账      | 访问报表、趋势分析 |
+-------------------+--------------------+--------------------+
```

### 1.2 数据库 vs 数据仓库 vs 数据湖

三者构成了现代数据平台的三大支柱，理解它们的区别至关重要：

```
                         数据处理光谱
    ┌──────────────────────────────────────────────────┐
    │                                                  │
    │  数据库          数据仓库          数据湖         │
    │  (Database)     (Data Warehouse)  (Data Lake)    │
    │                                                  │
    │  ┌──────┐      ┌──────────┐      ┌──────────┐   │
    │  │结构化│      │ 结构化   │      │ 结构化   │   │
    │  │ 数据 │      │ +清洗后  │      │ 半结构化 │   │
    │  │      │      │ +建模后  │      │ 非结构化 │   │
    │  └──────┘      └──────────┘      │ 原始数据 │   │
    │                                   └──────────┘   │
    │  MySQL/PG      Hive/ClickHouse   HDFS/S3/Iceberg│
    │  事务ACID       批量分析          Schema-on-Read  │
    │  行式存储       列式存储          任意格式        │
    │  实时操作       定时T+1           近实时/批量     │
    └──────────────────────────────────────────────────┘
```

**数据湖（Data Lake）** 存储所有原始数据（结构化、半结构化、非结构化），不预先定义 Schema。典型技术栈包括 HDFS、S3、Iceberg、Delta Lake。

**在数据平台项目中的对应关系：**

```
+-------------------+----------------------------------------+
|      组件         |           数据平台项目中用途              |
+-------------------+----------------------------------------+
| 数据库 (MySQL)    | 博客后端业务存储（用户、文章、文章）     |
| 数据仓库 (Hive)   | 博客数据分析（销售统计、热门文章排行）   |
| 数据湖 (Iceberg)  | 原始日志归档、支持实时和批处理查询       |
| 计算引擎 (Spark)  | ETL 数据清洗、机器学习推荐              |
| 实时引擎 (Flink)  | 实时文章流处理、实时热门排行            |
+-------------------+----------------------------------------+
```

## 二、数仓分层架构

### 2.1 为什么要分层

不分层的数据仓库会变成一团混乱的数据"沼泽"：

```
不分层的问题：
┌─────────────────────────────────────────────────┐
│                                                 │
│   原始数据 ──→ [一团乱麻] ──→ 报表输出           │
│                                                 │
│   问题：                                        │
│   - 原始数据与报表逻辑耦合                       │
│   - 数据重复计算，浪费资源                       │
│   - 一处出错，全部报表受影响                     │
│   - 无法追溯数据血缘                             │
│                                                 │
└─────────────────────────────────────────────────┘

分层后的优势：
┌─────────────────────────────────────────────────┐
│                                                 │
│   原始数据 → ODS → DWD → DWS → ADS → 报表       │
│                                                 │
│   优势：                                        │
│   - 每层职责明确，解耦复用                       │
│   - 中间结果可被多个下游复用                     │
│   - 问题定位快速，数据血缘清晰                   │
│   - 便于扩展新分析场景                           │
│                                                 │
└─────────────────────────────────────────────────┘
```

### 2.2 经典四层架构详解

```
  ┌─────────────────────────────────────────────────────────────┐
  │                     数据仓库分层架构                         │
  │                                                             │
  │   ┌─────┐    ┌─────┐    ┌─────┐    ┌─────┐                 │
  │   │ ADS │ ←  │ DWS │ ←  │ DWD │ ←  │ ODS │  ← 数据源      │
  │   └─────┘    └─────┘    └─────┘    └─────┘                 │
  │   应用层      汇总层      明细层      贴源层                  │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘
```

#### ODS 层（Operational Data Store，操作数据存储层）

- **别名**：贴源层、原始数据层、Staging 层
- **职责**：原样保存从数据源接入的原始数据，不做任何修改
- **特点**：数据结构通常与源系统一致，保留历史全量
- **存储周期**：长期保存（通常不删除）

```sql
-- ODS 层示例：博客原始访问日志
CREATE DATABASE IF NOT EXISTS ods;
USE ods;

-- 原始访问日志，与 Nginx 日志格式一致
CREATE EXTERNAL TABLE ods_access_log (
    raw_line STRING
)
STORED AS TEXTFILE
LOCATION '/user/hive/ods/access_log';

-- 原始博客文章，与 MySQL 同步的结构一致
CREATE EXTERNAL TABLE ods_product_article (
    id              BIGINT,
    title           STRING,
    content         STRING,
    author_id       BIGINT,
    category_id     BIGINT,
    status          INT,
    created_at      STRING,
    updated_at      STRING
)
STORED AS TEXTFILE
LOCATION '/user/hive/ods/product_article';
```

#### DWD 层（Data Warehouse Detail，明细数据层）

- **别名**：明细层、清洗层
- **职责**：对 ODS 层数据进行清洗（去重、空值处理、格式统一）、维度退化、标准化
- **特点**：采用维度建模，事实表与维度表分离
- **存储周期**：通常保留近 1-3 年

```sql
-- DWD 层示例：清洗后的访问明细事实表
CREATE DATABASE IF NOT EXISTS dwd;
USE dwd;

CREATE TABLE dwd_access_detail (
    access_id       STRING       COMMENT '访问唯一ID (UUID)',
    user_id         BIGINT       COMMENT '用户ID，未登录为NULL',
    article_id      BIGINT       COMMENT '文章ID',
    ip              STRING       COMMENT '访客IP',
    user_agent      STRING       COMMENT '浏览器UA',
    referer         STRING       COMMENT '来源URL',
    access_time     TIMESTAMP    COMMENT '访问时间',
    duration_sec    INT          COMMENT '停留秒数',
    is_mobile       BOOLEAN      COMMENT '是否移动端'
)
PARTITIONED BY (dt STRING COMMENT '日期分区 yyyy-MM-dd')
STORED AS ORC;
```

#### DWS 层（Data Warehouse Summary，汇总数据层）

- **别名**：轻度汇总层、服务层
- **职责**：基于 DWD 层，按主题域进行轻度聚合（按天/周/月、按用户/文章等维度）
- **特点**：数据粒度比 DWD 粗，是 ADS 层的直接数据来源
- **设计原则**：公共指标统一计算，避免重复

```sql
-- DWS 层示例：每日文章访问汇总
CREATE DATABASE IF NOT EXISTS dws;
USE dws;

CREATE TABLE dws_article_daily_stats (
    article_id      BIGINT       COMMENT '文章ID',
    pv_count        BIGINT       COMMENT '页面浏览量',
    uv_count        BIGINT       COMMENT '独立访客数',
    avg_duration    DOUBLE       COMMENT '平均停留时长(秒)',
    mobile_ratio    DOUBLE       COMMENT '移动端占比'
)
PARTITIONED BY (dt STRING COMMENT '统计日期 yyyy-MM-dd')
STORED AS ORC;
```

#### ADS 层（Application Data Store，应用数据层）

- **别名**：数据集市层、应用层、报表层
- **职责**：面向具体业务场景的高度聚合数据，直接供报表、BI 工具、API 使用
- **特点**：数据量最小，查询速度最快，面向最终结果

```sql
-- ADS 层示例：文章热度排行榜（供前端展示）
CREATE DATABASE IF NOT EXISTS ads;
USE ads;

CREATE TABLE ads_article_ranking (
    article_id      BIGINT       COMMENT '文章ID',
    title           STRING       COMMENT '文章标题',
    author_name     STRING       COMMENT '作者名',
    category_name   STRING       COMMENT '分类名',
    total_pv        BIGINT       COMMENT '近7天总PV',
    total_uv        BIGINT       COMMENT '近7天总UV',
    rank_score      DOUBLE       COMMENT '热度评分',
    rank_position   INT          COMMENT '排名'
)
STORED AS ORC;
```

## 三、维度建模

### 3.1 事实表与维度表

维度建模（Dimensional Modeling）是数据仓库之父 Ralph Kimball 提出的设计方法，核心思想是将数据分为两类：

```
┌─────────────────────────────────────────────────────────┐
│                    维度建模核心概念                      │
│                                                         │
│  ┌─────────────────────────────────────────────────┐    │
│  │  事实表 (Fact Table)                            │    │
│  │  - 记录业务过程中的"事件"或"度量"               │    │
│  │  - 通常包含：外键 + 数值度量                     │    │
│  │  - 数据量非常大（百万~百亿行）                   │    │
│  │  - 示例：访问记录、文章记录、交易记录             │    │
│  └─────────────────────────────────────────────────┘    │
│                                                         │
│  ┌─────────────────────────────────────────────────┐    │
│  │  维度表 (Dimension Table)                        │    │
│  │  - 描述事实发生的"环境"或"角度"                  │    │
│  │  - 通常包含：主键 + 描述性文本属性               │    │
│  │  - 数据量相对较小（千~百万行）                   │    │
│  │  - 示例：用户维度、文章维度、时间维度             │    │
│  └─────────────────────────────────────────────────┘    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**数据平台项目中的事实表和维度表：**

```

                            │
                            │           ┌────────────────────────┐

                               post_id│                        │
                                        │ - post_id (PK)      │
```

### 3.2 星型模型 vs 雪花模型

```
星型模型 (Star Schema)：
                    ┌──────────┐
                    │ dim_user │
                    └────┬─────┘
                         │
              ┌──────────┼──────────┐
              │          │          │
         ┌────┴───┐ ┌───┴────┐ ┌──┴──────────┐
         │dim_date│ │dim_cat │ │dim_article  │
         └────┬───┘ └───┬────┘ └──┬──────────┘
              │         │         │
              └─────────┼─────────┘
                        │
                 ┌──────┴──────┐
                 │  事实表     │
                 │ (Fact Table)│
                 └─────────────┘

  特点：所有维度表直接连接事实表，结构扁平
  优点：查询简单（JOIN 少）、性能好
  缺点：维度表有数据冗余


雪花模型 (Snowflake Schema)：
                    ┌──────────┐
                    │ dim_user │
                    └────┬─────┘
                         │ user_id
                    ┌────┴─────┐
                    │ dim_role │        ← 维度表再拆分
                    └──────────┘
              ┌──────────┼──────────┐
              │          │          │
         ┌────┴───┐ ┌───┴────┐ ┌──┴──────────┐
         │dim_date│ │dim_cat │ │dim_article  │
         └────┬───┘ └───┬────┘ └──┬──────────┘
              │         │         │
              │    ┌────┴────┐    │
              │    │dim_parent│    │  ← 分类再拆分父分类
              │    │_category │    │
              │    └─────────┘    │
              └─────────┼─────────┘
                        │
                 ┌──────┴──────┐
                 │  事实表     │
                 │ (Fact Table)│
                 └─────────────┘

  特点：维度表可以进一步拆分为更细粒度的子维度
  优点：减少数据冗余、结构规范
  缺点：查询需要更多 JOIN、性能略差
```

**数据平台项目推荐使用星型模型**，原因：
1. 查询频率高（前端实时展示排行榜），需要尽量减少 JOIN
2. 维度表数据量小，冗余存储成本低
3. Hive 作为批处理引擎，减少 JOIN 可以显著提升性能

## 四、Hive 在数仓中的角色

### 4.1 Hive 的定位

Hive 是建立在 HDFS 之上的数据仓库工具，它将 SQL 语句（HQL）翻译为 MapReduce/Tez/Spark 任务执行。在数仓体系中，Hive 扮演着以下核心角色：

```
┌────────────────────────────────────────────────────────────────┐

│  ┌─────────────────────────────────────────────────────────┐  │

│  │   Beeline / JDBC / Hive CLI / spark-sql                 │  │
│  └───────────────────────┬─────────────────────────────────┘  │
│                          │ HQL                                 │
│  ┌───────────────────────┴─────────────────────────────────┐  │
│  │                   Hive 引擎层                            │  │
│  │  ┌───────────┐  ┌───────────┐  ┌───────────┐           │  │
│  │  │  Parser   │→ │ Optimizer │→ │ Executor  │           │  │
```

### 4.2 Hive vs 传统数据库

```
+-------------------+-------------------+-------------------+
|      特性         |    Hive           |  MySQL/PostgreSQL |
+-------------------+-------------------+-------------------+
| 定位              | 数据仓库工具       | 关系型数据库       |
| 数据存储          | HDFS（文件系统）   | 本地磁盘           |
| 执行引擎          | MapReduce/Spark   | 自有执行引擎       |
| 事务支持          | 有限（ACID 表）   | 完整 ACID          |
| 索引              | 不支持传统索引     | B+Tree 索引        |
| 延迟              | 高（秒~分钟级）   | 低（毫秒级）       |
| 数据规模          | PB 级             | TB 级              |
| 更新/删除         | 仅 ACID 表支持    | 完整支持           |
| 子查询            | 有限支持           | 完整支持           |
| 扩展性            | 横向扩展（加节点） | 纵向扩展为主       |
+-------------------+-------------------+-------------------+
```

## 五、数据平台项目数仓设计完整示例

### 5.1 建库语句

```sql
-- 创建数仓各层对应的数据库
CREATE DATABASE IF NOT EXISTS ods   COMMENT '贴源层 - 原始数据';
CREATE DATABASE IF NOT EXISTS dwd   COMMENT '明细层 - 清洗后的明细数据';
CREATE DATABASE IF NOT EXISTS dws   COMMENT '汇总层 - 轻度聚合数据';
CREATE DATABASE IF NOT EXISTS ads   COMMENT '应用层 - 面向应用的汇总数据';
CREATE DATABASE IF NOT EXISTS dim   COMMENT '维度层 - 维度表';
```

### 5.2 ODS 层建表示例

```sql
-- ODS 层：原始博客文章（从 MySQL 同步）
CREATE EXTERNAL TABLE ods.ods_product_article (
    id              BIGINT          COMMENT '文章ID',
    title           STRING          COMMENT '文章标题',
    content         STRING          COMMENT '文章内容',
    author_id       BIGINT          COMMENT '作者ID',
    category_id     INT             COMMENT '分类ID',
    status          INT             COMMENT '状态 0下架 1在售 2售罄',
    view_count      BIGINT          COMMENT '访问量',
    like_count  INT             COMMENT '点赞数',
    created_at      STRING          COMMENT '创建时间',
    updated_at      STRING          COMMENT '更新时间'
)
COMMENT '原始博客文章 - 从MySQL同步'
PARTITIONED BY (dt STRING COMMENT '同步日期 yyyy-MM-dd')
ROW FORMAT DELIMITED FIELDS TERMINATED BY '\t'
STORED AS TEXTFILE
LOCATION '/user/hive/ods/product_article';
```

### 5.3 DWD 层建表示例

```sql
-- DWD 层：清洗后的文章发布事实表
```

## 六、总结

本篇我们学习了数据仓库的核心概念：

1. **数据仓库 vs 数据库 vs 数据湖**：三者定位不同，在数据平台项目中各有分工
2. **数仓分层架构**：ODS → DWD → DWS → ADS 四层架构，各层职责明确
3. **维度建模**：事实表记录度量，维度表描述环境，数据平台项目采用星型模型
4. **Hive 的角色**：作为构建在 HDFS 上的数仓工具，通过 HQL 实现大数据分析
