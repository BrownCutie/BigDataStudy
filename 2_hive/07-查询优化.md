# Hive 查询优化

> Hive 4.0.1 学习系列 - 第 7 篇
> 适用环境：macOS Apple Silicon / JDK 21 / Hive 4.0.1

## 一、EXPLAIN 执行计划详解

### 1.1 EXPLAIN 基础

`EXPLAIN` 是查询优化的起点，它展示 Hive 如何将 HQL 翻译为 MapReduce/Tez/Spark 任务：

```sql
-- 基本用法
EXPLAIN
SELECT author_id, COUNT(*) AS cnt
FROM dwd.dwd_article_clean
GROUP BY author_id;

-- EXTENDED：输出更详细的信息（包括抽象语法树）
EXPLAIN EXTENDED
SELECT author_id, COUNT(*) AS cnt
FROM dwd.dwd_article_clean
GROUP BY author_id;

-- FORMATTED：格式化输出（可读性最好）
EXPLAIN FORMATTED
SELECT author_id, COUNT(*) AS cnt
FROM dwd.dwd_article_clean
GROUP BY author_id;

-- CBO：显示基于成本优化后的执行计划
EXPLAIN CBO
SELECT author_id, COUNT(*) AS cnt
FROM dwd.dwd_article_clean
GROUP BY author_id;
```


## 二、MapJoin 原理和配置

### 2.1 Reduce Join 的问题

```
Reduce Join（默认 JOIN 方式）：

┌──────────────────────────────────────────────────────────────────┐
│                  Reduce Join 执行流程                             │
│                                                                  │
│  Map 阶段：                                                      │
│  ┌──────────┐    ┌──────────┐                                   │
│  │ 大表     │    │ 小表     │                                   │
│  │ (文章表) │    │ (分类表) │                                   │
│  │ 1GB      │    │ 1KB      │                                   │
│  └────┬─────┘    └────┬─────┘                                   │
│       │               │                                         │
│       │ Shuffle（全部数据通过网络传输到 Reduce）                │
│       ▼               ▼                                         │
│  ┌────────────────────────────────────┐                         │
│  │          Reduce 阶段                │                         │
│  │  按 JOIN Key 分组 → 执行 JOIN       │                         │
│  │                                    │                         │
│  │  问题：                            │                         │
│  │  • 所有数据都要经过 Shuffle         │                         │
│  │  • 网络传输量大（1GB + 1KB）       │                         │
│  │  • Reduce 可能成为瓶颈             │                         │
│  │  • 小表也被反复 Shuffle，浪费资源  │                         │
│  └────────────────────────────────────┘                         │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 2.2 MapJoin 原理

```
MapJoin 执行流程：

┌──────────────────────────────────────────────────────────────────┐
│                    MapJoin 执行流程                               │
│                                                                  │
│  任务开始前（小表处理）：                                        │
│  ┌──────────┐                                                   │
│  │ 小表     │                                                   │
│  │ (分类表) │                                                   │
│  │ 1KB      │                                                   │
│  └────┬─────┘                                                   │
│       │                                                          │
│       │ 读取到内存，构建 Hash Table                              │
│       ▼                                                          │
│  ┌──────────────────┐                                           │
│  │  Hash Table      │                                           │
│  │  1→大数据        │                                           │
│  │  2→实时计算      │                                           │
│  │  3→消息队列      │                                           │
│  │  4→搜索引擎      │                                           │
│  └──────────────────┘                                           │
│       │                                                          │
│       │ 广播到所有 Map Task                                     │
│       ▼                                                          │
│                                                                  │
│  Map 阶段：                                                      │
│  ┌──────────┐                                                   │
│  │ 大表     │                                                   │
│  │ (文章表) │                                                   │
│  │ 1GB      │                                                   │
│  └────┬─────┘                                                   │
│       │                                                          │
│       │ 逐行读取大表，在 Map 中直接 JOIN Hash Table              │
│       ▼                                                          │
│  ┌────────────────────────────────────┐                         │
│  │  Map Task                          │                         │
│  │                                    │                         │
│  │  for each row in 大表:             │                         │
│  │    key = row.category_id           │                         │
│  │    lookup Hash Table[key]          │                         │
│  │    output joined_row               │                         │
│  │                                    │                         │
│  │  优势：                            │                         │
│  │  • 无 Shuffle（无网络传输）        │                         │
│  │  • 无 Reduce 阶段                  │                         │
│  │  • 大表只读一次                    │                         │
│  │  • 速度极快                        │                         │
│  └────────────────────────────────────┘                         │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 2.3 MapJoin 配置

```sql
-- 开启自动 MapJoin 转换（默认已开启）
SET hive.auto.convert.join = true;

-- 小表阈值（默认 25MB，超过此大小不使用 MapJoin）
-- 如果小表较大，可以适当调大
SET hive.auto.convert.join.noconditionaltask.size = 50000000;  -- 50MB

-- 查看当前配置
SET hive.auto.convert.join;

-- Hive 4.x 默认行为：
-- 当小表大小 < hive.auto.convert.join.noconditionaltask.size 时
-- 自动将 Reduce Join 转换为 MapJoin
```

```sql
-- 数据分析项目 MapJoin 示例
-- dim_category 是小表（几行），dwd_article_clean 是大表
SELECT
    a.article_id,
    a.title,
    a.author_id,
    c.name AS category_name,
    a.view_count
FROM dwd.dwd_article_clean a
JOIN dim.dim_category c ON a.category_id = c.category_id
ORDER BY a.view_count DESC;

-- 在 EXPLAIN 中确认是否使用了 MapJoin
EXPLAIN
SELECT a.article_id, c.name AS category_name
FROM dwd.dwd_article_clean a
JOIN dim.dim_category c ON a.category_id = c.category_id;

-- 如果看到 "Map Join Operator" 或 "Map Join Alias"，
-- 说明 MapJoin 已生效
```

### 2.4 MapJoin 适用条件

```
┌──────────────────────────────────────────────────────────────────┐
│              MapJoin 使用条件                                     │
│                                                                  │
│  适用场景：                                                      │
│  ✓ 大表 JOIN 小表（小表 < 约 50MB）                             │
│  ✓ 维度表关联（dim 表通常很小）                                 │
│  ✓ 不等值 JOIN（IN / EXISTS）                                   │
│                                                                  │
│  不适用场景：                                                    │
│  ✗ 大表 JOIN 大表（内存放不下）                                 │
│  ✗ 小表超过阈值                                                 │
│                                                                  │
│  优化建议：                                                      │
│  1. 确保 hive.auto.convert.join = true                         │
│  2. 合理设置小表阈值                                            │
│  3. 大表放 LEFT，小表放 RIGHT（方便自动识别）                   │
│  4. 使用 /*+ MAPJOIN(table) */ Hint 强制指定                   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

```sql
-- 使用 Hint 强制 MapJoin
SELECT /*+ MAPJOIN(c) */
    a.article_id,
    c.name AS category_name
FROM dwd.dwd_article_clean a
JOIN dim.dim_category c ON a.category_id = c.category_id;
```

## 三、Sort Merge Bucket Map Join (SMB Join)

### 3.1 SMB Join 原理

当两个大表 JOIN 时，MapJoin 不适用（内存放不下）。如果两个表都按 JOIN Key 做了分桶和排序，可以使用 SMB Join：

```
┌──────────────────────────────────────────────────────────────────┐
│                SMB Join 执行流程                                  │
│                                                                  │
│  前提条件：                                                      │
│  • 两个表都 CLUSTERED BY (join_key)                             │
│  • 两个表的桶数相同或成倍数关系                                  │
│  • 两个表的数据按 join_key 排序                                  │
│                                                                  │
│  表 A（4 桶）：              表 B（4 桶）：                       │
│  ┌─────────┐                ┌─────────┐                        │
│  │bucket-0 │ ← key 范围0   │bucket-0 │ ← key 范围0            │
│  │ sorted  │                │ sorted  │                        │
│  ├─────────┤                ├─────────┤                        │
│  │bucket-1 │ ← key 范围1   │bucket-1 │ ← key 范围1            │
│  │ sorted  │                │ sorted  │                        │
│  ├─────────┤                ├─────────┤                        │
│  │bucket-2 │ ← key 范围2   │bucket-2 │ ← key 范围2            │
│  │ sorted  │                │ sorted  │                        │
│  ├─────────┤                ├─────────┤                        │
│  │bucket-3 │ ← key 范围3   │bucket-3 │ ← key 范围3            │
│  │ sorted  │                │ sorted  │                        │
│  └─────────┘                └─────────┘                        │
│       │                          │                               │
│       │ 只 JOIN 对应的桶        │                               │
│       │ （bucket-0 对 bucket-0）│                               │
│       ▼                          ▼                               │
│  ┌──────────────────────────────────────┐                      │
│  │  Bucket-0 JOIN Bucket-0             │                      │
│  │  Bucket-1 JOIN Bucket-1             │                      │
│  │  Bucket-2 JOIN Bucket-2             │                      │
│  │  Bucket-3 JOIN Bucket-3             │                      │
│  │                                      │                      │
│  │  优势：                              │                      │
│  │  • 每个 Map Task 只处理对应桶       │                      │
│  │  • 不需要全量 Shuffle               │                      │
│  │  • 排序后可以做 Merge Join           │                      │
│  │  • 大表 JOIN 大表的最优方案          │                      │
│  └──────────────────────────────────────┘                      │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 3.2 SMB Join 配置

```sql
-- 开启 SMB Join
SET hive.optimize.bucketmapjoin = true;              -- 开启分桶 Map Join
SET hive.optimize.bucketmapjoin.sortedmerge = true;  -- 开启 SMB Join
SET hive.input.format = org.apache.hadoop.hive.ql.io.BucketizedHiveInputFormat;

-- 创建分桶排序表
CREATE TABLE dwd.dwd_access_bucketed (
    access_id   STRING,
    user_id     BIGINT,
    article_id  BIGINT,
    duration_sec INT
)
CLUSTERED BY (article_id) SORTED BY (article_id) INTO 4 BUCKETS
STORED AS ORC;

CREATE TABLE dwd.dwd_article_bucketed (
    article_id  BIGINT,
    title       STRING,
    view_count  BIGINT
)
CLUSTERED BY (article_id) SORTED BY (article_id) INTO 4 BUCKETS
STORED AS ORC;

-- 写入分桶表时必须开启强制分桶
SET hive.enforce.bucketing = true;
SET hive.enforce.sorting = true;

-- 写入数据
INSERT OVERWRITE TABLE dwd.dwd_access_bucketed
SELECT access_id, user_id, article_id, duration_sec
FROM dwd.dwd_access_detail WHERE dt = '2026-04-15';

INSERT OVERWRITE TABLE dwd.dwd_article_bucketed
SELECT article_id, title, view_count
FROM dwd.dwd_article_clean;

-- 执行 SMB Join
SELECT /*+ MAPJOIN(b) */
    a.access_id, a.user_id, b.title, b.view_count
FROM dwd.dwd_access_bucketed a
JOIN dwd.dwd_article_bucketed b ON a.article_id = b.article_id;
```

## 四、谓词下推

### 4.1 谓词下推原理

谓词下推（Predicate Pushdown）是将 WHERE 过滤条件尽可能推到数据读取阶段执行，减少后续处理的数据量：

```
┌──────────────────────────────────────────────────────────────────┐
│                谓词下推 vs 不下推                                 │
│                                                                  │
│  不下推（先 JOIN 再过滤）：                                      │
│  ┌──────┐     ┌──────┐                                         │
│  │ 表 A │     │ 表 B │  全量数据 JOIN                           │
│  │100万行│     │500万行│  → 500万行（笛卡尔积的一部分）          │
│  └──┬───┘     └──┬───┘                                         │
│     │            │                                              │
│     └──── JOIN ──┘                                              │
│          │                                                      │
│          ▼                                                      │
│     ┌──────────┐                                                │
│     │ WHERE 过滤│  → 1000行（过滤掉99.98%）                     │
│     └──────────┘                                                │
│     问题：JOIN 处理了大量最终被丢弃的数据                        │
│                                                                  │
│  下推（先过滤再 JOIN）：                                         │
│  ┌──────┐     ┌──────┐                                         │
│  │ 表 A │     │ 表 B │                                          │
│  │100万行│     │500万行│                                         │
│  └──┬───┘     └──┬───┘                                         │
│     │ WHERE      │ WHERE                                        │
│     ▼ 过滤      ▼ 过滤                                          │
│  ┌──────┐     ┌──────┐                                         │
│  │5000行│     │2000行│  只 JOIN 过滤后的数据                   │
│  └──┬───┘     └──┬───┘                                         │
│     │            │                                              │
│     └──── JOIN ──┘                                              │
│          │                                                      │
│          ▼                                                      │
│     1000行                                                      │
│     优势：JOIN 的数据量大幅减少                                 │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 4.2 分区裁剪

分区裁剪是谓词下推的特殊形式，对于分区表至关重要：

```sql
-- 不走分区裁剪（全表扫描所有分区！）
SELECT * FROM dwd.dwd_access_detail
WHERE article_id = 1001;

-- 走分区裁剪（只扫描 dt='2026-04-15' 分区）
SELECT * FROM dwd.dwd_access_detail
WHERE dt = '2026-04-15' AND article_id = 1001;

-- 使用 BETWEEN 也支持分区裁剪
SELECT * FROM dwd.dwd_access_detail
WHERE dt BETWEEN '2026-04-10' AND '2026-04-15';

-- 使用 IN 也支持分区裁剪
SELECT * FROM dwd.dwd_access_detail
WHERE dt IN ('2026-04-14', '2026-04-15');

-- 使用函数会破坏分区裁剪！（常见错误）
-- 错误写法：
SELECT * FROM dwd.dwd_access_detail
WHERE SUBSTR(dt, 1, 7) = '2026-04';   -- ❌ 全表扫描

-- 正确写法：
SELECT * FROM dwd.dwd_access_detail
WHERE dt >= '2026-04-01' AND dt < '2026-05-01';  -- ✓ 分区裁剪
```

### 4.3 谓词下推配置

```sql
-- 开启谓词下推（默认已开启）
SET hive.optimize.ppd = true;

-- 开启分区裁剪（默认已开启）
SET hive.optimize.ppd.storage = true;

-- 验证谓词下推是否生效
EXPLAIN
SELECT a.article_id, a.title
FROM dwd.dwd_article_clean a
JOIN dim.dim_category c ON a.category_id = c.category_id
WHERE a.view_count > 3000 AND c.category_id = 1;

-- 在执行计划中，Filter 应该出现在 TableScan 之后、Join 之前
-- 如果 Filter 出现在 Join 之后，说明谓词下推未生效
```

## 五、并行执行和向量化查询

### 5.1 并行执行

Hive 可以将互不依赖的 Stage 并行执行，减少总执行时间：

```
┌──────────────────────────────────────────────────────────────────┐
│                并行执行 vs 串行执行                               │
│                                                                  │
│  串行执行：                                                      │
│  Stage-1 ───→ Stage-2 ───→ Stage-3 ───→ Stage-4                │
│  |<--- 5s --->|<--- 8s --->|<--- 3s --->|<--- 4s --->           │
│  总耗时：5 + 8 + 3 + 4 = 20 秒                                  │
│                                                                  │
│  并行执行（Stage-2 和 Stage-3 无依赖）：                        │
│  Stage-1 ───→ Stage-2 ───→ Stage-4                              │
│                ↘                                               │
│                 Stage-3                                         │
│  |<--- 5s --->|<--- 8s --->|<--- 4s --->                        │
│              |<--- 3s --->|                                     │
│  总耗时：5 + max(8, 3) + 4 = 17 秒（节省 3 秒）                 │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

```sql
-- 开启并行执行
SET hive.exec.parallel = true;

-- 最大并行 Stage 数（默认 8）
SET hive.exec.parallel.thread.number = 16;

-- 适用场景：
-- - UNION ALL 查询（多个独立子查询可以并行）
-- - Multi-Table Insert（多个 INSERT 目标可以并行写入）
```

### 5.2 向量化查询

向量化查询利用 CPU 的 SIMD（Single Instruction Multiple Data）指令，一次处理一批数据而非逐行处理：

```
非向量化（逐行处理）：
  for each row:                    → 慢
    process(row)

向量化（批量处理）：
  for each batch (1024 rows):      → 快
    process_batch(batch)           → 利用 SIMD 指令并行

性能提升：通常 2-5 倍
```

```sql
-- 开启向量化执行（默认已开启）
SET hive.vectorized.execution.enabled = true;

-- 开启向量化 GROUP BY（默认开启）
SET hive.vectorized.execution.reduce.enabled = true;

-- 注意：向量化执行要求：
-- - ORC 或 Parquet 格式
-- - 基本数据类型（不支持复杂类型 STRUCT/MAP 等）
-- - 简单表达式（不支持 UDF、复杂 CASE WHEN）
```

## 六、CBO（基于成本的优化器）

### 6.1 CBO 原理

```
┌──────────────────────────────────────────────────────────────────┐
│                CBO (Cost-Based Optimizer) 原理                   │
│                                                                  │
│  RBO (Rule-Based Optimizer，基于规则的优化器)：                   │
│  ┌──────────────────────────────────────────────────────┐       │
│  │ 按固定规则优化：                                     │       │
│  │ 1. 谓词下推                                         │       │
│  │ 2. MapJoin 自动转换                                 │       │
│  │ 3. 列裁剪                                           │       │
│  │ 问题：不考虑数据量和统计信息，可能选择次优方案        │       │
│  └──────────────────────────────────────────────────────┘       │
│                                                                  │
│  CBO (Cost-Based Optimizer，基于成本的优化器)：                  │
│  ┌──────────────────────────────────────────────────────┐       │
│  │ 基于统计信息计算每种执行方案的成本：                  │       │
│  │                                                      │       │
│  │  方案A: MapJoin  → 成本 = 小表大小 × 大表行数        │       │
│  │  方案B: ReduceJoin → 成本 = (大表+小表) × Shuffle   │       │
│  │                                                      │       │
│  │ 选择成本最低的方案                                  │       │
│  │ 需要：准确的表统计信息（ANALYZE）                    │       │
│  └──────────────────────────────────────────────────────┘       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 6.2 CBO 配置

```sql
-- 开启 CBO（Hive 4.x 默认已开启）
SET hive.cbo.enable = true;

-- 开启基于成本的 JOIN 重排序
SET hive.compute.query.using.stats = true;
SET hive.stats.fetch.column.stats = true;
SET hive.stats.fetch.partition.stats = true;

-- 使用 Calcite 优化器（Hive 4.x 默认）
SET hive.calcite.enable = true;
```

## 七、统计信息收集（ANALYZE）

### 7.1 为什么需要统计信息

CBO 优化器依赖准确的统计信息来选择最优执行计划。没有统计信息时，CBO 只能使用默认估算值（通常假设 1 行数据），导致优化失效。

### 7.2 收集统计信息

```sql
-- 收集表的统计信息（行数、大小等）
ANALYZE TABLE dwd.dwd_article_clean COMPUTE STATISTICS;

-- 收集列的统计信息（min、max、numNulls、distinct 值数量）
ANALYZE TABLE dwd.dwd_article_clean
COMPUTE STATISTICS FOR COLUMNS article_id, author_id, view_count;

-- 收集分区表的统计信息（指定分区）
ANALYZE TABLE dwd.dwd_access_detail
PARTITION (dt = '2026-04-15')
COMPUTE STATISTICS;

-- 收集分区表的列统计信息
ANALYZE TABLE dwd.dwd_access_detail
PARTITION (dt = '2026-04-15')
COMPUTE STATISTICS FOR COLUMNS article_id, user_id;

-- 查看统计信息
DESC FORMATTED dwd.dwd_article_clean;
-- 输出中查看：
--   numRows: 行数
--   rawDataSize: 原始数据大小
--   totalSize: 总大小
--   Column Statistics: 列统计信息
```

### 7.3 自动统计信息收集

```sql
-- 自动收集统计信息（在 INSERT 时自动收集）
SET hive.stats.autogather = true;         -- 默认开启
SET hive.stats.column.autogather = true;  -- 默认开启

-- 使用动态分区时，建议手动 ANALYZE
-- 因为动态分区的统计信息可能不准确
```

## 八、常见慢查询优化案例

### 8.1 案例1：全表扫描

```sql
-- 问题：没有分区条件，全表扫描
-- 慢查询
SELECT COUNT(*) FROM dwd.dwd_access_detail;       -- 扫描所有分区
SELECT * FROM dwd.dwd_access_detail WHERE user_id = 1001;  -- 全表扫描

-- 优化：添加分区条件
SELECT COUNT(*) FROM dwd.dwd_access_detail WHERE dt = '2026-04-15';
SELECT * FROM dwd.dwd_access_detail
WHERE dt = '2026-04-15' AND user_id = 1001;
```

### 8.2 案例2：SELECT *

```sql
-- 问题：SELECT * 读取所有列（列式存储无法利用列裁剪）
-- 慢查询
SELECT * FROM dwd.dwd_article_clean WHERE view_count > 3000;

-- 优化：只选择需要的列
SELECT article_id, title, view_count
FROM dwd.dwd_article_clean WHERE view_count > 3000;
```

### 8.3 案例3：JOIN 顺序不当

```sql
-- 问题：大表在前，小表在后，且没有利用 MapJoin
-- 慢查询
SELECT /*+ STREAMTABLE(a) */ a.*, b.name
FROM dwd.dwd_access_detail a    -- 大表（千万级）
JOIN dim.dim_user b ON a.user_id = b.user_id  -- 小表（百级）
WHERE a.dt = '2026-04-15';

-- 优化：小表在 JOIN 右侧，利用 MapJoin
SELECT /*+ MAPJOIN(b) */ a.access_id, b.nickname
FROM dwd.dwd_access_detail a
JOIN dim.dim_user b ON a.user_id = b.user_id
WHERE a.dt = '2026-04-15';
```

### 8.4 案例4：数据倾斜

```sql
-- 问题：某个 JOIN Key 的数据量远大于其他 Key
-- 例如：user_id = NULL 的用户特别多（未登录用户）
-- 导致某个 Reduce 处理大量数据，成为瓶颈

-- 优化方案1：过滤 NULL 值
SELECT a.*, b.nickname
FROM dwd.dwd_access_detail a
JOIN dim.dim_user b ON a.user_id = b.user_id AND a.user_id IS NOT NULL
WHERE a.dt = '2026-04-15';

-- 优化方案2：开启数据倾斜优化
SET hive.optimize.skewjoin = true;
SET hive.skewjoin.key = 100000;  -- 倾斜 Key 阈值

-- 优化方案3：使用 GROUP BY 时开启倾斜优化
SET hive.groupby.skewindata = true;
-- Hive 会自动将倾斜 Key 分散到多个 Reduce 处理
```

### 8.5 案例5：子查询未优化

```sql
-- 问题：使用 IN 子查询导致全表扫描
-- 慢查询
SELECT * FROM dwd.dwd_article_clean
WHERE author_id IN (SELECT user_id FROM dim.dim_user WHERE role = 'admin');

-- 优化：改为 JOIN
SELECT DISTINCT a.*
FROM dwd.dwd_article_clean a
JOIN dim.dim_user u ON a.author_id = u.user_id
WHERE u.role = 'admin';
```

### 8.6 优化检查清单

```
┌──────────────────────────────────────────────────────────────────┐
│                查询优化检查清单                                   │
│                                                                  │
│  □ 1. EXPLAIN 查看执行计划，确认无全表扫描                     │
│  □ 2. 分区表查询是否使用了分区裁剪                               │
│  □ 3. 是否只 SELECT 了需要的列（避免 SELECT *）                 │
│  □ 4. JOIN 是否利用了 MapJoin（小表广播）                       │
│  □ 5. WHERE 条件是否可以下推（谓词下推）                        │
│  □ 6. 是否有数据倾斜（检查 GROUP BY / JOIN Key 分布）           │
│  □ 7. 统计信息是否已收集（ANALYZE）                             │
│  □ 8. 文件格式是否为 ORC/Parquet（列式存储）                    │
│  □ 9. 是否开启了并行执行和向量化查询                             │
│  □ 10. GROUP BY 后是否使用 HAVING 而非 WHERE                    │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 九、总结

本篇学习了 Hive 查询优化的核心技术：

1. **EXPLAIN**：执行计划分析，Stage/Operator/Task 层次结构
2. **MapJoin**：小表广播到 Map 端，避免 Shuffle，适合大表 JOIN 小表
3. **SMB Join**：分桶排序的大表 JOIN 大表最优方案
4. **谓词下推**：将 WHERE 条件推到读取阶段，分区裁剪是最重要的形式
5. **并行执行**：互不依赖的 Stage 并行运行
6. **向量化查询**：利用 CPU SIMD 指令批量处理，性能提升 2-5 倍
7. **CBO**：基于统计信息选择最优执行计划
8. **统计信息**：ANALYZE 收集表和列的统计信息
9. **优化案例**：全表扫描、SELECT *、JOIN 顺序、数据倾斜等常见问题
