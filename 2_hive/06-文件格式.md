# Hive 文件格式与压缩

> Hive 4.0.1 学习系列 - 第 6 篇
> 适用环境：macOS Apple Silicon / JDK 21 / Hive 4.0.1

## 一、文件格式概述

Hive 中的数据最终以文件形式存储在 HDFS 上。选择合适的文件格式和压缩编码，直接影响存储成本和查询性能。

```
┌──────────────────────────────────────────────────────────────────┐
│                Hive 文件格式全景                                  │
│                                                                  │
│  ┌─────────────┐                                                │
│  │  行式存储   │  ← 一行数据连续存储                            │
│  │ Row-Based   │                                                │
│  ├─────────────┤                                                │
│  │ • TextFile  │  适合：全量扫描、数据加载                     │
│  │ • Sequence  │  适合：中间结果、Hadoop 生态兼容               │
│  │ • Avro      │  适合：Schema 演化、消息传输                  │
│  └─────────────┘                                                │
│                                                                  │
│  ┌─────────────┐                                                │
│  │  列式存储   │  ← 一列数据连续存储                            │
│  │ Col-Based   │                                                │
│  ├─────────────┤                                                │
│  │ • ORC       │  适合：Hive 生态，查询性能最优                 │
│  │ • Parquet   │  适合：Spark 生态，跨平台兼容                 │
│  └─────────────┘                                                │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 二、TextFile 格式详解

### 2.1 存储结构

TextFile 是 Hive 的默认格式，以纯文本形式存储数据，每行一条记录：

```
TextFile 存储方式（行式存储）：

文件内容（纯文本，可人工阅读）：
┌──────────────────────────────────────────────────────┐
│ 1001\tHive入门\t1001\t1\t3500\t42\n                  │
│ 1002\tSpark原理\t1001\t1\t5200\t78\n                 │
│ 1003\tFlink入门\t1002\t2\t4800\t65\n                 │
│ 1004\tHDFS架构\t1001\t1\t6100\t91\n                  │
└──────────────────────────────────────────────────────┘

一行中的所有列连续存储，用分隔符（\t / , / |）隔开
```

### 2.2 创建和使用

```sql
-- 创建 TextFile 表（默认格式，可以省略 STORED AS）
CREATE TABLE ods.ods_demo_article_text (
    id              INT,
    title           STRING,
    author_id       INT,
    category_id     INT,
    view_count      BIGINT,
    like_count      INT
)
ROW FORMAT DELIMITED
    FIELDS TERMINATED BY '\t'       -- 字段分隔符（默认 \001）
    LINES TERMINATED BY '\n'        -- 行分隔符（默认 \n）
STORED AS TEXTFILE;

-- 自定义分隔符
CREATE TABLE ods.ods_csv_import (
    id              INT,
    title           STRING,
    author_id       INT
)
ROW FORMAT SERDE 'org.apache.hadoop.hive.serde2.OpenCSVSerde'
WITH SERDEPROPERTIES (
    'separatorChar' = ',',
    'quoteChar'     = '"',
    'escapeChar'    = '\\'
)
STORED AS TEXTFILE;

-- 指定跳过 CSV 表头
STORED AS TEXTFILE
TBLPROPERTIES ('skip.header.line.count' = '1');
```

### 2.3 TextFile 优缺点

```
优点：
+ 可读性强：可以用 cat / head 直接查看
+ 兼容性好：任何系统都能处理纯文本
+ 加载方便：LOAD DATA 直接可用
+ 生态广泛：MySQL 导出、日志文件等天然适配

缺点：
- 存储空间大：无压缩或压缩率低
- 查询性能差：即使只查一列也要扫描整行
- 不支持列裁剪：无法跳过不需要的列
- 不支持谓词下推：无法在存储层过滤数据

适用场景：
→ ODS 层原始数据存储
→ 需要人工查看的数据
→ 与外部系统数据交换
```

## 三、ORC 格式详解

ORC（Optimized Row Columnar）是 Hive 专为 Hadoop 生态设计的列式存储格式，是 Hive 中最推荐的格式。

### 3.1 存储结构

```
ORC 文件内部结构：

┌──────────────────────────────────────────────────────────────┐
│                    ORC File                                  │
│                                                              │
│  ┌────────────┐  ┌────────────────────────────────────┐     │
│  │   File     │  │              Stripe 1               │     │
│  │  Footer    │  ├─────────────┬──────────────────────┤     │
│  │            │  │  Stripe     │  Column Chunk 0      │     │
│  │ - 文件元数据│  │  Footer    │  ┌──────┐ ┌──────┐   │     │
│  │ - 条数     │  │            │  │col_a │ │col_b │   │     │
│  │ - Schema   │  ├─────────────┤  │1001  │ │Hive  │   │     │
│  │ - 压缩信息 │  │  Index Data │  │1002  │ │Spark │   │     │
│  │ - Stripe   │  │  (min/max   │  │1003  │ │Flink │   │     │
│  │   列表     │  │   统计信息) │  └──────┘ └──────┘   │     │
│  └────────────┘  ├─────────────┤  Column Chunk 1      │     │
│                  │  Row Data   │  ┌──────┐ ┌──────┐   │     │
│  ┌────────────┐  │  (列式编码  │  │col_a │ │col_b │   │     │
│  │   Post     │  │   +压缩)    │  │3500  │ │张三  │   │     │
│  │  Script    │  │            │  │5200  │ │张三  │   │     │
│  │ (轻量级    │  │            │  │4800  │ │李四  │   │     │
│  │  脚本头)   │  │            │  └──────┘ └──────┘   │     │
│  └────────────┘  └─────────────┴──────────────────────┘     │
│                                                              │
│  ┌────────────────────────────────────────────────────┐     │
│  │              Stripe 2 ...                          │     │
│  └────────────────────────────────────────────────────┘     │
│                                                              │
└──────────────────────────────────────────────────────────────┘

Stripe（条带）：
  - ORC 文件被划分为多个 Stripe（默认 250MB）
  - 每个 Stripe 独立存储，可以并行读取
  - 包含三部分：Index Data + Row Data + Stripe Footer

Index Data（索引数据）：
  - 每列的 min/max 值
  - 行数统计
  - 用于谓词下推：WHERE col > 100 可以跳过不含该值的 Stripe

Row Data（行数据）：
  - 按列存储，每列独立编码和压缩
  - 使用字典编码 + Run-Length Encoding (RLE)
  - 支持轻量级索引，快速定位数据

File Footer（文件尾）：
  - 文件级别的 Schema、统计信息、Stripe 位置
  - 读取时先加载 Footer，再按需读取 Stripe
```

### 3.2 ORC 编码和压缩

```
ORC 编码方式：

1. 字典编码（Dictionary Encoding）
   ┌────────────────────────────────────────────┐
   │ 原始数据：Hive, Spark, Hive, Flink, Hive   │
   │                                            │
   │ 字典表：                                   │
   │   0 → Hive                                 │
   │   1 → Spark                                │
   │   2 → Flink                                │
   │                                            │
   │ 编码后：0, 1, 0, 2, 0                      │
   │ (用整数代替重复字符串，大幅减少存储)        │
   └────────────────────────────────────────────┘

2. Run-Length Encoding (RLE)
   ┌────────────────────────────────────────────┐
   │ 原始数据：1, 1, 1, 1, 2, 2, 3             │
   │                                            │
   │ RLE 编码：(1, 4), (2, 2), (3, 1)           │
   │ (值, 连续出现次数)                          │
   │ 适合排序数据和大量重复值                    │
   └────────────────────────────────────────────┘

3. 位压缩（Bit-Packing）
   - 将小整数紧凑存储到位中
   - 例如 0-7 的值只需 3 bit 存储
```

### 3.3 创建和使用

```sql
-- 创建 ORC 表
CREATE TABLE dwd.dwd_article_orc (
    article_id      BIGINT,
    title           STRING,
    author_id       BIGINT,
    category_id     INT,
    view_count      BIGINT,
    like_count      INT,
    publish_date    DATE
)
STORED AS ORC
TBLPROPERTIES ('orc.compress' = 'SNAPPY');

-- 指定 ORC 高级参数
CREATE TABLE dwd.dwd_article_orc_advanced (
    article_id      BIGINT,
    title           STRING
)
STORED AS ORC
TBLPROPERTIES (
    'orc.compress'              = 'ZSTD',       -- 压缩算法
    'orc.compress.size'         = '262144',     -- Stripe 大小（256KB）
    'orc.stripe.size'           = '67108864',   -- Stripe 大小（64MB）
    'orc.row.index.stride'      = '10000',      -- 索引步长（每10000行一个索引条目）
    'orc.create.index'          = 'true',       -- 创建索引
    'orc.bloom.filter.columns'  = 'article_id,title'  -- 布隆过滤器列
);

-- 从 TextFile 转换为 ORC
INSERT OVERWRITE TABLE dwd.dwd_article_orc
SELECT article_id, title, author_id, category_id,
       view_count, like_count, publish_date
FROM dwd.dwd_article_clean;
```

## 四、Parquet 格式详解

Parquet 是由 Twitter 和 Cloudera 共同开发的列式存储格式，是 Spark 和 Impala 生态的首选格式。

### 4.1 存储结构

```
Parquet 文件内部结构：

┌──────────────────────────────────────────────────────────────┐
│                    Parquet File                              │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │                    Row Group 1                        │   │
│  │  ┌──────────────┐  ┌──────────────────────────────┐  │   │
│  │  │  Column Chunk │  │  Column Chunk                │  │   │
│  │  │  (Column A)   │  │  (Column B)                  │  │   │
│  │  │              │  │                              │  │   │
│  │  │ ┌──────────┐ │  │  ┌──────────┐               │  │   │
│  │  │ │ Page 1   │ │  │  │ Page 1   │               │  │   │
│  │  │ │(Dict Page)│ │  │  │(Dict Page)│               │  │   │
│  │  │ └──────────┘ │  │  └──────────┘               │  │   │
│  │  │ ┌──────────┐ │  │  ┌──────────┐               │  │   │
│  │  │ │ Page 2   │ │  │  │ Page 2   │               │  │   │
│  │  │ │(Data Page)│ │  │  │(Data Page)│               │  │   │
│  │  │ └──────────┘ │  │  └──────────┘               │  │   │
│  │  └──────────────┘  └──────────────────────────────┘  │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │                    Row Group 2 ...                    │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │              File Metadata (Footer)                   │   │
│  │  - Schema (嵌套类型支持更好)                          │   │
│  │  - Row Group 位置                                    │   │
│  │  - 各列统计信息 (min/max/null_count)                 │   │
│  │  - Key Value Metadata                               │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
└──────────────────────────────────────────────────────────────┘

Row Group：类似 ORC 的 Stripe，每个 Row Group 包含所有列的数据
Page：列数据的最小存储单元，分为 Dictionary Page 和 Data Page
Footer：文件末尾，包含 Schema 和元数据
```

### 4.2 ORC vs Parquet

```
┌──────────────────────────────────────────────────────────────────┐
│                  ORC vs Parquet 对比                             │
│                                                                  │
│                    ORC                    Parquet                │
│  ┌────────────────────────┐   ┌────────────────────────┐       │
│  │ 开发者: Hortonworks    │   │ 开发者: Twitter/Cloudera│       │
│  │                        │   │                        │       │
│  │ 优势:                  │   │ 优势:                  │       │
│  │ + Hive 查询性能最优    │   │ + Spark 性能最优       │       │
│  │ + 内置索引更丰富       │   │ + 嵌套类型支持更好     │       │
│  │ + ACID 事务支持        │   │ + 跨语言生态更好       │       │
│  │ + 编码效率高           │   │ + 云存储兼容性更好     │       │
│  │                        │   │                        │       │
│  │ 劣势:                  │   │ 劣势:                  │       │
│  │ - 非 Hive 引擎支持一般 │   │ - Hive 性能不如 ORC    │       │
│  │ - 嵌套类型支持一般     │   │ - ACID 支持不如 ORC    │       │
│  └────────────────────────┘   └────────────────────────┘       │
│                                                                  │
│  选择建议：                                                      │
│  • 纯 Hive 数仓 → ORC（推荐）                                  │
│  • Hive + Spark 混用 → Parquet                                  │
│  • 需要多引擎共享 → Parquet                                     │
│  • 需要 ACID 事务 → ORC                                         │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 4.3 创建和使用

```sql
-- 创建 Parquet 表
CREATE TABLE dwd.dwd_article_parquet (
    article_id      BIGINT,
    title           STRING,
    author_id       BIGINT,
    category_id     INT,
    view_count      BIGINT,
    like_count      INT,
    publish_date    DATE
)
STORED AS PARQUET
TBLPROPERTIES ('parquet.compression' = 'SNAPPY');

-- 指定 Parquet 高级参数
CREATE TABLE dwd.dwd_article_parquet_advanced (
    article_id      BIGINT,
    title           STRING
)
STORED AS PARQUET
TBLPROPERTIES (
    'parquet.compression'        = 'ZSTD',
    'parquet.block.size'         = '134217728',  -- Row Group 大小（128MB）
    'parquet.page.size'          = '1048576',    -- Page 大小（1MB）
    'parquet.enable.dictionary'  = 'true'        -- 启用字典编码
);
```

## 五、Avro 格式详解

Avro 是 Apache 基金会开发的序列化系统，强调 Schema 演化和跨语言兼容性。

### 5.1 特点

```
┌──────────────────────────────────────────────────────────────────┐
│                    Avro 格式特点                                  │
│                                                                  │
│  核心优势：                                                      │
│  ┌──────────────────────────────────────────────────────┐       │
│  │ 1. Schema 与数据分离                                 │       │
│  │    Schema 存储在文件头部（JSON 格式）                │       │
│  │    读取时不需要额外提供 Schema                       │       │
│  │                                                      │       │
│  │ 2. Schema 演化支持                                   │       │
│  │    可以添加/删除字段而不影响旧数据读取               │       │
│  │    适合长期存储的数据                                 │       │
│  │                                                      │       │
│  │ 3. 行式存储 + 可选压缩                               │       │
│  │    适合写入密集的场景                                 │       │
│  │                                                      │       │
│  │ 4. 与 Kafka 集成良好                                 │       │
│  │    Kafka 常用 Avro 作为消息序列化格式                │       │
│  └──────────────────────────────────────────────────────┘       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 5.2 创建和使用

```sql
-- 创建 Avro 表（通过内联 Schema）
CREATE TABLE ods.ods_kafka_events (
    event_id        STRING,
    event_type      STRING,
    payload         STRING,
    event_time      STRING
)
STORED AS AVRO
TBLPROPERTIES (
    'avro.schema.literal' = '{
        "type": "record",
        "name": "KafkaEvent",
        "fields": [
            {"name": "event_id",   "type": "string"},
            {"name": "event_type", "type": "string"},
            {"name": "payload",    "type": "string"},
            {"name": "event_time", "type": "string"}
        ]
    }'
);

-- 或通过外部 Schema 文件
-- 'avro.schema.url' = 'hdfs:///user/hive/schemas/kafka_event.avsc'
```

## 六、SequenceFile 格式

SequenceFile 是 Hadoop 原生的二进制格式，适合作为 MapReduce 的中间结果存储。

```sql
-- 创建 SequenceFile 表
CREATE TABLE tmp_seq_file (
    key     STRING,
    value   STRING
)
STORED AS SEQUENCEFILE;

-- 特点：
-- - 二进制格式，不可人工阅读
-- - 支持压缩（块压缩和记录压缩）
-- - 适合 MapReduce 中间结果
-- - 现在已被 ORC/Parquet 取代，不推荐用于最终存储
```

## 八、如何选择合适的文件格式

### 8.1 按数仓分层选择

```
┌──────────────────────────────────────────────────────────────────┐
│              数仓各层文件格式选择建议                             │
│                                                                  │
│  ODS 层（原始数据层）：                                          │
│  ┌──────────────────────────────────────────────────────┐       │

│  ADS 层（应用数据层）：                                          │
│  ┌──────────────────────────────────────────────────────┐       │

│  │ 压缩：ZSTD                                            │       │
│  │ 原因：面向前端/报表，数据量最小，追求查询速度          │       │
│  └──────────────────────────────────────────────────────┘       │
│                                                                  │
│  DIM 层（维度表）：                                              │
│  ┌──────────────────────────────────────────────────────┐       │
│  │ 推荐：ORC + SNAPPY                                    │       │
│  │ 原因：数据量小，频繁 JOIN，需要快速读取                │       │

│                                                                  │
│  临时表：                                                        │
│  ┌──────────────────────────────────────────────────────┐       │
│  │ 推荐：TextFile（无需压缩，写入快）                     │       │
│  │ 原因：临时数据，用完即丢                              │       │
```

## 九、压缩编解码器对比

### 9.1 编解码器详解

```
┌──────────────────────────────────────────────────────────────────┐
│              压缩编解码器对比                                    │
│                                                                  │
│  ┌────────────────────────────────────────────────────────┐     │
│  │  GZIP                                                  │     │
│  │  ├── 压缩比：高 (~40% of 原始)                         │     │
│  │  ├── 压缩速度：中                                      │     │
│  │  ├── 解压速度：中                                      │     │
│  │  ├── 可分割：不支持（不可 Split）                       │     │
│  │  ├── CPU 消耗：中                                      │     │
│  │  └── 适用：数据归档、冷数据存储                        │     │
│  ├────────────────────────────────────────────────────────┤     │
│  │  SNAPPY                                                │     │
│  │  ├── 压缩比：中 (~50% of 原始)                         │     │
│  │  ├── 压缩速度：很快                                    │     │
│  │  ├── 解压速度：极快                                    │     │
│  │  ├── 可分割：不支持（但在 ORC/Parquet 中天然分块）     │     │
│  │  ├── CPU 消耗：低                                      │     │
│  │  └── 适用：频繁读写的热数据（DWD 层）                  │     │
│  ├────────────────────────────────────────────────────────┤     │
│  │  LZO                                                   │     │
│  │  ├── 压缩比：中低 (~55% of 原始)                       │     │
│  │  ├── 压缩速度：快                                      │     │
│  │  ├── 解压速度：极快                                    │     │
│  │  ├── 可分割：支持（需要额外建立索引）                  │     │
│  │  ├── CPU 消耗：低                                      │     │
│  │  └── 适用：需要可分割的大文件压缩                      │     │
│  │     注意：需要额外安装 native library                  │     │
│  ├────────────────────────────────────────────────────────┤     │
│  │  ZSTD (Zstandard)                                     │     │
│  │  ├── 压缩比：很高（~30-40% of 原始，可调节）           │     │
│  │  ├── 压缩速度：快（比 GZIP 快）                       │     │
│  │  ├── 解压速度：很快（比 GZIP 快数倍）                 │     │
│  │  ├── 可分割：支持                                      │     │
│  │  ├── CPU 消耗：低~中                                   │     │
│  │  ├── 可调级别：1(最快) ~ 22(最高压缩比)               │     │
│  │  └── 适用：存储成本敏感场景（DWS/ADS 层）              │     │
│  └────────────────────────────────────────────────────────┘     │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 9.3 压缩配置方法

```sql
-- 方法1：通过 TBLPROPERTIES 设置表级压缩
STORED AS ORC
TBLPROPERTIES ('orc.compress' = 'ZSTD');

-- 方法2：通过 TBLPROPERTIES 设置 Parquet 压缩
STORED AS PARQUET
TBLPROPERTIES ('parquet.compression' = 'SNAPPY');

-- 方法3：通过 SET 设置会话级默认压缩（对 TextFile 等非列式格式）
SET hive.exec.compress.output = true;
SET mapreduce.output.fileoutputformat.compress.codec = org.apache.hadoop.io.compress.ZStandardCodec;

-- 方法4：MapReduce 中间结果压缩（减少 Shuffle 数据量）
SET hive.exec.compress.intermediate = true;
SET mapreduce.map.output.compress.codec = org.apache.hadoop.io.compress.SnappyCodec;

-- 查看 ORC 文件的压缩信息
-- 通过 Hive 命令行查看
INSERT OVERWRITE LOCAL DIRECTORY '/tmp/orc_info'
ROW FORMAT DELIMITED FIELDS TERMINATED BY '\t'
SELECT INPUT__FILE__NAME, ORC_FILE_STATS(original_size, compressed_size, compression)
FROM dwd.dwd_article_orc LIMIT 1;
```

## 十、总结

本篇全面学习了 Hive 文件格式和压缩：

1. **TextFile**：默认行式格式，可读性好，适合 ODS 层
2. **ORC**：Hive 首选列式格式，压缩比高、查询快、支持 ACID
3. **Parquet**：Spark 生态首选，嵌套类型支持好
4. **Avro**：Schema 演化能力强，适合 Kafka 集成
5. **格式选择**：ODS 用 TextFile，DWD/DWS/ADS 用 ORC，混合引擎用 Parquet
6. **压缩选择**：热数据用 SNAPPY，冷数据/归档用 ZSTD，通用用 GZIP
7. **压缩配置**：TBLPROPERTIES、SET 参数、MapReduce 中间压缩
